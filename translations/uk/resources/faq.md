## Початок роботи

### П: Що таке Azure Developer CLI (azd)?
**В**: Azure Developer CLI (azd) — це орієнтований на розробника інструмент командного рядка, який прискорює час від початку розробки додатку до його запуску в Azure. Він забезпечує найкращі практики через шаблони та допомагає на всьому життєвому циклі розгортання.

### П: Чим azd відрізняється від Azure CLI?
**В**: 
- **Azure CLI**: універсальний інструмент для керування ресурсами Azure
- **azd**: інструмент, орієнтований на розробника, для робочих процесів розгортання додатків
- azd використовує Azure CLI внутрішньо, але надає більш високорівневі абстракції для типових сценаріїв розробки
- azd включає шаблони, управління середовищами та автоматизацію розгортання

### П: Чи потрібна Azure CLI для використання azd?
**В**: Так, azd потребує Azure CLI для автентифікації та деяких операцій. Спочатку встановіть Azure CLI, потім azd.

### П: Які мови програмування підтримує azd?
**В**: azd не залежить від мови. Працює з:
- Node.js/JavaScript/TypeScript
- Python
- .NET/C#
- Java
- Go
- PHP
- Статичні вебсайти
- Контейнеризовані додатки

### П: Чи можу я використовувати azd з існуючими проєктами?
**В**: Так! Ви можете:
1. Використати `azd init` для додавання конфігурації azd до існуючих проєктів
2. Адаптувати існуючі проєкти під структуру шаблонів azd
3. Створити власні шаблони на основі своєї архітектури

## Автентифікація та доступ

### П: Як автентифікуватися в Azure за допомогою azd?
**В**: Використовуйте `azd auth login`, що відкриє вікно браузера для автентифікації Azure. Для CI/CD сценаріїв — використовуйте сервісні принципали або керовані ідентичності.

### П: Чи можна використовувати azd з кількома підписками Azure?
**В**: Так. Використовуйте `azd env set AZURE_SUBSCRIPTION_ID <subscription-id>`, щоб вказати, яку підписку використовувати для кожного середовища.

### П: Які права потрібні для розгортання з azd?
**В**: Зазвичай потрібні:
- роль **Contributor** на групі ресурсів або підписці
- роль **User Access Administrator**, якщо розгортаєте ресурси, що вимагають призначення ролей
- конкретні права залежать від шаблону та ресурсів, що розгортаються

### П: Чи можна використовувати azd в CI/CD пайплайнах?
**В**: Безперечно! azd розроблений для інтеграції з CI/CD. Використовуйте сервісні принципали для автентифікації та встановлюйте змінні середовища для конфігурації.

### П: Як керувати автентифікацією в GitHub Actions?
**В**: Використовуйте дію Azure Login з обліковими даними сервісного принципала:
```yaml
- uses: azure/login@v1
  with:
    creds: ${{ secrets.AZURE_CREDENTIALS }}
- run: azd deploy --no-prompt
```

## Шаблони та проєкти

### П: Де знайти шаблони azd?
**В**: 
- Офіційні шаблони: [Azure-Samples/awesome-azd](https://github.com/Azure-Samples/awesome-azd)
- Шаблони спільноти: пошук на GitHub за "azd-template"
- Використовуйте `azd template list` для перегляду доступних шаблонів

### П: Як створити власний шаблон?
**В**: 
1. Почніть з існуючої структури шаблону
2. Змініть `azure.yaml`, файли інфраструктури та код додатку
3. Ретельно протестуйте за допомогою `azd up`
4. Опублікуйте на GitHub з відповідними тегами

### П: Чи можна використовувати azd без шаблону?
**В**: Так, використовуйте `azd init` у існуючому проєкті для створення необхідних конфігураційних файлів. Потрібно буде вручну налаштовувати `azure.yaml` і файли інфраструктури.

### П: Чим офіційні шаблони відрізняються від спільнотних?
**В**: 
- **Офіційні шаблони**: підтримуються Microsoft, регулярно оновлюються, мають повну документацію
- **Шаблони спільноти**: створені розробниками, можуть мати спеціалізовані сценарії, якість і підтримка різняться

### П: Як оновити шаблон у моєму проєкті?
**В**: Шаблони не оновлюються автоматично. Можна:
1. Вручну порівняти зміни і злити з вихідним шаблоном
2. Почати заново з `azd init`, використовуючи оновлений шаблон
3. Вибірково скопіювати поліпшення з оновлених шаблонів

## Розгортання та інфраструктура

### П: Які сервіси Azure може розгортати azd?
**В**: azd може розгортати будь-які сервіси Azure через Bicep/ARM шаблони, зокрема:
- App Services, Container Apps, Functions
- Бази даних (SQL, PostgreSQL, Cosmos DB)
- Сховища, Key Vault, Application Insights
- Ресурси мережі, безпеки та моніторингу

### П: Чи можна розгортати в кілька регіонів?
**В**: Так, налаштуйте кілька регіонів у Bicep-шаблонах і встановіть параметр location відповідно для кожного середовища.

### П: Як працювати з міграціями схеми бази даних?
**В**: Використовуйте хуки розгортання в `azure.yaml`:
```yaml
hooks:
  postdeploy:
    posix:
      run: ./scripts/migrate-database.sh
    windows:
      run: ./scripts/migrate-database.ps1
```

### П: Чи можна розгортати лише інфраструктуру без додатків?
**В**: Так, використовуйте `azd provision` для розгортання тільки інфраструктурних компонентів із шаблонів.

### П: Як розгортати на вже існуючі ресурси Azure?
**В**: Це складно і не підтримується напряму. Можна:
1. Імпортувати існуючі ресурси у Bicep-шаблони
2. Використовувати посилання на існуючі ресурси в шаблонах
3. Модифікувати шаблони для умовного створення або посилання на ресурси

### П: Чи можна використовувати Terraform замість Bicep?
**В**: Наразі azd переважно підтримує Bicep/ARM шаблони. Офіційної підтримки Terraform немає, але можуть бути рішення від спільноти.

## Конфігурація та середовища

### П: Як керувати різними середовищами (dev, staging, prod)?
**В**: Створюйте окремі середовища з `azd env new <назва-середовища>` і налаштуйте різні параметри для кожного:
```bash
azd env new development
azd env new staging  
azd env new production
```

### П: Де зберігаються конфігурації середовищ?
**В**: У папці `.azure` у каталозі вашого проєкту. Кожне середовище має власну папку з конфігураційними файлами.

### П: Як встановлювати конфігурацію, специфічну для середовища?
**В**: Використовуйте `azd env set` для налаштування змінних середовища:
```bash
azd env set AZURE_LOCATION eastus
azd env set DATABASE_TIER Basic
```

### П: Чи можна ділитися конфігураціями середовищ між членами команди?
**В**: Папка `.azure` містить конфіденційну інформацію і не повинна додаватися до системи контролю версій. Замість цього:
1. Документуйте необхідні змінні середовища
2. Використовуйте скрипти розгортання для налаштування середовищ
3. Використовуйте Azure Key Vault для конфіденційних налаштувань

### П: Як перевизначити значення за замовчуванням у шаблонах?
**В**: Встановіть змінні середовища, що відповідають параметрам шаблону:
```bash
azd env set LOCATION "West US 2"
azd env set SKU_NAME "B1"
```

## Усунення неполадок

### П: Чому не вдається виконати `azd up`?
**В**: Поширені причини:
1. **Проблеми з автентифікацією**: виконайте `azd auth login`
2. **Недостатні права**: перевірте призначення ролей в Azure
3. **Конфлікти імен ресурсів**: змініть AZURE_ENV_NAME
4. **Проблеми з квотами/пропускною спроможністю**: перевірте доступність у регіоні
5. **Помилки у шаблонах**: перевірте Bicep-шаблони на валідність

### П: Як налагоджувати помилки розгортання?
**В**: 
1. Використовуйте `azd deploy --debug` для детального виводу
2. Переглядайте історію розгортань в порталі Azure
3. Аналізуйте Журнал Активності в порталі Azure
4. Використовуйте `azd show` для відображення поточного стану середовища

### П: Чому мої змінні середовища не працюють?
**В**: Перевірте:
1. Імена змінних точно співпадають із параметрами шаблону
2. Значення коректно взяті в лапки, якщо містять пробіли
3. Вибране правильне середовище: `azd env select <environment>`
4. Змінні задані у правильному середовищі

### П: Як очистити невдалі розгортання?
**В**: 
```bash
azd down --force --purge
```
Це видаляє всі ресурси та конфігурації середовища.

### П: Чому додаток недоступний після розгортання?
**В**: Перевірте:
1. Розгортання завершилось успішно
2. Додаток працює (перевірте логи в порталі Azure)
3. Групи безпеки мережі дозволяють трафік
4. DNS/кастомні домени налаштовані правильно

## Витрати та білінг

### П: Скільки коштуватиме розгортання azd?
**В**: Вартість залежить від:
- Розгорнутих сервісів Azure
- Вибраних рівнів послуг/SKU
- Цінових відмінностей між регіонами
- Моделі використання

Використовуйте [Azure Pricing Calculator](https://azure.microsoft.com/pricing/calculator/) для розрахунку.

### П: Як контролювати витрати під час розгортання azd?
**В**: 
1. Використовуйте нижчі рівні SKU для середовищ розробки
2. Налаштуйте бюджети та сповіщення в Azure
3. Використовуйте `azd down` для видалення ресурсів, коли вони не потрібні
4. Обирайте відповідні регіони (вартість залежить від локації)
5. Використовуйте інструменти Azure Cost Management

### П: Чи є безкоштовні рівні для шаблонів azd?
**В**: Багато сервісів Azure мають безкоштовні рівні:
- App Service: є безкоштовний рівень
- Azure Functions: 1 млн безкоштовних виконань на місяць
- Cosmos DB: безкоштовний рівень з 400 RU/s
- Application Insights: перші 5 ГБ на місяць безкоштовно

Налаштовуйте шаблони з урахуванням таких безкоштовних рівнів.

### П: Як оцінити витрати перед розгортанням?
**В**: 
1. Перегляньте `main.bicep` в шаблоні, щоб дізнатися, які ресурси створюються
2. Використайте Azure Pricing Calculator із конкретними SKU
3. Спочатку розгорніть у середовищі розробки, щоб контролювати реальні витрати
4. Використовуйте Azure Cost Management для детального аналізу

## Кращі практики

### П: Які найкращі практики організації структури проєкту azd?
**В**: 
1. Тримайте код додатку окремо від інфраструктури
2. Використовуйте зрозумілі імена сервісів у `azure.yaml`
3. Реалізуйте коректну обробку помилок у скриптах збірки
4. Використовуйте конфігурацію, специфічну для середовища
5. Включайте повну документацію

### П: Як організувати кілька сервісів у azd?
**В**: Використовуйте рекомендовану структуру:
```
├── src/
│   ├── web/          # Frontend service
│   ├── api/          # Backend service  
│   └── worker/       # Background service
├── infra/            # Infrastructure templates
├── azure.yaml        # azd configuration
└── README.md         # Project documentation
```

### П: Чи варто комітити папку `.azure` у систему контролю версій?
**В**: **Ні!** Папка `.azure` містить конфіденційну інформацію. Додайте її до `.gitignore`:
```gitignore
.azure/
```

### П: Як працювати з секретами та конфіденційною конфігурацією?
**В**: 
1. Використовуйте Azure Key Vault для секретів
2. Посилайтесь на секрети Key Vault у конфігурації додатку
3. Ніколи не комітьте секрети у систему контролю версій
4. Використовуйте керовані ідентичності для автентифікації сервісів

### П: Який рекомендований підхід для CI/CD з azd?
**В**: 
1. Використовуйте окремі середовища для кожного етапу (dev/staging/prod)
2. Впроваджуйте автоматичне тестування перед розгортанням
3. Використовуйте сервісні принципали для автентифікації
4. Зберігайте конфіденційну конфігурацію у секретах/змінних пайплайнів
5. Використовуйте затвердження для розгортань у продакшн

## Розширені теми

### П: Чи можу я розширити azd власним функціоналом?
**В**: Так, через хуки розгортання в `azure.yaml`:
```yaml
hooks:
  predeploy:
    run: ./scripts/custom-setup.sh
  postdeploy:
    run: ./scripts/custom-config.sh
```

### П: Як інтегрувати azd із існуючими DevOps процесами?
**В**: 
1. Використовуйте команди azd у скриптах існуючих пайплайнів
2. Стандартизуйте шаблони azd у командах
3. Інтегруйте з існуючими системами моніторингу та сповіщення
4. Використовуйте JSON-вивід azd для інтеграції в пайплайни

### П: Чи можу я використовувати azd з Azure DevOps?
**В**: Так, azd працює з будь-якою системою CI/CD. Створіть пайплайни Azure DevOps, які використовують команди azd.

### П: Як зробити внесок у azd або створити спільнотні шаблони?
**В**:
1. **azd tool**: Внесок у [Azure/azure-dev](https://github.com/Azure/azure-dev)
2. **Templates**: Створюйте шаблони відповідно до [інструкцій щодо шаблонів](https://github.com/Azure-Samples/awesome-azd)
3. **Documentation**: Внесок в документацію у [MicrosoftDocs/azure-dev-docs](https://github.com/MicrosoftDocs/azure-dev-docs)

### Q: Який план розвитку azd?
**A**: Ознайомтеся з [офіційним планом розвитку](https://github.com/Azure/azure-dev/projects) щодо запланованих функцій та покращень.

### Q: Як мігрувати з інших інструментів розгортання на azd?
**A**: 
1. Проаналізуйте поточну архітектуру розгортання
2. Створіть еквівалентні шаблони Bicep
3. Налаштуйте `azure.yaml` відповідно до поточних сервісів
4. Ретельно тестуйте в середовищі розробки
5. Поступово мігруйте середовища
