## Початок роботи

### П: Що таке Azure Developer CLI (azd)?
**В**: Azure Developer CLI (azd) — це орієнтований на розробників інструмент командного рядка, який прискорює процес перенесення вашого застосунку з локального середовища розробки в Azure. Він надає найкращі практики через шаблони та допомагає з усім життєвим циклом розгортання.

### П: Чим azd відрізняється від Azure CLI?
**В**: 
- **Azure CLI**: Універсальний інструмент для керування ресурсами Azure
- **azd**: Інструмент, орієнтований на розробників, для робочих процесів розгортання застосунків
- azd використовує Azure CLI внутрішньо, але надає вищий рівень абстракції для типових сценаріїв розробки
- azd включає шаблони, керування середовищами та автоматизацію розгортання

### П: Чи потрібно встановлювати Azure CLI для використання azd?
**В**: Так, azd потребує Azure CLI для автентифікації та деяких операцій. Спочатку встановіть Azure CLI, а потім azd.

### П: Які мови програмування підтримує azd?
**В**: azd не залежить від мови програмування. Він працює з:
- Node.js/JavaScript/TypeScript
- Python
- .NET/C#
- Java
- Go
- PHP
- Статичними вебсайтами
- Контейнеризованими застосунками

### П: Чи можу я використовувати azd із наявними проєктами?
**В**: Так! Ви можете:
1. Використати `azd init`, щоб додати конфігурацію azd до наявних проєктів
2. Адаптувати наявні проєкти до структури шаблонів azd
3. Створити власні шаблони на основі вашої існуючої архітектури

## Автентифікація та доступ

### П: Як автентифікуватися в Azure за допомогою azd?
**В**: Використовуйте `azd auth login`, який відкриє вікно браузера для автентифікації в Azure. Для сценаріїв CI/CD використовуйте службові облікові записи (service principals) або керовані ідентичності.

### П: Чи можу я використовувати azd із кількома підписками Azure?
**В**: Так. Використовуйте `azd env set AZURE_SUBSCRIPTION_ID <subscription-id>`, щоб вказати, яку підписку використовувати для кожного середовища.

### П: Які дозволи потрібні для розгортання за допомогою azd?
**В**: Зазвичай потрібні:
- Роль **Contributor** для групи ресурсів або підписки
- Роль **User Access Administrator**, якщо розгортаються ресурси, які потребують призначення ролей
- Конкретні дозволи залежать від шаблону та ресурсів, що розгортаються

### П: Чи можу я використовувати azd у CI/CD конвеєрах?
**В**: Звісно! azd розроблений для інтеграції з CI/CD. Використовуйте службові облікові записи для автентифікації та налаштовуйте змінні середовища для конфігурації.

### П: Як обробляти автентифікацію в GitHub Actions?
**В**: Використовуйте дію Azure Login із обліковими даними службового облікового запису:
```yaml
- uses: azure/login@v1
  with:
    creds: ${{ secrets.AZURE_CREDENTIALS }}
- run: azd deploy --no-prompt
```

## Шаблони та проєкти

### П: Де я можу знайти шаблони azd?
**В**: 
- Офіційні шаблони: [Azure-Samples/awesome-azd](https://github.com/Azure-Samples/awesome-azd)
- Шаблони спільноти: Пошук у GitHub за "azd-template"
- Використовуйте `azd template list`, щоб переглянути доступні шаблони

### П: Як створити власний шаблон?
**В**: 
1. Почніть зі структури існуючого шаблону
2. Змініть `azure.yaml`, файли інфраструктури та код застосунку
3. Ретельно протестуйте за допомогою `azd up`
4. Опублікуйте на GitHub із відповідними тегами

### П: Чи можу я використовувати azd без шаблону?
**В**: Так, використовуйте `azd init` у наявному проєкті, щоб створити необхідні конфігураційні файли. Вам потрібно буде вручну налаштувати `azure.yaml` і файли інфраструктури.

### П: У чому різниця між офіційними та шаблонами спільноти?
**В**: 
- **Офіційні шаблони**: Підтримуються Microsoft, регулярно оновлюються, мають детальну документацію
- **Шаблони спільноти**: Створені розробниками, можуть мати спеціалізовані випадки використання, різну якість і підтримку

### П: Як оновити шаблон у моєму проєкті?
**В**: Шаблони не оновлюються автоматично. Ви можете:
1. Вручну порівняти та об’єднати зміни з вихідного шаблону
2. Почати з нуля за допомогою `azd init` із оновленим шаблоном
3. Вибірково перенести конкретні покращення з оновлених шаблонів

## Розгортання та інфраструктура

### П: Які сервіси Azure може розгортати azd?
**В**: azd може розгортати будь-які сервіси Azure через шаблони Bicep/ARM, включаючи:
- App Services, Container Apps, Functions
- Бази даних (SQL, PostgreSQL, Cosmos DB)
- Сховища, Key Vault, Application Insights
- Мережеві, безпекові та моніторингові ресурси

### П: Чи можу я розгортати в кількох регіонах?
**В**: Так, налаштуйте кілька регіонів у своїх шаблонах Bicep і встановіть відповідний параметр розташування для кожного середовища.

### П: Як обробляти міграції схем баз даних?
**В**: Використовуйте хуки розгортання в `azure.yaml`:
```yaml
hooks:
  postdeploy:
    posix:
      run: ./scripts/migrate-database.sh
    windows:
      run: ./scripts/migrate-database.ps1
```

### П: Чи можу я розгортати лише інфраструктуру без застосунків?
**В**: Так, використовуйте `azd provision`, щоб розгорнути лише компоненти інфраструктури, визначені у ваших шаблонах.

### П: Як розгортати в існуючі ресурси Azure?
**В**: Це складно й не підтримується безпосередньо. Ви можете:
1. Імпортувати існуючі ресурси у свої шаблони Bicep
2. Використовувати посилання на існуючі ресурси в шаблонах
3. Змінити шаблони, щоб умовно створювати або посилатися на ресурси

### П: Чи можу я використовувати Terraform замість Bicep?
**В**: Наразі azd переважно підтримує шаблони Bicep/ARM. Офіційної підтримки Terraform немає, хоча можуть існувати рішення від спільноти.

## Конфігурація та середовища

### П: Як керувати різними середовищами (dev, staging, prod)?
**В**: Створіть окремі середовища за допомогою `azd env new <environment-name>` і налаштуйте різні параметри для кожного:
```bash
azd env new development
azd env new staging  
azd env new production
```

### П: Де зберігаються конфігурації середовищ?
**В**: У папці `.azure` у вашій директорії проєкту. Кожне середовище має власну папку з конфігураційними файлами.

### П: Як налаштувати конфігурацію для конкретного середовища?
**В**: Використовуйте `azd env set`, щоб налаштувати змінні середовища:
```bash
azd env set AZURE_LOCATION eastus
azd env set DATABASE_TIER Basic
```

### П: Чи можу я ділитися конфігураціями середовищ із членами команди?
**В**: Папка `.azure` містить конфіденційну інформацію й не повинна бути додана до системи контролю версій. Натомість:
1. Документуйте необхідні змінні середовища
2. Використовуйте скрипти розгортання для налаштування середовищ
3. Використовуйте Azure Key Vault для конфіденційної конфігурації

### П: Як перевизначити значення за замовчуванням у шаблоні?
**В**: Встановіть змінні середовища, які відповідають параметрам шаблону:
```bash
azd env set LOCATION "West US 2"
azd env set SKU_NAME "B1"
```

## Усунення неполадок

### П: Чому `azd up` не працює?
**В**: Поширені причини:
1. **Проблеми з автентифікацією**: Виконайте `azd auth login`
2. **Недостатньо дозволів**: Перевірте свої призначення ролей у Azure
3. **Конфлікти імен ресурсів**: Змініть AZURE_ENV_NAME
4. **Проблеми з квотами/ємністю**: Перевірте доступність у регіоні
5. **Помилки в шаблоні**: Перевірте шаблони Bicep

### П: Як налагоджувати помилки розгортання?
**В**: 
1. Використовуйте `azd deploy --debug` для детального виводу
2. Перевірте історію розгортання в порталі Azure
3. Перегляньте Журнал активності в порталі Azure
4. Використовуйте `azd show`, щоб відобразити поточний стан середовища

### П: Чому мої змінні середовища не працюють?
**В**: Перевірте:
1. Імена змінних точно відповідають параметрам шаблону
2. Значення правильно взяті в лапки, якщо містять пробіли
3. Вибрано правильне середовище: `azd env select <environment>`
4. Змінні встановлені в правильному середовищі

### П: Як очистити невдалі розгортання?
**В**: 
```bash
azd down --force --purge
```
Це видалить усі ресурси та конфігурацію середовища.

### П: Чому мій застосунок недоступний після розгортання?
**В**: Перевірте:
1. Розгортання завершилося успішно
2. Застосунок працює (перевірте журнали в порталі Azure)
3. Групи безпеки мережі дозволяють трафік
4. DNS/кастомні домени налаштовані правильно

## Витрати та виставлення рахунків

### П: Скільки коштуватимуть розгортання azd?
**В**: Витрати залежать від:
- Ресурсів Azure, які розгортаються
- Обраних рівнів/планів сервісів
- Регіональних відмінностей у цінах
- Шаблонів використання

Використовуйте [Калькулятор цін Azure](https://azure.microsoft.com/pricing/calculator/) для оцінок.

### П: Як контролювати витрати в розгортаннях azd?
**В**: 
1. Використовуйте нижчі рівні сервісів для середовищ розробки
2. Налаштуйте бюджети та сповіщення Azure
3. Використовуйте `azd down`, щоб видалити ресурси, коли вони не потрібні
4. Обирайте відповідні регіони (ціни відрізняються залежно від розташування)
5. Використовуйте інструменти управління витратами Azure

### П: Чи є безкоштовні рівні для шаблонів azd?
**В**: Багато сервісів Azure пропонують безкоштовні рівні:
- App Service: Доступний безкоштовний рівень
- Azure Functions: 1 млн безкоштовних виконань на місяць
- Cosmos DB: Безкоштовний рівень із 400 RU/s
- Application Insights: Перші 5 ГБ/місяць безкоштовно

Налаштуйте шаблони для використання безкоштовних рівнів, якщо це можливо.

### П: Як оцінити витрати перед розгортанням?
**В**: 
1. Перегляньте `main.bicep` у шаблоні, щоб побачити, які ресурси створюються
2. Використовуйте Калькулятор цін Azure із конкретними рівнями сервісів
3. Спочатку розгорніть у середовищі розробки, щоб відстежити фактичні витрати
4. Використовуйте Azure Cost Management для детального аналізу витрат

## Найкращі практики

### П: Які найкращі практики для структури проєктів azd?
**В**: 
1. Тримайте код застосунку окремо від інфраструктури
2. Використовуйте зрозумілі імена сервісів у `azure.yaml`
3. Реалізуйте належну обробку помилок у скриптах збірки
4. Використовуйте конфігурацію, специфічну для середовища
5. Додавайте детальну документацію

### П: Як організувати кілька сервісів у azd?
**В**: Використовуйте рекомендовану структуру:
```
├── src/
│   ├── web/          # Frontend service
│   ├── api/          # Backend service  
│   └── worker/       # Background service
├── infra/            # Infrastructure templates
├── azure.yaml        # azd configuration
└── README.md         # Project documentation
```

### П: Чи варто додавати папку `.azure` до системи контролю версій?
**В**: **Ні!** Папка `.azure` містить конфіденційну інформацію. Додайте її до `.gitignore`:
```gitignore
.azure/
```

### П: Як обробляти секрети та конфіденційну конфігурацію?
**В**: 
1. Використовуйте Azure Key Vault для секретів
2. Посилайтеся на секрети Key Vault у конфігурації застосунку
3. Ніколи не додавайте секрети до системи контролю версій
4. Використовуйте керовані ідентичності для автентифікації між сервісами

### П: Який рекомендований підхід до CI/CD із azd?
**В**: 
1. Використовуйте окремі середовища для кожного етапу (dev/staging/pro
1. Проаналізуйте поточну архітектуру розгортання  
2. Створіть еквівалентні шаблони Bicep  
3. Налаштуйте `azure.yaml` відповідно до поточних сервісів  
4. Ретельно протестуйте в середовищі розробки  
5. Поступово мігруйте середовища
