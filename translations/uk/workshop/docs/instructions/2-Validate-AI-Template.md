## 1. Вступ

[Azure Developer CLI](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/) або `azd` — це відкритий командний інструмент, який спрощує робочий процес розробника при створенні та розгортанні додатків в Azure.

[AZD Шаблони](https://learn.microsoft.com/azure/developer/azure-developer-cli/azd-templates) — це стандартизовані репозиторії, які включають приклад коду додатків, ресурси _інфраструктури як коду_ та конфігураційні файли `azd` для цілісної архітектури рішення. Забезпечення інфраструктури стає таким простим, як команда `azd provision` — а команда `azd up` дозволяє за один крок забезпечити інфраструктуру **та** розгорнути ваш додаток!

В результаті запуск процесу розробки додатку може бути таким же простим, як знайти відповідний _AZD Starter шаблон_, який найбільше відповідає потребам вашого додатку та інфраструктури — а потім налаштувати репозиторій під вимоги вашого сценарію.

Перш ніж почати, переконаємося, що у вас встановлений Azure Developer CLI.

1. Відкрийте термінал VS Code і введіть цю команду:

      ```bash title="" linenums="0"
      azd version
      ```

1. Ви повинні побачити щось подібне!

      ```bash title="" linenums="0"
      azd version 1.27.1 (commit <current-build>)
      ```

**Ви тепер готові вибрати та розгорнути шаблон за допомогою azd**

## 3. Активація шаблону

Давайте спробуємо розгорнути цей шаблон і переконаємося, що він дійсний. Ми дотримуватимемося вказівок у розділі [Getting Started](https://github.com/Azure-Samples/get-started-with-ai-agents?tab=readme-ov-file#getting-started).

1. Виберіть робоче середовище для репозиторію шаблону:

      - **GitHub Codespaces**: Натисніть [це посилання](https://github.com/codespaces/new/Azure-Samples/get-started-with-ai-agents) та підтвердіть `Create codespace`
      - **Локальне клонування або dev контейнер**: Клонуйте `Azure-Samples/get-started-with-ai-agents` і відкрийте в VS Code

1. Почекайте, поки термінал VS Code буде готовий, потім введіть наступну команду:

   ```bash title="" linenums="0"
   azd up
   ```

Завершіть кроки робочого процесу, які це викличе:

1. Вам запропонують увійти в Azure — дотримуйтесь інструкцій для аутентифікації
1. Введіть унікальне ім'я середовища для себе — наприклад, я використав `nitya-mshack-azd`
1. Це створить папку `.azure/` — ви побачите підпапку з ім'ям середовища
1. Вам запропонують вибрати назву підписки — виберіть за замовчуванням
1. Вам запропонують вибрати локацію — використовуйте `East US 2`

Тепер почекайте, поки завершиться забезпечення. **Це займає 10-15 хвилин**

1. Коли все буде готово, ваша консоль покаже повідомлення ПРО УСПІХ на кшталт цього:
      ```bash title="" linenums="0"
      SUCCESS: Your up workflow to provision and deploy to Azure completed in 10 minutes 17 seconds.
      ```
1. Ваш Azure Портал тепер матиме групу ресурсів із такою назвою середовища:

      ![Infra](../../../../../translated_images/uk/02-provisioned-infra.46c706b14f56e0bf.webp)

1. **Ви тепер готові перевірити розгорнуту інфраструктуру та додаток**.

## 5. Перевірка агента

Azure Container App розгортає кінцеву точку, яка підключається до AI агента, розгорнутого в проекті Microsoft Foundry для цього шаблону. Давайте подивимось, що це означає.

1. Поверніться на сторінку _Overview_ вашої групи ресурсів у Azure Portal

1. Клікніть на ресурс `Microsoft Foundry` у цьому списку

1. Ви побачите це. Натисніть кнопку `Go to Microsoft Foundry Portal`.
   ![Foundry](../../../../../translated_images/uk/04-view-foundry-project.fb94ca41803f28f3.webp)

1. Ви побачите сторінку Foundry Project для вашого AI додатка
   ![Project](../../../../../translated_images/uk/05-visit-foundry-portal.d734e98135892d7e.webp)

1. Клікніть на `Agents` — ви побачите стандартного агента, якого розгорнуто у вашому проекті
   ![Agents](../../../../../translated_images/uk/06-visit-agents.bccb263f77b00a09.webp)

1. Виберіть його — і ви побачите деталі агента. Зауважте наступне:

      - Агент за замовчуванням використовує Пошук файлів (завжди)
      - У агента у полі `Knowledge` відзначено 32 файли, завантажені для пошуку файлів
      ![Agents](../../../../../translated_images/uk/07-view-agent-details.0e049f37f61eae62.webp)

1. Знайдіть пункт `Data+indexes` у лівому меню та натисніть для перегляду деталей.

      - Ви повинні побачити 32 файли даних, завантажені для знань.
      - Вони відповідатимуть 12 файлам клієнтів і 20 файлам продуктів у папці `src/files`
      ![Data](../../../../../translated_images/uk/08-visit-data-indexes.5a4cc1686fa0d19a.webp)

**Ви перевірили роботу агента!**

1. Відповіді агента засновані на знаннях із цих файлів.
1. Тепер ви можете ставити питання, пов’язані з цими даними, і отримувати обґрунтовані відповіді.
1. Наприклад: `customer_info_10.json` описує 3 покупки, зроблені "Amanda Perez"

Поверніться до вкладки в браузері з кінцевою точкою Container App і запитайте: `Які продукти має Amanda Perez?`. Ви побачите щось на кшталт цього:

![Data](../../../../../translated_images/uk/09-ask-in-aca.4102297fc465a4d5.webp)

## 7. Вбудована спостережуваність

Спостережуваність — це інструментування вашого додатка для генерації даних, які можна використати для розуміння, відлагодження й оптимізації його роботи. Щоб ознайомитись із цим:

1. Натисніть кнопку `View Run Info` — ви побачите цей вигляд. Це приклад [трасування агента](https://learn.microsoft.com/en-us/azure/ai-foundry/how-to/develop/trace-agents-sdk#view-trace-results-in-the-azure-ai-foundry-agents-playground) в дії. _Ви також можете отримати цей вигляд, натиснувши Thread Logs в головному меню_.

   - Оцініть кроки виконання і інструменти, залучені агентом
   - Зрозумійте загальну кількість токенів (проти використання вихідних токенів) для відповіді
   - Зрозумійте затримку і де витрачається час під час виконання

      ![Agent](../../../../../translated_images/uk/10-view-run-info.b20ebd75fef6a1cc.webp)

1. Клікніть на вкладку `Metadata`, щоб побачити додаткові атрибути запуску, які можуть бути корисними для налагодження проблем пізніше.

      ![Agent](../../../../../translated_images/uk/11-view-run-info-metadata.7966986122c7c2df.webp)


1. Клікніть на вкладку `Evaluations`, щоб побачити автоматичні оцінки відповіді агента. Вони включають оцінки безпеки (наприклад, самопошкодження) і специфічні оцінки агента (наприклад, розпізнавання намірів, дотримання завдання).

      ![Agent](../../../../../translated_images/uk/12-view-run-info-evaluations.ef25e4577d70efeb.webp)

1. І нарешті, клікніть на вкладку `Monitoring` у бічному меню.

      - Виберіть вкладку `Resource usage` на сторінці, що відкриється — і перегляньте метрики.
      - Відстежуйте використання додатку з погляду витрат (токенів) і навантаження (запитів).
      - Відстежуйте затримку додатку до першого байта (обробка вводу) і останнього байта (виводу).

      ![Agent](../../../../../translated_images/uk/13-monitoring-resources.5148015f7311807f.webp)

## 8. Змінні середовища

До цього моменту ми пройшли розгортання в браузері — і перевірили, що наша інфраструктура забезпечена, а застосунок працює. Але щоб працювати з кодом додатку _перш за все_, нам потрібно налаштувати локальне середовище розробки зі змінними, необхідними для роботи з цими ресурсами. Використання `azd` робить це простим.

1. Azure Developer CLI [використовує змінні середовища](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/manage-environment-variables?tabs=bash) для зберігання та керування конфігураційними налаштуваннями для розгортання додатків.

1. Змінні середовища зберігаються у `.azure/<env-name>/.env` — це обмежує їх для середовища з іменем `env-name`, використаного під час розгортання, і допомагає ізолювати середовища між різними цілями розгортання в одному репозиторії.

1. Змінні середовища автоматично завантажуються командою `azd` щоразу при виконанні конкретної команди (наприклад, `azd up`). Зверніть увагу, що `azd` не читає автоматично _керовані ОС_ змінні середовища (наприклад, встановлені в оболонці) — натомість використовуйте `azd set env` та `azd get env` для передачі інформації в рамках скриптів.


Давайте випробуємо кілька команд:

1. Отримайте всі змінні середовища, встановлені для `azd` у цьому середовищі:

      ```bash title="" linenums="0"
      azd env get-values
      ```
      
      Ви побачите щось подібне:

      ```bash title="" linenums="0"
      AZURE_AI_AGENT_DEPLOYMENT_NAME="gpt-4.1-mini"
      AZURE_AI_AGENT_NAME="agent-template-assistant"
      AZURE_AI_EMBED_DEPLOYMENT_NAME="text-embedding-3-small"
      AZURE_AI_EMBED_DIMENSIONS=100
      ...
      ```

1. Отримайте конкретне значення — наприклад, я хочу дізнатися, чи встановлено `AZURE_AI_AGENT_MODEL_NAME`

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```
      
      Ви побачите щось подібне — за замовчуванням воно не було встановлене!

      ```bash title="" linenums="0"
      ERROR: key 'AZURE_AI_AGENT_MODEL_NAME' not found in the environment values
      ```

1. Встановіть нову змінну середовища для `azd`. Тут ми оновлюємо ім'я моделі агента. _Примітка: будь-які зміни негайно відображатимуться у файлі `.azure/<env-name>/.env`.

      ```bash title="" linenums="0"
      azd env set AZURE_AI_AGENT_MODEL_NAME gpt-4.1
      azd env set AZURE_AI_AGENT_MODEL_VERSION 2025-04-14
      azd env set AZURE_AI_AGENT_DEPLOYMENT_CAPACITY 150
      ```

      Тепер ми повинні побачити, що значення встановлено:

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```

1. Зверніть увагу, що деякі ресурси є постійними (наприклад, розгортання моделей) і вимагатимуть більше, ніж просто `azd up` для примусового повторного розгортання. Спробуємо демонтувати оригінальне розгортання і повторно розгорнути з оновленими змінними середовища.

1. **Оновлення** Якщо ви раніше розгортали інфраструктуру за допомогою шаблону azd — ви можете _оновити_ стан ваших локальних змінних середовища на основі поточного стану розгортання Azure за допомогою цієї команди:

      ```bash title="" linenums="0"
      azd env refresh
      ```


      Це потужний спосіб _синхронізувати_ змінні оточення між двома або більше локальними середовищами розробки (наприклад, команда з кількома розробниками) - що дозволяє розгорнутій інфраструктурі слугувати як основа правди для стану змінних оточення. Члени команди просто _оновлюють_ змінні, щоб знову синхронізуватися.
