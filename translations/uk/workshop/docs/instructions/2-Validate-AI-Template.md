## 1. Вступ

[Azure Developer CLI](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/) або `azd` — це інструмент командного рядка з відкритим кодом, який оптимізує робочий процес розробника при створенні та розгортанні застосунків в Azure. 

[Шаблони AZD](https://learn.microsoft.com/azure/developer/azure-developer-cli/azd-templates) — це стандартизовані репозиторії, які містять приклад коду застосунку, ресурси _інфраструктури як коду_ та конфігураційні файли `azd` для цілісної архітектури рішення. Провізіювання інфраструктури стає простим, як команда `azd provision` — а команда `azd up` дозволяє одночасно провізіювати інфраструктуру **та** розгортати ваш застосунок!

У результаті процес запуску розробки вашого застосунку може бути таким же простим, як знайти відповідний _AZD Starter шаблон_, який найбільше відповідає потребам вашого застосунку та інфраструктури — а потім налаштувати репозиторій під ваші сценарії.

Перед початком упевнімося, що у вас встановлений Azure Developer CLI.

1. Відкрийте термінал у VS Code і введіть цю команду:

      ```bash title="" linenums="0"
      azd version
      ```

1. Ви маєте побачити приблизно ось що!

      ```bash title="" linenums="0"
      azd version 1.23.12 (commit <current-build>)
      ```

**Тепер ви готові вибрати та розгорнути шаблон за допомогою azd**

## 3. Активація шаблону

Спробуємо розгорнути цей шаблон та переконатися, що він дійсний. Керуватимемось інструкціями з розділу [Початок роботи](https://github.com/Azure-Samples/get-started-with-ai-agents?tab=readme-ov-file#getting-started).

1. Виберіть робоче середовище для репозиторію шаблону:

      - **GitHub Codespaces**: Натисніть [це посилання](https://github.com/codespaces/new/Azure-Samples/get-started-with-ai-agents) і підтвердіть `Create codespace`
      - **Локальне клонування або dev container**: Клонуйте `Azure-Samples/get-started-with-ai-agents` і відкрийте його в VS Code

1. Почекайте, доки термінал VS Code буде готовий, потім введіть таку команду:

   ```bash title="" linenums="0"
   azd up
   ```

Виконайте кроки робочого процесу, які вона запустить:

1. Вас попросять увійти в Azure — дотримуйтесь інструкцій для автентифікації
1. Введіть унікальне ім’я середовища, наприклад `nitya-mshack-azd`
1. Це створить папку `.azure/` — у ній буде підпапка з ім’ям вашого середовища
1. Вас попросять вибрати підписку — оберіть за замовчуванням
1. Вас попросять вибрати розташування - використовуйте `East US 2`

Тепер очікуйте завершення провізіювання. **Це триває 10-15 хвилин**

1. Після завершення ваша консоль покаже повідомлення УСПІХ, як ось це:
      ```bash title="" linenums="0"
      SUCCESS: Your up workflow to provision and deploy to Azure completed in 10 minutes 17 seconds.
      ```
1. У вашому Azure порталі з’явиться група ресурсів, провізійована з ім’ям цього середовища:

      ![Infra](../../../../../translated_images/uk/02-provisioned-infra.46c706b14f56e0bf.webp)

1. **Ви тепер готові перевірити розгорнуту інфраструктуру та застосунок**.

## 5. Перевірка агента

Azure Container App розгортає кінцеву точку, яка підключається до AI агента, провізійованого у проекті Microsoft Foundry для цього шаблону. Давайте розглянемо, що це означає.

1. Поверніться на сторінку _Огляд_ Azure Portal для вашої групи ресурсів

1. Клацніть ресурс `Microsoft Foundry` у цьому списку

1. Ви побачите це. Натисніть кнопку `Go to Microsoft Foundry Portal`. 
   ![Foundry](../../../../../translated_images/uk/04-view-foundry-project.fb94ca41803f28f3.webp)

1. Ви потрапите на сторінку проекту Foundry для вашого AI-застосунку
   ![Project](../../../../../translated_images/uk/05-visit-foundry-portal.d734e98135892d7e.webp)

1. Клацніть на `Agents` — ви побачите стандартного Агента, провізійованого у вашому проекті
   ![Agents](../../../../../translated_images/uk/06-visit-agents.bccb263f77b00a09.webp)

1. Оберіть його — та перегляньте деталі агента. Зверніть увагу на наступне:

      - Агент використовує File Search за замовчуванням (завжди)
      - У `Knowledge` агента вказано 32 завантажених файли (для пошуку у файлах)
      ![Agents](../../../../../translated_images/uk/07-view-agent-details.0e049f37f61eae62.webp)

1. Перейдіть у меню зліва до опції `Data+indexes` й натисніть для деталізації. 

      - Ви побачите 32 завантажені файли для знань.
      - Вони відповідатимуть 12 файлам з інформацією про клієнтів і 20 файлам з інформацією про продукти у папці `src/files` 
      ![Data](../../../../../translated_images/uk/08-visit-data-indexes.5a4cc1686fa0d19a.webp)

**Ви підтвердили роботу агента!** 

1. Відповіді агента базуються на знаннях з цих файлів. 
1. Тепер ви можете ставити питання, пов’язані з цими даними, і отримувати обґрунтовані відповіді.
1. Наприклад: `customer_info_10.json` описує 3 покупки, здійснені "Amanda Perez"

Поверніться у вкладку браузера з кінцевою точкою Container App і запитайте: `What products does Amanda Perez own?`. Ви побачите приблизно це:

![Data](../../../../../translated_images/uk/09-ask-in-aca.4102297fc465a4d5.webp)

## 7. Вбудована спостережуваність

Спостережуваність — це інструментування вашого застосунку для генерування даних, які можна використовувати для розуміння, налагодження та оптимізації його роботи. Щоб отримати уявлення про це:

1. Натисніть кнопку `View Run Info` — ви побачите цей вигляд. Це приклад [трасування агента](https://learn.microsoft.com/en-us/azure/ai-foundry/how-to/develop/trace-agents-sdk#view-trace-results-in-the-azure-ai-foundry-agents-playground) у дії. _Цей вигляд можна також отримати, натиснувши Thread Logs у верхньому меню_.

   - Ознайомтеся з кроками виконання та залученими інструментами агента
   - Зрозумійте загальну кількість токенів (у порівнянні з кількістю вихідних токенів) для відповіді
   - Зрозумійте затримки та де витрачається час під час виконання

      ![Agent](../../../../../translated_images/uk/10-view-run-info.b20ebd75fef6a1cc.webp)

1. Натисніть вкладку `Metadata`, щоб побачити додаткові атрибути виконання, які можуть бути корисними для налагодження проблем пізніше.   

      ![Agent](../../../../../translated_images/uk/11-view-run-info-metadata.7966986122c7c2df.webp)


1. Натисніть вкладку `Evaluations`, щоб переглянути автоматичні оцінювання відповіді агента. Вони включають оцінки безпеки (наприклад, самошкода) та агентно-специфічні оцінки (наприклад, розв’язання намірів, дотримання завдання).

      ![Agent](../../../../../translated_images/uk/12-view-run-info-evaluations.ef25e4577d70efeb.webp)

1. І нарешті, натисніть вкладку `Monitoring` у бічному меню.

      - Оберіть вкладку `Resource usage` на сторінці, що відкриється, і перегляньте метрики.
      - Відстежуйте використання застосунку в термінах витрат (токенів) і навантаження (запитів).
      - Відстежуйте затримку застосунку до першого байта (обробка вхідних даних) та останнього байта (вихід).

      ![Agent](../../../../../translated_images/uk/13-monitoring-resources.5148015f7311807f.webp)

## 8. Змінні середовища

Поки що ми проходили розгортання у браузері — і перевірили, що наша інфраструктура провізійована, а застосунок працездатний. Але щоб працювати з кодом застосунку _спочатку_, потрібно налаштувати локальне середовище розробки відповідними змінними, необхідними для роботи з цими ресурсами. Використання `azd` спрощує це.

1. Azure Developer CLI [використовує змінні середовища](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/manage-environment-variables?tabs=bash) для зберігання і керування конфігураційними налаштуваннями для розгортань застосунку.

1. Змінні середовища зберігаються у файлі `.azure/<env-name>/.env` — це обмежує їх дію на середовище з ім’ям `env-name`, яке використовується під час розгортання, і допомагає ізолювати середовища між різними цілями розгортання в одному репозиторії.

1. Змінні середовища автоматично завантажуються командою `azd` щоразу, коли вона виконує конкретну команду (наприклад, `azd up`). Зверніть увагу, що `azd` не зчитує автоматично _змінні середовища ОС_ (наприклад, встановлені у shell) — натомість використовуйте `azd set env` та `azd get env` для передачі інформації у скриптах.


Спробуємо кілька команд:

1. Отримати всі змінні середовища, встановлені для `azd` у цьому середовищі:

      ```bash title="" linenums="0"
      azd env get-values
      ```
      
      Ви побачите щось на кшталт:

      ```bash title="" linenums="0"
      AZURE_AI_AGENT_DEPLOYMENT_NAME="gpt-4.1-mini"
      AZURE_AI_AGENT_NAME="agent-template-assistant"
      AZURE_AI_EMBED_DEPLOYMENT_NAME="text-embedding-3-small"
      AZURE_AI_EMBED_DIMENSIONS=100
      ...
      ```

1. Отримати конкретне значення — наприклад, я хочу дізнатись, чи встановлено `AZURE_AI_AGENT_MODEL_NAME`

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```
      
      Ви побачите приблизно таке — за замовчуванням воно не встановлене!

      ```bash title="" linenums="0"
      ERROR: key 'AZURE_AI_AGENT_MODEL_NAME' not found in the environment values
      ```

1. Встановити нову змінну середовища для `azd`. Тут ми оновлюємо ім’я моделі агента. _Зверніть увагу: усі зміни одразу відобразяться у файлі `.azure/<env-name>/.env`.

      ```bash title="" linenums="0"
      azd env set AZURE_AI_AGENT_MODEL_NAME gpt-4.1
      azd env set AZURE_AI_AGENT_MODEL_VERSION 2025-04-14
      azd env set AZURE_AI_AGENT_DEPLOYMENT_CAPACITY 150
      ```

      Тепер ми маємо перевірити, що значення встановлено:

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```

1. Зверніть увагу, що деякі ресурси є постійними (наприклад, розгортання моделей) і вимагатимуть більше, ніж просто `azd up`, щоб примусити повторне розгортання. Спробуємо зруйнувати початкове розгортання та повторно розгорнути з оновленими змінними середовища.

1. **Оновлення** Якщо раніше ви розгортали інфраструктуру за допомогою шаблону azd — ви можете _оновити_ стан локальних змінних середовища на основі поточного стану вашого розгортання Azure за допомогою цієї команди:

      ```bash title="" linenums="0"
      azd env refresh
      ```

      Це потужний спосіб _синхронізувати_ змінні оточення між двома або більше локальними середовищами розробки (наприклад, команда з кількома розробниками) — дозволяючи розгорнутій інфраструктурі слугувати джерелом істини для стану змінних оточення. Члени команди просто _оновлюють_ змінні, щоб бути в синхронізації.
