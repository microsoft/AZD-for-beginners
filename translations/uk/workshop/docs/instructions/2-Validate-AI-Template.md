## 1. Вступ

[Azure Developer CLI](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/) або `azd` — це інструмент командного рядка з відкритим кодом, який спрощує робочий процес розробника під час створення та розгортання додатків в Azure.

[AZD Templates](https://learn.microsoft.com/azure/developer/azure-developer-cli/azd-templates) — це стандартизовані репозиторії, що включають зразки коду додатків, ресурси _інфраструктури як коду_ та конфігураційні файли `azd` для єдиної архітектури рішення. Процес створення інфраструктури стає таким же простим, як команда `azd provision` — а команда `azd up` дозволяє одночасно створити інфраструктуру **та** розгорнути ваш додаток!

Отже, початок розробки вашого додатка може бути таким простим, як пошук правильного _AZD Starter шаблону_, який найбільше відповідає вашим потребам у додатку та інфраструктурі — а потім налаштування репозиторію відповідно до ваших вимог.

Перед початком давайте переконаємось, що у вас встановлений Azure Developer CLI.

1. Відкрийте термінал VS Code і введіть цю команду:

      ```bash title="" linenums="0"
      azd version
      ```

1. Ви повинні побачити щось на кшталт цього!

      ```bash title="" linenums="0"
      azd version 1.19.0 (commit b3d68cea969b2bfbaa7b7fa289424428edb93e97)
      ```

**Тепер ви готові вибрати та розгорнути шаблон за допомогою azd**

## 3. Активація шаблону

Давайте спробуємо розгорнути цей шаблон і переконатися, що він дійсний. Дотримуймося інструкцій у розділі [Початок роботи](https://github.com/Azure-Samples/get-started-with-ai-agents?tab=readme-ov-file#getting-started).

1. Натисніть [це посилання](https://github.com/codespaces/new/Azure-Samples/get-started-with-ai-agents) — підтвердіть дію за замовчуванням `Create codespace`
1. Відкриється нова вкладка браузера — зачекайте, поки сесія GitHub Codespaces завантажиться
1. Відкрийте термінал VS Code у Codespaces — введіть таку команду:

   ```bash title="" linenums="0"
   azd up
   ```

Виконайте кроки робочого процесу, які це спричинить:

1. Вам запропонують увійти в Azure - дотримуйтесь інструкцій для аутентифікації
1. Введіть унікальне ім’я середовища — наприклад, я використав `nitya-mshack-azd`
1. Це створить папку `.azure/` — ви побачите підпапку з ім’ям середовища
1. Вам запропонують вибрати ім’я підписки — виберіть стандартну
1. Вам запропонують вибрати розташування — використовуйте `East US 2`

Тепер чекайте завершення розгортання. **Це займе 10-15 хвилин**

1. Коли буде виконано, у вашій консолі з’явиться повідомлення про УСПІХ, подібне до такого:
      ```bash title="" linenums="0"
      SUCCESS: Your up workflow to provision and deploy to Azure completed in 10 minutes 17 seconds.
      ```
1. У вашому порталі Azure тепер буде створено групу ресурсів із цим ім’ям середовища:

      ![Infra](../../../../../translated_images/uk/02-provisioned-infra.46c706b14f56e0bf.webp)

1. **Тепер ви готові перевірити розгорнуту інфраструктуру та додаток**.

1. Ви побачите інтерфейс фронтенду хостингового додатка, подібний до цього:

   ![App](../../../../../translated_images/uk/03-test-application.471910da12c3038e.webp)

## 5. Перевірка агента

Azure Container App розгортає кінцеву точку, яка підключається до AI агента, розгорнутого у проекті Microsoft Foundry для цього шаблону. Давайте подивимось, що це означає.

1. Поверніться на сторінку _Огляд_ групи ресурсів у порталі Azure

1. Клацніть на ресурс `Microsoft Foundry` у цьому списку

1. Ви повинні побачити це. Натисніть кнопку `Go to Microsoft Foundry Portal`.
   ![Foundry](../../../../../translated_images/uk/04-view-foundry-project.fb94ca41803f28f3.webp)

1. Ви побачите сторінку проекту Foundry для вашого AI-додатку
   ![Project](../../../../../translated_images/uk/05-visit-foundry-portal.d734e98135892d7e.webp)

1. Клацніть на `Agents` — ви бачите стандартного агента, розгорнутого у вашому проекті
   ![Agents](../../../../../translated_images/uk/06-visit-agents.bccb263f77b00a09.webp)

1. Виберіть його — ви побачите деталі агента. Зверніть увагу на наступне:

      - Агент за замовчуванням використовує File Search (завжди)
      - У агенті в розділі `Knowledge` вказано, що було завантажено 32 файли (для пошуку по файлах)
      ![Agents](../../../../../translated_images/uk/07-view-agent-details.0e049f37f61eae62.webp)

1. Знайдіть у лівому меню опцію `Data+indexes` і клацніть для деталей.

      - Ви повинні побачити 32 завантажені файли даних для знань.
      - Вони відповідають 12 файлам клієнтів і 20 файлам продуктів у папці `src/files`
      ![Data](../../../../../translated_images/uk/08-visit-data-indexes.5a4cc1686fa0d19a.webp)

**Ви перевірили роботу агента!**

1. Відповіді агента базуються на знаннях із цих файлів.
1. Тепер ви можете ставити питання, пов’язані з цими даними, і отримувати обґрунтовані відповіді.
1. Наприклад: `customer_info_10.json` описує 3 покупки, зроблені "Amanda Perez"

Поверніться у вкладку браузера з кінцевою точкою Container App і запитайте: `What products does Amanda Perez own?`. Ви повинні побачити щось подібне:

![Data](../../../../../translated_images/uk/09-ask-in-aca.4102297fc465a4d5.webp)

## 7. Вбудована спостережуваність

Спостережуваність передбачає інструментування вашого додатка для генерації даних, які можна використати для розуміння, налагодження та оптимізації його роботи. Щоб ознайомитись із цим:

1. Натисніть кнопку `View Run Info` — ви побачите цей вигляд. Це приклад [трасування агента](https://learn.microsoft.com/en-us/azure/ai-foundry/how-to/develop/trace-agents-sdk#view-trace-results-in-the-azure-ai-foundry-agents-playground) у дії. _Цей вигляд ви також можете отримати, натиснувши Thread Logs у верхньому меню_.

   - Ознайомтесь із кроками виконання та інструментами, що залучені агентом
   - Зрозумійте загальну кількість токенів (порівняно із витратами на вихідні токени) для відповіді
   - Дізнайтесь про затримку та місця, де витрачається час під час виконання

      ![Agent](../../../../../translated_images/uk/10-view-run-info.b20ebd75fef6a1cc.webp)

1. Натисніть вкладку `Metadata`, щоб побачити додаткові атрибути для запуску, які можуть бути корисними для налагодження проблем у майбутньому.

      ![Agent](../../../../../translated_images/uk/11-view-run-info-metadata.7966986122c7c2df.webp)


1. Натисніть вкладку `Evaluations`, щоб побачити автоматичні оцінки, зроблені на відповіді агента. Вони включають оцінки безпеки (наприклад, На шкоду собі) та агентоспецифічні оцінки (наприклад, Визначення наміру, Виконання завдання).

      ![Agent](../../../../../translated_images/uk/12-view-run-info-evaluations.ef25e4577d70efeb.webp)

1. І нарешті, натисніть вкладку `Monitoring` у бічному меню.

      - Виберіть вкладку `Resource usage` на відображеній сторінці — і перегляньте метрики.
      - Відстежуйте використання додатка з точки зору витрат (токени) та навантаження (запити).
      - Відстежуйте затримку додатка до першого байту (обробка вводу) і останнього байту (вивід).

      ![Agent](../../../../../translated_images/uk/13-monitoring-resources.5148015f7311807f.webp)

## 8. Змінні оточення

Поки що ми виконували розгортання в браузері — і перевірили, що наша інфраструктура створена, а додаток працює. Але щоб працювати з кодом додатка _спочатку_, нам потрібно налаштувати локальне середовище розробки відповідними змінними, необхідними для роботи з цими ресурсами. Використання `azd` робить це просто.

1. Azure Developer CLI [використовує змінні оточення](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/manage-environment-variables?tabs=bash) для збереження та керування конфігурацією розгортання додатків.

1. Змінні оточення зберігаються у `.azure/<env-name>/.env` — це обмежує їх дію до середовища `env-name`, використаного під час розгортання, і допомагає ізолювати середовища між різними цілями розгортання в одному репозиторії.

1. Змінні оточення автоматично завантажуються командою `azd` під час виконання конкретної команди (наприклад, `azd up`). Зверніть увагу, що `azd` не читає автоматично змінні оточення на рівні ОС (наприклад, встановлені в оболонці) — натомість використовуйте `azd set env` та `azd get env` для передачі інформації у скриптах.


Давайте спробуємо кілька команд:

1. Отримати всі змінні оточення, встановлені для `azd` у цьому середовищі:

      ```bash title="" linenums="0"
      azd env get-values
      ```
      
      Ви побачите щось на кшталт:

      ```bash title="" linenums="0"
      AZURE_AI_AGENT_DEPLOYMENT_NAME="gpt-4o-mini"
      AZURE_AI_AGENT_NAME="agent-template-assistant"
      AZURE_AI_EMBED_DEPLOYMENT_NAME="text-embedding-3-small"
      AZURE_AI_EMBED_DIMENSIONS=100
      ...
      ```

1. Отримати конкретне значення — наприклад, я хочу дізнатися, чи встановлено значення `AZURE_AI_AGENT_MODEL_NAME`

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```
      
      Ви побачите щось подібне — воно не було встановлено за замовчуванням!

      ```bash title="" linenums="0"
      ERROR: key 'AZURE_AI_AGENT_MODEL_NAME' not found in the environment values
      ```

1. Встановити нову змінну оточення для `azd`. Тут ми оновлюємо назву моделі агента. _Зверніть увагу: будь-які зміни негайно відображатимуться у файлі `.azure/<env-name>/.env`.

      ```bash title="" linenums="0"
      azd env set AZURE_AI_AGENT_MODEL_NAME gpt-4.1
      azd env set AZURE_AI_AGENT_MODEL_VERSION 2025-04-14
      azd env set AZURE_AI_AGENT_DEPLOYMENT_CAPACITY 150
      ```

      Тепер ми повинні побачити, що значення встановлено:

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```

1. Зверніть увагу, що деякі ресурси зберігаються постійно (наприклад, розгортання моделей) і потребують більше ніж одного `azd up` для примусового повторного розгортання. Давайте спробуємо видалити первинне розгортання і розгорнути знову зі зміненими змінними середовища.

1. **Оновлення** Якщо ви раніше розгортали інфраструктуру за допомогою шаблону azd — ви можете _оновити_ стан локальних змінних оточення на основі поточного стану вашого розгортання Azure за допомогою цієї команди:

      ```bash title="" linenums="0"
      azd env refresh
      ```

      Це потужний спосіб _синхронізувати_ змінні оточення між двома або більше локальними середовищами розробки (наприклад, команда з кількома розробниками) — що дозволяє розгорнутій інфраструктурі слугувати джерелом істини для стану змінних оточення. Члени команди просто _оновлюють_ змінні, щоб знову синхронізуватися.
