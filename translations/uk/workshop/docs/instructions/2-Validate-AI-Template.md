## 1. Вступ

[Azure Developer CLI](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/) або `azd` — це інструмент командного рядка з відкритим кодом, який спрощує робочий процес розробника при створенні та розгортанні додатків в Azure. 

[AZD Templates](https://learn.microsoft.com/azure/developer/azure-developer-cli/azd-templates) — це стандартизовані репозиторії, які включають приклади коду додатків, ресурси _infrastructure-as-code_ та конфігураційні файли `azd` для цілісної архітектури рішення. Підготовка інфраструктури стає настільки простою, як команда `azd provision`, а команда `azd up` дозволяє одночасно забезпечити інфраструктуру **та** розгорнути ваш додаток!

Отже, початок розробки вашого додатка може бути таким простим, як пошук відповідного _AZD Starter шаблону_, який найбільше підходить до ваших потреб в додатку та інфраструктурі, а потім налаштування репозиторію відповідно до вимог вашого сценарію.

Перед початком переконаємось, що у вас встановлений Azure Developer CLI.

1. Відкрийте термінал VS Code і введіть цю команду:

      ```bash title="" linenums="0"
      azd version
      ```

1. Ви повинні побачити щось подібне!

      ```bash title="" linenums="0"
      azd version 1.25.6 (commit <current-build>)
      ```

**Тепер ви готові обрати та розгорнути шаблон за допомогою azd**

## 3. Активізація шаблону

Спробуємо розгорнути цей шаблон і переконатися, що він дійсний. Дотримуємося вказівок у розділі [Getting Started](https://github.com/Azure-Samples/get-started-with-ai-agents?tab=readme-ov-file#getting-started).

1. Оберіть робоче середовище для репозиторію шаблону:

      - **GitHub Codespaces**: Натисніть [це посилання](https://github.com/codespaces/new/Azure-Samples/get-started-with-ai-agents) і підтвердіть `Create codespace`
      - **Локальне клонування чи dev container**: Клонуйте `Azure-Samples/get-started-with-ai-agents` і відкрийте у VS Code

1. Почекайте, доки термінал VS Code буде готовий, потім введіть наступну команду:

   ```bash title="" linenums="0"
   azd up
   ```

Виконайте кроки робочого процесу, які ця команда запустить:

1. Вас попросять увійти в Azure - дотримуйтесь інструкцій для автентифікації
1. Введіть унікальне ім’я середовища - наприклад, я використав `nitya-mshack-azd`
1. Це створить папку `.azure/` - ви побачите підпапку з ім’ям середовища
1. Вас попросять вибрати ім’я підписки - виберіть за замовчуванням
1. Вас попросять вибрати розташування - використовуйте `East US 2`

Тепер очікуйте, поки завершиться підготовка. **Це займає 10-15 хвилин**

1. Коли завершиться, ваша консоль покаже повідомлення SUCCESS подібне до цього:
      ```bash title="" linenums="0"
      SUCCESS: Your up workflow to provision and deploy to Azure completed in 10 minutes 17 seconds.
      ```
1. Ваша панель Azure тепер матиме підготовлену групу ресурсів з ім’ям того середовища:

      ![Infra](../../../../../translated_images/uk/02-provisioned-infra.46c706b14f56e0bf.webp)

1. **Тепер ви готові перевірити розгорнуту інфраструктуру та додаток**.

## 5. Перевірка агента

Azure Container App розгортає endpoint, який підключається до AI агента, підготовленого у проекті Microsoft Foundry для цього шаблону. Давайте подивимось, що це означає.

1. Поверніться на сторінку _Overview_ у Azure Portal для вашої групи ресурсів

1. Натисніть на ресурс `Microsoft Foundry` у цьому списку

1. Ви побачите це. Натисніть кнопку `Go to Microsoft Foundry Portal`. 
   ![Foundry](../../../../../translated_images/uk/04-view-foundry-project.fb94ca41803f28f3.webp)

1. Ви побачите сторінку проекту Foundry для вашого AI-додатку
   ![Project](../../../../../translated_images/uk/05-visit-foundry-portal.d734e98135892d7e.webp)

1. Натисніть на `Agents` — ви побачите стандартного агента, розгорнутого у вашому проекті
   ![Agents](../../../../../translated_images/uk/06-visit-agents.bccb263f77b00a09.webp)

1. Виберіть його — і ви побачите деталі агента. Зверніть увагу на наступне:

      - Агент за замовчуванням використовує File Search (завжди)
      - Поле `Knowledge` агента вказує, що завантажено 32 файли (для файлового пошуку)
      ![Agents](../../../../../translated_images/uk/07-view-agent-details.0e049f37f61eae62.webp)

1. Знайдіть опцію `Data+indexes` у меню зліва і натисніть для перегляду деталей.

      - Ви побачите 32 завантажених файли знань.
      - Вони відповідають 12 файлам клієнтів і 20 файлам продуктів у папці `src/files`
      ![Data](../../../../../translated_images/uk/08-visit-data-indexes.5a4cc1686fa0d19a.webp)

**Ви перевірили роботу агента!**

1. Відповіді агента базуються на знаннях із цих файлів.
1. Тепер ви можете ставити запитання, пов’язані з цими даними, і отримувати обґрунтовані відповіді.
1. Приклад: `customer_info_10.json` описує 3 покупки "Amanda Perez"

Поверніться до вкладки браузера з endpoint Container App і запитайте: `What products does Amanda Perez own?`. Ви побачите щось подібне:

![Data](../../../../../translated_images/uk/09-ask-in-aca.4102297fc465a4d5.webp)

## 7. Вбудована спостережливість

Спостережливість означає інструментування вашого додатку для збирання даних, які можна використовувати для розуміння, налагодження та оптимізації його роботи. Щоб відчути це:

1. Натисніть кнопку `View Run Info` — ви побачите це вікно. Це приклад [Agent tracing](https://learn.microsoft.com/en-us/azure/ai-foundry/how-to/develop/trace-agents-sdk#view-trace-results-in-the-azure-ai-foundry-agents-playground) у дії. _Цей вигляд також можна отримати, натиснувши Thread Logs у верхньому меню_.

   - Ознайомтеся з кроками запуску та інструментами, які використовує агент
   - Зрозумійте загальну кількість токенів (порівняно з використаними токенами відповіді)
   - Зрозумійте затримку та де витрачається час під час виконання

      ![Agent](../../../../../translated_images/uk/10-view-run-info.b20ebd75fef6a1cc.webp)

1. Натисніть вкладку `Metadata`, щоб побачити додаткові атрибути для запуску, які можуть бути корисними при налагодженні проблем пізніше.

      ![Agent](../../../../../translated_images/uk/11-view-run-info-metadata.7966986122c7c2df.webp)


1. Натисніть вкладку `Evaluations`, щоб побачити автоматичні оцінки відповіді агента. Вони включають оцінки безпеки (наприклад, Самошкодження) та специфічні для агента оцінки (наприклад, розпізнавання наміру, дотримання завдання).

      ![Agent](../../../../../translated_images/uk/12-view-run-info-evaluations.ef25e4577d70efeb.webp)

1. І нарешті, натисніть вкладку `Monitoring` у боковому меню.

      - Оберіть вкладку `Resource usage` на сторінці — і перегляньте метрики.
      - Відстежуйте використання додатку з точки зору витрат (токенів) та навантаження (запитів).
      - Відстежуйте затримку додатку до першого байту (обробка вхідних даних) і останнього байту (вивід).

      ![Agent](../../../../../translated_images/uk/13-monitoring-resources.5148015f7311807f.webp)

## 8. Змінні середовища

Поки що ми проходили розгортання через браузер і перевірили, що наша інфраструктура підготовлена і додаток працює. Але щоб працювати з додатком _з коду_, нам потрібно налаштувати локальне середовище розробки відповідними змінними, необхідними для роботи з цими ресурсами. Використання `azd` це спрощує.

1. Azure Developer CLI [використовує змінні середовища](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/manage-environment-variables?tabs=bash) для збереження та керування налаштуваннями конфігурації для розгортань додатків.

1. Змінні середовища зберігаються у `.azure/<env-name>/.env` — це обмежує їх до середовища `env-name`, яке використовувалося під час розгортання і допомагає ізолювати середовища між різними цілями розгортання в одному репозиторії.

1. Змінні середовища автоматично завантажуються командою `azd` щоразу, коли вона виконує певну команду (наприклад, `azd up`). Зверніть увагу, що `azd` не читає автоматично _змінні середовища ОС_ (наприклад, встановлені в оболонці) — замість цього використовуйте `azd set env` і `azd get env` для передачі інформації у скриптах.

Спробуємо кілька команд:

1. Отримати всі змінні середовища, встановлені для `azd` у цьому середовищі:

      ```bash title="" linenums="0"
      azd env get-values
      ```
      
      Ви побачите щось подібне:

      ```bash title="" linenums="0"
      AZURE_AI_AGENT_DEPLOYMENT_NAME="gpt-4.1-mini"
      AZURE_AI_AGENT_NAME="agent-template-assistant"
      AZURE_AI_EMBED_DEPLOYMENT_NAME="text-embedding-3-small"
      AZURE_AI_EMBED_DIMENSIONS=100
      ...
      ```

1. Отримати конкретне значення — наприклад, я хочу дізнатись, чи встановлено `AZURE_AI_AGENT_MODEL_NAME`

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```
      
      Ви побачите щось подібне — це не було встановлено за замовчуванням!

      ```bash title="" linenums="0"
      ERROR: key 'AZURE_AI_AGENT_MODEL_NAME' not found in the environment values
      ```

1. Встановити нову змінну середовища для `azd`. Тут ми оновлюємо ім’я моделі агента. _Зверніть увагу: будь-які зміни одразу відобразяться у файлі `.azure/<env-name>/.env`.

      ```bash title="" linenums="0"
      azd env set AZURE_AI_AGENT_MODEL_NAME gpt-4.1
      azd env set AZURE_AI_AGENT_MODEL_VERSION 2025-04-14
      azd env set AZURE_AI_AGENT_DEPLOYMENT_CAPACITY 150
      ```

      Тепер ми маємо переконатися, що значення встановлено:

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```

1. Зверніть увагу, що деякі ресурси є постійними (наприклад, розгортання моделей) і потребують чогось більшого, аніж просто `azd up` для примусового повторного розгортання. Спробуємо видалити оригінальне розгортання та повторно розгорнути зі зміненими змінними середовища.

1. **Оновлення** Якщо ви раніше розгортали інфраструктуру з використанням azd шаблону — ви можете _оновити_ стан вашого локального середовища змінних, спираючись на поточний стан вашого Azure розгортання за допомогою цієї команди:

      ```bash title="" linenums="0"
      azd env refresh
      ```

      Це потужний спосіб _синхронізувати_ змінні оточення між двома або більше локальними середовищами розробки (наприклад, команда з кількома розробниками) — дозволяючи розгорнутій інфраструктурі слугувати реальним джерелом істини для стану змінних оточення. Члени команди просто _оновлюють_ змінні, щоб знову синхронізуватися.
