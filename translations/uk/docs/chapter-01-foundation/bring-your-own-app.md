# Принесіть власний додаток - Додати azd до існуючого проєкту

**Навігація по розділу:**
- **📚 Головна курсу**: [AZD для початківців](../../README.md)
- **📖 Поточний розділ**: Розділ 1 - Основи та швидкий старт
- **⬅️ Попередній**: [Ваш перший проєкт](first-project.md)
- **➡️ Наступний**: [Dev Containers & Codespaces](dev-containers.md)

> Перевірено з `azd 1.25.6` у червні 2026.

## Вступ

У [Вашому першому проєкті](first-project.md) ви розгорнули додаток, почавши з шаблону. Але найчастіше у вас вже є додаток — Node.js API, сервіс Python Flask, .NET веб-додаток — який лежить у папці на вашому комп’ютері. Цей урок показує, як додати azd до вже наявного коду, щоб ви могли розгорнути його за допомогою `azd up`, без необхідності шаблону.

## Цілі навчання

До кінця цього уроку ви:
- Зрозумієте три способи почати проєкт azd
- Запустите `azd init` у вже існуючій кодовій базі
- Зрозумієте, що робить `azure.yaml` та папка `infra/` для вашого додатка
- Дізнаєтеся, коли дозволити azd згенерувати інфраструктуру, а коли писати її самому
- Розгорнете ваш наявний додаток в Azure за допомогою `azd up`

## Результати навчання

Після завершення цього уроку ви зможете:
- Ініціалізувати azd у проєкті, який у вас уже є
- Читати та редагувати базовий файл `azure.yaml`
- Згенерувати початкову інфраструктуру за допомогою `azd infra generate`
- Обрати підходящий хост Azure для вашого додатка
- Розгорнути та очистити власний застосунок

---

## Три способи почати проєкт azd

| Початкова точка | Команда | Коли використовувати |
|----------------|---------|-------------|
| **З шаблону** | `azd init --template <name>` | Навчання або початок нового додатка з перевіреного прикладу |
| **З вашого існуючого коду** | `azd init` (у папці вашого проєкту) | У вас уже є додаток і ви хочете його розгорнути |
| **З Git-репозиторію** | `azd init --from-code` (у клонованому репі) | Впровадження azd у наявний репозиторій |

Ви вже практикували перший варіант. Цей урок охоплює другий — найпоширенішу реальну ситуацію.

---

## Крок 1: Запустіть `azd init` у вашому проєкті

Відкрийте термінал **всередині папки вашого існуючого проєкту** і запустіть:

```bash
cd my-existing-app
azd init
```

azd запитає, як ви хочете ініціалізувати. Виберіть:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

Оберіть **"Use code in the current directory."**. azd потім сканує вашу папку, визначає мову та фреймворк і пропонує хост.

### Що виявляє azd

azd шукає сигнали на кшталт `package.json`, `requirements.txt`, `pom.xml`, `*.csproj` або `Dockerfile` і пропонує відповідний хост в Azure:

| Ваш додаток | Імовірний хост |
|----------|----------------------|
| Node.js / Python / .NET веб-додаток | Azure App Service або Container Apps |
| Контейнеризований додаток (`Dockerfile`) | Azure Container Apps |
| Function app | Azure Functions |
| Статичний сайт (збудований вихід React/Vue) | Azure Static Web Apps |

Підтвердіть виявлені сервіс(и), і azd згенерує потрібні файли.

---

## Крок 2: Зрозумійте, що створив azd

Після ініціалізації у вашому проєкті з’являться два нові елементи:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — визначення проєкту

Це серце проєкту azd. Мінімальний приклад виглядає так:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

Блок `services` — ключова частина: кожен запис відображає папку вашого коду на хост в Azure. Якщо ваш додаток має фронтенд і API, у вас буде два сервіси.

### `infra/` — ваші ресурси Azure як код

Папка `infra/` містить файли Bicep, які визначають ресурси Azure, потрібні вашому додатку (App Service, базу даних тощо). Не обов’язково писати їх вручну — azd генерує робочу відправну точку. Ви можете редагувати їх пізніше, щоб додати ресурси або посилити безпеку (розглянуто в [Розділі 4](../chapter-04-infrastructure/README.md)).

> **Порада:** Бажаєте побачити або налаштувати згенеровану інфраструктуру перед розгортанням? Запустіть `azd infra generate` (також доступно як `azd infra synth`), щоб записати IaC на диск для перегляду та контролю версій.

---

## Крок 3: Встановіть необхідні конфігурації

Якщо вашому додатку потрібні налаштування або секрети (рядок підключення, API-ключ), не хардкодьте їх. Використовуйте значення середовища:

```bash
# Створити середовище
azd env new dev

# Встановити несекретне значення
azd env set API_VERSION 1.0.0
```

Для реальних секретів зберігайте їх у Key Vault і посилайтеся на них з інфраструктури — див. [Розділ 3: Конфігурація та автентифікація](../chapter-03-configuration/authsecurity.md).

---

## Крок 4: Розгортання

Тепер використайте той самий робочий процес, який ви вже знаєте:

```bash
# Аутентифікуватися (необхідно для azd)
azd auth login

# Переглянути ресурси, які будуть створені
azd provision --preview

# Налаштувати інфраструктуру та розгорнути ваш код
azd up
```

Коли завершиться, azd виведе URL вашого додатка. Перевірте його так само, як будь-який додаток azd:

```bash
azd show           # показати кінцеві точки
azd monitor --logs # перевірте журнали, якщо потрібно
```

## Поширені проблеми при першому запуску

| Симптом | Імовірна причина | Виправлення |
|---------|--------------|-----|
| azd не виявив мій додаток | Відсутній маніфест (наприклад, `package.json`) | Додайте маніфест або оберіть хост вручну під час `azd init` |
| Помилка збірки під час `azd up` | Додаток потребує кроку збірки | Додайте `buildCommand`/`outputPath` під сервісом у `azure.yaml` |
| Додаток стартує, але повертає помилки | Відсутні конфіг/секрет | Встановіть значення за допомогою `azd env set` або налаштуйте Key Vault |
| Обрано неправильний хост | Автовизначення вгадало не той варіант | Відредагуйте `host:` у `azure.yaml` і повторно запустіть `azd up` |

Детальніше див. у [Розділі 7: Виправлення помилок](../chapter-07-troubleshooting/README.md).

---

## Очищення

```bash
azd down --force --purge
```

---

## Підсумок

- `azd init` → **"Use code in the current directory"** додає azd до додатка, який у вас уже є.
- `azure.yaml` відображає ваші папки коду на хости Azure; `infra/` визначає ресурси як Bicep.
- `azd infra generate` дозволяє переглянути або налаштувати згенеровану інфраструктуру.
- Після ініціалізації ваш існуючий додаток використовує той самий робочий процес `azd up` / `azd down`, що й додаток із шаблону.
