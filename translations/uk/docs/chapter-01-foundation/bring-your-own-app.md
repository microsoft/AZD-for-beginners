# Візьми Зі Собою Свій Додаток - Додавання azd до Існуючого Проєкту

**Навігація по Розділах:**

- **📖 Поточний розділ**: Розділ 1 - Основи та Швидкий Старт

- **➡️ Наступний**: [Dev Containers та Codespaces](dev-containers.md)

> Перевірено на `azd 1.27.1` у липні 2026 року.

## Вступ

У [Вашому Першому Проєкті](first-project.md) ви розгортали додаток, починаючи з шаблону. Але переважно у вас вже *є* додаток — Node.js API, Python Flask сервіс, .NET веб-додаток — що знаходиться у папці на вашому комп’ютері. Цей урок показує, як додати azd до цього існуючого коду, щоб ви могли розгортати його за допомогою `azd up`, без необхідності шаблону.

## Цілі Навчання

Наприкінці цього уроку ви:
- Зрозумієте три способи почати проєкт azd
- Запустите `azd init` всередині існуючого коду
- Зрозумієте, для чого потрібні `azure.yaml` та папка `infra/` у вашому додатку
- Дізнаєтеся, коли дозволити azd генерувати інфраструктуру, а коли писати власну
- Розгорнете свій наявний додаток в Azure за допомогою `azd up`

## Результати Навчання

Після завершення цього уроку ви зможете:
- Ініціалізувати azd у проєкті, який у вас вже є
- Читати та редагувати базовий файл `azure.yaml`
- Генерувати стартову інфраструктуру за допомогою `azd infra generate`
- Обрати відповідний Azure-хост для вашого додатка
- Розгортати та очищувати ваш власний додаток

---

## Три Способи Почати Проєкт azd

| Початкова точка | Команда | Коли використовувати |
|----------------|---------|-------------|
| **З шаблону** | `azd init --template <name>` | Навчання або створення нового додатка з перевіреного зразка |
| **З існуючого коду** | `azd init` (у вашій папці проєкту) | У вас вже є додаток і ви хочете його розгорнути |
| **З Git репозиторію** | `azd init --from-code` (у клонованому репозиторії) | Впровадження azd для існуючого репозиторію |

Ви вже практикували перший варіант. Цей урок охоплює другий — найпоширеніший реальний сценарій.

---

## Крок 1: Запустіть `azd init` у Вашому Проєкті

Відкрийте термінал **всередині папки вашого існуючого проєкту** і виконайте:

```bash
cd my-existing-app
azd init
```

azd запитає, як ви хочете ініціалізувати. Оберіть:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

Виберіть **"Використати код у поточній директорії."** azd сканує вашу папку, виявить вашу мову та фреймворк і запропонує хост.

### Що виявляє azd

azd шукає файли, такі як `package.json`, `requirements.txt`, `pom.xml`, `*.csproj` або `Dockerfile`, і пропонує відповідний Azure-хост:

| Ваш додаток | Ймовірно виявлений хост |
|----------|----------------------|
| Node.js / Python / .NET веб-додаток | Azure App Service або Container Apps |
| Додаток у контейнері (`Dockerfile`) | Azure Container Apps |
| Function додаток | Azure Functions |
| Статичний сайт (збірка React/Vue) | Azure Static Web Apps |

Підтвердіть виявлені сервіси, і azd створить необхідні файли.

---

## Крок 2: Зрозумійте, Що Згенерував azd

Після ініціалізації у вашому проєкті будуть дві нові речі:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — визначення проєкту

Це серце проєкту azd. Мінімальний вигляд:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

Блок `services` — це ключова частина: кожен запис відображає папку з вашим кодом на Azure-хост. Якщо ваш додаток має і фронтенд, і API, у вас буде два сервіси.

### `infra/` — ваші Azure-ресурси як код

Папка `infra/` містить файли Bicep, що визначають Azure-ресурси, які потрібні вашому додатку (App Service, базу даних тощо). Вам не потрібно писати їх вручну — azd генерує робочий стартовий набір. Ви *можете* редагувати їх пізніше, щоб додати ресурси або посилити безпеку (розглядається у [Розділі 4](../chapter-04-infrastructure/README.md)).

> **Підказка:** Хочете побачити або налаштувати згенеровану інфраструктуру перед розгортанням? Виконайте `azd infra generate` (також доступно як `azd infra synth`), щоб записати IaC на диск, і ви зможете її переглянути і контролювати версії.

---

## Крок 3: Встановіть Необхідну Конфігурацію

Якщо ваш додаток потребує налаштувань або секретів (рядок підключення, API-ключ), не хардкодьте їх. Використовуйте значення середовища:

```bash
# Створити середовище
azd env new dev

# Встановити не секретне значення
azd env set API_VERSION 1.0.0
```

Для справжніх секретів зберігайте їх у Key Vault і посилайтеся на них з вашої інфраструктури — див. [Розділ 3: Конфігурація та Аутентифікація](../chapter-03-configuration/authsecurity.md).

---

## Крок 4: Розгортання

Тепер використовуйте той самий робочий процес, який ви вже знаєте:

```bash
# Аутентифікуватися (потрібно для azd)
azd auth login

# Переглянути ресурси, які будуть створені
azd provision --preview

# Забезпечити інфраструктуру та розгорнути ваш код
azd up
```

Після завершення azd виведе URL вашого додатка. Перевірте його так само, як будь-який azd-додаток:

```bash
azd show           # показати кінцеві точки
azd monitor --logs # перевірте логи, якщо потрібно
```

## Часті Проблеми Під Час Першого Використання

| Симптом | Ймовірна причина | Вирішення |
|---------|--------------|-----|
| azd не виявив мій додаток | Відсутній маніфест (наприклад, `package.json`) | Додайте маніфест або оберіть хост вручну під час `azd init` |
| Збірка не вдається під час `azd up` | Додаток потребує кроку збірки | Додайте `buildCommand`/`outputPath` у сервісі у `azure.yaml` |
| Додаток стартує, але повертає помилки | Відсутня конфігурація/секрети | Встановіть значення через `azd env set` або підключіть Key Vault |
| Обрано неправильний хост | Невдала авто-детекція | Відредагуйте `host:` у `azure.yaml` і знову запустіть `azd up` |

Для більш детальної інформації дивіться [Розділ 7: Усунення неполадок](../chapter-07-troubleshooting/README.md).

---

## Очищення

```bash
azd down --force --purge
```

---

## Підсумок

- `azd init` → **"Використати код у поточній директорії"** додає azd до додатку, який у вас вже є.
- `azure.yaml` відображає ваші папки коду на Azure-хости; `infra/` визначає ресурси у форматі Bicep.
- `azd infra generate` дозволяє переглянути або налаштувати згенеровану інфраструктуру.
- Після ініціалізації ваш існуючий додаток використовує той самий робочий процес `azd up` / `azd down`, як і додаток на базі шаблону.
