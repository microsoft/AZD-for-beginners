- **⬅️ ಹಿಂದಿನ**: [ನಿಮ್ಮ ಮೊದಲ ಪ್ರಾಜೆಕ್ಟ್](first-project.md)

## ಪರಿಚಯ

[ನಿಮ್ಮ ಮೊದಲ ಪ್ರಾಜೆಕ್ಟ್](first-project.md) ನಲ್ಲಿ ನೀವು ಟೆಂಪ್ಲೇಟಿನಿಂದ ಪ್ರಾರಂಭಿಸಿ ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು ನಿಯೋಜಿಸಿದ್ದಿರಿ. ಆದರೆ ಬಹುತೇಕ ಸಂದರ್ಭಗಳಲ್ಲಿ ನಿಮ್ಮ ಬಳಿ ಈಗಾಗಲೇ ಒಂದು ಅಪ್ಲಿಕೇಶನ್ ಇದೆ — Node.js API, Python Flask ಸೇವೆ, .NET ವೆಬ್ ಅಪ್ಲಿಕೇಶನ್ — ಇದು ನಿಮ್ಮ ಯಂತ್ರ上的 ಫೋಲ್ಡರ್‌ನಲ್ಲಿ ಇರುತ್ತದೆ. ಈ ಪಾಠವು ಆ ಇರುವ ಕೋಡ್‌ಗೆ azd ಅನ್ನು ಸೇರಿಸುವ ವಿಧಾನವನ್ನು ತೋರಿಸುತ್ತದೆ zodat ನೀವು `azd up` ಬಳಸಿ ನಿಯೋಜಿಸಬಹುದು, ಯಾವುದೇ ಟೆಂಪ್ಲೇಟ್ ಅಗತ್ಯವಿಲ್ಲ.

## ಕಲಿಕಾ ಗುರಿಗಳು

ಈ ಪಾಠದ ಅಂತ್ಯಕ್ಕೆ ನೀವು:
- azd ಪ್ರಾಜೆಕ್ಟ್ ಅನ್ನು ಪ್ರಾರಂಭಿಸುವ ಮೂರು ವಿಧಗಳ ಬಗ್ಗೆ ಅರ್ಥ ಮಾಡಿಕೊಂಡಿರುತ್ತೀರಿ
- ಇರುವ ಕೋಡ್‌ಬೇಸ್‌ನೊಳಗೆ `azd init` ಅನ್ನು ಓಡಿಸಲು ಸಾಧ್ಯವಾಗುತ್ತದೆ
- `azure.yaml` ಮತ್ತು `infra/` ಫೋಲ್ಡರ್ ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನಿಗಾಗಿ ಏನು ಮಾಡುತ್ತವೆ ಎಂದು ಅರ್ಥವಾಗುತ್ತದೆ
- azd ಇನ್‌ಫ್ರಾಸ್ಟ್ರಕ್ಚರ್ ಅನ್ನು ಸ್ವಯಂ ರಚಿಸಲಿ ಅಥವಾ ನೀವು ಬರೆದಿರಲಿ ಎಂಬುದನ್ನು ಯಾವಾಗ ಆಯ್ಕೆ ಮಾಡಬೇಕೆಂದು ತಿಳಿದುಕೊಳ್ಳುತ್ತೀರಿ
- ನಿಮ್ಮ ಇರುವ ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು `azd up` ಬಳಸಿ Azure‌ಗೆ ನಿಯೋಜಿಸಬಹುದು

## ಕಲಿಕಾ ಫಲಿತಾಂಶಗಳು

ಈ ಪಾಠವನ್ನು ಪೂರ್ಣಗೊಳಿಸಿದ ನಂತರ, ನೀವು ಶಕ್ತರಾಗಿರುವಿರಿ:
- ನೀವು ಈಗಾಗಲೇ ಹೊಂದಿರುವ ಪ್ರಾಜೆಕ್ಟ್‌ನಲ್ಲಿ azd ಅನ್ನು ಆರಂಭಿಸಲು
- ಮೂಲಭೂತ `azure.yaml` ಫೈಲ್ ಓದಿ ಸಂಪಾದಿಸಲು
- `azd infra generate` ಬಳಸಿ ಪ್ರಾರಂಭಿಕ ಇನ್‌ಫ್ರಾ ರಚಿಸಲು
- ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್‌ಗೆ ಯೋಗ್ಯವಾದ Azure ಹೋಸ್ಟ್ ಆಯ್ಕೆಮಾಡಲು
- ನಿಮ್ಮ ಸ್ವಂತ ಅಪ್ಲಿಕೇಶನ್ ಅನ್ನು ನಿಯೋಜಿಸಿ ಮತ್ತು ತೆರವುಗೊಳಿಸಲು

## azd ಪ್ರಾಜೆಕ್ಟ್ ಪ್ರಾರಂಭಿಸುವ ಮೂರು ವಿಧಾನಗಳು

| Starting point | Command | When to use |
|----------------|---------|-------------|
| **ಟೆಂಪ್ಲೇಟಿನಿಂದ** | `azd init --template <name>` | ಅಭ್ಯಾಸಕ್ಕೆ, ಅಥವಾ ದೃಢಪಟ್ಟ ಮಾದರಿಯಿಂದ ಹೊಸ ಅಪ್ಲಿಕೇಶನ್ ಪ್ರಾರಂಭಿಸುವಾಗ |
| **ನಿಮ್ಮ ಇರುವ ಕೋಡ್‌ನಿಂದ** | `azd init` (in your project folder) | ನಿಮ್ಮ ಬಳಿ ಈಗಾಗಲೇ ಅಪ್ಲಿಕೇಶನ್ ಇದ್ದಿದ್ದು ಅದನ್ನು ನಿಯೋಜಿಸಲು ನೀವು ಬಯಸುವಾಗ |
| **Git ರೆपोದಿಂದ** | `azd init --from-code` (in a cloned repo) | ಇರುವ ರೆಪೊಗಳಿಗೆ azd ಅನ್ನು ತೆಗೆದುಕೊಳ್ಳುವಾಗ |

ನೀವು ಮೊದಲ ಆಯ್ಕೆಯನ್ನು ಈಗಾಗಲೇ ಅಭ್ಯಾಸ ಮಾಡಿದ್ದಾರೆ. ಈ ಪಾಠ ಎರಡನೆಯದನ್ನು — ಅತಿ ಸಾಮಾನ್ಯ ನೈಜ ಜಗತ್ತಿನ ಪರಿಸ್ಥಿತಿಯನ್ನು — ಕವರ್ಮಾಡುತ್ತದೆ.

## ಹಂತ 1: ನಿಮ್ಮ ಪ್ರಾಜೆಕ್ಟ್‌ನಲ್ಲಿ `azd init` ಓಡಿಸಿ

ಟರ್ಮಿನಲ್ ಅನ್ನು **ನಿಮ್ಮ ಇರುವ ಪ್ರಾಜೆಕ್ಟ್ ಫೋಲ್ಡರ್ ಒಳಗೆ** ತೆರೆಯಿರಿ ಮತ್ತು ಕೆಳಕಂಡದನ್ನು ಒಡಿಸಿ:

```bash
cd my-existing-app
azd init
```

azd ನಿಮಗೆ ಹೇಗೆ ಪ್ರಾರಂಭಿಸಬೇಕೆಂದು ಕೇಳುತ್ತದೆ. ಆಯ್ಕೆಮಾಡಿ:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

ಆಯ್ಕೆಮಾಡಿ **"Use code in the current directory."**. azd ನಂತರ ನಿಮ್ಮ ಫೋಲ್ಡರ್ ಅನ್ನು ಸ್ಕ್ಯಾನ್ ಮಾಡಿ, ನಿಮ್ಮ ಭಾಷೆ ಮತ್ತು ಫ್ರೇಮ್ವರ್ಕ್ ಅನ್ನು ಪತ್ತೆಮಾಡಿ, ಮತ್ತು ಒಂದು ಹೋಸ್ಟ್ ಅನ್ನು ಸೂಚಿಸುತ್ತದೆ.

### azd ಯಾವುದು ಪತ್ತೆಮಾಡುತ್ತದೆ

azd `package.json`, `requirements.txt`, `pom.xml`, `*.csproj`, ಅಥವಾ `Dockerfile` ಮುಂತಾದ ಸೂಚನೆಗಳನ್ನು ಹುಡುಕುತ್ತದೆ ಮತ್ತು ಹೊಂದಿಕೊಳ್ಳುವ Azure ಹೋಸ್ಟ್ ಅನ್ನು ಸೂಚಿಸುತ್ತದೆ:

| Your app | Likely detected host |
|----------|----------------------|
| Node.js / Python / .NET ವೆಬ್ ಅಪ್ಲಿಕೇಶನ್ | Azure App Service ಅಥವಾ Container Apps |
| ಕಂಟೈನರೈಜ್ಡ್ ಅಪ್ಲಿಕೇಶನ್ (`Dockerfile`) | Azure Container Apps |
| Function ಅಪ್ಲಿಕೇಶನ್ | Azure Functions |
| ಸ್ಟ್ಯಾಟಿಕ್ ಸೈಟ್ (React/Vue ಬಿಲ್ಡ್ ಔಟ್‌ಪುಟ್) | Azure Static Web Apps |

ಹುಡುಕಿದ ಸೇವೆ(ಗಳನ್ನು) ದೃಢೀಕರಿಸಿ, ಮತ್ತು azd ನಿಮಗೆ ಬೇಕಾದ ಫೈಲ್‌ಗಳನ್ನು ರಚಿಸುತ್ತದೆ.

## ಹಂತ 2: azd ಏನು ರಚಿಸಿತು ಎಂಬುದನ್ನು ಅರ್ಥಮಾಡಿಕೊಳ್ಳಿ

init ನಂತರ, ನಿಮ್ಮ ಪ್ರಾಜೆಕ್ಟ್‌ನಲ್ಲಿ ನೀವು ಎರಡು ಹೊಸ விஷಯಗಳನ್ನು ಹೊಂದಿರುತ್ತೀರಿ:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — ಪ್ರಾಜೆಕ್ಟ್ ವ್ಯಾಖ್ಯಾನ

ಇದು azd ಪ್ರಾಜೆಕ್ಟ್‌ನ ಹೃದಯವಾಗಿದೆ. ಕನಿಷ್ಠ ರೂಪವು ಈ ರೀತಿ ಕಾಣುತ್ತದೆ:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

`services` ಬ್ಲಾಕ್ ಪ್ರಮುಖ ಭಾಗವಾಗಿದೆ: ಪ್ರತಿ ಎಂಟ್ರಿಯು ನಿಮ್ಮ ಕೋಡ್‌ನ ಫೋಲ್ಡರ್ ಅನ್ನು Azure ಹೋಸ್ಟ್‌ಗೆ ಮ್ಯಾಪ್ ಮಾಡುತ್ತದೆ. ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್‌ನಲ್ಲಿ ಫ್ರಂಟ್‌ಎಂಡ್ ಮತ್ತು API ಎರಡೂ ಇದ್ದರೆ, ನೀವು ಎರಡು ಸೇವೆಗಳನ್ನಿರಿಸುತ್ತೀರಿ.

### `infra/` — ನಿಮ್ಮ Azure ಸಂಪನ್ಮೂಲಗಳನ್ನು ಕೋಡ್ ರೂಪದಲ್ಲಿ

`infra/` ಫೋಲ್ಡರ್ Bicep ಫೈಲ್‌ಗಳನ್ನು ধারণಿಸುತ್ತದೆ, ಅವು ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್‌ಗೆ ಬೇಕಾದ Azure ಸಂಪನ್ಮೂಲಗಳನ್ನು (App Service, ಡೇಟಾಬೇಸ್ ಮುಂತಾದವು) ವ್ಯಾಖ್ಯಾನಿಸುತ್ತವೆ. ನೀವು इन्हें ಕೈಯಿಂದ ಬರೆಯಬೇಕಾಗಿಲ್ಲ — azd ಕಾರ್ಯನಿರ್ವಹಿಸುವ ಪ್ರಾರಂಭಿಕ ಬಿಂದು ರಚಿಸುತ್ತದೆ. ನೀವು ನಂತರ ಅವುಗಳನ್ನು ಸಂಪಾದಿಸಿ ಸಂಪನ್ಮೂಲಗಳನ್ನು ಸೇರಿಸಬಹುದು ಅಥವಾ ಭದ್ರತೆ ಕಠಿಣಗೊಳಿಸಬಹುದು (ಇದು [ಅಧ್ಯಾಯ 4](../chapter-04-infrastructure/README.md) ನಲ್ಲಿ ಚರ್ಚಿಸಲಾಗಿದೆ).

> **ಟಿಪ್:** ನಿಯೋಜಿಸುವ ಮೊದಲು ರಚಿತ ಇನ್‌ಫ್ರಾಸ್ಟ್ರಕ್ಚರ್ ಅನ್ನು ನೋಡಬೇಕೇ ಅಥವಾ ಕಸ್ಟಮೈಸ್ ಮಾಡಬೇಕೇ ಎಂದು ಬಯಸಿದರೆ, `azd infra generate` (ಇದೇ ರೀತಿಯಲ್ಲಿ `azd infra synth` ಎಂಬಂತೆ ಲಭ್ಯ) ಅನ್ನು ಓಡಿಸಿ IaC ಅನ್ನು ಡಿಸ್ಕ್‌ಗೆ ಬರೆಯಿರಿ zodat ನೀವು ಅದನ್ನು ಪರಿಶೀಲಿಸಿ ವರ್ಸನ್-ಕಂಟ್ರೋಲ್ ಮಾಡಬಹುದು.

## ಹಂತ 3: ಅಗತ್ಯ ಹೊಂದಾಣಿಕೆಯನ್ನು ಸೆಟ್ ಮಾಡಿ

ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್‌ಗೆ ಸೆಟ್ಟಿಂಗ್‌ಗಳು ಅಥವಾ ಗುಪ್ತ ಮಾಹಿತಿಗಳು (ಒಂದು ಕನೆಕ್ಷನ್ ಸ್ಟ್ರಿಂಗ್, API ಕೀ) ಬೇಕಾದರೆ, ಅವುಗಳನ್ನು ಹಾರ್ಡ್‌ಕೋಡ್ ಮಾಡಬೆಕಾಗಿಲ್ಲ. ಪರಿಸರ ಮೌಲ್ಯಗಳನ್ನು ಬಳಸಿ:

```bash
# ಒಂದು ಪರಿಸರವನ್ನು ರಚಿಸಿ
azd env new dev

# ಗೌಪ್ಯವಲ್ಲದ ಮೌಲ್ಯವನ್ನು ಸೆಟ್ ಮಾಡಿ
azd env set API_VERSION 1.0.0
```

ವಾಸ್ತವಿಕ ಗುಪ್ತ ಮಾಹಿತಿಗಾಗಿ, ಅವುಗಳನ್ನು Key Vault ನಲ್ಲಿ ಸಂಗ್ರಹಿಸಿ ಮತ್ತು ನಿಮ್ಮ ಇನ್‌ಫ್ರಾಸ್ಟ್ರಕ್ಚರ್‌ನಿಂದ ಅವುಗಳನ್ನು ರೆಫರ್ ಮಾಡಿ — ನೋಡಿ [ಅಧ್ಯಾಯ 3: ಸಂರಚನೆ ಮತ್ತು ಪ್ರಾಮಾಣೀಕರಣ](../chapter-03-configuration/authsecurity.md).

## ಹಂತ 4: ನಿಯೋಜಿಸಿ

ಇಗೋ ನೀವು ಈಗಾಗಲೆ ತಿಳಿದಿರುವ ಅದೇ ವರ್ಕ್‌ಫ್ಲೋ ಅನ್ನು ಬಳಸಿ:

```bash
# ಪ್ರಮಾಣೀಕರಿಸಿ (azd ಗಾಗಿ ಅಗತ್ಯ)
azd auth login

# ರಚಿಸಲಾಗುವ ಸಂಪನ್ಮೂಲಗಳನ್ನು ಪೂರ್ವದೃಶ್ಯ ಮಾಡಿ
azd provision --preview

# ಮೂಲಸೌಕರ್ಯವನ್ನು ಒದಗಿಸಿ ಮತ್ತು ನಿಮ್ಮ ಕೋಡ್ ಅನ್ನು ನಿಯೋಜಿಸಿ
azd up
```

ಏಕೆಂದರೆ ಇದು ಮುಗಿದಾಗ, azd ನಿಮ್ಮ ಅಪ್ಲಿಕೇಶನ್‌ನ URL ಅನ್ನು ಮುದ್ರಿಸುತ್ತದೆ. ಯಾವುದೇ azd ಅಪ್ಲಿಕೇಶನ್‌ನಂತೆ ಅದನ್ನು ಪರಿಶೀಲಿಸಿ:

```bash
azd show           # ಎಂಡ್‌ಪಾಯಿಂಟ್‌ಗಳನ್ನು ತೋರಿಸಿ
azd monitor --logs # ಅಗತ್ಯವಿದ್ದರೆ ಲಾಗ್‌ಗಳನ್ನು ಪರಿಶೀಲಿಸಿ
```

## ಸಾಮಾನ್ಯ ಪ್ರಥಮ-ಸಮಯ ಸಮಸ್ಯೆಗಳು

| ಲಕ್ಷಣ | ಸಾಧ್ಯ ಕಾರಣ | ಪರಿಹಾರ |
|---------|--------------|-----|
| azd ನನ್ನ ಅಪ್‌ ಅನ್ನು ಪತ್ತೆಮಾಡಲಿಲ್ಲ | ಮನಿಫೆಸ್ಟ್ ಕಾಣೆಯದು (ಉದಾ., `package.json`) | ಮನಿಫೆಸ್ಟ್ ಸೇರಿಸಿ, ಅಥವಾ `azd init` ಸಮಯದಲ್ಲಿ ಹೋಸ್ಟ್ ಅನ್ನು ಕೈಯಿಂದ ಆಯ್ಕೆಮಾಡಿ |
| `azd up` ವೇಳೆ ಬಿಲ್ಡ್ ವಿಫಲವಾಗುವುದು | ಅಪ್ಲಿಕೇಶನ್‌ಗಾಗಿ ಕಟ್ಟುವ ಹಂತ ಅಗತ್ಯ | `azure.yaml`ದಲ್ಲಿನ ಸೇವೆಯಡಿ `buildCommand`/`outputPath` ಸೇರಿಸಿ |
| ಅಪ್ಲಿಕೇಶನ್ ಪ್ರಾರಂಭವಾಗುತ್ತದೆ ಆದರೆ ದೋಷಗಳನ್ನು ಹಿಂದಿರುಗಿಸುತ್ತದೆ | ಕಾನ್ಫಿಗ್/ಗುಪ್ತ ಮಾಹಿತಿ ಕಾಣೆಯದು | `azd env set` ಬಳಸಿ ಮೌಲ್ಯಗಳನ್ನು ಸೆಟ್ ಮಾಡಿ ಅಥವಾ Key Vault ಅನ್ನು ಜೋಡಿಸಿ |
| ತಪ್ಪಾದ ಹೋಸ್ಟ್ ಆಯ್ಕೆಮಾಡಲಾಗಿದೆ | ಸ್ವಯಂಚಾಲಿತ ಪತ್ತೆಮಾಡಿದ್ದು ಗಾಳಿಯಲ್ಲಿ ಊಹಿಸಿ | `azure.yaml` ನಲ್ಲಿ `host:` ಅನ್ನು ಸಂಪಾದಿಸಿ ಮತ್ತು ಪುನರುತ್ಪಾದನೆಗೂ `azd up` ಅನ್ನು ಓಡಿಸಿ |

ಕಡಿಮೆ ಹೆಚ್ಚು ತಿಳಿವಳಿಕೆಗಾಗಿ, ನೋಡಿ [ಅಧ್ಯಾಯ 7: ಸಮಸ್ಯೆ ಪರಿಹಾರ](../chapter-07-troubleshooting/README.md).

## ಸಾರಾಂಶ

- `azd init` → **"Use code in the current directory"** ನಿಮ್ಮ ಬಳಿ ಈಗಾಗಲೇ ಇರುವ ಅಪ್ಲಿಕೇಶನ್‌ಗೆ azd ಅನ್ನು ಸೇರಿಸುತ್ತದೆ.
- `azure.yaml` ನಿಮ್ಮ ಕೋಡ್ ಫೋಲ್ಡರ್‌ಗಳನ್ನು Azure ಹೋಸ್ಟ್‌ಗಳೊಂದಿಗೆ ಮ್ಯಾಪ್ ಮಾಡುತ್ತದೆ; `infra/` Bicep ರೂಪದಲ್ಲಿ ಸಂಪನ್ಮೂಲಗಳನ್ನು ವ್ಯಾಖ್ಯಾನಿಸುತ್ತದೆ.
- `azd infra generate` ರಚಿಸಲಾದ ಇನ್‌ಫ್ರಾಸ್ಟ್ರಕ್ಚರ್ ಅನ್ನು ಪರಿಶೀಲಿಸಲು ಅಥವಾ ಕಸ್ಟಮೈಸ್ ಮಾಡಲು ಅವಕಾಶ ಮಾಡಿಕೊಡುತ್ತದೆ.
- ಒಂದು ನಿರ್ಧಾರವಾದ ನಂತರ, ನಿಮ್ಮ ಇರುವ ಅಪ್ಲಿಕೇಶನ್ ಟೆಂಪ್ಲೇಟ್ ಆಧಾರಿತ ಅಪ್ಲಿಕೇಶನ್‌ನಷ್ಟೇ ಇದೇ `azd up` / `azd down` ವರ್ಕ್‌ಫ್ಲೋ ಅನ್ನು ಬಳಸುತ್ತದೆ.
