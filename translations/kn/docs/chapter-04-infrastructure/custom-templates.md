- **📖 ಪ್ರಸ್ತುತ ಅಧ್ಯಾಯ**: ಅಧ್ಯಾಯ 4 - ಒದಗಿಸುವಿಕೆಯ ಪ್ರಕ್ರಿಯೆ ಮತ್ತು ಬ್ಯಾಂಕಿಂಗ್
- **⬅️ ಹಿಂದಿನದು**: [ಡಿಪ್ಲಾಯ್ ಸಿಗ್ನಲ್ಸ್](deployment-guide.md)
- **🚀 ಮುಂದಿನ ಅಧ್ಯಾಯ**: [ಅಧ್ಯಾಯ 5: ಮಲ್ಟಿ-ಏಜೆಂಟ್ ಪರಿಹಾರಗಳು](../chapter-05-multi-agent/README.md)

## పరిచಯ

ಈವರೆಗೆ ನೀವು `azd init --template <name>` ಬಳಸಿ ಟೆಮ್ಪ್ಲೇಟುಗಳನ್ನು *ಬಳಸಿದ್ದೀರಿ*. ಆದರೆ ನಿಮ್ಮ ತಂಡ ಇಷ್ಟಪಡುವ ಯೋಜನೆಯ ರೂಪರೇಖೆ ಒಂದು ಸೃಷ್ಟಿಸಿದಾಗ—ಉದಾಹರಣೆಗೆ, ಕಂಟೇನರ್ ಅಪ್ಲಿಕೇಶನ್‍ಗೆ Cosmos DB ಮತ್ತು ಸರಿಯಾದ ಮಾನಿಟರಿಂಗ್ ಇದ್ದಾಗ—ನೀವು ಅದನ್ನು ನಿಮ್ಮದೇ ಪುನಃಬಳಕೆಗೆ ತಕ್ಕ ಟೆಮ್ಪ್ಲೇಟಾಗಿ ಮಾರ್ಪಡಿಸಲು ಬಯಸುತ್ತೀರಿ. ಟೆಮ್ಪ್ಲೇಟು ಎಂದರೆ azd ಓದಿಕೊಳ್ಳಲು ಗೊತ್ತಿರುವ ನಿಯಮಿತ ರಚನೆಯೊಂದಿಗೆ Git ರಿಪೊಸಿಟರಿಯಾಗಿದೆ. ಈ ಪಾಠವು ನಿಮಗೆ ನಿರ್ಮಿಸಲು, ಪರೀಕ್ಷಿಸಲು ಮತ್ತು (ಐಚ್ಛಿಕ) ಸಮುದಾಯ ಗ್ಯಾಲರಿ‌ಗೆ ಪ್ರಕಟಿಸಲು ಎಚ್ಚರಸಲು ಹೇಗೆ ಎಂಬುದನ್ನು ತೋರಿಸುತ್ತದೆ.

## ಕಲಿಕೆ ಗುರಿಗಳು

ಈ ಪಾಠದ ಅಂತ್ಯಕ್ಕೆ ನೀವು:
- "azd ಟೆಮ್ಪ್ಲೇಟು" ಎಂದು ಫೋಲ್ಡರ್ ಏಕೆ ಅಳವಾಗುತ್ತದೆ ಎಂಬುದರ ತಿಳುವಳಿಕೆ ಹೊಂದಿರುತ್ತೀರಿ
- ಅಗತ್ಯವಿರುವ ಕಡತಗಳು ಮತ್ತು ಫೋಲ್ಡರ್ ವಿನ್ಯಾಸವನ್ನು ತಿಳಿಯಿರಿ
- ಇತರರು ತಪ್ಪದೆ ಪುನಃಬಳಕೆ ಮಾಡಿಕೊಳ್ಳಬಹುದಾದ `azure.yaml` ಮತ್ತು `infra/` ಅನ್ನು ಸೇರಿಸಬಹುದು
- ನಿಮ್ಮ ಟೆಮ್ಪ್ಲೇಟನ್ನು ಸ್ಥಳೀಯವಾಗಿ ಪರೀಕ್ಷಿಸಿ ಹಂಚಿಕೊಳ್ಳುವುದಕ್ಕೆ ಮೊದಲು ಪರೀಕ್ಷಿಸಬಹುದು
- ಅದನ್ನು ಪ್ರಕಟಿಸಿ ಹಾಗೂ (ಐಚ್ಛಿಕವಾಗಿ) Awesome AZD ಗೆ ಸಲ್ಲಿಸಬಹುದು

## ಕಲಿಕೆಯ ಫಲಿತಾಂಶಗಳು

ಈ ಪಾಠವನ್ನು ಮುಗಿಸಿದ ನಂತರ, ನೀವು:
- ಹೊಸ ಟೆಮ್ಪ್ಲೇಟು ರಿಪೊಸಿಟರಿಯನ್ನು ಸ್ಕ್ಯಾಫೋಲ್ಡ್ ಮಾಡಬೇಡಿ
- ಎಲ್ಲಿ ಬೇಕಾದರೂ ಕಾರ್ಯನಿರ್ವಹಿಸುವಂತೆ ಮೂಲಸಂರಚನೆಯನ್ನು ಪ್ರಾಮಾಣೀಕರಿಸಿ
- `azd init` ಮತ್ತು `azd up` ಬಳಸಿ ಟೆಮ್ಪ್ಲೇಟನ್ನು ಮಾನ್ಯತೆಗೊಳಿಸಿ
- ಸಮುದಾಯ ಗ್ಯಾಲರಿ ಬೇಕಾದ ಮೆಟಾಡೇಟಾವನ್ನು ಸೇರಿಸಿ

## ಟೆಮ್ಪ್ಲೇಟು ಎಂದರೆ ಏನು?

azd ಟೆಮ್ಪ್ಲೇಟು ಎಂದರೆ ಕನಿಷ್ಠವಾಗಿಸಿಕೊಂಡರೆ **Git ರಿಪೊಸಿಟರಿ** ಆಗಿದ್ದು, ಇದರಲ್ಲಿ ಇದಾಗಿರಬೇಕು:

| ಕಡತ / ಫೋಲ್ಡರ್ | ಉದ್ದೇಶ | ಅಗತ್ಯವಿದೆಯೇ? |
|---------------|---------|-----------|
| `azure.yaml` | ಸೇವೆಗಳು, ಹೋಸ್ಟ್ ಗಳು ಮತ್ತು ಮೂಲಸಂರಚನೆ ಪ್ರೊವೈಡರ್ ವಿವರಣೆ | ✅ ಹೌದು |
| `infra/` | ಬೈಸಪ್, ಟೆರೆಫಾರ್ಮ್ ಅಥವಾ ಪಿಸಲಮಿ ಮೂಲಸಂರಚನೆ ಪ್ರೊವಿಷನ್ ಆಗುತ್ತದೆ | ✅ ಹೌದು |
| `src/` (ಅಥವಾ ನಿಮ್ಮ ಕೋಡ್) | ಅಪ್ಲಿಕೇಶನ್ ಕೋಡ್ azd ಮೂಲಕ ನಿಯೋಜಿತ | ✅ ಹೌದು |
| `README.md` | ಟೆಮ್ಪ್ಲೇಟು ಬಳಸುವ ವಿಧಾನ | ✅ ಶಿಫಾರಸು ಮಾಡಲಾಗಿದೆ |
| `.azdo/` ಅಥವಾ `.github/` | CI/CD ಪೈಪ್‌ಲೈನ್ ವ್ಯಾಖ್ಯಾನಗಳು | ಐಚ್ಛಿಕ |
| `.devcontainer/` | ಪುನಃ ಸೃಷ್ಟಿಸಬಹುದಾದ ಅಭಿವೃದ್ಧಿ ಪರಿಸರ | ಐಚ್ಛಿಕ |
| `azure.yaml` `metadata` | ಗ್ಯಾಲರಿ + ಟೆಲಿಮೆಟ್ರಿ ಮಾಹಿತಿ | ಐಚ್ಛಿಕ (ಪ್ರಕರಶಿಸಲು ಅಗತ್ಯ) |

ಇಲ್ಲಿ ಯಾವುದೇ ಮಾಯಾಜಾಲ ಇಲ್ಲ: ನೀವು `azd init --template you/your-repo` ಸಂಚಾಲಿಸಿದಾಗ, azd ರಿಪೊ ಕ್ಲೋನ್ ಮಾಡಿ `azure.yaml` ಓದುವದು.

## ಹಂತ 1: ರಿಪೊಸಿಟರಿಯನ್ನು ಸ್ಕ್ಯಾಫೋಲ್ಡ್ ಮಾಡುವುದು

ಫೋಲ್ಡರ್ ರಚನೆಯನ್ನು ಕೈಯಿಂದ ನಿರ್ಮಿಸಿ ಅಥವಾ ಕನಿಷ್ಠ ಟೆಮ್ಪ್ಲೇಟಿನಿಂದ ಪ್ರಾರಂಭಿಸಿ ಮತ್ತು ಅದನ್ನು ತಿದ್ದುಪಡಿ ಮಾಡಿ:

```bash
mkdir my-azd-template && cd my-azd-template
git init

# ಮಾನಕ ವಿನ್ಯಾಸವನ್ನು ರಚಿಸಿ
mkdir -p src infra
```

ಸಾಮಾನ್ಯವಾಗಿ ಪೂರ್ಣಗೊಂಡ ವಿನ್ಯಾಸ ಹೀಗೆ ಕಾಣುತ್ತದೆ:

```
my-azd-template/
├── azure.yaml              # ← template definition
├── README.md               # ← how to use it
├── infra/
│   ├── main.bicep          # entry point
│   ├── main.parameters.json
│   └── modules/
│       └── web.bicep
├── src/
│   └── app/                # your application code
└── .github/
    └── workflows/
        └── azure-dev.yml   # optional CI/CD
```

## ಹಂತ 2: `azure.yaml` ಬರೆಯಿರಿ

ಇದು ಟೆಮ್ಪ್ಲೇಟಿನ ಹೃದಯವಾಗಿದೆ. ಇದು azd ಬಳಸಿ ಏನು ನಿಯೋಜಿಸುವುದೆಂದು ಹೇಳುತ್ತದೆ:

```yaml
# azure.yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/Azure/azure-dev/main/schemas/v1.0/azure.yaml.json
name: my-azd-template
metadata:
  template: my-azd-template@1.0.0   # name@version — used by the gallery and telemetry
infra:
  provider: bicep                   # or "terraform" / "pulumi"
  path: infra
services:
  web:
    project: ./src/app
    language: js                    # js | python | dotnet | java | go
    host: containerapp              # appservice | containerapp | function | aks | staticwebapp
```

> `metadata.template` ಕ್ಷೇತ್ರವು ಗ್ಯಾಲರಿ ಟೆಲಿಮೆಟ್ರಿ ಬಳಕೆ ಎಣಿಕೆಗಾಗಿ ಬಳಸಲಾಗುತ್ತದೆ. `name@version` ಪದ್ಧತಿಯನ್ನು ಬಳಸಿ.

## ಹಂತ 3: ಮೂಲಸಂರಚನೆಯನ್ನು ಪಾರುಮೆಟರೈಸ್ ಮಾಡಿ

ಪುನಃಬಳಕೆಗೊಳ್ಳಬಹುದಾದ ಟೆಮ್ಪ್ಲೇಟಿಗಾಗಿ ಅತ್ಯಂತ ಮುಖ್ಯ ನಿಯಮ: **ಹೆಸರುಗಳು, ಪ್ರದೇಶಗಳು ಅಥವಾ ಸಬ್ಸ್ಕ್ರಿಪ್ಷನ್ ವಿಶಿಷ್ಟ ಮೌಲ್ಯಗಳನ್ನು ಎಂದಿಗೂ ಹಾರ್ಡ್‌ಕೋಡ್ ಮಾಡಬೇಡಿ.** ಪಾರಾಮೀಟರ್‌ಗಳು ಮತ್ತು ನಿಯಮಿತ Resource Token ಮಾದರಿಯನ್ನು ಬಳಸಿ ಅಷ್ಟು ಟೆಮ್ಪ್ಲೇಟು ಯಾರದಾದರೂ ಸಬ್ಸ್ಕ್ರಿಪ್ಷನ್‌ನಲ್ಲಿ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ.

```bicep
// infra/main.bicep
targetScope = 'resourceGroup'

@minLength(1)
@description('Primary location for all resources')
param location string = resourceGroup().location

@minLength(1)
@maxLength(64)
@description('Name of the environment (used to generate resource names)')
param environmentName string

// Generate globally-unique, stable names from the environment + subscription
var resourceToken = uniqueString(subscription().id, environmentName, location)
var tags = { 'azd-env-name': environmentName }

module web 'modules/web.bicep' = {
  name: 'web'
  params: {
    name: 'app-${resourceToken}'
    location: location
    tags: tags
  }
}

output SERVICE_WEB_ENDPOINT_URL string = web.outputs.uri
```

ಈ ಟೆಮ್ಪ್ಲೇಟನ್ನು ಸ್ನೇಹಪೂರ್ವಕವಾಗಿಸುವ ಎರಡು ವಿಷಯಗಳು:

1. **`azd-env-name` ಟ್ಯಾಗ್** — azd ಎಲ್ಲ ಪರಿಸರಗಳ ಸಂಪನ್ಮೂಲಗಳನ್ನು ಟ್ರ್ಯಾಕ್ ಮಾಡಿ ಶುಚಿಗೊಳಿಸಲು ಇದನ್ನು ಉಪಯೋಗಿಸುತ್ತದೆ.
2. **`uniqueString(...)` ರಿಸೋರ್ಸ್ ಟೋಕನ್** — ಹೆಸರಿನ ಸಂಘರ್ಷ ತಪ್ಪಿಸಲು ಸ್ಥಿರ, ಜಾಗತಿಕವಾಗಿ ವಿಶಿಷ್ಟ ಉಪಸರ್ಗವನ್ನು ಉತ್ಪಾದಿಸುತ್ತದೆ.

ಹೆಸರಿನ ಮ್ಯಾಚಿಂಗ್ ಪಾರಾಮೀಟರ್ಸ್ ಕಡತವನ್ನು ಒದಗಿಸಿ, ಅದು ಪರಿಸರದಿಂದ azd ಒಳಗಾಗಿ ಮೌಲ್ಯಗಳನ್ನು ಓದುತ್ತದೆ:

```json
// infra/main.parameters.json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    "environmentName": { "value": "${AZURE_ENV_NAME}" },
    "location": { "value": "${AZURE_LOCATION}" }
  }
}
```

azd ಸ್ವಯಂಚಾಲಿತವಾಗಿ ಪ್ರಸ್ತುತ ಪರಿಸರದಿಂದ `${AZURE_ENV_NAME}` ಮತ್ತು `${AZURE_LOCATION}` ಅನ್ನು ಬದಲಾಯಿಸುತ್ತದೆ.

## ಹಂತ 4: ನಿಮ್ಮ ಟೆಮ್ಪ್ಲೇಟನ್ನು ಸ್ಥಳೀಯವಾಗಿ ಪರೀಕ್ಷಿಸಿ

ಹಂಚಿಕೊಳ್ಳುವುದಕ್ಕೆ ಮೊದಲು, ಟೆಮ್ಪ್ಲೇಟು ಸ್ವಚ್ಛ ಸ್ಥಿತಿಯಿಂದ ಕೆಲಸ ಮಾಡುತ್ತದೆ ಎಂದು ಸಾಬೀತು ಪಡಿಸಿ.

**ಸ್ಥಳೀಯ ಫೋಲ್ಡರ್‌ನಿಂದ ಪರೀಕ್ಷಿಸಿ** (Git ಪುಷ್ ಅಗತ್ಯವಿಲ್ಲ):

```bash
# ಖಾಲಿ ಡೈರೆಕ್ಟರಿಗೆ, ನಿಮ್ಮ ಸ್ಥಳೀಯ ಟೆಂಪ್ಲೇಟ್ ಪಾತ್ ಅನ್ನು ಬಳಸಿ ಪ್ರಾರंभಿಸಿ
mkdir /tmp/test-run && cd /tmp/test-run
azd init --template /path/to/my-azd-template

# ಪ್ರಾವೀಣ್ಯಮೂಡಿಸಿ + ಕೊನೆಯಲ್ಲಿ ನಿಯೋಜಿಸಿ
azd auth login
azd up
```

**ಆಮೇಲೆ ಟೆರ್ಡೌನ್ ಪರೀಕ್ಷಿಸಿ**—ಒಳ್ಳೆಯ ಟೆಮ್ಪ್ಲೇಟು ಸಂಪೂರ್ಣವಾಗಿ ಶುದ್ಧೀಕರಿಸುತ್ತದೆ:

```bash
azd down --force --purge
```

`azd down` ಉಳಿಸಿದರೆ, ನೀವು ಬಹುಶಃ ಸಂಪನ್ಮೂಲದಲ್ಲಿ `azd-env-name` ಟ್ಯಾಗ್ ಅನ್ನು ಮರೆತಿರಬಹುದು.

> **ಸಲಹೆ:** ಮೊದಲು `azd provision --preview` ಅನ್ನು ಸಂಚಾಲಿಸಿ. ಇದು ಏನಾಗಬಹುದು ಎಂಬುದನ್ನು ಪರೀಕ್ಷಿಸುತ್ತದೆ ಮತ್ತು ಸಂಪನ್ಮೂಲ ಸೃಷ್ಟಿಸುವ ಮೊದಲು ಟೆಮ್ಪ್ಲೇಟಿನ ದೋಷಗಳನ್ನು ಪರಿಚಯಿಸುತ್ತದೆ.

## ಹಂತ 5: ಟೆಮ್ಪ್ಲೇಟನ್ನು ಪ್ರಕಟಿಸಿ

ರಿಪೊ GitHub ಗೆ (ಜನತೆಗೆ ಬಳಸಲು ಬಯಸಿದರೆ ಸಾರ್ವಜನಿಕವಾಗಿ) ಪுஷ್ ಮಾಡಿ:

```bash
gh repo create my-azd-template --public --source=. --push
```

ಈಗ ಯಾರೂ ಅದನ್ನು ಬಳಸಬಹುದು:

```bash
azd init --template your-github-username/my-azd-template
```

## ಹಂತ 6 (ಐಚ್ಛಿಕ): Awesome AZD ಗೆ ಸಲ್ಲಿಸಿ

[Awesome AZD ಗ್ಯಾಲರಿ](https://azure.github.io/awesome-azd/) ಸಮುದಾಯ ಟೆಮ್ಪ್ಲೇಟುಗಳನ್ನು ಪಟ್ಟಿ ಮಾಡುತ್ತದೆ. ಪಟ್ಟಿ ಆಗಬೇಕಾದ ನಿಮ್ಮ ರಿಪೊ ಈ ಸಂಗತಿಗಳು ಒಳಗೊಂಡಿರಬೇಕು:

- ✅ prerequisites, ವಾಸ್ತುಶಿಲ್ಪ ಚಿತ್ತರಣೆ ಮತ್ತು ವೆಚ್ಚದ ಟಿಪ್ಪಣಿಯೊಂದಿಗೆ ಸ್ಪಷ್ಟ `README.md`
- ✅ `metadata.template` ಹೊಂದಿರುವ ಕಾರ್ಯನಿರ್ವಹಿಸುವ `azure.yaml`
- ✅ ಫ್ರೆಶ್ ಸಬ್ಸ್ಕ್ರಿಪ್ಷನ್‍ನಲ್ಲಿ ಸ್ವಚ್ಛವಾಗಿ ಪ್ರೊವಿಷನ್ ಆಗುವ ಮೂಲಸಂರಚನೆ
- ✅ `LICENSE` ಕಡತ
- ✅ (ಶಿಫಾರಸು) ಒನ್-ಕ್ಲಿಕ್ ಕೋಡ್‌ಸ್ಪೇಸ್‌ಗಳಿಗೆ `.devcontainer/`

Awesome AZD ರಿಪೊಸಿಟರಿಯಲ್ಲಿನ ಕೊಡುಗೆ ಮಾರ್ಗದರ್ಶನವನ್ನು ಅನುಸರಿಸಿ, ನೂತನ ಟೆಮ್ಪ್ಲೇಟನ್ನು ಗ್ಯಾಲರಿ ಡೇಟಾ ಕಡತಕ್ಕೆ ಸೇರಿಸಿ ಒಂದು ಪುಲ್ ರಿಕ್ವಸ್ಟ್ ತೆರೆಯಿರಿ.

---

## ಸಾಮಾನ್ಯ ತಪ್ಪುಗಳು

| ತಪ್ಪು | ಪರಿಹಾರ |
|---------|-----|
| ಹಾರ್ಡ್‌ಕೋಡ್ ಮಾಡಿದ ಸಂಪನ್ಮೂಲ ಹೆಸರುಗಳು | `uniqueString()` ರಿಸೋರ್ಸ್ ಟೋಕನ್ ಬಳಸಿ |
| `azd down` ಸಂಪನ್ಮೂಲಗಳನ್ನು ಬಿಟ್ಟುಕೊಡುತ್ತದೆ | ಪ್ರತಿಯೊಂದು ಸಂಪನ್ಮೂಲ (ಅಥವಾ ಸಂಪನ್ಮೂಲ ಗುಂಪು) `azd-env-name` ಟ್ಯಾಗ್ ಮಾಡಿ |
| ಟೆಮ್ಪ್ಲೇಟು ನಿಮಗೆ ಕೆಲಸ ಮಾಡುತ್ತದೆ, ಇತರರಿಗೆ ವಿಫಲವಾಗಿದೆ | ಸಬ್ಸ್ಕ್ರಿಪ್ಷನ್ ಐಡಿ, ಟೆನಂಟ್ ಐಡಿ ಮತ್ತು ಪ್ರದೇಶ ಕಲ್ಪನೆಗಳನ್ನು ತೆಗೆದುಹಾಕಿ |
| ಔಟ್‌ಪುಟ್‌ಗಳು ಆಪ್‌ನಲ್ಲಿ ಸಂಪರ್ಕಿಸಲ್ಪಟ್ಟಿಲ್ಲ | `SERVICE_<NAME>_ENDPOINT_URL` ಮತ್ತು ಇತರ `AZURE_*` ಔಟ್‌ಪುಟ್‌ಗಳನ್ನು ರಫ್ತು ಮಾಡಿ |
| ಗ್ಯಾಲರಿ ಸಲ್ಲಿಕೆ ತಿರಸ್ಕರಿಸಲಾಗಿದೆ | `README.md`, `LICENSE`, ಮತ್ತು `metadata.template` ಸೇರಿಸಿ |

---

## ಸಾರಾಂಶ

- ಟೆಮ್ಪ್ಲೇಟು ಎಂದರೆ ಗಿಟ್ ರಿಪೊ ಅದು `azure.yaml`, `infra/` ಮತ್ತು ನಿಮ್ಮ ಕೋಡ್ ಹೊಂದಿರುತ್ತದೆ.
- ಎಲ್ಲವನ್ನೂ ಪಾರಾಮೀಟರೈಸ್ ಮಾಡಿ—ಹೆಸರುಗಳು, ಪ್ರದೇಶಗಳು ಮತ್ತು IDಗಳು—ಹಾಗಾಗಿ ಎಲ್ಲಿಗೆ ಬೇಕಾದರೂ ಕಾರ್ಯನಿರ್ವಹಿಸುತ್ತದೆ.
- `azd down` ಕೆಲಸ ಮಾಡಲು ಹುದುಗಿಯೂ ಉತ್ತಮ ಟ್ಯಾಗ್ ಮಾಡಿಕೊಂಡು ಸಂಪನ್ಮೂಲಗಳನ್ನು `azd-env-name` ಟ್ಯಾಗ್ ಮಾಡಿ.
- ಪ್ರಕಟಿಸುವ ಮೊದಲು `azd init --template <local-path>` ಬಳಸಿ ಸ್ಥಳೀಯವಾಗಿ ಪರೀಕ್ಷಿಸಿ.
- Awesome AZD ಗೆ ಸಲ್ಲಿಸಲು ಮೆಟಾಡೇಟಾ ಮತ್ತು README ಸೇರಿಸಿ.
