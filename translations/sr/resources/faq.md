## Почетак

### П: Шта је Azure Developer CLI (azd)?
**О**: Azure Developer CLI (azd) је алатка за командну линију усмерена на програмере која убрзава време потребно за пребацивање апликације из локалног развојног окружења у Azure. Пружа најбоље праксе кроз шаблоне и помаже у целом животном циклусу имплементације.

### П: По чему се azd разликује од Azure CLI?
**О**: 
- **Azure CLI**: Алатка опште намене за управљање Azure ресурсима
- **azd**: Алатка усмерена на програмере за радне токове имплементације апликација
- azd интерно користи Azure CLI, али пружа апстракције вишег нивоа за уобичајене развојне сценарије
- azd укључује шаблоне, управљање окружењима и аутоматизацију имплементације

### П: Да ли ми је потребан Azure CLI за коришћење azd?
**О**: Да, azd захтева Azure CLI за аутентификацију и неке операције. Прво инсталирајте Azure CLI, а затим azd.

### П: Које програмске језике подржава azd?
**О**: azd је независан од језика. Ради са:
- Node.js/JavaScript/TypeScript
- Python
- .NET/C#
- Java
- Go
- PHP
- Статичке веб странице
- Апликације у контејнерима

### П: Могу ли користити azd са постојећим пројектима?
**О**: Да! Можете:
1. Користити `azd init` за додавање azd конфигурације постојећим пројектима
2. Прилагодити постојеће пројекте да одговарају структури azd шаблона
3. Креирати прилагођене шаблоне на основу постојеће архитектуре

## Аутентификација и приступ

### П: Како да се аутентификујем са Azure користећи azd?
**О**: Користите `azd auth login`, који ће отворити прозор у претраживачу за Azure аутентификацију. За CI/CD сценарије, користите сервисне принципале или управљане идентитете.

### П: Могу ли користити azd са више Azure претплата?
**О**: Да. Користите `azd env set AZURE_SUBSCRIPTION_ID <subscription-id>` да наведете коју претплату желите да користите за свако окружење.

### П: Које дозволе су ми потребне за имплементацију са azd?
**О**: Типично вам је потребно:
- **Contributor** улога на групи ресурса или претплати
- **User Access Administrator** ако имплементирате ресурсе који захтевају додељивање улога
- Специфичне дозволе варирају у зависности од шаблона и ресурса који се имплементирају

### П: Могу ли користити azd у CI/CD процесима?
**О**: Апсолутно! azd је дизајниран за интеграцију са CI/CD. Користите сервисне принципале за аутентификацију и подесите променљиве окружења за конфигурацију.

### П: Како да управљам аутентификацијом у GitHub Actions?
**О**: Користите Azure Login акцију са акредитивима сервисног принципала:
```yaml
- uses: azure/login@v1
  with:
    creds: ${{ secrets.AZURE_CREDENTIALS }}
- run: azd deploy --no-prompt
```

## Шаблони и пројекти

### П: Где могу пронаћи azd шаблоне?
**О**: 
- Званични шаблони: [Azure-Samples/awesome-azd](https://github.com/Azure-Samples/awesome-azd)
- Шаблони заједнице: Претражите GitHub за "azd-template"
- Користите `azd template list` за преглед доступних шаблона

### П: Како да креирам прилагођени шаблон?
**О**: 
1. Почните са постојећом структуром шаблона
2. Измените `azure.yaml`, инфраструктурне датотеке и код апликације
3. Темељно тестирајте са `azd up`
4. Објавите на GitHub-у са одговарајућим ознакама

### П: Могу ли користити azd без шаблона?
**О**: Да, користите `azd init` у постојећем пројекту да креирате неопходне конфигурационе датотеке. Мораћете ручно да конфигуришете `azure.yaml` и инфраструктурне датотеке.

### П: Која је разлика између званичних и шаблона заједнице?
**О**: 
- **Званични шаблони**: Одржава их Microsoft, редовно ажурирани, са свеобухватном документацијом
- **Шаблони заједнице**: Креирани од стране програмера, могу имати специјализоване случајеве употребе, различитог квалитета и одржавања

### П: Како да ажурирам шаблон у свом пројекту?
**О**: Шаблони се не ажурирају аутоматски. Можете:
1. Ручно упоредити и спојити измене из изворног шаблона
2. Почети испочетка са `azd init` користећи ажурирани шаблон
3. Изабрати специфична побољшања из ажурираних шаблона

## Имплементација и инфраструктура

### П: Које Azure услуге може azd да имплементира?
**О**: azd може имплементирати било коју Azure услугу кроз Bicep/ARM шаблоне, укључујући:
- App Services, Container Apps, Functions
- Базе података (SQL, PostgreSQL, Cosmos DB)
- Складиштење, Key Vault, Application Insights
- Мрежне, безбедносне и мониторинг ресурсе

### П: Могу ли имплементирати у више региона?
**О**: Да, конфигуришите више региона у вашим Bicep шаблонима и подесите параметар локације за свако окружење.

### П: Како да управљам миграцијама шема базе података?
**О**: Користите deployment hooks у `azure.yaml`:
```yaml
hooks:
  postdeploy:
    posix:
      run: ./scripts/migrate-database.sh
    windows:
      run: ./scripts/migrate-database.ps1
```

### П: Могу ли имплементирати само инфраструктуру без апликација?
**О**: Да, користите `azd provision` за имплементацију само инфраструктурних компоненти дефинисаних у вашим шаблонима.

### П: Како да имплементирам у постојеће Azure ресурсе?
**О**: Ово је сложено и није директно подржано. Можете:
1. Увезти постојеће ресурсе у ваше Bicep шаблоне
2. Користити референце постојећих ресурса у шаблонима
3. Изменити шаблоне да условно креирају или референцирају ресурсе

### П: Могу ли користити Terraform уместо Bicep-а?
**О**: Тренутно, azd примарно подржава Bicep/ARM шаблоне. Terraform подршка није званично доступна, иако могу постојати решења из заједнице.

## Конфигурација и окружења

### П: Како да управљам различитим окружењима (dev, staging, prod)?
**О**: Креирајте одвојена окружења са `azd env new <environment-name>` и конфигуришите различита подешавања за свако:
```bash
azd env new development
azd env new staging  
azd env new production
```

### П: Где се чувају конфигурације окружења?
**О**: У `.azure` фасцикли унутар вашег директоријума пројекта. Свако окружење има своју фасциклу са конфигурационим датотекама.

### П: Како да подесим конфигурацију специфичну за окружење?
**О**: Користите `azd env set` за подешавање променљивих окружења:
```bash
azd env set AZURE_LOCATION eastus
azd env set DATABASE_TIER Basic
```

### П: Могу ли делити конфигурације окружења са члановима тима?
**О**: `.azure` фасцикла садржи осетљиве информације и не би требало да се додаје у верзионисање. Уместо тога:
1. Документујте потребне променљиве окружења
2. Користите скрипте за подешавање окружења
3. Користите Azure Key Vault за осетљиву конфигурацију

### П: Како да заменим подразумеване вредности шаблона?
**О**: Подесите променљиве окружења које одговарају параметрима шаблона:
```bash
azd env set LOCATION "West US 2"
azd env set SKU_NAME "B1"
```

## Отклањање проблема

### П: Зашто `azd up` не ради?
**О**: Уобичајени узроци:
1. **Проблеми са аутентификацијом**: Покрените `azd auth login`
2. **Недовољне дозволе**: Проверите своје Azure улоге
3. **Конфликти у именовању ресурса**: Промените AZURE_ENV_NAME
4. **Проблеми са квотама/капацитетом**: Проверите доступност региона
5. **Грешке у шаблону**: Валидација Bicep шаблона

### П: Како да дебагујем грешке у имплементацији?
**О**: 
1. Користите `azd deploy --debug` за детаљан излаз
2. Проверите историју имплементације у Azure порталу
3. Прегледајте Activity Log у Azure порталу
4. Користите `azd show` за приказ тренутног стања окружења

### П: Зашто моје променљиве окружења не раде?
**О**: Проверите:
1. Имена променљивих тачно одговарају параметрима шаблона
2. Вредности су правилно наведене ако садрже размаке
3. Одабрано је исправно окружење: `azd env select <environment>`
4. Променљиве су подешене у исправном окружењу

### П: Како да очистим неуспеле имплементације?
**О**: 
```bash
azd down --force --purge
```
Ово уклања све ресурсе и конфигурацију окружења.

### П: Зашто моја апликација није доступна након имплементације?
**О**: Проверите:
1. Да ли је имплементација успешно завршена
2. Да ли апликација ради (проверите логове у Azure порталу)
3. Да ли мрежне безбедносне групе дозвољавају саобраћај
4. Да ли су DNS/прилагођени домени правилно конфигурисани

## Трошкови и наплата

### П: Колико ће коштати azd имплементације?
**О**: Трошкови зависе од:
- Azure услуга које се имплементирају
- Изабраних нивоа услуга/SKU-ова
- Регионалних разлика у ценама
- Шаблона коришћења

Користите [Azure Pricing Calculator](https://azure.microsoft.com/pricing/calculator/) за процене.

### П: Како да контролишем трошкове у azd имплементацијама?
**О**: 
1. Користите ниже нивое услуга за развојна окружења
2. Подесите Azure буџете и обавештења
3. Користите `azd down` за уклањање ресурса када нису потребни
4. Изаберите одговарајуће регионе (трошкови варирају по локацији)
5. Користите Azure Cost Management алате

### П: Да ли постоје бесплатне опције за azd шаблоне?
**О**: Многе Azure услуге нуде бесплатне нивое:
- App Service: Доступан бесплатан ниво
- Azure Functions: 1М бесплатних извршења месечно
- Cosmos DB: Бесплатан ниво са 400 RU/s
- Application Insights: Првих 5GB/месечно бесплатно

Конфигуришите шаблоне да користе бесплатне нивое где је могуће.

### П: Како да проценим трошкове пре имплементације?
**О**: 
1. Прегледајте `main.bicep` шаблона да видите који се ресурси креирају
2. Користите Azure Pricing Calculator са специфичним SKU-овима
3. Прво имплементирајте у развојно окружење да пратите стварне трошкове
4. Користите Azure Cost Management за детаљну анализу трошкова

## Најбоље праксе

### П: Које су најбоље праксе за структуру azd пројекта?
**О**: 
1. Држите код апликације одвојено од инфраструктуре
2. Користите смислена имена услуга у `azure.yaml`
3. Примените правилно руковање грешкама у скриптама за изградњу
4. Користите конфигурацију специфичну за окружење
5. Укључите свеобухватну документацију

### П: Како да организујем више услуга у azd?
**О**: Користите препоручену структуру:
```
├── src/
│   ├── web/          # Frontend service
│   ├── api/          # Backend service  
│   └── worker/       # Background service
├── infra/            # Infrastructure templates
├── azure.yaml        # azd configuration
└── README.md         # Project documentation
```

### П: Да ли треба да додам `.azure` фасциклу у верзионисање?
**О**: **Не!** `.azure` фасцикла садржи осетљиве информације. Додајте је у `.gitignore`:
```gitignore
.azure/
```

### П: Како да управљам тајнама и осетљивом конфигурацијом?
**О**: 
1. Користите Azure Key Vault за тајне
2. Референцирајте тајне из Key Vault-а у конфигурацији апликације
3. Никада не додајте тајне у верзионисање
4. Користите управљане идентитете за аутентификацију између услуга

### П: Који је препоручени приступ за CI/CD са azd?
**О**: 
1. Користите одвојена окружења за
1. Анализирајте тренутну архитектуру распореда  
2. Креирајте еквивалентне Bicep шаблоне  
3. Конфигуришите `azure.yaml` да одговара тренутним сервисима  
4. Темељно тестирајте у развојном окружењу  
5. Постепено мигрирајте окружења
