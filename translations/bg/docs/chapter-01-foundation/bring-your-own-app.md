## Въведение

В [Вашият първи проект](first-project.md) разгръщахте приложение, като започвахте от шаблон. Но повечето пъти вече *имате* приложение — Node.js API, Python Flask услуга, .NET web приложение — намиращо се в папка на вашата машина. Този урок показва как да добавите azd към съществуващия код, за да можете да го разгръщате с `azd up`, без шаблон.

## Цели на обучението

В края на този урок ще можете:
- Да разберете трите начина за стартиране на azd проект
- Да изпълните `azd init` в съществуващ код
- Да разберете какво правят `azure.yaml` и папката `infra/` за вашето приложение
- Да знаете кога да позволите на azd да генерира инфраструктура и кога да пишете своя собствена
- Да разгръщате съществуващото си приложение в Azure с `azd up`

## Изходни резултати от обучението

След приключване на урока ще можете:
- Да инициализирате azd в проект, който вече имате
- Да прочетете и редактирате основен файл `azure.yaml`
- Да генерирате стартова инфраструктура с `azd infra generate`
- Да изберете подходящ хост в Azure за вашето приложение
- Да разгръщате и почиствате собственото си приложение

## Три начина за стартиране на azd проект

| Начална точка | Команда | Кога да се използва |
|-------------|---------|--------------------|
| **От шаблон** | `azd init --template <name>` | За обучение или започване на ново приложение от доказан пример |
| **От съществуващия ви код** | `azd init` (в папката на проекта) | Вече имате приложение и искате да го разгръщате |
| **От Git хранилище** | `azd init --from-code` (в клонирано хранилище) | Приемане на azd за съществуващо хранилище |

Вече сте практикували първата опция. Този урок разглежда втората — най-честият реален сценарий.

## Стъпка 1: Изпълнете `azd init` в проекта си

Отворете терминал **вътре в папката на съществуващия проект** и изпълнете:

```bash
cd my-existing-app
azd init
```

azd ще попита как искате да инициализирате. Изберете:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

Изберете **"Използвай код в текущата директория."** azd сканира папката, открива езика и рамката, и предлага хост.

### Какво открива azd

azd търси сигнали като `package.json`, `requirements.txt`, `pom.xml`, `*.csproj` или `Dockerfile` и предлага съответстващ Azure хост:

| Вашето приложение | Вероятно открит хост |
|------------------|---------------------|
| Node.js / Python / .NET web приложение | Azure App Service или Container Apps |
| Контейнеризирано приложение (`Dockerfile`) | Azure Container Apps |
| Функционално приложение | Azure Functions |
| Статичен сайт (React/Vue изход за билд) | Azure Static Web Apps |

Потвърдете откритите услуги и azd създава необходимите файлове.

## Стъпка 2: Разберете какво създаде azd

След инициализацията ще имате две нови неща в проекта:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — дефиниция на проекта

Това е сърцето на azd проекта. Минимален пример изглежда така:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

Блокът `services` е ключовата част: всяка позиция свързва папка от вашия код с Azure хост. Ако приложението ви има и фронтенд и API, ще имате две услуги.

### `infra/` — вашите Azure ресурси като код

Папката `infra/` съдържа Bicep файлове, които дефинират Azure ресурсите, необходими за вашето приложение (App Service, база данни и др.). Не е нужно да ги пишете ръчно — azd генерира работеща начална точка. Можете да ги редактирате по-късно, за да добавяте ресурси или да засилите сигурността (описано в [Глава 4](../chapter-04-infrastructure/README.md)).

> **Съвет:** Искате ли да видите или персонализирате генерираната инфраструктура преди разгръщане? Изпълнете `azd infra generate` (достъпно и като `azd infra synth`), за да изведете IaC на диск, за да можете да я прегледате и контролирате версията ѝ.

## Стъпка 3: Задайте необходимата конфигурация

Ако приложението ви се нуждае от настройки или тайни (напр. низ за връзка, API ключ), не ги кодирайте директно. Използвайте стойности от средата:

```bash
# Създайте среда
azd env new dev

# Задайте стойност, която не е тайна
azd env set API_VERSION 1.0.0
```

При истински тайни ги съхранявайте в Key Vault и ги реферирайте от вашата инфраструктура — вижте [Глава 3: Конфигурация и автентикация](../chapter-03-configuration/authsecurity.md).

## Стъпка 4: Разгръщане

Сега използвайте същия работен поток, който вече знаете:

```bash
# Удостоверяване (изисква се за azd)
azd auth login

# Прегледайте ресурсите, които ще бъдат създадени
azd provision --preview

# Осигурете инфраструктура и разположете своя код
azd up
```

Когато приключи, azd извежда URL на приложението ви. Проверете го по същия начин, както при всяко azd приложение:

```bash
azd show           # покажи краища на точките
azd monitor --logs # проверете логовете ако е необходимо
```

## Често срещани проблеми при първото използване

| Симптом | Вероятна причина | Решение |
|---------|-----------------|---------|
| azd не откри приложението ми | Липсва манифест (напр. `package.json`) | Добавете манифеста или изберете хоста ръчно при `azd init` |
| Сграждането се проваля по време на `azd up` | Приложението се нуждае от стъпка за билд | Добавете `buildCommand`/`outputPath` под услугата в `azure.yaml` |
| Приложението стартира, но връща грешки | Липсва конфигурация/тайна | Задайте стойности с `azd env set` или свържете Key Vault |
| Избран е грешен хост | Автоматичното откриване се е объркало | Редактирайте `host:` в `azure.yaml` и пуснете отново `azd up` |

За повече информация вижте [Глава 7: Отстраняване на неизправности](../chapter-07-troubleshooting/README.md).

## Почистване

```bash
azd down --force --purge
```

---

## Резюме

- `azd init` → **"Използвай код в текущата директория"** добавя azd към приложение, което вече имате.
- `azure.yaml` свързва папките с кода с Azure хостове; `infra/` дефинира ресурсите чрез Bicep.
- `azd infra generate` ви позволява да преглеждате или персонализирате генерираната инфраструктура.
- След инициализация, вашето съществуващо приложение използва същия работен поток `azd up` / `azd down`, както приложение базирано на шаблон.
