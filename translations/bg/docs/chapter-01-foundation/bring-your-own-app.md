## Въведение

В [Вашия първи проект](first-project.md) разположихте приложение, като започнахте от шаблон. Но в повечето случаи вече имате приложение — Node.js API, Python Flask услуга, .NET уеб приложение — намиращо се в папка на вашия компютър. Този урок показва как да добавите azd към този съществуващ код, за да можете да го разположите с `azd up`, без да е необходим шаблон.

## Учебни цели

Към края на този урок ще:
- Разберете трите начина за стартиране на проект с azd
- Изпълните `azd init` в съществуваща кодова база
- Разберете какво правят `azure.yaml` и папката `infra/` за вашето приложение
- Знаете кога да позволите на azd да генерира инфраструктура и кога да напишете собствена
- Разположите съществуващото си приложение в Azure с `azd up`

## Резултати от обучението

След завършване на този урок ще можете да:
- Инициализирате azd в проект, който вече имате
- Четете и редактирате основен файл `azure.yaml`
- Генерирате начална инфраструктура с `azd infra generate`
- Изберете подходящ хост в Azure за вашето приложение
- Разположите и почистите собственото си приложение

## Три начина за стартиране на проект с azd

| Starting point | Command | When to use |
|----------------|---------|-------------|
| **От шаблон** | `azd init --template <name>` | За учене или за започване на ново приложение от доказан пример |
| **От съществуващия ви код** | `azd init` (in your project folder) | Вече имате приложение и искате да го разположите |
| **От Git хранилище** | `azd init --from-code` (in a cloned repo) | Въвеждане на azd в съществуващо хранилище |

Вече практикувахте първата опция. Този урок обхваща втората — най-често срещаният сценарий в реалния свят.

## Стъпка 1: Изпълнете `azd init` в своя проект

Отворете терминал **вътре в папката на съществуващия си проект** и изпълнете:

```bash
cd my-existing-app
azd init
```

azd ще попита как искате да инициализирате. Изберете:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

Изберете **"Използвайте кода в текущата директория."** azd след това сканира папката ви, разпознава езика и фреймуърка, и предлага хост.

### Какво открива azd

azd търси сигнали като `package.json`, `requirements.txt`, `pom.xml`, `*.csproj`, или `Dockerfile`, и предлага подходящ хост в Azure:

| Вашето приложение | Вероятно открит хост |
|----------|----------------------|
| Node.js / Python / .NET уеб приложение | Azure App Service или Container Apps |
| Контейнеризирано приложение (`Dockerfile`) | Azure Container Apps |
| Функционно приложение | Azure Functions |
| Статичен сайт (изход на билд от React/Vue) | Azure Static Web Apps |

Потвърдете откритата услуга(и), и azd генерира необходимите файлове.

## Стъпка 2: Разберете какво е създал azd

След инициализация ще имате две нови неща в проекта си:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — дефиницията на проекта

Това е сърцето на един azd проект. Един минимален пример изглежда така:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

Блокът `services` е ключовата част: всяка запис съпоставя папка с вашия код към хост в Azure. Ако вашето приложение има както фронтенд, така и API, ще имате две услуги.

### `infra/` — вашите Azure ресурси като код

Папката `infra/` съдържа Bicep файлове, които дефинират Azure ресурсите, необходими за вашето приложение (App Service, базата данни и т.н.). Не е необходимо да ги пишете ръчно — azd генерира работна начална точка. Можете да ги редактирате по-късно, за да добавите ресурси или да затегнете сигурността (обсъжда се в [Глава 4](../chapter-04-infrastructure/README.md)).

> **Съвет:** Искате ли да видите или персонализирате генерираната инфраструктура преди разполагането? Стартирайте `azd infra generate` (също достъпно като `azd infra synth`), за да запишете IaC на диск, така че да можете да го прегледате и поставите под контрол на версиите.

## Стъпка 3: Задайте необходимата конфигурация

Ако приложението ви се нуждае от настройки или тайни (например connection string, API ключ), не ги вграждайте в кода. Използвайте стойности от средата:

```bash
# Създайте среда
azd env new dev

# Задайте стойност, която не е поверителна
azd env set API_VERSION 1.0.0
```

За истински тайни ги съхранявайте в Key Vault и ги реферирайте от вашата инфраструктура — вижте [Глава 3: Конфигурация и удостоверяване](../chapter-03-configuration/authsecurity.md).

## Стъпка 4: Разположете

Сега използвайте същия работен процес, който вече знаете:

```bash
# Удостоверете се (задължително за azd)
azd auth login

# Прегледайте ресурсите, които ще бъдат създадени
azd provision --preview

# Осигурете инфраструктурата и внедрете кода си
azd up
```

Когато приключи, azd показва URL на вашето приложение. Проверете го по същия начин както при всяко azd приложение:

```bash
azd show           # покажи крайни точки
azd monitor --logs # проверете логовете при необходимост
```

## Чести проблеми при първото използване

| Симптом | Вероятна причина | Решение |
|---------|--------------|-----|
| azd не откри приложението ми | Липсва манифест (например `package.json`) | Добавете манифеста, или изберете хоста ръчно по време на `azd init` |
| Сглобяването (Build) не успява по време на `azd up` | Приложението се нуждае от стъпка за билд | Добавете `buildCommand`/`outputPath` под услугата в `azure.yaml` |
| Приложението стартира, но връща грешки | Липсва конфигурация/тайна | Задайте стойности с `azd env set` или свържете Key Vault |
| Избран е неправилен хост | Автоматичното откриване е направило предположение | Редактирайте `host:` в `azure.yaml` и пуснете отново `azd up` |

За повече вижте [Глава 7: Отстраняване на неизправности](../chapter-07-troubleshooting/README.md).

## Почистване

```bash
azd down --force --purge
```

---

## Резюме

- `azd init` → **"Използвайте кода в текущата директория"** добавя azd към приложение, което вече имате.
- `azure.yaml` съпоставя папките с кода ви към хостове в Azure; `infra/` дефинира ресурсите като Bicep.
- `azd infra generate` ви позволява да прегледате или персонализирате генерираната инфраструктура.
- След инициализация съществуващото ви приложение използва точно същия работен процес `azd up` / `azd down`, както приложение, базирано на шаблон.
