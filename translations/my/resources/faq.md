## မိတ်ဆက်

ဒီဖော်ပြချက်ပြည့်စုံတဲ့ FAQ က Azure Developer CLI (azd) နှင့် Azure ပေါ်သို့ တင်သွင်းမှုများအကြောင်း မကြာခဏမေးလေ့ရှိတဲ့မေးခွန်းများရဲ့ ဖြေကြားချက်တွေကို ပေးပါသည်။ ပုံမှန် ပြဿနာများအတွက် မြန်ဆန်ဖြေရှင်းနည်းများကို ရှာဖွေနိုင်ပြီး၊ အကောင်းဆုံးလမ်းလေ့များကို နားလည်ပြီး azd အကြောင်းအရာများနှင့် လုပ်ငန်းစဉ်များကိုရှင်းလင်းစေပါသည်။

## လေ့လာရန် ရည်မှန်းချက်များ

ဒီ FAQ ကို ကြည့်ရှုခြင်းဖြင့် သင်သည်:
- Azure Developer CLI နဲ့ ပတ်သက်တဲ့ မကြာခဏမေးလေ့ရှိတဲ့ မေးခွန်းများနှင့် ပြဿနာများအတွက် အမြန်ဖြေရှင်းချက်များ ရှာနိုင်ပါမည်
- အဓိက အသိပညာနှင့် အသုံးအနှုန်းများကို Q&A ပုံစံဖြင့် နားလည်နိုင်ပါမည်
- မကြာခဏကြုံတွေ့လေ့ရှိသော ပြဿနာများနှင့် အမှားအခြေအနေများအတွက် troubleshooting ဖြေရှင်းနည်းများကို လက်လှမ်းမီနိုင်ပါသည်
- အကောင်းဆုံးလေ့ကျင့်မှုများကို တိုးတက်အောင် ပြဿနာများအားဖြင့် သင်ယူနိုင်ပါသည်
- ထူးချွန်သည့် လုပ်ဆောင်ချက်များနှင့် စွမ်းရည်များကို ကျွမ်းကျင်သူအဆင့် မေးခွန်းများမှတဆင့် ရှာဖွေနိုင်ပါသည်
- ကုန်ကျစရိတ်၊ ဘေးလုံခြုံမှုနှင့် တင်သွင်းရန် မဟာဗျူဟာများကို တိတိကျကျ ကိုးကားလေ့လာနိုင်ပါသည်

## သင်ယူပြီးရမည့် ရလဒ်များ

FAQ ကို အကြိမ်ကြိမ် ကိုးကားလေ့လာခြင်းဖြင့် သင်သည်:
- ပေးထားသော ဖြေရှင်းနည်းများကို အသုံးပြု၍ Azure Developer CLI ရဲ့ ပုံမှန် ပြဿနာများကို လွတ်လပ်စွာ ဖြေရှင်းနိုင်မည်
- တင်သွင်းမှု မဟာဗျူဟာများနှင့် ဖော်ပြချက်များအပေါ် သတ်မှတ်ချက်ဖြင့် ဆုံးဖြတ်ချက်များချနိုင်မည်
- azd နှင့် အခြား Azure ကိရိယာများနှင့် ဝန်ဆောင်မှုများအကြား ဆက်နွယ်မှုကို နားလည်နိုင်မည်
- အသိုင်းအဝိုင်း အတွေ့အကြုံနှင့် ကျွမ်းကျင်သူ အကြံပေးချက်များအပေါ် အကောင်းဆုံး လေ့ကျင့်မှုများကို လက်တွဲအသုံးပြုနိုင်မည်
- အတည်ပြုခြင်း၊ တင်သွင်းခြင်းနှင့် ဖော်ပြချက်ပြဿနာများကို ထိရောက်စွာ troubleshooting ပြုလုပ်နိုင်မည်
- FAQ အကြံပေးချက်များကို အသုံးပြု၍ ကုန်ကျစရိတ်နှင့် ဖျော်ဖြေရေးစွမ်းဆောင်ရည်များကို အကောင်းဆုံးပြုလုပ်နိုင်မည်

## စတင်ရန်

### **မေး:** Azure Developer CLI (azd) ဆိုတာဘာလဲ?
**ဖြေ**: Azure Developer CLI (azd) သည် developer ဦးတည်ထားသော command-line ကိရိယာတစ်ခုဖြစ်ပြီး သင်၏ application ကို local ဖန်တီးရေးပတ်ဝန်းကျင်မှ Azure သို့ ရောက်အောင် လျင်မြန်စေသည်။ templates များဖြင့် အကောင်းဆုံးလေ့ကျင့်မှုများကို ပေးပြီး တင်သွင်းမှု စဉ်ဆက်ကို အကူအညီပေးပါသည်။

### **မေး:** azd ကို Azure CLI ထက် မည်ကဲ့သို့ ကွာခြားသနည်း?
**ဖြေ**: 
- **Azure CLI**: Azure အရင်းအမြစ်များကို စီမံခန့်ခွဲရန် အထွေထွေ အသုံးချသော ကိရိယာ
- **azd**: application တင်သွင်းရေး လုပ်ငန်းစဉ်များအတွက် developer အလေးပေးထားသော ကိရိယာ
- azd သည် အတွင်းပိုင်းတွင် Azure CLI ကို အသုံးပြုသော်လည်း စာရင်းဝင်သူများအတွက် အဆင့်မြင့် abstraction များကို ပေးသည်
- azd တွင် templates, ပတ်ဝန်းကျင် စီမံခန့်ခွဲမှုနှင့် တင်သွင်းမှု အလိုအလျောက်လုပ်ဆောင်မှုများ ပါဝင်သည်

### **မေး:** azd ကို အသုံးပြုရန် Azure CLI ကို ထည့်သွင်းထားရန် လိုအပ်ပါသလား?
**ဖြေ**: ဟုတ်ပါတယ်၊ azd သည် အတည်ပြုခြင်းနှင့် တချို့ လုပ်ဆောင်ချက်များအတွက် Azure CLI ကို လိုအပ်ပါသည်။ ပထမဦးဆုံး Azure CLI ကို ထည့်သွင်းပြီးနောက် azd ကို ထည့်သွင်းပါ။

### **မေး:** azd က ဘာ Programming ဘာသာစကားတွေကို ပံ့ပိုးသလဲ?
**ဖြေ**: azd သည် ဘာသာစကားကိုမရွေး ပါသည်။ အောက်ပါများနှင့် လက်တွေ့အလုပ်လုပ်နိုင်ပါသည် -
- Node.js/JavaScript/TypeScript
- Python
- .NET/C#
- Java
- Go
- PHP
- Static websites
- Containerized applications

### **မေး:** azd ကို ရှိပြီးသား project များနှင့် အသုံးပြုနိုင်မလား?
**ဖြေ**: ဟုတ်ပါတယ်! သင်သည် အောက်ပါအတိုင်းလုပ်နိုင်ပါသည် -
1. `azd init` ကို အသုံးပြု၍ ရှိပြီးသား project များတွင် azd ဖိုင်များထည့်သွင်းရန်
2. ရှိပြီးသား project များကို azd template ဖွဲ့စည်းပုံနှင့် ကိုက်ညီအောင် ပြင်ဆင်ရန်
3. မူလ معم အဆောက်အအုံအပေါ် အခြေခံပြီး custom template များ ဖန်တီးရန်

## အတည်ပြုခြင်းနှင့် ဝင်ရောက်ခွင့်

### **မေး:** azd အသုံးပြု၍ Azure တွင် မည်သို့ အတည်ပြုမည်နည်း?
**ဖြေ**: `azd auth login` ကို အသုံးပြုပါ؛ ၎င်းသည် Azure အတည်ပြုခြင်းအတွက် browser တစ်ခုဖွင့်ပါလိမ့်မည်။ CI/CD အခြေအနေများတွင် service principal သို့မဟုတ် managed identities ကို အသုံးပြုပါ။

### **မေး:** azd ကို အမျိုးမျိုးသော Azure subscription များနှင့် အသုံးပြုနိုင်မလား?
**ဖြေ**: ဟုတ်ပါတယ်။ environment တစ်ခုချင်းစီအတွက် subscription ကို သတ်မှတ်ရန် `azd env set AZURE_SUBSCRIPTION_ID <subscription-id>` ကို အသုံးပြုပါ။

### **မေး:** azd ဖြင့် တင်သွင်းနိုင်ရန် ဘယ်လို အခွင့်အရေးများ လိုအပ်သလဲ?
**ဖြေ**: အများအားဖြင့် လိုအပ်သည်မှာ -
- အရင်းအမြစ် group သို့မဟုတ် subscription ပေါ်တွင် **Contributor** role
- role assignments များ လုပ်ရန်လိုလျှင် **User Access Administrator**
- အထူးသတ်မှတ်ခွင့်များသည် template နှင့် တင်သွင်းမည့် အရင်းအမြစ်ပေါ် မူတည်၍ ကွဲပြားနိုင်ပါသည်

### **မေး:** azd ကို CI/CD pipeline များတွင် အသုံးပြုနိုင်မလား?
**ဖြေ**: တင်းတိမ်စွာ! azd သည် CI/CD အောင်မြင်စေရန် ဒီဇိုင်းဆွဲထားပါသည်။ အတည်ပြုခြင်းအတွက် service principal များကို အသုံးပြု၍ ဖော်ပြချက်များအတွက် environment variables များ သတ်မှတ်ပါ။

### **မေး:** GitHub Actions တွင် အတည်ပြုခြင်းကို မည်သို့ ကိုင်တွယ်မလဲ?
**ဖြေ**: service principal အသုံးပြုပြီး Azure Login action ကို အသုံးပြုပါ -
```yaml
- uses: azure/login@v1
  with:
    creds: ${{ secrets.AZURE_CREDENTIALS }}
- run: azd deploy --no-prompt
```

## Templates & Projects

### **မေး:** azd templates များကို ဘယ်မှာ ရှာမလဲ?
**ဖြေ**: 
- အခြေခံ templates: [Azure-Samples/awesome-azd](https://github.com/Azure-Samples/awesome-azd)
- အသိုင်းအဝိုင်း templates: GitHub တွင် "azd-template" ကို ရှာဖွေပါ
- ရရှိနိုင်သော templates များကို စူးစမ်းရန် `azd template list` ကို အသုံးပြုပါ

### **မေး:** custom template ကို မည်သို့ ဖန်တီးမလဲ?
**ဖြေ**: 
1. ရှိပြီးသား template ဖွဲ့စည်းပုံနှင့် စတင်ပါ
2. `azure.yaml`, infrastructure ဖိုင်များနှင့် application ကုဒ်ကို ပြင်ဆင်ပါ
3. `azd up` ဖြင့် သေချာစွာ စမ်းသပ်ပါ
4. သင့် GitHub တွင် သင့်တင့်သော tag များဖြင့် ထုတ်ဝေပါ

### **မေး:** template မရှိဘဲ azd ကို အသုံးပြုနိုင်မလား?
**ဖြေ**: ဟုတ်ပါတယ်၊ ရှိပြီးသား project တွင် `azd init` ကို အသုံးပြု၍ လိုအပ်သည့် ဖိုင်များကို ဖန်တီးနိုင်သည်။ `azure.yaml` နှင့် infrastructure ဖိုင်များကို လက်ဖြင့် ပြင်ဆင်ရန် လိုပါမည်။

### **မေး:** အဖွဲ့အစည်းမှ ထုတ်ဝေသည့် templates နှင့် အသိုင်းအဝိုင်း templates တွင် မည်သို့ ကွာခြားသနည်း?
**ဖြေ**: 
- **Official templates**: Microsoft မှ ထိန်းသိမ်းထုတ်ဝေပြီး ပုံမှန်အားဖြင့် update များရှိပြီး ဖော်ပြချက်များ ပြည့်စုံသည်
- **Community templates**: ဖန်တီးသူများက ဆောင်ရွက်ထားပြီး အထူးသဖြင့် အသုံးအရာများရှိနိုင်သည်၊ အရည်အသွေးနှင့် ထိန်းသိမ်းမှုကွဲပြားနိုင်သည်

### **မေး:** ကဏ္ဍတွင် template ကို မည်သည့်နည်းဖြင့် update ပြုလုပ်မလဲ?
**ဖြေ**: templates များကို အလိုအလျောက် update မလုပ်ပါ။ သင်သည် -
1. အရင်းအမြစ် template မှ ပြောင်းလဲချက်များကို လက်ဖြင့် နှိုင်းယှဉ်ပြီး merge ပြုလုပ်ပါ
2. update လုပ်ထားသော template ဖြင့် `azd init` လုပ်၍ အသစ်စတင်ပါ
3. update လုပ်ထားသော templates များမှ ထူးခြားသော တိုးတက်မှုများကို cherry-pick လုပ်ပါ

## တင်သွင်းမှုနှင့် အျခေခံအဆောက်အအုံ

### **မေး:** azd က Azure ၏ မည်သည့်ဝန်ဆောင်မှုများကို တင်သွင်းနိုင်သနည်း?
**ဖြေ**: azd သည် Bicep/ARM templates များဖြင့် Azure ၏ မည်သည့်ဝန်ဆောင်မှုမှ ထည့်သွင်း၍ တင်သွင်းနိုင်ပါသည်၊ ထိုတွင် ပါဝင်သည်များမှာ -
- App Services, Container Apps, Functions
- Databases (SQL, PostgreSQL, Cosmos DB)
- Storage, Key Vault, Application Insights
- Networking, security, နှင့် monitoring အရင်းအမြစ်များ

### **မေး:** မျိုးစုံဒေသများသို့ တင်သွင်းနိုင်မလား?
**ဖြေ**: ဟုတ်ပါတယ်၊ သင့် Bicep template များတွင် မျိုးစုံဒေသများကို ဖော်ပြပြီး environment တစ်ခုချင်းစီအတွက် location parameter ကို သက်ဆိုင်ရာအတိုင်း သတ်မှတ်ပါ။

### **မေး:** database schema migrations များကို မည်သို့ ကိုင်တွယ်မလဲ?
**ဖြေ**: `azure.yaml` ထဲရှိ deployment hooks များကို အသုံးပြုပါ -
```yaml
hooks:
  postdeploy:
    posix:
      run: ./scripts/migrate-database.sh
    windows:
      run: ./scripts/migrate-database.ps1
```

### **မေး:** များသောအချက်အလက်နှင့် applications မပါဘဲ အဆောက်အအုံပဲ တင်သွင်းလိုပါသလား?
**ဖြေ**: ဟုတ်ပါတယ်၊ သင်၏ templates တွင် သတ်မှတ်ထားသော infrastructure အစိတ်အပိုင်းများကို တင်သွင်းရန် `azd provision` ကို အသုံးပြုပါ။

### **မေး:** ရှိပြီးသား Azure အရင်းအမြစ်များသို့ မည်သို့ တင်သွင်းမလဲ?
**ဖြေ**: ၎င်းသည် စုစုပေါင်းရှုပ်ထွေးပြီး တိုက်ရိုက် ထောက်ပံ့ထားသည့် အရာမဟုတ်ပါ။ သင်သည် -
1. ရှိပြီးသားအရင်းအမြစ်များကို သင့် Bicep template များထဲသို့ import ပြုလုပ်ပါ
2. templates ထဲတွင် ရှိပြီးသား resource reference များကို အသုံးပြုပါ
3. resources များကို အခြေအနေအလိုက် ဖန်တီးမည် သို့မဟုတ် reference မည်ဟု conditional ပြင်ဆင်ရန် templates များကို ပြင်ဆင်ပါ

### **မေး:** Terraform ကို Bicep အစား အသုံးပြုလို့ ရမလား?
**ဖြေ**: လက်ရှိတွင် azd သည် အဓိကအားဖြင့် Bicep/ARM templates များကို ပံ့ပိုးသည်။ Terraform တွင် ပံ့ပိုးမှုကို တရားဝင် အနေဖြင့် မပေးထားသေးပါ၊ သို့သော် အသိုင်းအဝိုင်း ဖြေရှင်းနည်းများ ရှိနိုင်သည်။

## ဖော်ပြချက်များ နှင့် ပတ်ဝန်းကျင်များ

### **မေး:** ကွဲပြားသည့် ပတ်ဝန်းကျင်များ (dev, staging, prod) ကို မည်သို့ စီမံမလဲ?
**ဖြေ**: `azd env new <environment-name>` ကို အသုံးပြု၍ သီးခြားပတ်ဝန်းကျင်များ ဖန်တီးပြီး အဖွဲ့ခြားသော ဆက်တင်များကို သတ်မှတ်ပါ။
```bash
azd env new development
azd env new staging  
azd env new production
```

### **မေး:** ပတ်ဝန်းကျင် ဖော်ပြချက်များကို ဘယ်ကို သိမ်းဆည်းထားလဲ?
**ဖြေ**: သင့် project directory အတွင်းရှိ `.azure` ဖိုလ်ဒါတွင် သိမ်းဆည်းထားသည်။ ပတ်ဝန်းကျင်တိုင်းတွင် ဖိုင်ဖိုင်များဖြင့် မိမိ၏ ဖိုလ်ဒါ ရှိပါသည်။

### **မေး:** ပတ်ဝန်းကျင်အလိုက် ဖော်ပြချက်များကို မည်သို့ သတ်မှတ်မလဲ?
**ဖြေ**: ပတ်ဝန်းကျင် အပြောင်းအလဲများကို သတ်မှတ်ရန် `azd env set` ကို အသုံးပြုပါ -
```bash
azd env set AZURE_LOCATION eastus
azd env set DATABASE_TIER Basic
```

### **မေး:** အဖွဲ့ဝင်များအချင်းချင်း ပတ်ဝန်းကျင် ဖော်ပြချက်များကို မျှဝေလိုပါက?
**ဖြေ**: `.azure` ဖိုလ်ဒါတွင် တွေ့ရသည့် အချက်အလက်များသည် အနူးအနက် အာရုံစိုက်စရာဖြစ်၍ version control တွင် commit မလုပ်သင့်ပါ။ အစား -
1. လိုအပ်သော environment variables များကို စာရွက်အဖြစ် သတ်မှတ်၍ ဆဲလ်အဖြစ် မှတ်သားပါ
2. ပတ်ဝန်းကျင်များကို စတင်ဖွင့်ရန် deployment script များကို အသုံးပြုပါ
3. အဓိက ဖော်ပြချက်များအတွက် Azure Key Vault ကို အသုံးပြုပါ

### **မေး:** template default များကို မည်သို့ override ပြုလုပ်မလဲ?
**ဖြေ**: template parameter များနှင့် ကိုက်ညီသော environment variables များကို သတ်မှတ်ပါ။
```bash
azd env set LOCATION "West US 2"
azd env set SKU_NAME "B1"
```

## ပြဿနာရှာဖွေရေး (Troubleshooting)

### **မေး:** `azd up` မအောင်မြင်တာဘာကြောင့်လဲ?
**ဖြေ**: ပုံမှန် ဖြစ်ရပ်များမှာ -
1. **Authentication issues**: `azd auth login` ကို ပြန်လည် လုပ်ဆောင်ပါ
2. **Insufficient permissions**: Azure role assignments များကို စစ်ဆေးပါ
3. **Resource naming conflicts**: AZURE_ENV_NAME ကို ပုံသေနည်းပြောင်းပါ
4. **Quota/capacity issues**: ဒေသဆိုင်ရာ ရရှိနိုင်မှုကို စစ်ဆေးပါ
5. **Template errors**: Bicep template များကို validate ပြုလုပ်ပါ

### **မေး:** တင်သွင်းမှု လက်ခတ်ချုပ်တွေကို မည်သို့ debug ပြုလုပ်မလဲ?
**ဖြေ**: 
1. တိကျသော output များအတွက် `azd deploy --debug` ကို အသုံးပြုပါ
2. Azure portal တွင် deployment history ကို စစ်ဆေးပါ
3. Azure portal ၏ Activity Log ကို ကြည့်ပါ
4. လက်ရှိ environment အခြေအနေကို ပြသရန် `azd show` ကို အသုံးပြုပါ

### **မေး:** မိမိ၏ environment variables မည်သို့ အလုပ်မလုပ်နေသလဲ?
**ဖြေ**: စစ်ဆေးရန် -
1. Variable နာမည်များသည် template parameter များနှင့် တိတိကျကျ ကိုက်ညီထားပါသလား
2. စာသားများတွင် အားလုံး များပါက စကားဝိုင်းများအတွင်း ကောင်းကင် quote ထည့်ထားပါသလား
3. ပတ်ဝန်းကျင် ရွေးချယ်ထားသည်: `azd env select <environment>`
4. Variables များကို သတ်မှတ်ထားသော ပတ်ဝန်းကျင်တွင် သတ်မှတ်ထားပါသလား

### **မေး:** ပြင်ဆင်မှု မအောင်မြင်သော သတ်မှတ်ချက်များကို မည်သို့ ရှင်းလင်းမလဲ?
**ဖြေ**: 
```bash
azd down --force --purge
```
ဤအရာသည် စုစုပေါင်းသော အရင်းအမြစ်များနှင့် ပတ်ဝန်းကျင် ဖော်ပြချက်များအား ဖယ်ရှားပါသည်။

### **မေး:** တင်သွင်းပြီးနောက် အက်ပလီကေးရှင်းကို ဝင်ရောက်မရချင်တာဘာကြောင့်လဲ?
**ဖြေ**: စစ်ဆေးပါ -
1. တင်သွင်းမှု အောင်မြင်စွာပြီးစီးခဲ့ပါသလား
2. အက်ပလီကေးရှင်း လည်ပတ်နေပါသလား (Azure portal မှ logs များကို စစ်ဆေးပါ)
3. Network security groups များ traffic ကို ခွင့်ပြုနေပါသလား
4. DNS/custom domain များကို မှန်ကန်စွာ ပြင်ဆင်ထားပါသလား

## ကုန်ကျစရိတ်နှင့် ဘေလ်လ်

### **မေး:** azd တင်သွင်းမှုများ ကုန်ကျမယ်ဆိုရင် ဘယ်လောက်ကျမလဲ?
**ဖြေ**: ကုန်ကျစရိတ်များသည် အောက်ပါပေါ် မူတည်ပါသည် -
- တင်သွင်းသည့် Azure ဝန်ဆောင်မှုများ
- ရွေးချယ်ထားသည့် service tier/SKU များ
- ဒေသအလိုက် စျေးနှုန်း ကွာခြားမှုများ
- အသုံးချမှု ပုံစံများ

တွက်ချက်ရန် [Azure Pricing Calculator](https://azure.microsoft.com/pricing/calculator/) ကို အသုံးပြုပါ။

### **မေး:** azd တင်သွင်းမှုများတွင် ကုန်ကျစရိတ် ကို မည်သို့ ထိန်းချုပ်မလဲ?
**ဖြေ**: 
1. ဖွံ့ဖြိုးရေး ပတ်ဝန်းကျင်များအတွက် အနိမ့် tier SKUs များကို အသုံးပြုပါ
2. Azure budgets နှင့် အကြောင်းကြားချက်များကို သတ်မှတ်ပါ
3. မလိုအပ်သောအခါ `azd down` ကို အသုံးပြု၍ အရင်းအမြစ်များ ဖယ်ရှားပါ
4. သင့်တည်နေရာသင့်ဖြင့် သင့်တင်သွင်းရန် ချိတ်ဆက်ပါ (ဒေသအလိုက် ကုန်ကျစရိတ်ကွာခြားသည့်)
5. Azure Cost Management ကိရိယာများကို အသုံးပြုပါ

### **မေး:** azd templates များအတွက် အခမဲ့ tier ရှိပါသလား?
**ဖြေ**: Azure ဝန်ဆောင်မှုများ များစွာတွင် အခမဲ့ tier ရှိသည် -
- App Service: အခမဲ့ tier ရရှိနိုင်သည်
- Azure Functions: 1M အခမဲ့ executions/လ
- Cosmos DB: 400 RU/s ဖြင့် အခမဲ့ tier
- Application Insights: ပထမ 5GB/လ အခမဲ့

လက်ရှိရရှိနိုင်သည့်နေရာများတွင် free tiers ကို အသုံးပြုရန် templates များကို ပြင်ဆင်ပါ။

### **မေး:** တင်သွင်းမလုပ်ခင် ကုန်ကျစရိတ်ကို မည်သို့ ခန့်မှန်းမလဲ?
**ဖြေ**: 
1. ဖန်တီးထားသော `main.bicep` ကို ကြည့်၍ မည်သည့် resources များဖန်တီးမည်ကို ကြည့်ပါ
2. သတ်မှတ်ထားသော SKUs များနှင့်အတူ Azure Pricing Calculator ကို အသုံးပြုပါ
3. ပထမဦးဆုံး development environment သို့ တင်သွင်း၍ လက်တွေ့ ကုန်ကျစရိတ်ကို စောင့်ကြည့်ပါ
4. အသေးစိတ် ကုန်ကျစရိတ်ခွဲခြမ်းစိတ်ဖြာရန် Azure Cost Management ကို အသုံးပြုပါ


### **မေး:** `.azure` ဖိုလ်ဒါကို version control သို့ commit မလုပ်သင့်ပါသလား?
**ဖြေ**: **မလုပ်ပါနဲ့!** `.azure` ဖိုလ်ဒါတွင် အလွန်ထိခိုက်မည့် အချက်အလက်များ ပါရှိသည်။ `.gitignore` ထဲသို့ ထည့်ပါ -
```gitignore
.azure/
```

### **မေး:** secret များနှင့် ထိပါတ်စရာ ဖော်ပြချက်များကို မည်သို့ ကိုင်တွယ်မလဲ?
**ဖြေ**: 
1. secret များအတွက် Azure Key Vault ကို အသုံးပြုပါ
2. application ဖော်ပြချက်များတွင် Key Vault secret များကို ရည်ညွှန်းပါ
3. secret များကို ever သို့တောင် version control တွင် commit မလုပ်ပါနှင့်
4. ဝန်ဆောင်မှုမှ ဝန်ဆောင်မှုသို့ အတည်ပြုရန် managed identities ကို အသုံးပြုပါ

### **မေး:** azd နှင့် CI/CD အတွက် အကြံပြုနည်းမှာ ဘာလဲ?
**ဖြေ**: 
1. အထွတ်အထူး အဆင့်များအတွက် အလိုက်သင့်ပတ်ဝန်းကျင်များ အသုံးပြုပါ (dev/staging/prod)
2. တင်သွင်းမီ automated testing များကို ထည့်သွင်းပါ
3. အတည်ပြုခြင်းအတွက် service principals များကို အသုံးပြုပါ
4. pipeline secrets/variables တွင် ထိပါတ်ဖော်ပြချက်များကို ထည့်သိမ်းပါ
5. production တင်သွင်းမှုများအတွက် အသိအမှတ်ပြု gate များ ထည့်သွင်းပါ

## အဆင့်မြင့် ခေါင်းစဉ်များ

### **မေး:** azd ကို custom functionality များဖြင့် တိုးချဲ့နိုင်မလား?
**ဖြေ**: ဟုတ်ပါတယ်၊ `azure.yaml` ထဲရှိ deployment hooks များမှတဆင့် ဆက်လက်တိုးချဲ့နိုင်ပါသည် -
```yaml
hooks:
  predeploy:
    run: ./scripts/custom-setup.sh
  postdeploy:
    run: ./scripts/custom-config.sh
```

### **မေး:** azd ကို ရှိပြီးသား DevOps လုပ်ငန်းစဉ်များနှင့် မည်သို့ ပေါင်းစည်းမလဲ?
**ဖြေ**: 
1. ရှိပြီးသား pipeline script များတွင် azd command များကို အသုံးပြုပါ
2. အဖွဲ့များအဆင့်တစ်ခုလျှင် azd templates များကို စံသတ်မှတ်ပါ
3. ရှိပြီးသား monitoring နှင့် alerting အတွင်းတြင် ပေါင်းစည်းပါ
4. pipeline ပေါင်းစည်းမှုအတွက် azd ၏ JSON output ကို အသုံးပြုပါ

### **မေး:** azd ကို Azure DevOps နှင့် အသုံးပြုလို့ ရမလား?
**ဖြေ**: ဟုတ်ပါတယ်၊ azd သည် မည်သည့် CI/CD စနစ်နှင့်မဆို အလုပ်လုပ်နိုင်ပါသည်။ azd command များကို အသုံးပြုသော Azure DevOps pipeline များကို ဖန်တီးပါ။

### **မေး:** azd ကို ပံ့ပိုးရန် သို့မဟုတ် အသိုင်းအဝိုင်း templates များ ဖန်တီးရန် ဦးစီးပေးရန် မည်သို့ လုပ်မလဲ?
**ဖြေ**:
1. **azd tool**: [Azure/azure-dev](https://github.com/Azure/azure-dev) သို့ ပါဝင် လှုံ့ဆော်ပါ
2. **Templates**: [နမူနာ လမ်းညွှန်ချက်များ](https://github.com/Azure-Samples/awesome-azd) အတိုင်း နမူနာများ ဖန်တီးပါ
3. **Documentation**: [MicrosoftDocs/azure-dev-docs](https://github.com/MicrosoftDocs/azure-dev-docs) တွင် စာရွက်စာတမ်းများ ထောက်ပံ့ပါ

### Q: azd အတွက် လမ်းပြဇယား ဘာတွေလဲ?
**A**: စီစဉ်ထားသော လုပ်ဆောင်ချက်များနှင့် တိုးတက်မှုများအတွက် [တရားဝင် လမ်းပြဇယား](https://github.com/Azure/azure-dev/projects) ကို ကြည့်ပါ။

### Q: အခြား ဖြန့်ချိမှု ကိရိယာများမှ azd သို့ ဘယ်လို ရွေ့ပြောင်းမလဲ?
**A**: 
1. လက်ရှိ ဖြန့်ချိမှု ဖွဲ့စည်းပုံကို သုံးသပ်ပါ
2. တူညီသော Bicep နမူနာများ ဖန်တီးပါ
3. လက်ရှိ ဝန်ဆောင်မှုများနှင့် ကိုက်ညီအောင် `azure.yaml` ကို ပြင်ဆင်ပါ
4. ဖွံ့ ဖြိုးရေး ပတ်ဝန်းကျင်တွင် လုံလောက်စွာ စမ်းသပ်ပါ
5. ပတ်ဝန်းကျင်များကို အဆင့်လိုက် ရွေ့ပြောင်းပါ
