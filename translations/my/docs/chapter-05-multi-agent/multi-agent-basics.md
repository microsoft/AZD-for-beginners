## နိဒါန်း

အရင်အခန်းများတွင် သင်တစ်ခုတည်းသော အက်ပလီကေးရှင်းကို တပ်ဆင်ခဲ့ပြီး၊ အခန်း ၂ တွင် သင်တစ်ခုတည်း AI အေးဂျင့်ကို တပ်ဆင်ခဲ့သည်။ ဒီသင်ခန်းစာက အဆင့်နောက်တစ်ဆင့်ကို ယူသည်။ එထဲမှာ «multi-agent system» ဖြစ်ပြီး အထူးပြု အေးဂျင့်များစွာသည် အတူတကွ ပူးပေါင်း၍ တစ်ဦးတစ်ယောက် တာဝန်မထမ်းဆောင်နိုင်သော ပြဿနာကို ဖြေရှင်းသည်။

စတင်သူများအတွက် သတင်းကောင်းကောင်းမှာကတော့: **သင်သည် အမိန့်အသစ်များ မလိုအပ်ပါ။** Multi-agent ဖြေရှင်းချက်သည် azd project ဖြစ်သည်။ သင်သည် `azd init`, `azd up`, စမ်းသပ်ခြင်း နှင့် `azd down` လုပ်ဆောင်မည်။ သင်သိပြီးသား workflow အသစ် မဟုတ်ပါ။ ပြောင်းလဲသည့် အချက်မှာ အက်ပလီကေးရှင်းအတွင်းရှိ *ပုံသဏ္ဍာန်* ဖြစ်သည်။

## သင်ယူရမည့် ရည်မှန်းချက်များ

ဒီသင်ခန်းစာကုန်ဆုံးစဉ်တွင် သင်သည်:
- "multi-agent" ဆိုသည်မှာ ဘာလဲ၊ ၎င်းသည် မည်သည့်အခါတွင် ပို၍ရှုပ်ထွေးမှုကို တန်ဖိုးပေးသလဲကို နားလည်ပါမည်။
- Multi-agent စနစ်မှ စံပြ အခန်းကဏ္ဍများ (orchestrator + specialist) ကို အသိအမှတ်ပြုပါမည်။
- `azd up` ဖြင့် တကယ် အသုံးပြုနိုင်သော multi-agent template တစ်ခုကို တပ်ဆင်ပါမည်။
- Multi-agent အက်ပလီကေးရှင်းအတွက် သက်ဆိုင်သော Azure အရင်းအမြစ်များကို နားလည်ပါမည်။
- ဖြေရှင်းချက်ကို ဘေးကင်းစွာ သေချာမှု၊ စိတ်တိုင်းကျ ပြုပြင်မှုနှင့် ရှင်းလင်းမှုလုပ်နည်းများကို သိရှိပါမည်။

## သင်ယူပြီးရရှိမှုများ

သင်ခန်းစာပြီးဆုံးလျှင် သင်ပြုလုပ်နိုင်မည့် အရာများမှာ:
- တစ်ဦးတည်းသော အေးဂျင့်နှင့် multi-agent စနစ်၏ မတူညီမှုကို ရှင်းလင်းဖော်ပြနိုင်သည်။
- တစ်ဦးတည်း အေးဂျင့်တွင် ကိရိယာများပါရှိခြင်း နှင့် multi-agent ဒီဇိုင်းတစ်ခုကို ရွေးချယ်နိုင်သည်။
- azd ဖြင့် multi-agent template ကို များစွာ စမ်းသပ်၍ တပ်ဆင်နိုင်သည်။
- အေးဂျင့်တစ်ဦးချင်းလည်ပတ်ရာနေရာနှင့် ၎င်းတို့ ဆက်သွယ်နည်းကို ဖော်ထုတ်နိုင်သည်။
- ဆက်လက်ကြေးဆောင်မှုများမဖြစ်စေရန် အရင်းအမြစ်အားလုံးကို ရှင်းလင်းစင်ကြယ်စွာ ဖယ်ရှားနိုင်သည်။

## Multi-Agent စနစ်ဆိုသည်မှာဘာလဲ?

တစ်ဦးတည်း AI အေးဂျင့်သည် မော်ဒယ်တစ်ခုသာဖြစ်ပြီး ညွှန်ကြားချက်နှင့် (တစ်ချို့ကိရိယာများပါရှိနိုင်သည်) ပါရှိသည်။ ၎င်းသည် အာရုံစူးစိုက် လုပ်ငန်းများအတွက် ကောင်းစွာ အလုပ်လုပ်သည်။ သို့သော် တာဝန်ကြီးလာခဲ့ပါက - သုတေသန၊ ပြီးမှရေးသားခြင်း၊ ပြင်ဆင်ခြင်းနှင့် မှန်ကန်မှုစစ်ဆေးခြင်း - အားလုံးကို တစ်ခုတည်းသော prompt တစ်ခုထဲ ထည့်သွင်းသည်မှာ အေးဂျင့်ကို သက်သာမှုနည်းပြီး၊ ယုံကြည်စိတ်ချမှုနည်း၊ အမှားရှာရခက် ဖြစ်စေသည်။

**multi-agent စနစ်** သည် လုပ်ငန်းကို အထူးပြု အလုပ်တစ်ခုပြုလုပ်သော specialist များသို့ ခွဲခြား၍ orchestrator မှ ညှိနှိုင်းပေးသည်။

```mermaid
graph TD
    User([အသုံးပြုသူ တောင်းဆိုချက်]) --> Orchestrator[အော်ကဲစထ်ရေးတာအေးဂျင့်<br/>အလုပ်စီမံကိန်းရေးဆွဲပြီး လမ်းညွှန်သည်]
    Orchestrator --> Researcher[သုတေသနအေးဂျင့်<br/>အချက်အလက်စုဆောင်းသည်]
    Orchestrator --> Writer[ကိုယ်ရေးအေးဂျင့်<br/>အကြောင်းအရာ စျေးဆွဲသည်]
    Orchestrator --> Editor[တည်းဖြတ်အေးဂျင့်<br/>သုံးသပ်ပြင်ဆင်သည်]
    Researcher --> Orchestrator
    Writer --> Orchestrator
    Editor --> Orchestrator
    Orchestrator --> Result([နောက်ဆုံးဖြေရှင်းချက်])
```

### သင်မြင်တွေ့လေ့ရှိသူ နှစ်ရာထူး

| ရာထူး | တာဝန် | ဥပမာ |
|------|-----|---------|
| **Orchestrator** | *နောက်လိုက်ဖြစ်မည့်*အရာကို ဆုံးဖြတ်၍ အေးဂျင့်များအကြား လုပ်ငန်းလမ်းညွှန်သည် | " ပထမဆုံး သုတေသန၊ ပြီးမှရေး၊ ပြီးမှပြင်" |
| **Specialist** | အထူးပြု တာဝန်တစ်ခုကို လုပ်ပြီး ရလဒ်ပြန်လည်ပေးသည် | သက်ဆိုင်သော အချက်အလက်တင်ဆက်သည့် "အတတ်ပညာရှင်" |

### သင်တွင် အေးဂျင့်များ များစွာ လိုအပ်ပါသလား?

ရိုးရိုးစောင့်ကြည့်ပါ။ Multi-agent ကို **သာ** အသုံးပြုပြီး အောက်ပါအချက်ထဲမှ တစ်ခုမှန်ကန်ပါကသာ:

- ✅ လုပ်ငန်းတွင် **ကွဲပြားသောအဆင့်များ** ရှိပြီး အဲဒီအဆင့်များတွင် အမိန့်ညွှန်ကြားချက်ကွာခြားမှုရှိသည် (သုတေသန vs အရေးအသားရေးခြင်း vs ပြန်လည်သုံးသပ်ခြင်း)
- ✅ အထူးပြုများကို **တပြိုင်နက်** လုပ်နိုင်ရန် စိတ်ကူး
- ✅ အဆင့်အလိုက် **ကိရိယာ သို့မဟုတ် ဒေတာအရင်းအမြစ်ကွာခြားမှုများ** လိုအပ်သည်
- ✅ အဆင့်တစ်ခုစီကို **လွတ်လပ်စွာ စမ်းသပ်နှင့် အမှားရှာစစ်ဆေးနိုင်ရန်** လိုအပ်သည်

သင်၏ လုပ်ငန်းမှာ တစ်ကြိမ်ဖြေကြားရမည့်မေးခွန်းတစ်ခု သို့မဟုတ် ရိုးရှင်းသော ကိရိယာခေါ်ဆိုမှု တစ်ခုပါက၊ **တစ်ဦးတည်း အေးဂျင့်နှင့် ကိရိယာများ** (အခန်း ၂) သည် ရိုးရှင်း၊ စျေးသက်သာပြီး လုပ်ငန်းလုပ်ငန်းခွဲရလွယ်သည်။

> **စတင်သူအတွက် အကြံပြုချက်:** "အေးဂျင့်များ ပိုမိုခြင်း" ဟာ "ကောင်းမွန်ခြင်း" မဟုတ်ပါ။ အေးဂျင့်တစ်ရပ်ရပ်သည် နောက်ကျမှု၊ ကုန်ကျစရိတ်နှင့် ကြည့်ရှုဖို့ လိုအပ်ချက် အသစ်တစ်ခုဖြစ်သည်။ ပြဿနာသည် သေချာခွဲခြားထားပါကသာ အေးဂျင့်များ ထည့်သွင်းပါ။

## Azure အတွက် Multi-Agent များ ဆောက်လုပ်ဖို့ နည်းလမ်း နှစ်မျိုး

| နည်းလမ်း | အရာ | သင့်တော်သောအမှု |
|----------|-----------|----------|
| **တစ်ဦးတည်း အေးဂျင့် + ကိရိယာများ** | Functions/tools များကို ခေါ်သုံးသော Foundry အေးဂျင့် တစ်ခု | ရိုးရိုးရှင်းရှင်း workflow များ၊ စတင်သူများအတွက် |
| **အေးဂျင့်များ များစွာ နှင့် ညှိနှိုင်းမှု** | Orchestrator နှင့် အေးဂျင့်များ များစွာ | ကွဲပြားသော အဆင့်များ၊ တပြိုင်နက်လုပ်ငန်းများ၊ အထူးပြုမှုများ |

ဒီသင်ခန်းစာက ဒုတိယနည်းလမ်းကို အခြေခံကာ **အဆင်သင့် template** ကို အသုံးပြုပြီး သင်၏ပုဂ္ဂိုလ်ရေး multi-agent စနစ်ကို တည်ဆောက်မတိုင်မှီ တကယ်အသုံးပြုနိုင်သော multi-agent စနစ်ကို မြင်နားလည်စေသည်။

## လက်တွေ့လုပ်ရင်း: လုပ်ဆောင်နိုင်သော Multi-Agent App တပ်ဆင်ခြင်း

ကျွန်ုပ်တို့သည် ဖော်ပြပေးမည့် Contoso Creative Writer သည် Azure မှ တရားဝင်နမူနာဖြစ်ပြီး multi-agent အေးဂျင့်များ (သုတေသနဆောင်, စာရေးသူ, အယ်ဒီတာ) ကို ပူးပေါင်းအသုံးပြု၍ ဆောင်းပါးတစ်ပုဒ် ထုတ်လုပ်သည်။ ၎င်းသည် multi-agent အက်ပလီကေးရှင်း အကြောင်းကို နားလည်ရလွယ်သော အခန်းကဏ္ဍများ ပါဝင်သည်မှ ဖြစ်၍ ပထမဆုံး multi-agent app အနေနှင့် အထူးသင့်တော်သည်။

### အဆင့် ၁: Template ကို စတင်ဖန်တီးခြင်း

```bash
# အလုပ်လုပ်မယ့် ဖိုလ်ဒါတစ်ခု ဖန်တီးပါ
mkdir creative-writer && cd creative-writer

# တရားဝင် multi-agent အတွက် မူပိုင်ဖောင်တမ်းမှ စတင်ပါ
azd init --template contoso-creative-writer
```

> အချိန်မရွေး အခြား multi-agent templates များကို [Awesome AZD AI gallery](https://azure.github.io/awesome-azd/?tags=ai) တွင် လေ့လာနိုင်သည်။ စတင်သူဆိုင်ရာ ရွေးချယ်စရာများမှာ `get-started-with-ai-agents` နှင့် `azure-ai-travel-agents` တို့ပါဝင်သည်။

### အဆင့် ၂: အတည်ပြုခြင်း

```bash
# azd workflow များအတွက်လိုအပ်သည်
azd auth login
```

### အဆင့် ၃: ပတ်ဝန်းကျင် ဖန်တီးခြင်း

```bash
azd env new dev
```

### အဆင့် ၄: ကြည့်ရှုခြင်း၊ ထို့နောက် တပ်ဆင်ခြင်း

```bash
# ဘာတွေဖန်တီးမယ်ဆိုတာ အကုန်ချရန်မတိုင်မှီ ကြည့်ရှုပါ (အကြံပြုထားသည်)
azd provision --preview

# အဆောက်အအုံကို အသင့်ပြင်ပြီး အေးဂျင့်အားလုံးကို တစ်ချက်တည်း ဖြန့်ချိပါ
azd up
```

`azd up` သည် subscription နှင့် region တိုကို မေးမြန်းပြီး Azure အရင်းအမြစ်များကို ဖန်တီးပြီး အက်ပလီကေးရှင်းကို တပ်ဆင်မည်။ AI တပ်ဆင်မှုများမှာ ရိုးရှင်းသော ဝက်ဘ် အက်ပလီကေးရှင်းထက်ပိုမိုချိန်ယူတတ်သည်။ အကြီးစား မော်ဒယ်များ တပ်ဆင်ပါက deploy timeout ကို တိုးချဲ့နိုင်သည်:

```bash
azd deploy --timeout 1800
```

> **ကုန်ကျစရိတ်နှင့် တာဝန်ရှိမှုအတွက် သတိပေးချက်**: Multi-agent app များ သည် quota ကို သုံးစွဲပြီး ကုန်ကျစရိတ်ရှိစေသည်။ `azd up` သည် မော်ဒယ် quota အပေါ်မအောင်မြင်လျှင် [AI Troubleshooting](../chapter-07-troubleshooting/ai-troubleshooting.md) တွင် ဒေသနှင့် quota ပြုပြင်နေရာများကို ကြည့်ပါ၊ အခန်း ၆ [Capacity Planning](../chapter-06-pre-deployment/capacity-planning.md) တွင်လည်း ကြည့်ပါ။

## သင်တပ်ဆင်ခဲ့သည်ကို နားလည်ခြင်း

ဒီလို multi-agent app ပုံမှန်အားဖြင့် အောက်ပါဇယားတွင် ဖေါ်ပြထားသည့် တာဝန်ဆိုင်ရာများနှင့် တိုက်ရိုက်ဆက်စပ်နေသော Azure အရင်းအမြစ်များ စုစည်း တပ်ဆင်သည့်အခါဖြစ်သည်။

| အရင်းအမြစ် | ရည်ရွယ်သည့် အကြောင်းရင်း |
|----------|----------------|
| **Microsoft Foundry / မော်ဒယ်များ** | အေးဂျင့်တစ်ဦးစီ အသုံးပြုသည့် ဘာသာစကား မော်ဒယ်များ ချန်လှပ်ရန် |
| **Azure AI Search** | သုတေသနဆောင်အေးဂျင့်အတွက် အခြေခံ ဒေတာများ ရှာဖွေမှုပိုင်းတွင် အထောက်အကူဖြစ်စေသည် |
| **Container Apps** (သို့) App Service | Orchestrator နှင့် အေးဂျင့် ကုတ်များ ပျံ့နှံ့ရန် ပလက်ဖောင်း |
| **Cosmos DB** (အချို့နမူနာများတွင်) | အေးဂျင့်များအကြား အမျိုးမျိုးသော ဟိုက်ရန်မှတ်ဉာဏ်ဖြစ်သည့် memory/နေရာများ သိမ်းဆည်းထားရန် |
| **Application Insights** | အေးဂျင့်အကြား စဉ်ဆက်မပြတ်သော အမိန့်တောင်းဆိုမှုကို လိုက်လံကြည့်ရှုနိုင်ရန် |

### အေးဂျင့်များ ဘယ်လို ဆက်သွယ်လဲ

azd multi-agent များတွင် များသောအားဖြင့် **orchestrator သည် သင့်အက်ပလီကေးရှင်းကုတ်ထဲတွင် လည်ပတ်သည်** (ဥပမာ အဖြစ် Semantic Kernel သို့မဟုတ် Microsoft Agent Framework ကဲ့သို့သော မျက်နှာကြပ် framework အသုံးပြုသည်)။ orchestrator သည် specialist အေးဂျင့်တစ်ဦးချင်းစီကို ခေါ်ဆိုပြီး ရလဒ်များကို ကောင်းစွာ စုပေါင်းပြီး နောက်ဆုံးဖြေရှင်းချက် ပေးသည်။ အေးဂျင့်များသည် အောက်ပါဖြင့် context ကိုမျှဝေသည်:

- **လုပ်ဆောင်ချက်/ကိရိယာခေါ်ဆိုမှုများ** — orchestrator သည် specialist တစ်ဦးကိုခေါ်ပြီး ရလဒ်တစ်ခု ပြန်လည်ရသည်
- **မျှဝေထားသောမှတ်ဉာဏ်** — ဒေတာ့ဘေ့(သို့မဟုတ် Cosmos DB) တွင် အေးဂျင့်နှစ်ဦးစလုံး ဖတ်ရလွယ်သော ဆက်သွယ်ရေးအချက်အလက် သိမ်းဆည်းထားသည်
- **စာတိုက်/ပေါ်လိုက်ခြင်း** — ပိုမိုချိတ်ဆက်မှု အားနည်းမှုရှိရန်အတွက်၊ အေးဂျင့်များသည် queue သို့မဟုတ် Service Bus မှတဆင့် ဆက်သွယ်သည်

> **အမှားရှာခြင်းတွင် အရေးကြီးမှု**: အဆင့်တိုင်းဟာ သီးခြားဖြစ်သောကြောင့် Application Insights သည် ခေတ်နောက်ကျသော်လည်း မဆက်မစပ် သို့မဟုတ် မအောင်မြင်သော အေးဂျင့်အား သတိပြု၍ ကြည့်ရှုနိုင်သည်။ ဒါဟာ တာဝန်များကို အေးဂျင့်များ တိုက်ရိုက် ခွဲခြားသပ်ရပ်စေခြင်း၏ အကြီးမားဆုံးအကြောင်းတစ်ခု ဖြစ်သည်။

---

## တပ်ဆင်ပြီးပြီဆိုတာ စစ်ဆေးပါ

စနစ်သည် တကယ့်အလုပ်လုပ်နေမှု သေချာခံယူရန်:

```bash
# တပ်ဆင်ပြီးသော endpoint များကို ပြပါ
azd show

# အက်ပ်၏ ကြည့်ရှုမှု dashboard ကို ဖွင့်ပါ
azd monitor

# တစ်စုံတစ်ရာ မှားနေသည်ထင်ရင် log များကို ဆက်လက်ကြည့်ရှုပါ
azd monitor --logs
```

ပြီးလျှင် `azd show` မှ apps URL ကို ဖွင့်ကာ အေးဂျင့်အားလုံး ပါဝင်စေသော တောင်းဆိုမှုတစ်ခုကို စမ်းပါ (Creative Writer အတွက် သတ္တိ့တစ်ခုအပေါ် ဆောင်းပါးတို တစ်ပုဒ် ရေးချင်ဟု မေးပါ)။ Application Insights **transaction search** တွင် သုတေသနဆောင်, စာရေးသူ, အယ်ဒီတာအဆင့်များကို ရှုမြင်နိုင်ပါလိမ့်မည်။

**အောင်မြင်မှု အညွှန်းစံ:**
- ✅ `azd show` မှ ရောက်ရှိနိုင်သော endpoint ကို ဖော်ပြသည်။
- ✅ တောင်းဆိုမှုတစ်ခုသည် အဆင့်မျိုးစုံဖြတ်သန်းသော ရလဒ်ကို ထုတ်ပေးသည်။
- ✅ Application Insights မှ အေးဂျင့်အဆင့် များစွာအတွက် လမ်းကြောင်းများကို ပြသသည်။

---

## စိတ်တိုင်းကျ ပြင်ဆင်ခြင်း: အေးဂျင့်တစ်ခု ထည့်ခြင်း သို့မဟုတ် ပြင်ဆင်ခြင်း

အေးဂျင့် တစ်ခုစီသည် ညွှန်ကြားချက် များနှင့် ကိရိယာများသာ ရှိသောကြောင့် ပြင်ဆင်ရလွယ်သည်။

၁။ Template ထဲရှိ အေးဂျင့် ပြဋ္ဌာန်းချက်များကို ရှာပါ (ပုံမှန်အားဖြင့် `prompts/`, `agents/` သို့မဟုတ် `*.prompty` ဖိုင်များ)။
၂။ အေးဂျင့်တစ်ခု၏ ညွှန်ကြားချက်များအား လိုအပ်သလို ချိန်ညှိပါ။ ဥပမာ၊ အယ်ဒီတာအေးဂျင့်အား တိကျသော အသံသွင်းမှု သို့မဟုတ် စကားလုံးအရေအတွက် သတ်မှတ်ရန် ပြောပါ။
၃။ ကုဒ်ကိုသာ ပြန်လည်တပ်ဆင်ပါ (အခြေခံဖွဲ့စည်းမှု မပြောင်းလဲပါ)။

   ```bash
   azd deploy
   ```

ဆက်လက်၍ မိမိ၏ *manifest* မှ အေးဂျင့်များကို ဆောက်လုပ်လိုပါက agent extension နှင့် အပြီးအစီး လည်ပတ်မှုများကို အသုံးပြုပါ:

```bash
azd extension install azure.ai.agents
azd ai agent init -m agent-manifest.yaml
azd up
azd ai agent invoke      # စတင်စစ်ဆေးခြင်း၊ တုံ့ပြန်ချိန်နှင့်အတူ
```

အခန်း ၂: Agents (../chapter-02-ai-development/agents.md) နှင့် [AZD AI CLI reference](../chapter-08-production/production-ai-practices.md#azd-ai-cli-commands-and-extensions) တွင် `invoke`, `eval generate`, `optimize`, `delete` အပါအဝင် အေးဂျင့် လည်ပတ်မှု လမ်းညွှန်ချက်များ ရှိသည်။

---

## ရှင်းလင်းခြင်း

Multi-agent app များတွင် အသုံးပြုသူ ကုန်ကျစရိတ်ရှိသော ဝန်ဆောင်မှုများ များစွာ ပါဝင်သည်။ လုပ်ငန်းပြီးလျှင် အားလုံးကို ရှင်းလင်းပါ:

```bash
azd down --force --purge
```

`--purge` ဖလက်ကိုလည်း soft-delete ထားသော AI အရင်းအမြစ်များ (Foundry/Azure AI Services အကောင့်များကဲ့သို့) ကို ဖယ်ရှားပေးသောကြောင့် နောက်တစ်ကြိမ် ပြန်တပ်ဆင်ခြင်းကို အနှောင့်အယှက် မဖြစ်စေပါနှင့်၊ ကုန်ကျစရိတ် ခံစားမှု မရှိစေရ။

## ထုတ်လုပ်ရေး Multi-Agent စနစ်အကြောင်း မှတ်ချက်

ဒီ repo တွင် ပါရှိသည့် [Retail Multi-Agent Solution](../../examples/retail-scenario.md) သည် **နည်းပညာ ရေးဆွဲမှု ဖောင်ပြား** ဖြစ်ပြီး တစ်ချက်တွင် template မဟုတ်ပါ၊ ထုတ်လုပ်ရေး retail စနစ် တည်ဆောက်မှု ကို ရေးဆွဲထားသည် (ပြီးပြည့်စုံမွမ်းမံမှုများ ကြီးမားသည်ကို ရှင်းလင်းပြထားသည်)။ ၎င်းကို ဒီမှာ တပ်ဆင်ပြီးသော နမူနာ အပြီးတွင် ဒီဇိုင်း စာရွက်အဖြစ် အသုံးပြုပါ။ ထုတ်လုပ်ရေးအတွက် (ပြန်လည်တည်ဆောက်မှု၊ ကုန်ကျစရိတ်၊ နေကြတတ်မှု၊ အုပ်ချုပ်မှု) အခန်း ၈: Production AI Practices (../chapter-08-production/production-ai-practices.md) တွင် ဆက်လက်လေ့လာပါ။

## အနှစ်ချုပ်

- Multi-agent စနစ်သည် တာဝန်အထူးပြုများကို orchestrator တစ်ဦးက ညှိနှိုင်းပေးကာ ခွဲခြားထားသည်။
- မိမိ၏ တာဝန်သည် ကွဲပြားသောအဆင့်များ၊ တပြိုင်နက် လုပ်ငန်းများ သို့မဟုတ် အဆင့်တစ်ခုချင်းစီကို ကိရိယာကွာခြားမှုရှိပါက သာမန် ဗဟိုထား အေးဂျင့် တစ်ခုထက်သုံးပါ။
- azd workflow သည် မပြောင်းလဲပါ: `azd init` → `azd up` → စမ်းသပ် → `azd down`။
- `contoso-creative-writer` ကဲ့သို့ တကယ်အသုံးပြုနိုင်သော template ဖိုင်မှတဆင့် multi-agent app ကို ကြည့်ရှု၍ စိတ်တိုင်းကျ ပြင်ဆင်နိုင်သည်။
- Application Insights မှ agent များအကြား လမ်းကြောင်းကြည့်ရှုနိုင်မှုသည် multi-agent ဒီဇိုင်း၏ အကျိုးရှိစေမှုကြီးတစ်ခုဖြစ်သည်။
