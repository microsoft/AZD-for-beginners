# உங்கள் சொந்த azd வார்ப்புருவை உருவாக்குதல்

**அத்தியாய வழிகாட்டி:**
- **📚 பாடநெறி முகப்பு**: [AZD தொட beginners](../../README.md)
- **📖 தற்போதைய அத்தியாயம்**: அத்தியாயம் 4 - கட்டமைப்பான குறியீடு மற்றும் நிறுவல்
- **⬅️ முன்**: [நிறுவல் வழிகாட்டி](deployment-guide.md)
- **🚀 அடுத்த அத்தியாயம்**: [அத்தியாயம் 5: பன்முக பிரிந்த தீர்வுகள்](../chapter-05-multi-agent/README.md)

> `azd 1.27.1` இல் 2026 ஜூலை மாதத்தில் சரிபார்க்கப்பட்டது.

## அறிமுகம்

இதுவரை நீங்கள் `azd init --template <name>` உடன் வார்ப்புருக்களை *பயன்படுத்தி* உள்ளீர்கள். ஆனால் உங்கள் குழு விரும்பும் திட்ட அமைப்பை—உதாரணமாக, Cosmos DB உடன் ஒரு Container App மற்றும் சரியான கண்காணிப்புடன்—உங்கள் சொந்தமாய் மீண்டும் பயன்படுத்தக்கூடிய வார்ப்புருவாக மாற்ற விரும்புவீர்கள். ஒரு வார்ப்புரு என்பது azd எப்படி படிக்கும் என்று கணிக்கக்கூடிய கட்டமைப்பை கொண்ட Git சேமிப்பகமாகும். இந்த பாடம், அதை ஆரம்பத்திலிருந்து எப்படி கட்டுவது, சோதிப்பது, மற்றும் (ஆவலுக்கு) சமூக கண்காட்சியில் வெளியிடுவது பற்றி கொடுக்கிறது.

## கற்றல் இலக்குகள்

இந்த பாடம் முடிவில், நீங்கள்:
- ஒரு கோப்புறை எப்படி "azd வார்ப்புரு" ஆக இருக்க வேண்டும் என்பதை புரிந்துகொள்வீர்கள்
- தேவையான கோப்புகள் மற்றும் கோப்புறையமைப்பை அறிந்து கொள்வீர்கள்
- மற்றோர் பேர் பயன்படு கொள்ள `azure.yaml` மற்றும் `infra/` சேர்க்கலாம்
- உங்கள் வார்ப்புருவை பகிர்தலுக்கு முன் உள்ளகமாக சோதிக்கலாம்
- வெளியிட்டு (ஆவலுக்கு) Awesome AZD க்குள் சமர்ப்பிக்கலாம்

## கற்றல் விளைவுகள்

இந்த பாடம் முடிந்த பிறகு, நீங்கள் முடியும்:
- புதிய வார்ப்புரு சேமிப்பகத்தை அமைத்தல்
- எந்த சந்தாவிலும் வேலை செய்யும் படி கட்டமைப்பை தொலைநிலை நிரலை மாற்றுதல்
- `azd init` மற்றும் `azd up` மூலம் வார்ப்புருவை சரிபார்த்தல்
- சமூக கண்காட்சிக்கு தேவையான விவரக்கோவையைச் சேர்த்தல்

## வார்ப்புரு என்றால் என்ன, உண்மையில்?

azd வார்ப்புரு என்பது குறைந்தபட்சம் உள்ள **ஒரு Git சேமிப்பகம்** ஆகும்:

| கோப்பு / கோப்புறை | நோக்கம் | தேவைபாடு? |
|---------------|---------|-----------|
| `azure.yaml` | சேவைகள், ஹோஸ்ட்கள் மற்றும் கட்டமைப்பு வழங்குநரை விவரிக்கிறது | ✅ ஆமாம் |
| `infra/` | வளங்களை ஏற்படுத்தும் Bicep, Terraform, அல்லது Pulumi | ✅ ஆமாம் |
| `src/` (அல்லது உங்கள் குறியீடு) | azd நிறுவும் பயன்பாட்டு குறியீடு | ✅ ஆமாம் |
| `README.md` | வார்ப்புருவைப் பயன்படுத்துவது எப்படி என்பதை விளக்கும் | ✅ உறுதியான பரிந்துரை |
| `.azdo/` அல்லது `.github/` | CI/CD குழாய் வரையறைகள் | விருப்பமானது |
| `.devcontainer/` | மறுபடியும் உருவாக்கக்கூடிய டெவ் சூழல் | விருப்பமானது |
| `azure.yaml` `metadata` | கண்காட்சி + தொலைபேசி தகவல் | விருப்பமானது (வெளியிட தேவையானது) |

இத்துடன் யாரும் மந்திரம் இல்லை: `azd init --template you/your-repo` இயக்கும் போது, azd அந்த சேமிப்பகத்தை கிளோன் செய்து `azure.yaml`-ஐ வாசிக்கிறது.

## படி 1: சேமிப்பகம் அமைத்தல்

கோப்புறை அமைப்பை கைமுறையாக உருவாக்கவோ அல்லது குறைந்தபட்ச வார்ப்புருவில் இருந்து தொடங்கி திருத்தவோ செய்க:

```bash
mkdir my-azd-template && cd my-azd-template
git init

# சாதாரண அமைப்பை உருவாக்கவும்
mkdir -p src infra
```

ஒரு பொதுவான முடிவடைந்த அமைப்பு இதுபோன்று இருக்கும்:

```
my-azd-template/
├── azure.yaml              # ← template definition
├── README.md               # ← how to use it
├── infra/
│   ├── main.bicep          # entry point
│   ├── main.parameters.json
│   └── modules/
│       └── web.bicep
├── src/
│   └── app/                # your application code
└── .github/
    └── workflows/
        └── azure-dev.yml   # optional CI/CD
```

## படி 2: `azure.yaml` எழுதுதல்

இது வார்ப்புருவின் இதயம். இது azd என்ன நிறுவ வேண்டும் மற்றும் எப்படிச் செய்ய வேண்டும் என்பதை கூறுகிறது:

```yaml
# azure.yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/Azure/azure-dev/main/schemas/v1.0/azure.yaml.json
name: my-azd-template
metadata:
  template: my-azd-template@1.0.0   # name@version — used by the gallery and telemetry
infra:
  provider: bicep                   # or "terraform" / "pulumi"
  path: infra
services:
  web:
    project: ./src/app
    language: js                    # js | python | dotnet | java | go
    host: containerapp              # appservice | containerapp | function | aks | staticwebapp
```

> `metadata.template` புலம் கண்காட்சி தொலைபேசி விவரக்கோவை பயன்படுத்தி பயன்பாட்டை எண்ணுகிறது. `name@version` வழக்கம் பின்பற்றவும்.

## படி 3: கட்டமைப்பை அளவுருக்கள் செய்யவும்

மீண்டும் பயன்படக்கூடிய வார்ப்புருவுக்கான மிக முக்கிய விதி: **எப்போதும் பெயர்கள், பிரதேசங்கள், அல்லது சந்தா விவரங்களை கடுமையாக சேர்க்க வேண்டாம்.** அளவுருக்களைவும் வள அடையாள பாணியையும் பயன்படுத்தி ஒரே வார்ப்புரு எல்லோருக்கும் வேலை செய்யும்.

```bicep
// infra/main.bicep
targetScope = 'resourceGroup'

@minLength(1)
@description('Primary location for all resources')
param location string = resourceGroup().location

@minLength(1)
@maxLength(64)
@description('Name of the environment (used to generate resource names)')
param environmentName string

// Generate globally-unique, stable names from the environment + subscription
var resourceToken = uniqueString(subscription().id, environmentName, location)
var tags = { 'azd-env-name': environmentName }

module web 'modules/web.bicep' = {
  name: 'web'
  params: {
    name: 'app-${resourceToken}'
    location: location
    tags: tags
  }
}

output SERVICE_WEB_ENDPOINT_URL string = web.outputs.uri
```

இரண்டு அம்சங்கள் இந்த வார்ப்புருவை பயிற்சிக்கு ஏற்றவையாக்குகின்றன:

1. **`azd-env-name` குறிச்சொல்** — azd இதைப் பயன்படுத்தி ஒவ்வொரு சூழலுக்கும் வளங்களை கண்காணிக்கவும் மற்றும் சுத்தம் செய்யவும்.
2. **`uniqueString(...)` வள அடையாள களஞ்சியம்** — பெயர்கள் ஒன்றோடொன்று மோதாமல் முழுலோகம் முழுவதும் தனித்துவமான தகுதிகொண்ட முடிவை உருவாக்குகிறது.

சூழலுக்குள் azd ஊக்குவிக்கும் மதிப்புகளை வாசிக்கும் பொருத்தப்பட்ட அளவுருக்கள் கோப்பை வழங்கவும்:

```json
// infra/main.parameters.json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    "environmentName": { "value": "${AZURE_ENV_NAME}" },
    "location": { "value": "${AZURE_LOCATION}" }
  }
}
```

azd தற்போதைய சூழலிலிருந்து `${AZURE_ENV_NAME}` மற்றும் `${AZURE_LOCATION}` ஆகியவற்றை தானாக மாற்றிக் கொள்கிறது.

## படி 4: உங்கள் வார்ப்புருவை உள்ளகமாக சோதிக்கவும்

பகிருவதற்கு முன், வார்ப்புரு தெளிவான நிலையில் சரியாக வேலை செய்கிறது என்று நிரூபியுங்கள்.

**உள்ளக கோப்புறை இலிருந்து சோதிக்கவும்** (Git புஷ் தேவையில்லை):

```bash
# காலி அடைவை இருந்து, உங்கள் உள்ளூர் வார்ப்புரு பாதையை பயன்படுத்தி துவக்கம் செய்யவும்
mkdir /tmp/test-run && cd /tmp/test-run
azd init --template /path/to/my-azd-template

# ஒரே நேரத்தில் ஏற்பாடு செய்து செயல்படுத்தவும்
azd auth login
azd up
```

**பின்னர் பார்வையிடல் சோதனை செய்க**— ஒரு நல்ல வார்ப்புரு முழுமையாக சுத்தம் செய்யும்:

```bash
azd down --force --purge
```

`azd down` பின்பு வளங்களே இருக்கின், வளத்தில் `azd-env-name` குறிச்சொல்லை தவறவிட்டிருக்க வாய்ப்பு.

> **அடக்கம்:** முதலில் `azd provision --preview` இயக்கவும். இதன் மூலம் என்ன நடக்கும் என்று கணிக்கவும், வளம் உருவாக்கப்படும்வரை வார்ப்புரு பிழைகள் தெரியவரும்.

## படி 5: வார்ப்புருவை வெளியிடவும்

சேமிப்பகத்தை GitHub இல் புஷ் செய்யவும் (பசியி இருப்பின் மற்றவர்கள் பயன்படுத்த கூடிய வகையில்):

```bash
gh repo create my-azd-template --public --source=. --push
```

இப்போது யாரும் இதைப் பயன்படுத்தலாம்:

```bash
azd init --template your-github-username/my-azd-template
```

## சுருக்கம்

- ஒரு வார்ப்புரு என்பது போல `azure.yaml`, `infra/`, மற்றும் உங்கள் குறியீடு கொண்ட Git சேமிப்பகம் மட்டுமே.
- எல்லாவற்றையும் அளவுருக்கள் செய்யுங்கள்—பெயர்கள், பிரதேசங்கள் மற்றும் ஐடிகள்—எங்கு வேண்டுமானாலும் இயங்க.
- எப்போதும் வளங்களை `azd-env-name` குறிச்சொல்லுடன் குறிச்செள்ளுங்கள், அதனால் `azd down` வேலை செய்கிறது.
- வெளியிடுவதற்கு முன் உள்ளகமாக `azd init --template <local-path>` கொண்டு சோதிக்கவும்.
- Awesome AZD இல் சமர்ப்பிக்க metadata மற்றும் README சேர்க்கவும்.
