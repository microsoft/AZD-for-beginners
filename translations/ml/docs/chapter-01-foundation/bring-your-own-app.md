## പരിചയം

[നിങ്ങളുടെ ആദ്യത്തെ പ്രോജക്ട്](first-project.md) ൽ നിങ്ങൾ ഒരു ടെം‌പ്ലേറ്റിൽ നിന്നുതന്നെ ആപ്പ് ഡിപ്ലോയുചെയ്‍തു. എന്നാൽ പലതിനും നിങ്ങൾക്ക് ഇതിനകം തന്നെ ഒരു ആപ്പ് ഉണ്ടാകാം — ഒരു Node.js API, Python Flask സർവീസ്, .NET വെബ് ആപ്പ് — നിങ്ങളുടെ മെഷീനിലുള്ള ഒരു ഫോൾഡറിലൊന്നായി. ഈ പാഠം ആ നിലവിലുള്ള കോഡിൽ azd എങ്ങനെ ചേർക്കാമെന്ന് കാണിക്കുന്നു, അതിലൂടെ `azd up` ഉപയോഗിച്ച് ഡിപ്ലോയുചെയ്യാം, ടെം‌പ്ലേറ്റ് ആവശ്യമില്ല.

## പഠന ലക്ഷ്യങ്ങൾ

ഈ പാഠം പൂർത്തിയാക്കിയാൽ, നിങ്ങൾക്ക് സാധിക്കුന്നു:
- azd പ്രോജക്ട് തുടങ്ങാനുള്ള മൂന്ന് വഴികൾ മനസ്സിലാക്കുക
- നിലവിലുള്ള കോഡ്‌ബേസിന്റെ ഉള്ളിലായി `azd init` ഓടിക്കുക
- `azure.yaml`യും `infra/` ഫോൾഡറും നിങ്ങളുടെ ആപ്പിനായി എന്താണെന്നു മനസ്സിലാക്കുക
- azd ഇന്റ്രാസ്‌ട്രക്ചർ ജനറേറ്റ് ചെയ്യണമെന്നോ സ്വയം എഴുതണമെന്നോ എപ്പോഴെന്ന് തിരിച്ചറിയുക
- നിങ്ങളുടെ നിലവിലുള്ള ആപ്പ് `azd up` ഉപയോഗിച്ച് Azure-ലേക്ക് ഡിപ്ലോയുചെയ്യുക

## പഠന ഫലങ്ങൾ

ഈ പാഠം കഴിഞ്ഞാൽ, നിങ്ങൾക്ക് കഴിയും:
- 이미 ഉള്ള പ്രോജക്ടിൽ azd ആരംഭിക്കുക
- ഒരു അടിസ്ഥാന `azure.yaml` ഫയൽ വായിക്കുകയും തെരഞ്ഞെടുക്കുകയും ചെയ്യുക
- `azd infra generate` ഉപയോഗിച്ച് സ്റ്റാർട്ടർ ഇന്റ്രാസ്‌ട്രക്ചർ ജനറേറ്റ് ചെയ്യുക
- നിങ്ങളുടെ ആപ്പിനായി അനുയായമായ Azure ഹോസ്റ്റ് തിരഞ്ഞെടുക്കുക
- നിങ്ങളുടെ സ്വന്തം അപ്ലിക്കേഷൻ ഡിപ്ലോയും ക്ലീൻഅപ്പും ചെയ്യുക

## azd പ്രോജക്ട് തുടങ്ങാനുള്ള മൂന്ന് മാർഗങ്ങൾ

| Starting point | Command | When to use |
|----------------|---------|-------------|
| **From a template** | `azd init --template <name>` | പഠിക്കാൻ, അഥവാ ഒരു പരിശോദിച്ച സാമ്പിളിൽ നിന്നു പുതിയ ആപ്പ് തുടങ്ങുമ്പോൾ |
| **From your existing code** | `azd init` (in your project folder) | ഇതിനകം ഒരു ആപ്പ് ഉണ്ടാകുമ്പോഴും അതിനെ ഡിപ്ലോയുചെയ്യാൻ ആഗ്രഹിക്കുമ്പോൾ |
| **From a Git repo** | `azd init --from-code` (in a cloned repo) | നിലവിലുള്ള റീപോസിറ്ററിയിൽ azd സ്വീകരിക്കുമ്പോൾ |

നിങ്ങൾ കഴിയുന്നത് ആദ്യ ഓപ്ഷൻ വളരെ അഭ്യാസമാക്കിയിട്ടുണ്ട്. ഇതിൽ രണ്ടാമത്തെ — യഥാർത്ഥ ലോകത്തിലെ ഏറ്റവും സാധാരണമായ സാഹചര്യം — ആണ് ഇവിടം.

## Step 1: നിങ്ങളുടെ പ്രോജക്ടിൽ `azd init` ഓടിക്കുക

നിലവിലുള്ള പ്രോജക്ട് ഫോൾഡറിന്റെ ഉള്ളിൽ ഒരു ടെർമിനൽ തുറന്ന് ഓടിക്കുക:

```bash
cd my-existing-app
azd init
```

azd നിങ്ങളെ എങ്ങനെ ഇൻഷിയലൈസ്ട് ചെയ്യണമെന്നു ചോദിക്കും. തിരഞ്ഞെടുക്കുക:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

തിരഞ്ഞെടുക്കുക **"നിലവിലുള്ള ഡയറക്ടറിയിലെ കോഡ് ഉപയോഗിക്കുക."** azd പിന്നീട് നിങ്ങളുടെ ഫോൾഡർ സ്കാൻ ചെയ്ത് ഭാഷയും ഫ്രെയിംവർക്കും കണ്ടെത്തി ഒരു ഹോസ്റ്റ് നിർദേശിക്കും.

### azd എന്തെല്ലാം കണ്ടെത്തുന്നു

azd `package.json`, `requirements.txt`, `pom.xml`, `*.csproj`, അല്ലെങ്കിൽ `Dockerfile` പോലെയുള്ള സൂചനകൾ നോക്കുന്നു, കൂടെ അനുയോജ്യമായ ഒരു Azure ഹോസ്റ്റ് നിർദ്ദേശിക്കും:

| Your app | Likely detected host |
|----------|----------------------|
| Node.js / Python / .NET web app | Azure App Service or Container Apps |
| Containerized app (`Dockerfile`) | Azure Container Apps |
| Function app | Azure Functions |
| Static site (React/Vue build output) | Azure Static Web Apps |

കണ്ടെത്തിയ സേവന(ങ്ങൾ) സ്ഥിരീകരിക്കുക, azd നിങ്ങൾക്ക് ആവശ്യമായ ഫയലുകൾ scaffolding ചെയ്യുന്നു.

## Step 2: azd 무엇 സൃഷ്ടിച്ചു എന്ന് മനസ്സിലാക്കുക

init കഴിഞ്ഞ്, നിങ്ങളുടെ പ്രോജക്ടിൽ രണ്ട് പുതിയ കാര്യങ്ങൾ ഉണ്ടാകും:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — പ്രോജക്ട് നിർവചനമാണ്

ഇത് azd പ്രോജക്ടിന്റെ ഹൃദയമാണ്. ഒരു കുറഞ്ഞതരം ഫയൽ ഇങ്ങിനെ തോന്നും:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

`services` ബ്ലോക്ക് പ്രധാന ഘടകമാണ്: ഓരോ എൻട്രിയും നിങ്ങളുടെ കോഡിന്റെ ഒരു ഫോൾഡറിനെ ഒരു Azure ഹോസ്റ്റുമായി മാപ്പ് ചെയ്യുന്നു. നിങ്ങളുടെ ആപ്പിന് ഫ്രണ്ട്‌എൻഡ് ഉളളതും API ഉളളതുമാണെങ്കിൽ, നിങ്ങൾക്ക് രണ്ട് സേവനങ്ങളും ഉണ്ടാകും.

### `infra/` — കോഡായി നിങ്ങളുടെ Azure റിസോഴ്സുകൾ

`infra/` ഫോൾഡറിൽ നിങ്ങളുടെ ആപ്പിനാവശ്യമായ Azure റിസോഴ്സുകൾ നിർവചിക്കുന്ന Bicep ഫയലുകൾ ఉంటాయి (App Service, ഡാറ്റാബേസ് എന്നിവ). നിങ്ങൾക്ക് ഇവ കൈവഴങ്ങി എഴുതേണ്ടതില്ല — azd ഒരു പ്രവർത്തനക്ഷമമായ സ്റ്റാർട്ടിംഗ് പോയിന്റ് ജനറേറ്റ് ചെയ്യുന്നു. പിന്നീട് റിസോഴ്‌സുകൾ ചേർക്കാനോ സുരക്ഷ കൂടുതൽ കടുത്തതാക്കി മാറ്റാനോ നിങ്ങൾക്ക് ഇവ എഡിറ്റ് ചെയ്യാവുന്നതാണ് (ഇത് [Chapter 4](../chapter-04-infrastructure/README.md) ൽ വിശദീകരിച്ചിരിക്കുന്നു).

> **Tip:** ഡിപ്ലോയു ചെയ്യുന്നതിന് മുമ്പായി ജനറേറ്റുചെയ്‌ത ഇന്റ്രാസ്ട്രക്ചർ കാണാൻ അല്ലെങ്കിൽ കസ്റ്റമൈസ് ചെയ്യാൻ ആഗ്രഹിക്കുമോ? IaC ഡിസ്കിലേക്ക് എഴുതാൻ `azd infra generate` (അല്ലെങ്കിൽ `azd infra synth`) ഓടിക്കുക, അവ പരിശോധിച്ചതും വേർഷൻ-കൺട്രോളിനായി സംഭสีക്കാവുന്നതുമാണ്.

## Step 3: ആവശ്യമായ കോൺഫിഗറേഷൻ സജ്ജമാക്കുക

നിങ്ങളുടെ ആപ്പിന് സെറ്റിംഗുകളും രഹസ്യങ്ങളും (കണക്ഷൻ സ്റ്റ്രിങ്, API കീ മുതലായവ) ആവശ്യമുണ്ടെങ്കിൽ അവ ഹാർഡ്‌കോഡ് ചെയ്യരുത്. പരിസ്ഥിതി മൂല്യങ്ങൾ ഉപയോഗിക്കുക:

```bash
# ഒരു പരിസ്ഥിതി സൃഷ്ടിക്കുക
azd env new dev

# ഒരു രഹസ്യമല്ലാത്ത മൂല്യം സജ്ജമാക്കുക
azd env set API_VERSION 1.0.0
```

വാസ്തവമായ രഹസ്യങ്ങൾ Key Vault-ൽ സൂക്ഷിച്ച് അവയെ നിങ്ങളുടെ ഇൻഫ്രാസ്ട്രക്ചറിൽ നിന്ന് റഫറൻസ്ചെയ്യുക — നോക്കുക [Chapter 3: Configuration & Authentication](../chapter-03-configuration/authsecurity.md).

## Step 4: ഡിപ്ലോയുചെയ്യുക

ഇപ്പോൾ നിങ്ങൾക്ക് അറിയാവുന്ന സമാന വർക്ക്‌ഫ്ലോ ഉപയോഗിക്കുക:

```bash
# പ്രാമാണീകരിക്കുക (azd-ക്കായി ആവശ്യമാണ്)
azd auth login

# സൃഷ്ടിക്കപ്പെടുന്ന റിസോഴ്‌സുകളുടെ മുൻദർശനം
azd provision --preview

# ഇൻഫ്രാസ്ട്രക്ചർ ഒരുക്കുകയും നിങ്ങളുടെ കോഡ് വിന്യസിക്കുകയും ചെയ്യുക
azd up
```

മుగിഞ്ഞപ്പോൾ, azd നിങ്ങളുടെ ആപ്പിന്റെ URL പ്രിന്റ് ചെയ്യും. സാധാരണ azd ആപ്പ് പോലെ തന്നെ അത് സ്ഥിരീകരിക്കുക:

```bash
azd show           # എൻഡ്‌പോയിന്റുകൾ കാണിക്കുക
azd monitor --logs # ആവശ്യമായെങ്കിൽ ലോഗുകൾ പരിശോധിക്കുക
```

## ആദ്യത്തേത് നേരിടുന്ന സാധാരണ പ്രശ്നങ്ങൾ

| ലക്ഷണം | സാധ്യതയുള്ള കാരണം | പരിഹാരം |
|---------|------------------------------|-----|
| azd എന്റെ ആപ്പ് കണ്ടെത്തിയില്ല | മാനിഫെസ്റ്റ് നഷ്ടമാണ് (ഉദാഹരണം: `package.json`) | മാനിഫെസ്റ്റ് ചേർക്കുക, അല്ലെങ്കിൽ `azd init` സമയത്ത് ഹോസ്റ്റ് കൈമേൽമാറ്റം തിരഞ്ഞെടുക്കുക |
| `azd up` സമയത്ത് ബിൽഡ് പരാജയപ്പെടുന്നു | ആപ്പിന് ഒരു ബിൽഡ് സ്റ്റെപ്പ് ആവശ്യമാണ് | `azure.yaml`-ഇൽ സേവനത്തിന് കീഴിൽ `buildCommand`/`outputPath` ചേർക്കുക |
| ആപ്പ് സ്റ്റാർട്ട് ചെയ്യുന്നു പക്ഷേ പൊറുതിവേൽ λά.error കൾ കാണിക്കുന്നു | കോൺഫിഗ്/രഹസ്യം ഇല്ല | `azd env set` ഉപയോഗിച്ച് മൂല്യങ്ങൾ സജ്ജമാക്കുക അല്ലെങ്കിൽ Key Vault ബന്ധിപ്പിക്കുക |
| തെറ്റായ ഹോസ്റ്റ് തിരഞ്ഞെടുക്കപ്പെട്ടു | ഓട്ടോ-ഡിറ്റക്ഷൻ സംശയിച്ചു | `azure.yaml`-ഇൽ `host:` എഡിറ്റ് ചെയ്ത് വീണ്ടും `azd up` ഓടിക്കുക |

കൂടുതലിനായി, നോക്കുക [Chapter 7: Troubleshooting](../chapter-07-troubleshooting/README.md).

## സംഗ്രഹം

- `azd init` → **"നിലവിലുള്ള ഡയറക്ടറിയിലെ കോഡ് ഉപയോഗിക്കുക"** എന്നത് ഇതിനകം ഉള്ള ഒരു ആപ്പിലേക്ക് azd ചേർക്കുന്നു.
- `azure.yaml` നിങ്ങളുടെ കോഡ് ഫോള്ഡറുകൾ Azure ഹോസ്റ്റുകളിലേക്ക് മാപ്പ് ചെയ്യുന്നു; `infra/` Bicep ആയി റിസോഴ്‌സുകൾ നിർവ്വചിക്കുന്നു.
- `azd infra generate` നിങ്ങളെ ജനറേറ്റ് ചെയ്ത ഇന്റ്രാസ്ട്രക്ചർ പരിശോധിക്കാനും കസ്റ്റമൈസ് ചെയ്യാനും അനുവദിക്കുന്നു.
- ഒരു തവണ ഇൻഷിയലൈസ് ചെയ്താൽ, നിങ്ങളുടെ നിലവിലുള്ള ആപ്പ് ടെം‌പ്ലേറ്റ് അടിസ്ഥാനമാക്കിയുള്ള ആപ്പുപോലെ തന്നെ `azd up` / `azd down` വർക്ക്‌ഫ്ലോ ഉപയോഗിക്കും.
