## പരിചയം

ഇതുവരെ നിങ്ങൾ `azd init --template <name>` ഉപയോഗിച്ച് ടെംപ്ലേറ്റുകൾ *ഉപയോഗിച്ചിട്ടുണ്ട്*. എന്നാൽ നിങ്ങളുടെ ടീം ഇഷ്ടപ്പെടുന്ന ഒരു പ്രോജക്ട് ഒരുക്കം ഉണ്ടാക്കിയാൽ—ഉദാഹരണത്തിന്, ഒരു കോൺടെയ്‌നർ ആപ്പ് koosmos DBയും ശരിയായ მონിറ്ററിങ്ങും ഉള്ളതായി—നിങ്ങൾ അതിനെ നിങ്ങളുടെ സ്വന്തം പുനരുപയോഗയോഗ്യമായ ടെംപ്ലേറ്റായി മാറിക്കാൻ ആഗ്രഹിക്കും. ഒരു ടെംപ്ലേറ്റ് അഥവാ Git റിപോസിറ്ററിയാണ്, azd സ്കാൻ ചെയ്യാൻ കഴിയുന്ന നിർദ്ദേശിത ഘടനയോടുകൂടിയതാണ്. ഈ പാഠം ആദ്യം മുതൽ ഒരു ടെംപ്ലേറ്റ് എങ്ങനെ നിർമ്മിക്കാം, പരീക്ഷിക്കാം, (ഓപ്ഷണലായി) സമൂഹ ഗാലറിയിലേക്ക് പ്രക്രിയ ചെയ്യാം എന്നത് കാണിക്കുന്നു.

- ഒരു ഫോൾഡർ "azd ടെംപ്ലേറ്റ്" ആക്കുന്നത് എന്താണെന്ന് മനസ്സിലാക്കുക
- ആവശ്യമായ ഫയലുകളും ഫോൾഡർ ഘടനയും അറിയുക
- മറ്റുള്ളവർക്ക് പുനരുപയോഗിക്കാവുന്ന `azure.yaml` ഒപ്പം `infra/` ചേർക്കുക
- പങ്കുവക്കുന്നതിനു മുൻപ് നിങ്ങളുടെ ടെംപ്ലേറ്റ് ലോകൽ ആയി ടെസ്റ്റ് ചെയ്യുക
- അത് പ്രസിദ്ധീകരിക്കുക ഒപ്പം (ഓപ്ഷണലായി) Awesome AZD-യിലേക്ക് സമർപ്പിക്കുക

## പഠന ഫലം

ഈ പാഠം പൂർത്തിയാക്കിയാൽ, നിങ്ങൾക്ക് കഴിയും:
- പുതിയ ഒരു ടെംപ്ലേറ്റ് റിപോസിറ്ററി നിർമ്മിക്കുക

- `azd init` ഒപ്പം `azd up` ഉപയോഗിച്ച് ടെംപ്ലേറ്റ് സാധൂകരിക്കുക
- സമൂഹ ഗാലറി ആവശ്യമായ മെറ്റാഡേറ്റ ചേര്ക്കുക

## ടെംപ്ലേറ്റ്란 무엇인가?

azd ടെംപ്ലേറ്റ് എന്ന് പറയുന്നത് **Git റിപോസിറ്ററി** ആണ്, അതിൽ കുറഞ്ഞത് ഇതൊക്കെ ഉണ്ടാകണം:

| ഫയൽ / ഫോൾഡർ | പ്രവർത്തനം | ആവശ്യമാണ്? |
|---------------|---------|-----------|
| `azure.yaml` | സേവനങ്ങൾ, ഹോസ്റ്റുകൾ, ഇൻഫ്രാ പ്രൊവൈഡർ വിവരിക്കുന്നു | ✅ ആവശ്യമാണ് |
| `infra/` | ബൈസിപ്പ്, ടെറഫോം, അല്ലെങ്കിൽ പൂൽമി ഉപയോഗിച്ച് റിസോഴ്‌സുകൾ ഒരുക്കുന്നു | ✅ ആവശ്യമാണ് |
| `src/` (അല്ലെങ്കിൽ നിങ്ങളുടെ കോഡ്) | ആപ്ലിക്കേഷൻ കോഡ് azd ഡിപ്പ്ലോയ്മെന്റ് നടത്തുന്നു | ✅ ആവശ്യമാണ് |
| `README.md` | ടെംപ്ലേറ്റ് എങ്ങനെ ഉപയോഗിക്കാമെന്ന് | ✅ ബലമായി ശുപാർശ ചെയ്യുന്നു |
| `.azdo/` അല്ലെങ്കിൽ `.github/` | CI/CD പൈപ്പ്‌ലൈൻ നിർവചനങ്ങൾ | ഐച്ഛികം |
| `.devcontainer/` | പുനരുകരിക്കാൻ കഴിയുന്ന ഡെവ് പരിസ്ഥിതി | ഐച്ഛികം |
| `azure.yaml` `metadata` | ഗാലറി + ടെലിമെട്രി വിവരങ്ങൾ | ഐച്ഛികം (പബ്ലിഷ് ചെയ്യാൻ ആവശ്യമാണ്) |

ഇത് ഒരു മാജിക് ഒന്നുമല്ല: നിങ്ങൾ `azd init --template you/your-repo` എന്നത് ഓടിക്കുമ്പോൾ, azd റിപോ ക്ലോൺ ചെയ്തു `azure.yaml` വായിക്കുന്നു.

## എടപ്പ് 1: റിപോസിറ്ററി Scaffold ചെയ്യുക

ഫോൾഡർ ഘടന കൈമുതൽ സൃഷ്ടിക്കുക അല്ലെങ്കിൽ കുറഞ്ഞ ടെംപ്ലേറ്റ് ഇല്ലാതെ അത് ആരംഭിച്ച് എഡിറ്റ് ചെയ്യുക:

```bash
mkdir my-azd-template && cd my-azd-template
git init

# സ്റ്റാൻഡേർഡ് ലേഔട്ട് സൃഷ്ടിക്കുക
mkdir -p src infra
```

സാധാരണമായ ഒരു പൂർത്തിയായ ഘടന ഇതുപോലെ ആയിരിക്കും:

```
my-azd-template/
├── azure.yaml              # ← template definition
├── README.md               # ← how to use it
├── infra/
│   ├── main.bicep          # entry point
│   ├── main.parameters.json
│   └── modules/
│       └── web.bicep
├── src/
│   └── app/                # your application code
└── .github/
    └── workflows/
        └── azure-dev.yml   # optional CI/CD
```

## എടപ്പ് 2: `azure.yaml` എഴുതുക

ഇത് ടെംപ്ലേറ്റിന്റെ ഹൃദയം ആണ്. ഇത് azd-ന് എന്തൊക്കെ ഡിപ്പ്ലോയ് ചെയ്യാമെന്ന് പറയുന്നു:

```yaml
# azure.yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/Azure/azure-dev/main/schemas/v1.0/azure.yaml.json
name: my-azd-template
metadata:
  template: my-azd-template@1.0.0   # name@version — used by the gallery and telemetry
infra:
  provider: bicep                   # or "terraform" / "pulumi"
  path: infra
services:
  web:
    project: ./src/app
    language: js                    # js | python | dotnet | java | go
    host: containerapp              # appservice | containerapp | function | aks | staticwebapp
```

> `metadata.template` ഫീൽഡ് ഗാലറി ടെലിമെട്രി ഉപയോഗിച്ച് ഉപയോഗം എണ്ണാൻ ആണ്. `name@version` രൂപം ഉപയോഗിക്കുക.

## എടപ്പ് 3: ഇൻഫ്രാസ്ട്രക്ചർ പാരാമറ്ററൈസ് ചെയ്യുക

പുനരുപയോഗം സാധ്യമാക്കാനുള്ള ഏറ്റവും പ്രധാന നിയമം: **നാമങ്ങൾ, പ്രദേശങ്ങൾ, സബ്സ്ക്രിപ്ഷൻ-സ്പെസിഫിക് മൂല്യങ്ങൾ ഒരിക്കലും ഹാർഡ്കോഡ് ചെയ്യരുത്.** പാരാമീറ്ററുകളും റിസോഴ്‌സ് ടോക്കൺ പാറ്റേണും ഉപയോഗിക്കുക, അതിനാൽ ഒരേ ടെംപ്ലേറ്റ് ആരുടേയും സബ്സ്ക്രിപ്ഷനിലും പ്രവർത്തിക്കും.

```bicep
// infra/main.bicep
targetScope = 'resourceGroup'

@minLength(1)
@description('Primary location for all resources')
param location string = resourceGroup().location

@minLength(1)
@maxLength(64)
@description('Name of the environment (used to generate resource names)')
param environmentName string

// Generate globally-unique, stable names from the environment + subscription
var resourceToken = uniqueString(subscription().id, environmentName, location)
var tags = { 'azd-env-name': environmentName }

module web 'modules/web.bicep' = {
  name: 'web'
  params: {
    name: 'app-${resourceToken}'
    location: location
    tags: tags
  }
}

output SERVICE_WEB_ENDPOINT_URL string = web.outputs.uri
```

ഈ ടെംപ്ലേറ്റ് സൗഹൃദമാക്കുന്ന രണ്ട് കാരണങ്ങൾ:

1. **`azd-env-name` ടാഗ്** — azd പരിസ്ഥിതിയനുസരിച്ച് റിസോഴ്‌സുകൾ ട്രാക്ക് ചെയ്യാനും ക്ലീൻ ചെയ്യാനും ഇത് ഉപയോഗിക്കുന്നു.
2. **`uniqueString(...)` റിസോഴ്‌സ് ടോക്കൺ** — സ്ഥിരവും ആഗോളമായി അതുല്യമായ സഫ്ഫിക്സ് ഉൽപ്പാദിപ്പിക്കുന്നു, അത്ാൽ പേരുകൾ തട്ടിപ്പില്ലാതെ നിലനിൽക്കും.

പരിസ്ഥിതിയിൽ നിന്നും azd ചേർക്കുന്ന മൂല്യങ്ങൾ വായിക്കാനുള്ള പാരാമീറ്ററുകൾ ഉള്ള ഫയൽ കൊടുക്കുക:

```json
// infra/main.parameters.json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    "environmentName": { "value": "${AZURE_ENV_NAME}" },
    "location": { "value": "${AZURE_LOCATION}" }
  }
}
```

azd സ്വയം നിലവിലെ പരിസ്ഥിതിയിൽ നിന്നും `${AZURE_ENV_NAME}` ഒപ്പം `${AZURE_LOCATION}` പകരുന്നു.

## എടപ്പ് 4: നിങ്ങളുടെ ടെംപ്ലേറ്റ് ലോകലായി ടെസ്റ്റ് ചെയ്യുക

പങ്കുവക്കുന്നതിന് മുൻപ്, ടെംപ്ലേറ്റ് ശുദ്ധമായ അവസ്ഥയിൽ പ്രവർത്തിക്കുന്നതായി തെളിയിക്കുക.

**ലോകൽ ഫോൾഡറിൽ നിന്ന് ടെസ്റ്റ് ചെയ്യുക** (Git push ആവശ്യമില്ല):

```bash
# ഒരു ശൂന്യ ഡയറക്ടറിയിൽ നിന്ന്, നിങ്ങളുടെ പ്രാദേശിക ടെംപ്ലേറ്റ് പാത്ത് ഉപയോഗിച്ച് ഇൻഷ്യലൈസ് ചെയ്യുക
mkdir /tmp/test-run && cd /tmp/test-run
azd init --template /path/to/my-azd-template

# പ്രൊവിഷൻ + എൻഡ് ടു എൻഡ് ഡിപ്ലോയ് ചെയ്യുക
azd auth login
azd up
```

**തുടർന്ന് ക്ലീൻ അപ് ടെസ്റ്റ് ചെയ്യുക**—നല്ല ടെംപ്ലേറ്റ് പൂർണ്ണമായി ക്ലീൻ ചെയ്യണം:

```bash
azd down --force --purge
```

`azd down` റിസോഴ്സുകൾ വിട്ട് പോയാൽ, നിങ്ങൾ ഒരു റിസോഴ്‌സിൽ `azd-env-name` ടാഗ് ചേർക്കാൻ മറന്നിരിക്കാം.

> **ടിപ്പ്:** ആദ്യം `azd provision --preview` ഓടിക്കുക. ഇത് എന്ത് സംഭവിക്കാൻ പോകുന്നു എന്ന് കാണിക്കുന്നു, ടെംപ്ലേറ്റ് പിശകുകൾ ആദ്യത്തേ തന്നെ നൽകുന്നു.

## എടപ്പ് 5: ടെംപ്ലേറ്റ് പ്രസിദ്ധീകരിക്കുക

റിപോസിറ്ററി GitHub-ലേക്ക് പുഷ് ചെയ്യുക (മറ്റുള്ളവർ ഉപയോഗിക്കണമെങ്കിൽ പൊതു):

```bash
azd init --template your-github-username/my-azd-template
```

- ✅ (ശുപാർശ) ഒരു ക്ലിക്ക് കോഡ്സ്പേസുകൾക്ക് `.devcontainer/`

Awesome AZD റിപ്പൊസിറ്ററിയിലെ [സംവരണഗൈഡ്](https://github.com/Azure/awesome-azd) അനുസരിച്ച്, ഗാലറിയുടെ ഡാറ്റ ഫയലിൽ നിങ്ങളുടെ ടെംപ്ലേറ്റ് കൂട്ടിച്ചേർക്കുന്ന ഒരു പുൾ റിക്വസ്റ്റ് തുറന്ന് സമർപ്പിക്കുക.

## പൊതുവായ പിഴവുകൾ

| പിഴവ് | പരിഹാരം |
|---------|-----|
| ഹാർഡ്കോഡ് ചെയ്ത റിസോഴ്‌സ് നാമങ്ങൾ | `uniqueString()` റിസോഴ്‌സ് ടോക്കൺ ഉപയോഗിക്കുക |
| `azd down` റിസോഴ്‍സുകൾ വിടുന്നു | എല്ലാ റിസോഴ്‌സുകൾക്കും (അഥവാ റിസോഴ്‌സ് ഗ്രൂപ്പിനും) `azd-env-name` ടാഗ് ചേർക്കുക |
| ടെംപ്ലേറ്റ് നിങ്ങൾക്ക് പ്രവർത്തിച്ച് മറ്റുള്ളവർക്കായി പരാജയപ്പെടുന്നു | സബ്സ്ക്രിപ്ഷൻ ഐഡികൾ, ടെനന്റ് ഐഡികൾ, പ്രദേശ അനുമാനങ്ങൾ നീക്കം ചെയ്യുക |
| ഔട്ട്‌പുട്ടുകൾ ആപ്പിൽ കൂട്ടിച്ചേർക്കാത്തത് | `SERVICE_<NAME>_ENDPOINT_URL` ഒപ്പം മറ്റുള്ള `AZURE_*` ഔട്ട്‌പുട്ടുകൾ എക്സ്പോർട്ട് ചെയ്യുക |
| ഗാലറി സമർപ്പണം തള്ളിടുന്നു | `README.md`, `LICENSE`, ഒപ്പം `metadata.template` ചേർക്കുക |

## സംഗ്രഹം

- ഒരു ടെംപ്ലേറ്റ് Git റിപ്പൊ മാത്രം ഉള്ളതാണ്, അതിൽ `azure.yaml`, `infra/`, ഒപ്പം നിങ്ങളുടെ കോഡ് ഉൾപ്പെടുന്നു.
- എല്ലാം പാരാമറ്ററൈസ് ചെയ്യുക - പേരുകൾ, പ്രദേശങ്ങൾ, ഐഡികൾ - അതിനാൽ ഏതു സ്ഥലം കൊണ്ടും പ്രവർത്തിക്കും.
- എല്ലാ റിസോഴ്‌സുകൾക്കും `azd-env-name` ടാഗ് ചേർക്കുക, അതിനാൽ `azd down` ശരിയായി പ്രവർത്തിക്കും.
- പ്രസിദ്ധീകരിക്കുന്നതിന് മുൻപ് `azd init --template <local-path>` ഉപയോഗിച്ച് ലോകലായി ടെസ്റ്റ് ചെയ്യുക.
- Awesome AZD-യിലേക്ക് സമർപ്പിക്കാൻ മെറ്റാഡേറ്റയും READMEയും ചേർക്കുക.
