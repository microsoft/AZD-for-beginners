## പരിചയപ്പെടുത്തൽ

ഇതുവരെ നിങ്ങൾ `azd init --template <name>` ഉപയോഗിച്ച് ടേംപ്ലേറ്റുകൾ *ഉപയോഗിച്ചു*. എന്നാൽ നിങ്ങളുടെ ടീം ഇഷ്ടപ്പെടുന്ന ഒരു പ്രോജക്ട് ലേഔട്ട് — ഉദാഹരണത്തിന്, Cosmos DB കൂടേണ്ട ഒരു Container App കൂടാതെ ശരിയായ മോണിറ്ററിംഗ് — ഉണ്ടാക്കിയ ശേഷം, അതിനെ നിങ്ങളുടെ സ്വന്തം പുനരുപയോഗിക്കാവുന്ന ടെംപ്ലേറ്റായി മാറ്റാൻ ആഗ്രഹിക്കും. ഒരു ടെംപ്ലേറ്റ് azd വായിക്കാനാവുന്ന പ്രവചനീയ ഘടനയുള്ള ഒരു Git റെപ്പോസിറ്ററിയ ғана ആണ്. ഈ പാഠം നിങ്ങളെ ശൂന്യസ്ഥിതിയിൽ നിന്നു തുടങ്ങിവച്ച് ഒന്ന് നിർമ്മിക്കുക, പരിശോധിക്കുക, (ഐച്ഛികമായി) സമൂഹ ഗ്യാലറിയിലേക്ക് പ്രസിദ്ധീകരിക്കുക എന്നിവ എങ്ങനെ ചെയ്യാമെന്നത് കാണിക്കുന്നു.

## പഠന ലക്ഷ്യങ്ങൾ

ഈ പാഠം പൂർത്തിയാക്കിയപ്പോൾ, നിങ്ങൾ:
- ഫോൾഡർ എന്തുകൊണ്ട് "azd ടെംപ്ലേറ്റ്" എന്നറിയും
- ആവശ്യമായ ഫയൽ/ഫോൾഡർ ലേഔട്ട് അറിയാം
- മറ്റുള്ളവര്‍ക്ക് പുനഃഉപയോഗിക്കാവുന്ന `azure.yaml` සහ `infra/` ചേർക്കും
- പങ്കുവയ്‌ക്കുന്നതിന് മുമ്പ് ടെംപ്ലേറ്റ് ലോക്കലായി പരീക്ഷിക്കും
- അതിനെ പ്രസിദ്ധീകരിക്കുകയും (ഐച്ഛികമായി) Awesome AZD-ലേക്ക് സമർപ്പിക്കുകയും ചെയ്യാം

## പഠനഫലം

ഈ പാഠം പൂർത്തിയാക്കിയതിനു ശേഷം, നിങ്ങൾക്ക് സാധിക്കാം:
- പുതിയ ടെംപ്ലേറ്റ് റെപ്പോ സ്കാഫോൾഡ് ചെയ്യുക
- ഇൻഫ്രാസ്ട്രക്ചർ എല്ലാ സബ്‌സ്‌ക്രിപ്ഷനുകളിലേയ്ക്കും പ്രവർത്തിക്കാൻ പാരാമറ്ററൈസ് ചെയ്യുക
- `azd init` және `azd up` ഉപയോഗിച്ച് ടെംപ്ലേറ്റ് സ്ഥിരീകരിക്കുക
- കമ്മ്യൂണിറ്റി ഗ്യാലറി ആവശ്യപ്പെടുന്ന മെറ്റഡേറ്റ ചേർക്കുക

## ടേംപ്ലേറ്റ് എന്നത് യഥാർത്ഥത്തിൽ എന്താണ്?

azd ടെംപ്ലേറ്റ് എന്നത് കുറഞ്ഞത് ഇതൊക്കെയാണ് ഉള്ളടക്കം ഉള്ള ഒരു Git റെപ്പോസിറ്ററി:

| ഫയൽ / ഫോൾഡർ | ഉദ്ദേശം | ആവശ്യമാണ്? |
|---------------|---------|-----------|
| `azure.yaml` | സേവനങ്ങൾ, ഹോസ്റ്റുകൾ, ഇൻഫ്രാ പ്രൊവൈഡർ എന്നിവയെ വിവരിക്കുന്നു | ✅ അതെ |
| `infra/` | റിസോഴ്‌സുകൾ പ്രോവിഷൻ ചെയ്യുന്ന Bicep, Terraform, അല്ലെങ്കിൽ Pulumi | ✅ അതെ |
| `src/` (അഥവാ നിങ്ങളുടെ കോഡ്) | azd ഡിപ്പ്ലോയ്ചെയ്യുന്ന അപ്ലിക്കേഷൻ കോഡ് | ✅ അതെ |
| README.md | ടെംപ്ലേറ്റ് എങ്ങനെ ഉപയോഗിക്കാം എന്നത് | ✅ ശക്തമായി ശുപാർശ ചെയ്യുന്നു |
| `.azdo/` or `.github/` | CI/CD പൈപ്‌ലൈൻ നിർവചനങ്ങൾ | ഐച്ഛികം |
| `.devcontainer/` | പുനര നിർമ്മിക്കാവുന്ന ഡെവ് പരിസ്ഥിതി | ഐച്ഛികം |
| `azure.yaml` `metadata` | ഗ്യാലറി + ടെലിമെട്രി വിവരങ്ങൾ | ഐച്ഛികം (പ്രസിദ്ധീകരിക്കാൻ ആവശ്യമാണ്) |

ഇവിടെ പ്രത്യേകമായ ഒന്നുമില്ല: നിങ്ങൾ `azd init --template you/your-repo` चलിക്കുമ്പോൾ azd റെപ്പോ ക്ലോൺ ചെയ്ത് `azure.yaml` വായിക്കുന്നു.

## പടി 1: റെപ്പോസിറ്ററി സ്കാഫോൾഡ് ചെയ്യുക

ഫോൾഡർ ഘടന കൈയോടെ സൃഷ്ടിക്കുക അല്ലെങ്കിൽ ഒരു ലഘു ടെംപ്ലേറ്റ് നിന്ന് തുടങ്ങുകയും തിരുത്തുകയും ചെയ്യുക:

```bash
mkdir my-azd-template && cd my-azd-template
git init

# സാധാരണ ലേഔട്ട് സൃഷ്ടിക്കുക
mkdir -p src infra
```

സാധാരണയായി അവസാനത്തിലുള്ള ലേഔട്ട് ഇങ്ങനെ കാണപ്പെടുന്നു:

```
my-azd-template/
├── azure.yaml              # ← template definition
├── README.md               # ← how to use it
├── infra/
│   ├── main.bicep          # entry point
│   ├── main.parameters.json
│   └── modules/
│       └── web.bicep
├── src/
│   └── app/                # your application code
└── .github/
    └── workflows/
        └── azure-dev.yml   # optional CI/CD
```

## പടി 2: `azure.yaml` എഴുതുക

ഇത് ടേംപ്ലേറ്റിന്റെ ഹൃദയമാണ്. ഇത് azd എന്താണ് വിന്യസിക്കണമെന്ന് എങ്ങനെ ചെയ്യാമെന്നതും പറയുന്നു:

```yaml
# azure.yaml
# yaml-language-server: $schema=https://raw.githubusercontent.com/Azure/azure-dev/main/schemas/v1.0/azure.yaml.json
name: my-azd-template
metadata:
  template: my-azd-template@1.0.0   # name@version — used by the gallery and telemetry
infra:
  provider: bicep                   # or "terraform" / "pulumi"
  path: infra
services:
  web:
    project: ./src/app
    language: js                    # js | python | dotnet | java | go
    host: containerapp              # appservice | containerapp | function | aks | staticwebapp
```

> `metadata.template` ഫീൽഡ് ഗ്യാലറി ടെലിമെട്രി ഉപയോഗിച്ച് ഉപയോഗം എണ്ണുമ്പോൾ ഉപയോഗിക്കുന്നു. `name@version` കോൺവെൻഷൻ ഉപയോഗിക്കുക.

## പടി 3: ഇൻഫ്രാസ്ട്രക്ചർ പാരാമറ്ററൈസ് ചെയ്യുക

പുനരുപയോഗയോഗ്യമായ ഒരു ടെംപ്ലേറ്റ് ഉണ്ടാക്കാൻ ഏറ്റവും പ്രധാനമായ ഒരു നിയമം: **നാമങ്ങൾ, പ്രദേശങ്ങൾ, അല്ലെങ്കിൽ സബ്‌സ്‌ക്രിപ്ഷൻ-നിർഭരമായ മൂല്യങ്ങൾ നിശ്ചിതമായി ഹാർഡ്കോഡ് ചെയ്യരുത്.** പാരാമീറ്ററുകളും റിസോഴ്‌സ് ടോക്കൺ പാറ്റേണും ഉപയോഗിച്ച് ഒക്കെ മാറ്റുക ताकि ഒരു ടെംപ്ലേറ്റ് ഏവരുടെയും സബ്‌സ്‌ക്രിപ്ഷനിലും പ്രവർത്തിക്കൂ.

```bicep
// infra/main.bicep
targetScope = 'resourceGroup'

@minLength(1)
@description('Primary location for all resources')
param location string = resourceGroup().location

@minLength(1)
@maxLength(64)
@description('Name of the environment (used to generate resource names)')
param environmentName string

// Generate globally-unique, stable names from the environment + subscription
var resourceToken = uniqueString(subscription().id, environmentName, location)
var tags = { 'azd-env-name': environmentName }

module web 'modules/web.bicep' = {
  name: 'web'
  params: {
    name: 'app-${resourceToken}'
    location: location
    tags: tags
  }
}

output SERVICE_WEB_ENDPOINT_URL string = web.outputs.uri
```

ഇതെല്ലാ കാരണങ്ങൾ ഈ ടെംപ്ലേറ്റ്-ഫ്രണ്ട്ലിയാക്കുന്നു:

1. **`azd-env-name` ടാഗ്** — ഓരോ എൻവയോൺമെന്റിനും അനുയോജ്യമായി റിസോഴ്‌സുകൾ ട്രാക്കും, ക്ലീൻ അപ്പ് ചെയ്യുന്നതിന് azd ഇത് ഉപയോഗിക്കുന്നു.
2. **`uniqueString(...)` resource token** — സ്ഥിരതയുള്ള, ഗ്ലോബലി-അദ്വിതീയ സഫ്ഫിക്സ് ഉത്പാദിപ്പിക്കുന്നു, അതോടെ നാമങ്ങൾ പൊരുത്തപ്പെടാതെ ബഹിഷ്കരിക്കപ്പെടാവുന്നില്ല.

പരിസ്ഥിതിയിൽ നിന്ന് azd ചേർക്കുന്ന മൂല്യങ്ങൾ വായിക്കുന്ന പൊരുത്തമുള്ള ഒരു പാരാമീറ്ററുകൾ ഫയൽ നൽകുക:

```json
// infra/main.parameters.json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    "environmentName": { "value": "${AZURE_ENV_NAME}" },
    "location": { "value": "${AZURE_LOCATION}" }
  }
}
```

azd നിലവിലെ പരിസ്ഥിതിയിൽ നിന്ന് `${AZURE_ENV_NAME}` and `${AZURE_LOCATION}` സ്വയമേവ സബ്സ്റ്റിറ്റ്യൂട്ട് ചെയ്യുന്നു.

## പടി 4: ലോക്കലായി നിങ്ങളുടെ ടെംപ്ലേറ്റ് പരീക്ഷിക്കുക

പങ്കുവയ്‌ക്കുന്നതിനു മുമ്പ്, ശൂന്യനിലയിൽ നിന്ന് ടെംപ്ലേറ്റ് പ്രവർത്തിക്കുന്നുണ്ടെന്ന് തെളിയിക്കുക.

**ലോക്കൽ ഫോൾഡറിൽ നിന്ന് പരീക്ഷിക്കുക** (Git push ആവശ്യമാണ് അല്ല):

```bash
# ശൂന്യമായ ഡയറക്ടറിയിൽ നിന്ന്, നിങ്ങളുടെ ലോക്കൽ ടെംപ്ലേറ്റ് പാത ഉപയോഗിച്ച് ആരംഭിക്കുക
mkdir /tmp/test-run && cd /tmp/test-run
azd init --template /path/to/my-azd-template

# ആരംഭം മുതൽ അവസാനം വരെ പ്രൊവിഷൻ ചെയ്ത് വിന്യസിക്കുക
azd auth login
azd up
```

**അടുത്തে teardown പരീക്ഷിക്കുക**—ഒരു നല്ല ടെംപ്ലേറ്റ് പൂർണ്ണമായി ക്ലീൻ ചെയ്യണം:

```bash
azd down --force --purge
```

`azd down` റിസോഴ്‌സുകൾ വിട്ട് പോകുകയാണെങ്കിൽ, നിങ്ങൾ ഏതെങ്കിലും റിസോഴ്‌സിൽ `azd-env-name` ടാഗ് ഒന്നുകൂടി ചേർക്കാൻ മറന്നിരിക്കാം.

> **ഉപദേശം:** ആദ്യം `azd provision --preview` റൺ ചെയ്യുക. ഇത് ഒരു what-if നടത്തുകയും ടെംപ്ലേറ്റിന്റെ എററുകൾ റിസോഴ്‌സ് സൃഷ്ടിക്കാതെ മുൻപ് ചൂണ്ടിക്കാണിക്കുകയും ചെയ്യും.

## പടി 5: ടെംപ്ലേറ്റ് പ്രസിദ്ധീകരിക്കുക

റെപ്പോസിറ്ററി GitHub-ലേക്ക് പുഷ് ചെയ്യുക (മറ്റ് ആളുകൾ ഉപയോഗിക്കാൻ പൊതു ആക്കുകയാണെങ്കിൽ public):

```bash
gh repo create my-azd-template --public --source=. --push
```

ഏവർവരും ഇപ്പോൾ ഇത് ഉപയോഗിക്കാം:

```bash
azd init --template your-github-username/my-azd-template
```

## പടി 6 (ഐച്ഛികം): Awesome AZD-ലേക്ക് സമർപ്പിക്കുക

[Awesome AZD ഗ്യാലറി](https://azure.github.io/awesome-azd/) കമ്മ്യൂണിറ്റി ടേംപ്ലേറ്റുകൾ ലിസ്റ്റ് ചെയ്യുന്നു. ലിസ്റ്റിൽപ്പെടാൻ നിങ്ങളുടെ റെപ്പോയിൽ താഴെവരുന്നവ അടങ്ങിയിരിക്കണം:

- ✅ prerequisites, ഒരു ആർക്കിടെക്ചർ ഡയാഗ്രാം, ചിലവ് കുറിച്ചുള്ള കുറിപ്പുകൾ എന്നിവയുള്ള വ്യക്തമായ `README.md`
- ✅ `metadata.template` ഉള്ള പ്രവർത്തനക്ഷമമായ `azure.yaml`
- ✅ പുതിയ സബ്‌സ്‌ക്രിപ്ഷനിൽ ശുദ്ധമായി പ്രോവിഷൻ ചെയ്യുന്ന ഇൻഫ്രാസ്ട്രക്ചർ
- ✅ ഒരു `LICENSE` ഫയൽ
- ✅ (ശുപാർശ ചെയ്യപ്പെടുന്നു) ഒറ്റ ക്ലിക്കിൽ Codespaces സജ്ജമാക്കാൻ `.devcontainer/`

ഗ്യാലറിയുടെ ഡാറ്റാ ഫയലിൽ നിങ്ങളുടെ ടെംപ്ലേറ്റ് ചേർക്കുന്ന ഒരു_PULL_REQUEST തുറന്ന് സമർപ്പിക്കൂ, [Awesome AZD repository](https://github.com/Azure/awesome-azd) ലെ സംഭാവന മാർഗ്ഗനിർദ്ദേശം പാലിച്ചുകൊണ്ട്.

---

## സാധാരണ പിഴച്ച കോൺദർശനങ്ങൾ

| പിഴവ് | പരിഹാരം |
|---------|-----|
| ഹാർഡ്കോഡ് ചെയ്ത റിസോഴ്‌സ് നാമങ്ങൾ | `uniqueString()` resource token ഉപയോഗിക്കുക |
| `azd down` റിസോഴ്‌സുകൾ അവശേഷിക്കുന്നു | എല്ലാ റിസോഴ്‌സിനും (അഥവാ റിസോഴ്‌സ് ഗ്രൂപ്പിനും) `azd-env-name` ടാഗ് ചേർക്കുക |
| ടെംപ്ലേറ്റ് നിങ്ങളുടെ പ്രക്രിയയിൽ ജോലി ചെയ്യുന്നു, മറ്റുള്ളവർക്കുള്ളത്ത് പരാജയപ്പെടുന്നു | സബ്‌സ്‌ക്രिप്ഷൻ IDകൾ, ടെനന്റ് IDകൾ, പ്രദേശ സമ്ബന്ധമായ നിലനില്പ് വ്യവഹാരങ്ങൾ എന്നിവ നീക്കംചെയ്യുക |
| ഔട്ട്പുട്ടുകൾ അപ്ലിക്കേഷനിലേക്ക് തിരിഞ്ഞിട്ടില്ല | `SERVICE_<NAME>_ENDPOINT_URL` എന്നിവയും മറ്റ് `AZURE_*` ഔട്ട്പുട്ടുകളുംExport ചെയ്യുക |
| ഗ്യാലറി സമർപ്പണം തള്ളപ്പെട്ടു | `README.md`, `LICENSE`, மற்றும் `metadata.template` ചേർക്കുക |

---

## സർവസംഹാരം

- ഒരു ടെംപ്ലേറ്റ് എന്നത് `azure.yaml`, `infra/`, നിങ്ങളുടെ കോഡ് എന്നിവയുള്ള ഒരു Git റെപ്പോ മാത്രമാണ്.
- എല്ലാം പാരാമറ്ററൈസ് ചെയ്യുക — നാമങ്ങൾ, പ്രദേശങ്ങൾ, IDകൾ — അതിലൂടെ അത് എവിടെയും ഓടും.
- എല്ലായ്പ്പോഴും റിസോഴ്‌സുകൾ `azd-env-name` ഉപയോഗിച്ച് ടാഗ് ചെയ്യുക, അതോടെ `azd down` പ്രവർത്തിക്കും.
- പ്രസിദ്ധപ്പെടുത്തുന്നതിന് മുൻപ് `azd init --template <local-path>` ഉപയോഗിച്ച് ലോക്കലായി പരിശോധിക്കുക.
- ഗ്യാലറിയിലേക്ക് സമർപ്പിക്കാൻ മെറ്റാഡേറ്റയും README ചേർക്കുക.
