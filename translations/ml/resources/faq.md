## परिचय

यह व्यापक FAQ Azure Developer CLI (azd) और Azure तैनातियों से संबंधित सबसे सामान्य प्रश्नों के उत्तर प्रदान करता है। सामान्य समस्याओं के त्वरित समाधान खोजें, सर्वोत्तम प्रथाओं को समझें, और azd अवधारणाओं और वर्कफ़्लो पर स्पष्टीकरण प्राप्त करें।

## सीखने के लक्ष्य

इस FAQ की समीक्षा करके, आप:
- सामान्य Azure Developer CLI प्रश्नों और समस्याओं के त्वरित उत्तर पाएंगे
- व्यावहारिक प्रश्नोत्तर प्रारूप के माध्यम से प्रमुख अवधारणाओं और शब्दावली को समझेंगे
- बार-बार आने वाली समस्याओं और त्रुटि परिदृश्यों के लिए समस्या-निवारण समाधान तक पहुँचेंगे
- अनुकूलन के बारे में अक्सर पूछे जाने वाले प्रश्नों के माध्यम से सर्वोत्तम प्रथाएं सीखेंगे
- विशेषज्ञ-स्तर के प्रश्नों के माध्यम से उन्नत विशेषताएँ और क्षमताएं खोजेंगे
- लागत, सुरक्षा, और तैनाती रणनीति मार्गदर्शन को प्रभावी रूप से संदर्भित करेंगे

- प्रमाणीकरण, तैनाती, और कॉन्फ़िगरेशन समस्याओं का प्रभावी ढंग से निवारण करें

## शुरुआत

### प्रश्न: Azure Developer CLI (azd) क्या है?
**उत्तर**: Azure Developer CLI (azd) एक डेवलपर-केंद्रित कमांड-लाइन उपकरण है जो आपकी एप्लिकेशन को स्थानीय विकास पर्यावरण से Azure तक पहुँचाने का समय तेज करता है। यह टेम्प्लेट के माध्यम से सर्वोत्तम प्रथाएं प्रदान करता है और पूरी तैनाती जीवनचक्र में मदद करता है।

### प्रश्न: azd Azure CLI से कैसे भिन्न है?
**उत्तर**: 
- **Azure CLI**: Azure संसाधनों का प्रबंधन करने के लिए सामान्य प्रयोजन उपकरण
- **azd**: एप्लिकेशन तैनाती वर्कफ़्लो के लिए डेवलपर-केंद्रित उपकरण
- azd आंतरिक रूप से Azure CLI का उपयोग करता है लेकिन सामान्य विकास परिदृश्यों के लिए उच्च-स्तरीय अमूर्तता प्रदान करता है
- azd टेम्प्लेट, पर्यावरण प्रबंधन, और तैनाती स्वचालन शामिल करता है

### प्रश्न: क्या मुझे azd का उपयोग करने के लिए Azure CLI स्थापित करना होगा?
**उत्तर**: हाँ, azd प्रमाणीकरण और कुछ संचालन के लिए Azure CLI की आवश्यकता होती है। पहले Azure CLI स्थापित करें, फिर azd स्थापित करें।

### प्रश्न: azd कौन-कौनसी प्रोग्रामिंग भाषाओं का समर्थन करता है?
**उत्तर**: azd भाषा-स्वतंत्र है। यह निम्न के साथ काम करता है:
- Node.js/JavaScript/TypeScript
- Python
- .NET/C#
- Java
- Go
- PHP
- स्थिर वेबसाइटें
- कंटेनरीकृत एप्लिकेशन

### प्रश्न: क्या मैं वर्तमान परियोजनाओं के साथ azd का उपयोग कर सकता हूँ?
**उत्तर**: हाँ! आप या तो:
1. `azd init` का उपयोग करके मौजूदा परियोजनाओं में azd कॉन्फ़िगरेशन जोड़ सकते हैं
2. मौजूदा परियोजनाओं को azd टेम्प्लेट संरचना के अनुसार अनुकूलित कर सकते हैं
3. अपने मौजूदा वास्तुकला के आधार पर कस्टम टेम्प्लेट बना सकते हैं

## प्रमाणीकरण और पहुँच

### प्रश्न: मैं azd का उपयोग करके Azure के साथ कैसे प्रमाणीकरण करूँ?
**उत्तर**: `azd auth login` का उपयोग करें जो Azure प्रमाणीकरण के लिए एक ब्राउज़र विंडो खोलेगा। CI/CD परिदृश्यों के लिए, सेवा प्रधान या प्रबंधित पहचान का उपयोग करें।

### प्रश्न: क्या मैं azd को कई Azure सदस्यताओं के साथ उपयोग कर सकता हूँ?
**उत्तर**: हाँ। प्रत्येक पर्यावरण के लिए उपयोग करने वाली सदस्यता निर्दिष्ट करने के लिए `azd env set AZURE_SUBSCRIPTION_ID <subscription-id>` का उपयोग करें।

### प्रश्न: azd के साथ तैनाती के लिए मुझे कौन-कौन सी अनुमतियाँ आवश्यक हैं?
**उत्तर**: आमतौर पर आपको चाहिए:
- संसाधन समूह या सदस्यता पर **Contributer** भूमिका
- यदि संसाधनों के लिए भूमिका असाइनमेंट आवश्यक है तो **User Access Administrator**
- विशिष्ट अनुमतियाँ टेम्प्लेट और तैनात किए जाने वाले संसाधनों के अनुसार भिन्न हो सकती हैं

### प्रश्न: क्या मैं azd को CI/CD पाइपलाइनों में उपयोग कर सकता हूँ?
**उत्तर**: बिलकुल! azd CI/CD एकीकरण के लिए डिज़ाइन किया गया है। प्रमाणीकरण के लिए सेवा प्रधान का उपयोग करें और कॉन्फ़िगरेशन के लिए पर्यावरण चर सेट करें।

### प्रश्न: GitHub Actions में प्रमाणीकरण कैसे संभालूँ?
**उत्तर**: सेवा प्रधान प्रमाण पत्र के साथ Azure Login कार्रवाई का उपयोग करें:
```yaml
- uses: azure/login@v1
  with:
    creds: ${{ secrets.AZURE_CREDENTIALS }}
- run: azd deploy --no-prompt
```

## टेम्पलेट और परियोजनाएँ

### प्रश्न: मैं azd टेम्प्लेट कहाँ पा सकता हूँ?
**उत्तर**: 
- आधिकारिक टेम्प्लेट: [Azure-Samples/awesome-azd](https://github.com/Azure-Samples/awesome-azd)
- समुदाय टेम्प्लेट: GitHub पर "azd-template" खोजें
- उपलब्ध टेम्प्लेट ब्राउज़ करने के लिए `azd template list` का उपयोग करें

### प्रश्न: मैं कस्टम टेम्प्लेट कैसे बनाऊँ?
**उत्तर**: 
1. मौजूदा टेम्प्लेट संरचना से शुरुआत करें
2. `azure.yaml`, अवसंरचना फ़ाइलें, और एप्लिकेशन कोड संशोधित करें
3. `azd up` के साथ पूरी तरह परीक्षण करें
4. उपयुक्त टैग के साथ GitHub पर प्रकाशित करें

### प्रश्न: क्या मैं टेम्पलेट के बिना azd का उपयोग कर सकता हूँ?
**उत्तर**: हाँ, आवश्यक कॉन्फ़िगरेशन फ़ाइलें बनाने के लिए मौजूदा परियोजना में `azd init` का उपयोग करें। आपको मैन्युअल रूप से `azure.yaml` और अवसंरचना फ़ाइलों को कॉन्फ़िगर करना होगा।

### प्रश्न: आधिकारिक और समुदाय टेम्प्लेट में क्या अंतर है?
**उत्तर**: 
- **आधिकारिक टेम्प्लेट**: माइक्रोसॉफ्ट द्वारा रखरखाव, नियमित अपडेट, व्यापक दस्तावेज़ीकरण
- **समुदाय टेम्प्लेट**: डेवलपर्स द्वारा बनाए गए, विशिष्ट उपयोग मामलों के लिए हो सकते हैं, गुणवत्ता और रखरखाव में अंतर

### प्रश्न: मेरी परियोजना में टेम्प्लेट को कैसे अपडेट करें?
**उत्तर**: टेम्प्लेट स्वचालित रूप से अपडेट नहीं होते। आप:
1. स्रोत टेम्प्लेट से परिवर्तनों की मैन्युअल तुलना और विलय कर सकते हैं
2. अद्यतित टेम्प्लेट का उपयोग करके `azd init` से नई शुरुआत कर सकते हैं
3. अद्यतित टेम्प्लेट से विशिष्ट सुधार चुन सकते हैं

## तैनाती और अवसंरचना

### प्रश्न: azd किनAzure सेवाओं को तैनात कर सकता है?
**उत्तर**: azd Bicep/ARM टेम्प्लेट के माध्यम से किसी भी Azure सेवा को तैनात कर सकता है, जिसमें शामिल हैं:
- App Services, Container Apps, Functions
- डेटाबेस (SQL, PostgreSQL, Cosmos DB)
- स्टोरेज, Key Vault, Application Insights
- नेटवर्किंग, सुरक्षा, और निगरानी संसाधन

### प्रश्न: क्या मैं कई क्षेत्रों में तैनाती कर सकता हूँ?
**उत्तर**: हाँ, अपने Bicep टेम्प्लेट में कई क्षेत्रों को कॉन्फ़िगर करें और प्रत्येक पर्यावरण के लिए स्थान पैरामीटर ठीक से सेट करें।

### प्रश्न: मैं डेटाबेस स्कीमा माइग्रेशन कैसे संभालूँ?
**उत्तर**: `azure.yaml` में तैनाती हुक का उपयोग करें:
```yaml
hooks:
  postdeploy:
    posix:
      run: ./scripts/migrate-database.sh
    windows:
      run: ./scripts/migrate-database.ps1
```

### प्रश्न: क्या मैं केवल अवसंरचना बिना एप्लिकेशन के तैनात कर सकता हूँ?
**उत्तर**: हाँ, केवल आपके टेम्प्लेट में परिभाषित अवसंरचना घटकों को तैनात करने के लिए `azd provision` का उपयोग करें।

### प्रश्न: मैं मौजूदा Azure संसाधनों पर तैनाती कैसे करूँ?
**उत्तर**: यह जटिल है और सीधे समर्थित नहीं है। आप:
1. मौजूदा संसाधनों को अपने Bicep टेम्प्लेट में आयात कर सकते हैं
2. टेम्प्लेट में मौजूदा संसाधन संदर्भों का उपयोग कर सकते हैं
3. टेम्प्लेट को संसाधनों को सशर्त रूप से बनाने या संदर्भित करने के लिए संशोधित कर सकते हैं

### प्रश्न: क्या मैं Bicep के बजाय Terraform का उपयोग कर सकता हूँ?
**उत्तर**: वर्तमान में, azd मुख्य रूप से Bicep/ARM टेम्प्लेट का समर्थन करता है। Terraform समर्थन आधिकारिक रूप से उपलब्ध नहीं है, हालांकि समुदाय के हल हो सकते हैं।

## कॉन्फ़िगरेशन और पर्यावरण

### प्रश्न: मैं विभिन्न पर्यावरण (dev, staging, prod) का प्रबंधन कैसे करूँ?
**उत्तर**: `azd env new <environment-name>` के साथ अलग-अलग पर्यावरण बनाएं और प्रत्येक के लिए विभिन्न सेटिंग्स कॉन्फ़िगर करें:
```bash
azd env new development
azd env new staging  
azd env new production
```

### प्रश्न: पर्यावरण कॉन्फ़िगरेशन कहाँ संगृहीत होती हैं?
**उत्तर**: आपकी परियोजना निर्देशिका में `.azure` फ़ोल्डर में। प्रत्येक पर्यावरण की अपनी फ़ोल्डर होती है जिसमें कॉन्फ़िगरेशन फ़ाइलें होती हैं।

### प्रश्न: मैं पर्यावरण-विशिष्ट कॉन्फ़िगरेशन कैसे सेट करूँ?
**उत्तर**: पर्यावरण चर कॉन्फ़िगर करने के लिए `azd env set` का उपयोग करें:
```bash
azd env set AZURE_LOCATION eastus
azd env set DATABASE_TIER Basic
```

### प्रश्न: क्या मैं टीम सदस्यों के बीच पर्यावरण कॉन्फ़िगरेशन साझा कर सकता हूँ?
**उत्तर**: `.azure` फ़ोल्डर संवेदनशील जानकारी रखता है और इसे संस्करण नियंत्रण में कमिट नहीं किया जाना चाहिए। इसके बजाय:
1. आवश्यक पर्यावरण चर दस्तावेज़ित करें
2. पर्यावरण स्थापित करने के लिए तैनाती स्क्रिप्ट का उपयोग करें
3. संवेदनशील कॉन्फ़िगरेशन के लिए Azure Key Vault का उपयोग करें

### प्रश्न: मैं टेम्पलेट डिफ़ॉल्ट्स को कैसे ओवरराइड करूँ?
**उत्तर**: टेम्पलेट पैरामीटर से मेल खाने वाले पर्यावरण चर सेट करें:
```bash
azd env set LOCATION "West US 2"
azd env set SKU_NAME "B1"
```

## समस्या निवारण

### प्रश्न: `azd up` विफल क्यों हो रहा है?
**उत्तर**: सामान्य कारण:
1. **प्रमाणीकरण समस्याएँ**: रन करें `azd auth login`
2. **अनुचित अनुमतियाँ**: अपने Azure भूमिका असाइनमेंट देखें
3. **संसाधन नामकरण संघर्ष**: AZURE_ENV_NAME बदलें
4. **कोटा/क्षमता समस्याएँ**: क्षेत्रीय उपलब्धता जांचें
5. **टेम्प्लेट त्रुटियाँ**: Bicep टेम्प्लेट मान्य करें

### प्रश्न: मैं तैनाती विफलताओं का डिबग कैसे करूँ?
**उत्तर**: 
1. विस्तृत आउटपुट के लिए `azd deploy --debug` का उपयोग करें
2. Azure पोर्टल तैनाती इतिहास देखें
3. Azure पोर्टल में एक्टिविटी लॉग देखें
4. वर्तमान पर्यावरण स्थिति दिखाने के लिए `azd show` का उपयोग करें

### प्रश्न: मेरे पर्यावरण चर काम क्यों नहीं कर रहे हैं?
**उत्तर**: जांचें:
1. चर के नाम टेम्प्लेट पैरामीटर से बिल्कुल मेल खाते हों
2. मान ठीक से उद्धृत हों यदि उनमें स्थान हो
3. पर्यावरण चुना गया हो: `azd env select <environment>`
4. चर सही पर्यावरण में सेट किए गए हों

### प्रश्न: मैं असफल तैनाती कैसे साफ करूँ?
**उत्तर**: 
```bash
azd down --force --purge
```
यह सभी संसाधन और पर्यावरण कॉन्फ़िगरेशन को हटा देता है।

### प्रश्न: मेरी एप्लिकेशन तैनाती के बाद सुलभ क्यों नहीं है?
**उत्तर**: जांचें:
1. तैनाती सफलतापूर्वक पूरी हुई
2. एप्लिकेशन चल रही है (Azure पोर्टल में लॉग देखें)
3. नेटवर्क सुरक्षा समूह ट्रैफ़िक की अनुमति देता है
4. DNS/कस्टम डोमेन सही ढंग से कॉन्फ़िगर किए गए हैं

## लागत और बिलिंग

### प्रश्न: azd तैनातियों की लागत कितनी होगी?
**उत्तर**: लागत इस पर निर्भर करती है:
- तैनात Azure सेवाएं
- सेवा टीयर/एसकेयू चयन
- क्षेत्रीय मूल्य भिन्नताएं
- उपयोग पैटर्न

[Azure Pricing Calculator](https://azure.microsoft.com/pricing/calculator/) का उपयोग करें अनुमान के लिए।

### प्रश्न: मैं azd तैनातियों में लागत कैसे नियंत्रित करूँ?
**उत्तर**: 
1. विकास पर्यावरण के लिए निचले स्तर के SKU का उपयोग करें
2. Azure बजट और अलर्ट सेट करें
3. जब आवश्यक न हो तो संसाधन हटाने के लिए `azd down` का उपयोग करें
4. उपयुक्त क्षेत्र चुनें (लागत स्थान के अनुसार अलग-अलग होती है)
5. Azure Cost Management उपकरणों का उपयोग करें

### प्रश्न: क्या azd टेम्प्लेट के लिए मुफ्त स्तर विकल्प हैं?
**उत्तर**: कई Azure सेवाएं मुफ्त स्तर प्रदान करती हैं:
- App Service: मुफ्त स्तर उपलब्ध
- Azure Functions: 1M मुफ्त निष्पादन/महीना
- Cosmos DB: 400 RU/s के साथ मुफ्त स्तर
- Application Insights: पहली 5GB/महीना मुफ्त

जहाँ संभव हो टेम्प्लेट को मुफ्त स्तर का उपयोग करने के लिए कॉन्फ़िगर करें।

### प्रश्न: मैं तैनाती से पहले लागत का अनुमान कैसे लगाऊँ?
**उत्तर**: 
1. देखें कि टेम्प्लेट के `main.bicep` में कौन से संसाधन बनाए जाते हैं
2. विशिष्ट SKU के साथ Azure Pricing Calculator का उपयोग करें
3. वास्तविक लागत की निगरानी के लिए पहले विकास पर्यावरण में तैनात करें
4. विस्तृत लागत विश्लेषण के लिए Azure Cost Management का उपयोग करें

## सर्वोत्तम प्रथाएँ

### प्रश्न: azd परियोजना संरचना के लिए सर्वोत्तम प्रथाएँ क्या हैं?
**उत्तर**: 
1. एप्लिकेशन कोड को अवसंरचना से अलग रखें
2. `azure.yaml` में अर्थपूर्ण सेवा नामों का उपयोग करें
3. बिल्ड स्क्रिप्ट में उचित त्रुटि हैंडलिंग लागू करें
4. पर्यावरण-विशिष्ट कॉन्फ़िगरेशन का उपयोग करें
5. व्यापक दस्तावेज़ शामिल करें

### प्रश्न: मुझे azd में कई सेवाओं का संगठन कैसे करना चाहिए?
**उत्तर**: अनुशंसित संरचना का उपयोग करें:
```
├── src/
│   ├── web/          # Frontend service
│   ├── api/          # Backend service  
│   └── worker/       # Background service
├── infra/            # Infrastructure templates
├── azure.yaml        # azd configuration
└── README.md         # Project documentation
```

### प्रश्न: क्या मुझे `.azure` फ़ोल्डर संस्करण नियंत्रण में कमिट करना चाहिए?
**उत्तर**: **नहीं!** `.azure` फ़ोल्डर में संवेदनशील जानकारी होती है। इसे `.gitignore` में जोड़ें:
```gitignore
.azure/
```

### प्रश्न: मैं रहस्यों और संवेदनशील कॉन्फ़िगरेशन को कैसे संभालूँ?
**उत्तर**: 
1. रहस्यों के लिए Azure Key Vault का उपयोग करें
2. एप्लिकेशन कॉन्फ़िगरेशन में Key Vault रहस्यों का संदर्भ लें
3. कभी भी रहस्यों को संस्करण नियंत्रण में कमिट न करें
4. सेवा-से-सेवा प्रमाणीकरण के लिए प्रबंधित पहचान का उपयोग करें

### प्रश्न: azd के साथ CI/CD के लिए अनुशंसित दृष्टिकोण क्या है?
**उत्तर**: 
1. प्रत्येक चरण (dev/staging/prod) के लिए अलग-अलग पर्यावरण का उपयोग करें
2. तैनाती से पहले स्वचालित परीक्षण लागू करें
3. प्रमाणीकरण के लिए सेवा प्रधानों का उपयोग करें
4. पाइपलाइन रहस्यों/चर में संवेदनशील कॉन्फ़िगरेशन संग्रहीत करें
5. उत्पादन तैनाती के लिए अनुमोदन गेट लागू करें

## उन्नत विषय

### प्रश्न: क्या मैं कस्टम कार्यक्षमता के साथ azd का विस्तार कर सकता हूँ?
**उत्तर**: हाँ, `azure.yaml` में तैनाती हुक के माध्यम से:
```yaml
hooks:
  predeploy:
    run: ./scripts/custom-setup.sh
  postdeploy:
    run: ./scripts/custom-config.sh
```

### प्रश्न: मैं मौजूदा DevOps प्रक्रियाओं के साथ azd को कैसे एकीकृत करूँ?
**उत्तर**: 
1. मौजूदा पाइपलाइन स्क्रिप्ट में azd आदेशों का उपयोग करें
2. टीमों में azd टेम्प्लेट मानकीकृत करें
3. मौजूदा निगरानी और अलर्टिंग के साथ एकीकृत करें
4. पाइपलाइन एकीकरण के लिए azd के JSON आउटपुट का उपयोग करें

### प्रश्न: क्या मैं azd को Azure DevOps के साथ उपयोग कर सकता हूँ?
**उत्तर**: हाँ, azd किसी भी CI/CD प्रणाली के साथ काम करता है। azd आदेशों का उपयोग करने वाले Azure DevOps पाइपलाइनों का निर्माण करें।

### प्रश्न: मैं azd में योगदान कैसे दे सकता हूँ या समुदाय टेम्प्लेट कैसे बना सकता हूँ?
**उत्तर**:
1. **azd tool**: [Azure/azure-dev](https://github.com/Azure/azure-dev) ൽ സംഭാവന ചെയ്യുക  
2. **ടെംപ്ലേറ്റുകള്‍**: [template guidelines](https://github.com/Azure-Samples/awesome-azd) അനുസരിച്ചു ടെംപ്ലേറ്റുകള്‍ സൃഷ്ടിക്കുക  
3. **ഡോക്യൂമെൻറേഷൻ**: [MicrosoftDocs/azure-dev-docs](https://github.com/MicrosoftDocs/azure-dev-docs) ൽ ഡോക്യുമെന്റുകള്‍ക്ക് സംഭാവന ചെയ്യുക

### Q: മറ്റു ഡിസ്ട്രിബ്യൂഷൻ ടൂളുകളിൽ നിന്നും azd ലേക്ക് എങ്ങനെ മാറാം?  
**A**:

3. നിലവിലെ സർവീസുകൾക്ക് പൊരുത്തപ്പെടാൻ `azure.yaml` കോൺഫിഗർ ചെയ്യുക  
4. ഡെവലപ്പ്മെന്റ് გარემത്തിൽ സമഗ്രമായി പരിശോധന നടത്തുക
