# إحضار تطبيقك الخاص - إضافة azd إلى مشروع موجود

**تنقل الفصل:**
- **📚 الصفحة الرئيسية للدورة**: [AZD للمبتدئين](../../README.md)
- **📖 الفصل الحالي**: الفصل 1 - الأساسيات والبداية السريعة
- **⬅️ السابق**: [مشروعك الأول](first-project.md)
- **➡️ التالي**: [حاويات التطوير ومساحات الأكواد](dev-containers.md)

> تم التحقق باستخدام `azd 1.27.1` في يوليو 2026.

## مقدمة

في [مشروعك الأول](first-project.md) قمت بنشر تطبيق بدءًا من قالب. لكن في معظم الأحيان لديك بالفعل تطبيق—واجهة برمجة تطبيقات Node.js، خدمة Python Flask، تطبيق ويب .NET—موجود في مجلد على جهازك. توضح هذه الدرسة كيفية إضافة azd إلى هذا الكود الموجود حتى تتمكن من نشره باستخدام `azd up`، دون الحاجة إلى قالب.

## أهداف التعلم

بنهاية هذا الدرس، ستتمكن من:
- فهم الطرق الثلاث لبدء مشروع azd
- تشغيل `azd init` داخل قاعدة كود موجودة
- فهم ماذا يفعل `azure.yaml` ومجلد `infra/` لتطبيقك
- معرفة متى تترك azd لتوليد البنية التحتية مقابل كتابة الخاص بك
- نشر تطبيقك الموجود على Azure باستخدام `azd up`

## نتائج التعلم

بعد إكمال هذا الدرس، ستكون قادرًا على:
- تهيئة azd في مشروع تمتلكه بالفعل
- قراءة وتحرير ملف `azure.yaml` أساسي
- توليد بنية تحتية ابتدائية باستخدام `azd infra generate`
- اختيار مضيف Azure المناسب لتطبيقك
- نشر وتنظيف تطبيقك الخاص

## ثلاث طرق لبدء مشروع azd

| نقطة البداية | الأمر | متى تستخدمه |
|----------------|---------|-------------|
| **من قالب** | `azd init --template <name>` | التعلم، أو بدء تطبيق جديد من نموذج مثبت |
| **من الكود الموجود لديك** | `azd init` (في مجلد مشروعك) | لديك تطبيق وتريد نشره |
| **من مستودع Git** | `azd init --from-code` (في مستودع منسوخ) | اعتماد azd لمستودع موجود |

لقد مارست الخيار الأول بالفعل. يغطي هذا الدرس الخيار الثاني—وهو السيناريو الأكثر شيوعًا في العالم الحقيقي.

## الخطوة 1: تشغيل `azd init` في مشروعك

افتح طرفية **داخل مجلد مشروعك الموجود** وشغّل:

```bash
cd my-existing-app
azd init
```

سيطلب منك azd كيفية التهيئة. اختر:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

اختر **"استخدام الكود في الدليل الحالي."** ثم يعمل azd على مسح مجلدك، يكتشف لغتك والإطار المستخدم، ويقترح مضيفاً.

### ما يكتشفه azd

يبحث azd عن إشارات مثل `package.json`، `requirements.txt`، `pom.xml`، `*.csproj`، أو `Dockerfile`، ويقترح مضيف Azure مطابق:

| تطبيقك | المضيف المحتمل المكتشف |
|----------|----------------------|
| تطبيق Node.js / Python / .NET ويب | خدمة تطبيقات Azure أو تطبيقات الحاوية |
| تطبيق محوّل لحاوية (`Dockerfile`) | تطبيقات حاوية Azure |
| تطبيق الدالة | وظائف Azure |
| موقع ثابت (مخرجات بناء React/Vue) | تطبيقات ويب Azure الثابتة |

أكد الخدمة (الخدمات) المكتشفة، ثم ينشئ azd الهيكلية والملفات التي تحتاجها.

## الخطوة 2: فهم ما أنشأه azd

بعد التهيئة، سيكون لديك شيئين جديدين في مشروعك:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — تعريف المشروع

هذا هو قلب مشروع azd. وشكله الأساسي كما يلي:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

كتلة `services` هي الجزء الرئيسي: كل مدخل يربط مجلد الكود الخاص بك بمضيف Azure. إذا كان لتطبيقك واجهة أمامية وواجهة برمجة تطبيقات، سيكون لديك خدمتان.

### `infra/` — موارد Azure الخاصة بك ككود

مجلد `infra/` يحتوي على ملفات Bicep التي تعرف موارد Azure التي يحتاجها تطبيقك (خدمة التطبيقات، قاعدة البيانات، إلخ). لست مضطرًا لكتابة هذه الملفات يدويًا — يقوم azd بإنشاء نقطة انطلاق جاهزة. يمكنك *تحريرها* لاحقًا لإضافة موارد أو تعزيز الأمان (مغطى في [الفصل 4](../chapter-04-infrastructure/README.md)).

> **نصيحة:** هل تريد رؤية البنية التحتية المولدة أو تخصيصها قبل النشر؟ شغّل `azd infra generate` (متوفر أيضًا كـ `azd infra synth`) لكتابة بنية البنية التحتية ككود على القرص لتتمكن من مراجعتها والتحكم في إصداراتها.

## الخطوة 3: تعيين التكوين المطلوب

إذا كان تطبيقك يحتاج لإعدادات أو أسرار (مثل سلسلة اتصال، مفتاح API)، لا تبرمجها مباشرة. استخدم القيم البيئية:

```bash
# إنشاء بيئة
azd env new dev

# تعيين قيمة غير سرية
azd env set API_VERSION 1.0.0
```

للأسرار الحقيقية، خزّنها في Key Vault وارجع إليها من بنيتك التحتية—انظر [الفصل 3: التكوين والمصادقة](../chapter-03-configuration/authsecurity.md).

## الخطوة 4: النشر

استخدم الآن سير العمل ذاته الذي تعرفه:

```bash
# المصادقة (مطلوبة لـ azd)
azd auth login

# معاينة الموارد التي سيتم إنشاؤها
azd provision --preview

# توفير البنية التحتية ونشر الكود الخاص بك
azd up
```

عند الانتهاء، يطبع azd عنوان URL لتطبيقك. تحقق منه بنفس طريقة أي تطبيق azd:

```bash
azd show           # عرض النقاط النهائية
azd monitor --logs # تحقق من السجلات إذا لزم الأمر
```

## مشكلات شائعة في المرة الأولى

| العرض | السبب المحتمل | الحل |
|---------|--------------|-----|
| azd لم يكتشف تطبيقي | نقص ملف التعريف (مثل `package.json`) | أضف الملف أو اختر المضيف يدويًا أثناء `azd init` |
| فشل البناء أثناء `azd up` | التطبيق يحتاج خطوة بناء | أضف `buildCommand`/`outputPath` تحت الخدمة في `azure.yaml` |
| التطبيق يبدأ لكنه يعيد أخطاء | نقص التكوين/السر | عيّن القيم باستخدام `azd env set` أو اربط Key Vault |
| تم اختيار مضيف خطأ | التخمين التلقائي | حرر `host:` في `azure.yaml` وأعد تشغيل `azd up` |

للمزيد، راجع [الفصل 7: استكشاف الأخطاء وإصلاحها](../chapter-07-troubleshooting/README.md).

## التنظيف

```bash
azd down --force --purge
```

---

## الملخص

- `azd init` → **"استخدام الكود في الدليل الحالي"** يضيف azd إلى تطبيق تمتلكه بالفعل.
- `azure.yaml` يربط مجلدات الكود بمضيفات Azure؛ `infra/` يعرّف الموارد باستخدام Bicep.
- `azd infra generate` يسمح لك بمراجعة أو تخصيص البنية التحتية المولدة.
- بمجرد التهيئة، يستخدم التطبيق الموجود نفس سير العمل الخاص بـ `azd up` / `azd down` كتطبيق قائم على قالب.
