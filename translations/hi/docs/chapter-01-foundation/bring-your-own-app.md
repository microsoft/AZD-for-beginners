# अपनी खुद की ऐप लेकर आएं - एक मौजूदा परियोजना में azd जोड़ें

**अध्याय नेविगेशन:**
- **📚 पाठ्यक्रम घर**: [शुरुआती के लिए AZD](../../README.md)
- **📖 वर्तमान अध्याय**: अध्याय 1 - आधार और त्वरित प्रारंभ
- **⬅️ पिछला**: [आपकी पहली परियोजना](first-project.md)
- **➡️ अगला**: [डेव कंटेनर और कोडस्पेसेस](dev-containers.md)

> जुलाई 2026 में `azd 1.27.1` के खिलाफ पुष्टि की गई।

## परिचय

[आपकी पहली परियोजना](first-project.md) में आपने एक टेम्पलेट से शुरू करके एक ऐप डिप्लॉय की थी। लेकिन ज्यादातर समय आपके पास पहले से ही एक ऐप होता है—Node.js API, Python Flask सेवा, .NET वेब ऐप—जो आपकी मशीन पर एक फ़ोल्डर में रखा होता है। यह पाठ दिखाता है कि मौजूदा कोड में azd कैसे जोड़ें ताकि आप इसे `azd up` के साथ तैनात कर सकें, बिना किसी टेम्पलेट की आवश्यकता के।

## सीखने के लक्ष्य

इस पाठ के अंत तक, आप:
- azd परियोजना शुरू करने के तीन तरीकों को समझेंगे
- मौजूदा कोडबेस के अंदर `azd init` चलाएंगे
- समझेंगे कि `azure.yaml` और `infra/` फ़ोल्डर आपके ऐप के लिए क्या करते हैं
- जानेंगे कि कब azd को इन्फ्रास्ट्रक्चर जनरेट करने दें और कब खुद लिखें
- अपने मौजूदा ऐप को `azd up` से Azure पर तैनात करेंगे

## सीखने के परिणाम

इस पाठ को पूरा करने के बाद, आप सक्षम होंगे:
- ऐसी परियोजना में azd इनिशियलाइज़ करना जो आपके पास पहले से है
- एक बुनियादी `azure.yaml` फ़ाइल पढ़ना और संपादित करना
- `azd infra generate` से स्टार्टिंग इन्फ्रास्ट्रक्चर जनरेट करना
- अपने ऐप के लिए उपयुक्त Azure होस्ट चुनना
- अपनी स्वयं की एप्लिकेशन को तैनात करना और साफ़ करना

---

## azd परियोजना शुरू करने के तीन तरीके

| शुरुआत बिंदु | आदेश | कब उपयोग करें |
|----------------|---------|-------------|
| **एक टेम्पलेट से** | `azd init --template <name>` | सीखने के लिए, या एक प्रमाणित नमूने से नया ऐप शुरू करने के लिए |
| **अपने मौजूदा कोड से** | `azd init` (अपने परियोजना फ़ोल्डर में) | आपके पास पहले से एक ऐप है और आप इसे तैनात करना चाहते हैं |
| **एक Git रिपॉजिटरी से** | `azd init --from-code` (क्लोन किए गए रिपॉजिटरी में) | मौजूदा रिपॉजिटरी के लिए azd अपनाने के लिए |

आपने पहले विकल्प का अभ्यास किया। यह पाठ दूसरे विकल्प को कवर करता है—सबसे सामान्य वास्तविक-विश्व परिदृश्य।

## चरण 1: अपनी परियोजना में `azd init` चलाएं

अपने मौजूदा परियोजना फ़ोल्डर **के अंदर एक टर्मिनल खोलें** और चलाएं:

```bash
cd my-existing-app
azd init
```

azd पूछेगा कि आप इसे कैसे इनिशियलाइज़ करना चाहते हैं। चुनें:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

चुनें **"वर्तमान निर्देशिका में कोड का उपयोग करें।"** इसके बाद azd आपका फ़ोल्डर स्कैन करता है, आपकी भाषा और फ्रेमवर्क का पता लगाता है, और एक होस्ट प्रस्तावित करता है।

### azd क्या पहचानता है

azd संकेतों की तलाश करता है जैसे `package.json`, `requirements.txt`, `pom.xml`, `*.csproj`, या `Dockerfile`, और एक मेल खाता Azure होस्ट सुझाता है:

| आपका ऐप | संभावित पहचाना गया होस्ट |
|----------|----------------------|
| Node.js / Python / .NET वेब ऐप | Azure App Service या Container Apps |
| कंटेनरयुक्त ऐप (`Dockerfile`) | Azure Container Apps |
| फ़ंक्शन ऐप | Azure Functions |
| स्थिर साइट (React/Vue बिल्ड आउटपुट) | Azure Static Web Apps |

पहचानी गई सेवा(ओं) की पुष्टि करें, और azd आवश्यक फ़ाइलें तैयार करता है।

## चरण 2: समझें कि azd ने क्या बनाया

इनिशियलाइज़ेशन के बाद, आपकी परियोजना में दो नई चीजें होंगी:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — परियोजना परिभाषा

यह azd परियोजना का हृदय है। एक न्यूनतम फ़ाइल इस प्रकार दिखती है:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

`services` ब्लॉक मुख्य हिस्सा है: प्रत्येक प्रविष्टि आपके कोड के एक फ़ोल्डर को Azure होस्ट से मैप करती है। यदि आपके ऐप में फ्रंटएंड और API दोनों हैं, तो आपके पास दो सेवाएं होंगी।

### `infra/` — आपके Azure संसाधन कोड के रूप में

`infra/` फ़ोल्डर में Bicep फ़ाइलें होती हैं जो आपके ऐप को आवश्यक Azure संसाधनों को परिभाषित करती हैं (App Service, डेटाबेस आदि)। आपको इन्हें हाथ से लिखने की जरूरत नहीं—azd एक कार्यशील प्रारंभिक बिंदु जनरेट करता है। आप बाद में इन्हें संपादित कर सकते हैं संसाधन जोड़ने या सुरक्षा कड़ी करने के लिए (जो [अध्याय 4](../chapter-04-infrastructure/README.md) में कवर किया गया है)।

> **सुझाव:** तैनाती से पहले जनरेट की गई इन्फ्रास्ट्रक्चर देखना या कस्टमाइज़ करना चाहते हैं? `azd infra generate` चलाएं (जो `azd infra synth` के रूप में भी उपलब्ध है) ताकि IaC डिस्क पर लिखा जाए और आप समीक्षा व संस्करण नियंत्रण कर सकें।

## चरण 3: आवश्यक कॉन्फ़िगरेशन सेट करें

यदि आपके ऐप को सेटिंग्स या रहस्य (जैसे कनेक्शन स्ट्रिंग, API कुंजी) की आवश्यकता है, तो उन्हें हार्डकोड न करें। पर्यावरण मानों का उपयोग करें:

```bash
# एक वातावरण बनाएं
azd env new dev

# एक गैर-गुप्त मान सेट करें
azd env set API_VERSION 1.0.0
```

असली रहस्यों के लिए, उन्हें Key Vault में संग्रहित करें और अपने इन्फ्रास्ट्रक्चर से संदर्भित करें—देखें [अध्याय 3: कॉन्फ़िगरेशन और प्रमाणीकरण](../chapter-03-configuration/authsecurity.md)।

---

## चरण 4: तैनात करें

अब वही कार्यप्रवाह उपयोग करें जिसे आप पहले से जानते हैं:

```bash
# प्रमाणित करें (azd के लिए आवश्यक)
azd auth login

# जिन संसाधनों का निर्माण होगा उनका पूर्वावलोकन करें
azd provision --preview

# आधारभूत संरचना तैयार करें और अपना कोड तैनात करें
azd up
```

जब यह समाप्त हो जाए, azd आपके ऐप का URL प्रिंट करता है। इसे किसी भी azd ऐप की तरह सत्यापित करें:

```bash
azd show           # एंडपॉइंट दिखाएँ
azd monitor --logs # आवश्यकता होने पर लॉग जांचें
```

## सामान्य पहली बार की समस्याएं

| लक्षण | संभावित कारण | सुधार |
|---------|--------------|-----|
| azd ने मेरा ऐप नहीं पहचाना | मैनिफेस्ट गायब है (जैसे `package.json`) | मैनिफेस्ट जोड़ें, या `azd init` के दौरान हैंड में होस्ट चुनें |
| `azd up` के दौरान बिल्ड फेल | ऐप को बिल्ड चरण की आवश्यकता है | सेवा के तहत `azure.yaml` में `buildCommand`/`outputPath` जोड़ें |
| ऐप शुरू होता है लेकिन त्रुटियाँ लौटाता है | कॉन्फिग/रहस्य गायब है | `azd env set` से मान सेट करें या Key Vault जोड़ें |
| गलत होस्ट चुना गया | स्वचालित पहचान गलत थी | `azure.yaml` में `host:` को संपादित करें और पुनः `azd up` चलाएं |

अधिक जानकारी के लिए देखें [अध्याय 7: समस्या निवारण](../chapter-07-troubleshooting/README.md)।

## साफ़ करें

```bash
azd down --force --purge
```

## सारांश

- `azd init` → **"वर्तमान निर्देशिका में कोड का उपयोग करें"** आपके पास पहले से मौजूद ऐप में azd जोड़ता है।
- `azure.yaml` आपके कोड फ़ोल्डरों को Azure होस्ट से मैप करता है; `infra/` संसाधनों को Bicep के रूप में परिभाषित करता है।
- `azd infra generate` आपको जनरेट की गई इन्फ्रास्ट्रक्चर की समीक्षा या कस्टमाइज़ेशन का मौका देता है।
- एक बार इनिशियलाइज़ हो जाने के बाद, आपका मौजूदा ऐप टेम्पलेट-आधारित ऐप की तरह ही `azd up` / `azd down` कार्यप्रवाह का उपयोग करता है।
