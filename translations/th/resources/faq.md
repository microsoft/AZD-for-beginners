## บทนำ

FAQ ฉบับสมบูรณ์นี้ให้คำตอบสำหรับคำถามที่พบบ่อยเกี่ยวกับ Azure Developer CLI (azd) และการปรับใช้ Azure ค้นหาวิธีแก้ไขปัญหาอย่างรวดเร็วสำหรับปัญหาทั่วไป เข้าใจแนวทางปฏิบัติที่ดีที่สุด และรับความชัดเจนเกี่ยวกับแนวคิดและกระบวนการทำงานของ azd

## เป้าหมายการเรียนรู้

โดยการทบทวน FAQ นี้ คุณจะ:
- พบคำตอบอย่างรวดเร็วสำหรับคำถามและปัญหาเกี่ยวกับ Azure Developer CLI
- เข้าใจแนวคิดและคำศัพท์สำคัญผ่านรูปแบบคำถามและคำตอบที่ใช้งานได้จริง
- เข้าถึงวิธีแก้ไขปัญหาสำหรับปัญหาซ้ำซากและกรณีข้อผิดพลาด
- เรียนรู้แนวทางปฏิบัติที่ดีที่สุดผ่านคำถามที่พบบ่อยเกี่ยวกับการปรับแต่ง
- ค้นพบฟีเจอร์และความสามารถขั้นสูงผ่านคำถามระดับผู้เชี่ยวชาญ
- อ้างอิงคำแนะนำเรื่องต้นทุน ความปลอดภัย และกลยุทธ์การปรับใช้ได้อย่างมีประสิทธิภาพ

## ผลลัพธ์การเรียนรู้

ด้วยการอ้างอิง FAQ นี้เป็นประจำ คุณจะสามารถ:
- แก้ไขปัญหาทั่วไปของ Azure Developer CLI ได้อย่างอิสระโดยใช้วิธีแก้ไขที่ให้ไว้
- ตัดสินใจอย่างมีข้อมูลเกี่ยวกับกลยุทธ์และการตั้งค่าการปรับใช้
- เข้าใจความสัมพันธ์ระหว่าง azd กับเครื่องมือและบริการ Azure อื่นๆ
- ใช้แนวทางปฏิบัติที่ดีที่สุดตามประสบการณ์ชุมชนและคำแนะนำของผู้เชี่ยวชาญ
- แก้ไขปัญหาการตรวจสอบสิทธิ์ การปรับใช้ และการกำหนดค่าได้อย่างมีประสิทธิภาพ
- ปรับแต่งต้นทุนและประสิทธิภาพด้วยข้อมูลเชิงลึกและคำแนะนำจาก FAQ

## เริ่มต้น

### Q: Azure Developer CLI (azd) คืออะไร?
**A**: Azure Developer CLI (azd) คือเครื่องมือบรรทัดคำสั่งเน้นนักพัฒนาที่ช่วยเร่งเวลาในการพาแอปพลิเคชันจากสภาพแวดล้อมการพัฒนาในเครื่องไปยัง Azure มันให้แนวทางปฏิบัติที่ดีที่สุดผ่านแม่แบบและช่วยในวงจรการปรับใช้ทั้งหมด

### Q: azd แตกต่างจาก Azure CLI อย่างไร?
**A**: 
- **Azure CLI**: เครื่องมือทั่วไปสำหรับการจัดการทรัพยากร Azure
- **azd**: เครื่องมือเน้นนักพัฒนาสำหรับเวิร์กโฟลว์การปรับใช้แอปพลิเคชัน
- azd ใช้ Azure CLI ภายในแต่ให้การนามธรรมระดับสูงสำหรับสถานการณ์พัฒนาทั่วไป
- azd รวมแม่แบบ การจัดการสภาพแวดล้อม และการทำงานอัตโนมัติของการปรับใช้

### Q: จำเป็นต้องติดตั้ง Azure CLI เพื่อใช้ azd ไหม?
**A**: ใช่ azd ต้องการ Azure CLI สำหรับการตรวจสอบสิทธิ์และบางการดำเนินการ ติดตั้ง Azure CLI ก่อนแล้วจึงติดตั้ง azd

### Q: azd รองรับภาษาโปรแกรมอะไรบ้าง?
**A**: azd ไม่ขึ้นกับภาษา มันทำงานกับ:
- Node.js/JavaScript/TypeScript
- Python
- .NET/C#
- Java
- Go
- PHP
- เว็บไซต์แบบสแตติก
- แอปพลิเคชันแบบคอนเทนเนอร์

### Q: ฉันสามารถใช้ azd กับโครงการที่มีอยู่ได้ไหม?
**A**: ใช่! คุณสามารถ:
1. ใช้ `azd init` เพื่อเพิ่มการตั้งค่า azd ในโครงการที่มีอยู่
2. ปรับโครงสร้างโครงการที่มีให้ตรงกับแม่แบบ azd
3. สร้างแม่แบบที่กำหนดเองตามสถาปัตยกรรมที่มีอยู่ของคุณ

## การตรวจสอบสิทธิ์ & การเข้าถึง

### Q: ฉันตรวจสอบสิทธิ์กับ Azure โดยใช้ azd อย่างไร?
**A**: ใช้ `azd auth login` ซึ่งจะเปิดหน้าต่างเบราว์เซอร์สำหรับการตรวจสอบสิทธิ์ Azure สำหรับสถานการณ์ CI/CD ให้ใช้ service principals หรือ managed identities

### Q: ฉันใช้ azd กับการสมัครใช้ Azure หลายบัญชีได้ไหม?
**A**: ได้ ใช้ `azd env set AZURE_SUBSCRIPTION_ID <subscription-id>` เพื่อระบุการสมัครที่ใช้สำหรับแต่ละสภาพแวดล้อม

### Q: ฉันต้องการสิทธิ์อะไรในการปรับใช้ด้วย azd?
**A**: โดยทั่วไปจำเป็นต้องมี:
- บทบาท **Contributor** บนกลุ่มทรัพยากรหรือการสมัครใช้
- บทบาท **User Access Administrator** หากปรับใช้ทรัพยากรที่ต้องการการกำหนดบทบาท
- สิทธิ์เฉพาะจะแตกต่างกันตามแม่แบบและทรัพยากรที่ปรับใช้

### Q: ฉันใช้ azd ใน pipeline CI/CD ได้ไหม?
**A**: ได้แน่นอน! azd ออกแบบมาเพื่อการรวมกับ CI/CD ใช้ service principals สำหรับการตรวจสอบสิทธิ์และตั้งค่าตัวแปรสภาพแวดล้อมสำหรับการกำหนดค่า

### Q: ฉันจัดการการตรวจสอบสิทธิ์ใน GitHub Actions อย่างไร?
**A**: ใช้ Azure Login action กับข้อมูลรับรองของ service principal:
```yaml
- uses: azure/login@v1
  with:
    creds: ${{ secrets.AZURE_CREDENTIALS }}
- run: azd deploy --no-prompt
```

## แม่แบบ & โครงการ

### Q: ฉันหแม่แบบ azd ได้ที่ไหน?
**A**: 
- แม่แบบอย่างเป็นทางการ: [Azure-Samples/awesome-azd](https://github.com/Azure-Samples/awesome-azd)
- แม่แบบจากชุมชน: ค้นหาใน GitHub โดยใช้คำว่า "azd-template"
- ใช้ `azd template list` เพื่อเรียกดูแม่แบบที่มีอยู่

### Q: ฉันสร้างแม่แบบกำหนดเองได้อย่างไร?
**A**: 
1. เริ่มจากโครงสร้างแม่แบบที่มีอยู่
2. แก้ไข `azure.yaml` ไฟล์โครงสร้างพื้นฐาน และโค้ดแอปพลิเคชัน
3. ทดสอบอย่างละเอียดด้วย `azd up`
4. เผยแพร่ใน GitHub พร้อมป้ายกำกับที่เหมาะสม

### Q: ฉันใช้ azd โดยไม่ต้องใช้แม่แบบได้ไหม?
**A**: ได้ ใช้ `azd init` ในโปรเจกต์ที่มีอยู่เพื่อสร้างไฟล์การกำหนดค่าที่จำเป็น คุณจะต้องตั้งค่า `azure.yaml` และไฟล์โครงสร้างพื้นฐานด้วยตนเอง

### Q: แตกต่างระหว่างแม่แบบทางการและแม่แบบชุมชนอย่างไร?
**A**: 
- **แม่แบบทางการ**: ดูแลโดย Microsoft อัปเดตอย่างสม่ำเสมอ มีเอกสารครอบคลุม
- **แม่แบบชุมชน**: สร้างโดยนักพัฒนา อาจมีกรณีใช้งานเฉพาะ เจาะจงคุณภาพและการดูแลรักษาที่แตกต่างกัน

### Q: ฉันจะอัปเดตแม่แบบในโปรเจกต์ของฉันอย่างไร?
**A**: แม่แบบไม่ได้รับการอัปเดตอัตโนมัติ คุณสามารถ:
1. เปรียบเทียบและรวมการเปลี่ยนแปลงจากแม่แบบต้นทางด้วยตนเอง
2. เริ่มใหม่ด้วย `azd init` โดยใช้แม่แบบที่อัปเดตแล้ว
3. เลือกเฉพาะการปรับปรุงบางส่วนจากแม่แบบที่อัปเดต

## การปรับใช้ & โครงสร้างพื้นฐาน

### Q: azd ปรับใช้บริการ Azure อะไรได้บ้าง?
**A**: azd สามารถปรับใช้บริการ Azure ใดๆ ผ่านแม่แบบ Bicep/ARM รวมถึง:
- App Services, Container Apps, Functions
- ฐานข้อมูล (SQL, PostgreSQL, Cosmos DB)
- Storage, Key Vault, Application Insights
- เครือข่าย ความปลอดภัย และทรัพยากรการตรวจสอบ

### Q: ฉันปรับใช้ไปยังหลายภูมิภาคได้ไหม?
**A**: ได้ ตั้งค่าภูมิภาคหลายแห่งในแม่แบบ Bicep ของคุณและตั้งค่าพารามิเตอร์ location ให้เหมาะสมสำหรับแต่ละสภาพแวดล้อม

### Q: ฉันจัดการการเปลี่ยนแปลงสเกมาในฐานข้อมูลอย่างไร?
**A**: ใช้ deployment hooks ใน `azure.yaml`:
```yaml
hooks:
  postdeploy:
    posix:
      run: ./scripts/migrate-database.sh
    windows:
      run: ./scripts/migrate-database.ps1
```

### Q: ฉันปรับใช้เฉพาะโครงสร้างพื้นฐานได้ไหมโดยไม่ใช้แอปพลิเคชัน?
**A**: ได้ ใช้ `azd provision` เพื่อปรับใช้เฉพาะส่วนโครงสร้างพื้นฐานที่กำหนดในแม่แบบของคุณ

### Q: ฉันปรับใช้กับทรัพยากร Azure ที่มีอยู่แล้วได้ไหม?
**A**: เรื่องนี้ซับซ้อนและไม่รองรับโดยตรง คุณสามารถ:
1. นำเข้าทรัพยากรที่มีอยู่ลงในแม่แบบ Bicep
2. อ้างอิงทรัพยากรที่มีอยู่ในแม่แบบ
3. แก้ไขแม่แบบเพื่อสร้างหรืออ้างอิงทรัพยากรตามเงื่อนไข

### Q: ฉันใช้ Terraform แทน Bicep ได้ไหม?
**A**: ปัจจุบัน azd รองรับแม่แบบ Bicep/ARM เป็นหลัก การรองรับ Terraform ยังไม่มีอย่างเป็นทางการ แม้จะมีวิธีแก้ชุมชนบ้างก็ตาม

## การตั้งค่า & สภาพแวดล้อม

### Q: ฉันจัดการกับสภาพแวดล้อมต่างๆ (dev, staging, prod) อย่างไร?
**A**: สร้างสภาพแวดล้อมแยกต่างหากด้วย `azd env new <environment-name>` และตั้งค่าการกำหนดค่าที่แตกต่างกันสำหรับแต่ละสภาพแวดล้อม:
```bash
azd env new development
azd env new staging  
azd env new production
```

### Q: การตั้งค่าสภาพแวดล้อมเก็บไว้ที่ไหน?
**A**: ที่โฟลเดอร์ `.azure` ภายในไดเรกทอรีโครงการของคุณ แต่ละสภาพแวดล้อมมีโฟลเดอร์และไฟล์กำหนดค่าของตัวเอง

### Q: ฉันตั้งค่าการกำหนดค่าสำหรับสภาพแวดล้อมเฉพาะได้อย่างไร?
**A**: ใช้ `azd env set` เพื่อตั้งค่าตัวแปรสภาพแวดล้อม:
```bash
azd env set AZURE_LOCATION eastus
azd env set DATABASE_TIER Basic
```

### Q: ฉันแชร์การตั้งค่าสภาพแวดล้อมระหว่างสมาชิกทีมได้ไหม?
**A**: โฟลเดอร์ `.azure` มีข้อมูลละเอียดอ่อนและไม่ควรบันทึกในระบบควบคุมเวอร์ชัน ให้ทำดังนี้แทน:
1. เอกสารตัวแปรสภาพแวดล้อมที่จำเป็น
2. ใช้สคริปต์ปรับใช้ในการตั้งค่าสภาพแวดล้อม
3. ใช้ Azure Key Vault สำหรับการตั้งค่าที่ละเอียดอ่อน

### Q: ฉันจะเปลี่ยนค่าเริ่มต้นของแม่แบบอย่างไร?
**A**: ตั้งค่าตัวแปรสภาพแวดล้อมที่สอดคล้องกับพารามิเตอร์ในแม่แบบ:
```bash
azd env set LOCATION "West US 2"
azd env set SKU_NAME "B1"
```

## การแก้ไขปัญหา

### Q: ทำไม `azd up` ถึงล้มเหลว?
**A**: สาเหตุทั่วไป:
1. **ปัญหาการตรวจสอบสิทธิ์**: รัน `azd auth login`
2. **สิทธิ์ไม่เพียงพอ**: ตรวจสอบการมอบหมายบทบาทใน Azure ของคุณ
3. **การชนกันของชื่อทรัพยากร**: เปลี่ยน AZURE_ENV_NAME
4. **ปัญหาโควตา/ความจุ**: ตรวจสอบความพร้อมใช้งานตามภูมิภาค
5. **ข้อผิดพลาดของแม่แบบ**: ตรวจสอบความถูกต้องของแม่แบบ Bicep

### Q: ฉันดีบักความล้มเหลวของการปรับใช้อย่างไร?
**A**: 
1. ใช้ `azd deploy --debug` เพื่อตรวจสอบรายละเอียดการทำงาน
2. ตรวจสอบประวัติการปรับใช้ใน Azure portal
3. ดูบันทึกกิจกรรมใน Azure portal
4. ใช้ `azd show` เพื่อแสดงสถานะปัจจุบันของสภาพแวดล้อม

### Q: ทำไมตัวแปรสภาพแวดล้อมของฉันถึงไม่ทำงาน?
**A**: ตรวจสอบว่า:
1. ชื่อตัวแปรตรงกับพารามิเตอร์แม่แบบอย่างแม่นยำ
2. ค่าถูกอ้างอิงอย่างถูกต้องหากมีช่องว่าง
3. เลือกสภาพแวดล้อมถูกต้องด้วย `azd env select <environment>`
4. ตั้งค่าตัวแปรในสภาพแวดล้อมที่ถูกต้องแล้ว

### Q: ฉันทำความสะอาดการปรับใช้ที่ล้มเหลวอย่างไร?
**A**: 
```bash
azd down --force --purge
```
สิ่งนี้จะลบทรัพยากรทั้งหมดและการตั้งค่าสภาพแวดล้อม

### Q: ทำไมแอปพลิเคชันของฉันเข้าถึงไม่ได้หลังจากปรับใช้?
**A**: ตรวจสอบว่า:
1. การปรับใช้เสร็จสมบูรณ์สำเร็จ
2. แอปพลิเคชันกำลังทำงาน (ดูบันทึกใน Azure portal)
3. กลุ่มความปลอดภัยเครือข่ายอนุญาตการรับส่งข้อมูล
4. DNS/โดเมนที่กำหนดเองตั้งค่าอย่างถูกต้อง

## ต้นทุน & การเรียกเก็บเงิน

### Q: การปรับใช้ azd จะมีค่าใช้จ่ายเท่าไหร่?
**A**: ค่าใช้จ่ายขึ้นอยู่กับ:
- บริการ Azure ที่ปรับใช้
- ระดับบริการ/SKU ที่เลือก
- ความแตกต่างราคาตามภูมิภาค
- รูปแบบการใช้งาน

ใช้ [เครื่องคิดเลขค่าใช้จ่าย Azure](https://azure.microsoft.com/pricing/calculator/) เพื่อคาดการณ์

### Q: ฉันควบคุมต้นทุนในการปรับใช้ azd ได้อย่างไร?
**A**: 
1. ใช้ SKU ระดับต่ำกว่าสำหรับสภาพแวดล้อมการพัฒนา
2. ตั้งงบประมาณ Azure และการแจ้งเตือน
3. ใช้ `azd down` เพื่อลบทรัพยากรเมื่อไม่ต้องการ
4. เลือกภูมิภาคที่เหมาะสม (ราคาจะต่างกันตามตำแหน่ง)
5. ใช้เครื่องมือจัดการต้นทุน Azure

### Q: มีตัวเลือกระดับฟรีสำหรับแม่แบบ azd หรือไม่?
**A**: บริการ Azure หลายรายการมีระดับฟรี:
- App Service: มีระดับฟรี
- Azure Functions: 1 ล้านการทำงานฟรีต่อเดือน
- Cosmos DB: ระดับฟรีที่ 400 RU/s
- Application Insights: ใช้ฟรี 5GB แรกต่อเดือน

ตั้งค่าแม่แบบให้ใช้ระดับฟรีเมื่อมี

### Q: ฉันประเมินต้นทุนก่อนปรับใช้อย่างไร?
**A**: 
1. ตรวจสอบ `main.bicep` ของแม่แบบเพื่อดูทรัพยากรที่จะถูกสร้าง
2. ใช้เครื่องคิดเลขค่าใช้จ่ายพร้อม SKU เฉพาะ
3. ปรับใช้ในสภาพแวดล้อมพัฒนาก่อนเพื่อตรวจสอบต้นทุนจริง
4. ใช้ Azure Cost Management วิเคราะห์ต้นทุนอย่างละเอียด

## แนวทางปฏิบัติที่ดีที่สุด

### Q: แนวทางปฏิบัติที่ดีที่สุดสำหรับโครงสร้างโปรเจกต์ azd คืออะไร?
**A**: 
1. แยกโค้ดแอปพลิเคชันออกจากโครงสร้างพื้นฐาน
2. ใช้ชื่อบริการที่มีความหมายใน `azure.yaml`
3. ใช้การจัดการข้อผิดพลาดอย่างเหมาะสมในสคริปต์ build
4. ตั้งค่าการกำหนดค่าสภาพแวดล้อมเฉพาะ
5. รวมเอกสารประกอบอย่างครบถ้วน

### Q: ฉันควรจัดการบริการหลายตัวใน azd อย่างไร?
**A**: ใช้โครงสร้างที่แนะนำ:
```
├── src/
│   ├── web/          # Frontend service
│   ├── api/          # Backend service  
│   └── worker/       # Background service
├── infra/            # Infrastructure templates
├── azure.yaml        # azd configuration
└── README.md         # Project documentation
```

### Q: ฉันควรบันทึกโฟลเดอร์ `.azure` ลงระบบควบคุมเวอร์ชันไหม?
**A**: **ไม่ควร!** โฟลเดอร์ `.azure` มีข้อมูลที่ละเอียดอ่อน ให้เพิ่มลงใน `.gitignore`:
```gitignore
.azure/
```

### Q: ฉันจัดการความลับและการตั้งค่าระดับละเอียดอ่อนได้อย่างไร?
**A**: 
1. ใช้ Azure Key Vault สำหรับเก็บความลับ
2. อ้างอิงความลับใน Key Vault ในการตั้งค่าแอปพลิเคชัน
3. ไม่ควรบันทึกความลับลงระบบควบคุมเวอร์ชัน
4. ใช้ managed identities สำหรับการตรวจสอบสิทธิ์ระหว่างบริการ

### Q: แนวทางแนะนำสำหรับ CI/CD กับ azd คืออะไร?
**A**: 
1. ใช้สภาพแวดล้อมแยกสำหรับแต่ละขั้นตอน (dev/staging/prod)
2. ทำการทดสอบอัตโนมัติก่อนปรับใช้
3. ใช้ service principals สำหรับการตรวจสอบสิทธิ์
4. เก็บการตั้งค่าระดับละเอียดอ่อนในความลับ/ตัวแปรของ pipeline
5. ใช้การอนุมัติสำหรับการปรับใช้ในสภาพแวดล้อม production

## หัวข้อขั้นสูง

### Q: ฉันขยาย azd ด้วยฟังก์ชันกำหนดเองได้ไหม?
**A**: ได้ ผ่าน deployment hooks ใน `azure.yaml`:
```yaml
hooks:
  predeploy:
    run: ./scripts/custom-setup.sh
  postdeploy:
    run: ./scripts/custom-config.sh
```

### Q: ฉันผสาน azd เข้ากับกระบวนการ DevOps ที่มีอยู่ได้อย่างไร?
**A**: 
1. ใช้คำสั่ง azd ในสคริปต์ pipeline ที่มีอยู่
2. มาตรฐานแม่แบบ azd ในทีมต่างๆ
3. ผสานเข้ากับการตรวจสอบและแจ้งเตือนที่มีอยู่
4. ใช้ผลลัพธ์ JSON ของ azd สำหรับการรวมกับ pipeline

### Q: ฉันใช้ azd กับ Azure DevOps ได้ไหม?
**A**: ได้ azd ทำงานได้กับระบบ CI/CD ทุกประเภท สร้าง pipeline ใน Azure DevOps ที่ใช้คำสั่ง azd

### Q: ฉันจะมีส่วนร่วมใน azd หรือสร้างแม่แบบชุมชนอย่างไร?
**A**: 

1. **azd tool**: ร่วมพัฒนาใน [Azure/azure-dev](https://github.com/Azure/azure-dev)
2. **Templates**: สร้างแม่แบบตาม [template guidelines](https://github.com/Azure-Samples/awesome-azd)
3. **Documentation**: ร่วมพัฒนาคู่มือที่ [MicrosoftDocs/azure-dev-docs](https://github.com/MicrosoftDocs/azure-dev-docs)

### Q: แผนงานสำหรับ azd คืออะไร?
**A**: ตรวจสอบ [official roadmap](https://github.com/Azure/azure-dev/projects) สำหรับฟีเจอร์และการปรับปรุงที่วางแผนไว้

### Q: ฉันจะย้ายจากเครื่องมือ deploy อื่นมาใช้ azd ได้อย่างไร?
**A**: 
1. วิเคราะห์สถาปัตยกรรมการ deploy ปัจจุบัน
2. สร้างแม่แบบ Bicep ที่เทียบเท่า
3. กำหนดค่า `azure.yaml` ให้ตรงกับบริการปัจจุบัน
4. ทดสอบอย่างละเอียดในสภาพแวดล้อมพัฒนา
5. ย้ายสภาพแวดล้อมทีละขั้นตอน
