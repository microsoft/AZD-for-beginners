## บทนำ

ใน [โปรเจกต์แรกของคุณ](first-project.md) คุณได้ปรับใช้แอปโดยเริ่มจากเทมเพลต แต่ส่วนใหญ่แล้วคุณมักจะ *มี* แอปอยู่แล้ว—เช่น API ของ Node.js, บริการ Python Flask, เว็บแอป .NET—ที่อยู่ในโฟลเดอร์บนเครื่องของคุณ บทเรียนนี้จะแสดงวิธีเพิ่ม azd ให้กับโค้ดที่มีอยู่เพื่อให้คุณสามารถปรับใช้ด้วย `azd up` ได้โดยไม่ต้องใช้เทมเพลต

## เป้าหมายการเรียนรู้

เมื่อจบบทเรียนนี้ คุณจะ:
- เข้าใจสามวิธีในการเริ่มโปรเจกต์ azd
- รัน `azd init` ภายในฐานโค้ดที่มีอยู่แล้ว
- เข้าใจว่า `azure.yaml` และโฟลเดอร์ `infra/` ทำอะไรให้กับแอปของคุณ
- รู้ว่าเมื่อใดควรให้ azd สร้างโครงสร้างพื้นฐานให้เทียบกับการเขียนเอง
- ปรับใช้แอปที่มีอยู่ของคุณไปยัง Azure ด้วย `azd up`

## ผลลัพธ์การเรียนรู้

หลังจากทำบทเรียนนี้เสร็จ คุณจะสามารถ:
- เริ่มต้น azd ในโปรเจกต์ที่คุณมีอยู่แล้ว
- อ่านและแก้ไขไฟล์ `azure.yaml` พื้นฐาน
- สร้างโครงสร้างพื้นฐานเริ่มต้นด้วย `azd infra generate`
- เลือกโฮสต์ Azure ที่เหมาะสมสำหรับแอปของคุณ
- ปรับใช้และลบแอปของคุณเอง

---

## สามวิธีในการเริ่มโปรเจกต์ azd

| Starting point | Command | When to use |
|----------------|---------|-------------|
| **จากเทมเพลต** | `azd init --template <name>` | เรียนรู้ หรือเริ่มแอปใหม่จากตัวอย่างที่พิสูจน์แล้ว |
| **จากโค้ดที่มีอยู่ของคุณ** | `azd init` (ในโฟลเดอร์โปรเจกต์ของคุณ) | คุณมีแอปอยู่แล้วและต้องการปรับใช้ |
| **จากรีโป Git** | `azd init --from-code` (ในรีโปที่คลอนมา) | นำ azd มาใช้กับรีโปที่มีอยู่แล้ว |

คุณได้ฝึกตัวเลือกแรกไปแล้ว บทเรียนนี้ครอบคลุมตัวเลือกที่สอง—ซึ่งเป็นสถานการณ์จริงที่พบบ่อยที่สุด

## ขั้นตอนที่ 1: รัน `azd init` ในโปรเจกต์ของคุณ

เปิดเทอร์มินัล **ภายในโฟลเดอร์โปรเจกต์ที่มีอยู่ของคุณ** แล้วรัน:

```bash
cd my-existing-app
azd init
```

azd จะถามว่าคุณต้องการเริ่มต้นอย่างไร เลือก:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

เลือก **"ใช้โค้ดในไดเรกทอรีปัจจุบัน"** จากนั้น azd จะสแกนโฟลเดอร์ของคุณ ตรวจจับภาษาและเฟรมเวิร์ก แล้วแนะนำโฮสต์

### สิ่งที่ azd ตรวจจับ

azd มองหาสัญญาณเช่น `package.json`, `requirements.txt`, `pom.xml`, `*.csproj` หรือ `Dockerfile` และจะแนะนำโฮสต์ Azure ที่ตรงกัน:

| แอปของคุณ | โฮสต์ที่น่าจะถูกตรวจจับ |
|----------|----------------------|
| Node.js / Python / .NET web app | Azure App Service หรือ Container Apps |
| แอปที่ใช้คอนเทนเนอร์ (`Dockerfile`) | Azure Container Apps |
| แอปฟังก์ชัน | Azure Functions |
| ไซต์สแตติก (ผลลัพธ์การ build ของ React/Vue) | Azure Static Web Apps |

ยืนยันบริการที่ตรวจพบ แล้ว azd จะสร้างโครงร่างไฟล์ที่คุณต้องการให้

## ขั้นตอนที่ 2: เข้าใจสิ่งที่ azd สร้างให้

หลังจาก init คุณจะมีสองอย่างใหม่ในโปรเจกต์ของคุณ:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — คำจำกัดความของโปรเจกต์

นี่คือหัวใจของโปรเจกต์ azd หนึ่งไฟล์แบบขั้นต่ำจะมีลักษณะดังนี้:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

บล็อก `services` คือส่วนสำคัญ: แต่ละรายการแมปโฟลเดอร์โค้ดของคุณไปยังโฮสต์ของ Azure หากแอปของคุณมีทั้ง frontend และ API คุณจะมีสองบริการ

### `infra/` — ทรัพยากร Azure ของคุณในรูปแบบโค้ด

โฟลเดอร์ `infra/` เก็บไฟล์ Bicep ที่กำหนดทรัพยากร Azure ที่แอปของคุณต้องการ (เช่น App Service, ฐานข้อมูล ฯลฯ) คุณไม่จำเป็นต้องเขียนไฟล์เหล่านี้ด้วยมือ—azd จะสร้างจุดเริ่มต้นที่ใช้งานได้ให้ คุณ *สามารถ* แก้ไขภายหลังเพื่อเพิ่มทรัพยากรหรือเสริมความปลอดภัย (จะกล่าวถึงใน [บทที่ 4](../chapter-04-infrastructure/README.md))

> **เคล็ดลับ:** ต้องการดูหรือปรับแต่งโครงสร้างพื้นฐานที่สร้างขึ้นก่อนปรับใช้หรือไม่? รัน `azd infra generate` (หรือ `azd infra synth`) เพื่อเขียน IaC ลงดิสก์เพื่อให้คุณตรวจทานและควบคุมเวอร์ชันได้

## ขั้นตอนที่ 3: ตั้งค่าคอนฟิกที่จำเป็น

หากแอปของคุณต้องการการตั้งค่าหรือความลับ (เช่น connection string, คีย์ API) อย่าใส่แบบ hardcode ให้ใช้ค่าจาก environment:

```bash
# สร้างสภาพแวดล้อม
azd env new dev

# ตั้งค่าที่ไม่เป็นความลับ
azd env set API_VERSION 1.0.0
```

สำหรับความลับจริงๆ ให้เก็บไว้ใน Key Vault และอ้างอิงจากโครงสร้างพื้นฐานของคุณ—ดู [บทที่ 3: การกำหนดค่า & การพิสูจน์ตัวตน](../chapter-03-configuration/authsecurity.md)

## ขั้นตอนที่ 4: ปรับใช้

ตอนนี้ใช้เวิร์กโฟลว์เดียวกับที่คุณรู้แล้ว:

```bash
# ยืนยันตัวตน (จำเป็นสำหรับ azd)
azd auth login

# ดูตัวอย่างทรัพยากรที่จะถูกสร้าง
azd provision --preview

# จัดเตรียมโครงสร้างพื้นฐานและปรับใช้โค้ดของคุณ
azd up
```

เมื่อเสร็จแล้ว azd จะแสดง URL ของแอปคุณ ยืนยันด้วยวิธีเดียวกับแอป azd อื่นๆ:

```bash
azd show           # แสดงจุดสิ้นสุด
azd monitor --logs # ตรวจสอบบันทึกหากจำเป็น
```

## ปัญหาทั่วไปในครั้งแรกที่ใช้งาน

| อาการ | สาเหตุที่น่าจะเป็น | วิธีแก้ |
|---------|--------------|-----|
| azd ไม่ตรวจจับแอปของฉัน | ขาด manifest (เช่น `package.json`) | เพิ่ม manifest หรือเลือกโฮสต์ด้วยตนเองระหว่าง `azd init` |
| การ build ล้มเหลวระหว่าง `azd up` | แอปต้องการขั้นตอนการ build | เพิ่ม `buildCommand`/`outputPath` ภายใต้ service ใน `azure.yaml` |
| แอปเริ่มแต่คืนค่าข้อผิดพลาด | ขาดคอนฟิก/ความลับ | ตั้งค่าด้วย `azd env set` หรือเชื่อมต่อกับ Key Vault |
| เลือกโฮสต์ผิด | การตรวจจับอัตโนมัติคาดเดาผิด | แก้ไข `host:` ใน `azure.yaml` แล้วรัน `azd up` อีกครั้ง |

สำหรับข้อมูลเพิ่มเติม ดู [บทที่ 7: การแก้ไขปัญหา](../chapter-07-troubleshooting/README.md)

## ล้างทรัพยากร

```bash
azd down --force --purge
```

## สรุป

- `azd init` → **"ใช้โค้ดในไดเรกทอรีปัจจุบัน"** เพิ่ม azd ให้กับแอปที่คุณมีอยู่แล้ว
- `azure.yaml` แมปโฟลเดอร์โค้ดของคุณไปยังโฮสต์ของ Azure; `infra/` กำหนดทรัพยากรเป็น Bicep
- `azd infra generate` ช่วยให้คุณตรวจทานหรือปรับแต่งโครงสร้างพื้นฐานที่สร้างขึ้น
- เมื่อตั้งค่าแล้ว แอปที่มีอยู่ของคุณจะใช้เวิร์กโฟลว์ `azd up` / `azd down` เดียวกันกับแอปที่เริ่มจากเทมเพลต
