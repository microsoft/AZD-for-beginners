# นำแอปของคุณมาเอง - เพิ่ม azd ให้กับโปรเจกต์ที่มีอยู่

**การนำทางบทเรียน:**
- **📚 หน้าแรกของหลักสูตร**: [AZD สำหรับผู้เริ่มต้น](../../README.md)
- **📖 บทปัจจุบัน**: บทที่ 1 - พื้นฐาน & เริ่มต้นอย่างรวดเร็ว
- **⬅️ ก่อนหน้า**: [โปรเจกต์แรกของคุณ](first-project.md)
- **➡️ ถัดไป**: [Dev Containers & Codespaces](dev-containers.md)

> ตรวจสอบกับ `azd 1.27.1` ในเดือนกรกฎาคม 2026 แล้ว

## บทนำ

ใน [โปรเจกต์แรกของคุณ](first-project.md) คุณได้ปรับใช้แอปโดยเริ่มจากเทมเพลต แต่ส่วนใหญ่มักจะ *มี* แอปอยู่แล้ว—เช่น Node.js API, บริการ Python Flask, เว็บแอป .NET—ซึ่งอยู่ในโฟลเดอร์บนเครื่องของคุณ บทเรียนนี้แสดงวิธีการเพิ่ม azd ลงในโค้ดที่มีอยู่เพื่อให้คุณสามารถปรับใช้ด้วย `azd up` โดยไม่ต้องใช้เทมเพลต

## เป้าหมายการเรียนรู้

เมื่อจบบทเรียนนี้คุณจะ:
- เข้าใจสามวิธีในการเริ่มโปรเจกต์ azd
- รัน `azd init` ภายในฐานโค้ดที่มีอยู่
- เข้าใจว่า `azure.yaml` และโฟลเดอร์ `infra/` ทำหน้าที่อะไรสำหรับแอปของคุณ
- รู้ว่าเมื่อไหร่ควรให้ azd สร้าง infrastructure หรือเขียนเอง
- ปรับใช้แอปที่มีอยู่ของคุณไปยัง Azure ด้วย `azd up`

## ผลลัพธ์การเรียนรู้

หลังจากทำบทเรียนนี้เสร็จ คุณจะสามารถ:
- เริ่มต้น azd ในโปรเจกต์ที่คุณมีอยู่แล้ว
- อ่านและแก้ไขไฟล์ `azure.yaml` เบื้องต้น
- สร้าง infrastructure เริ่มต้นด้วย `azd infra generate`
- เลือกโฮสต์ Azure ที่เหมาะสมสำหรับแอปของคุณ
- ปรับใช้และล้างแอปพลิเคชันของคุณเอง

---

## สามวิธีเริ่มโปรเจกต์ azd

| จุดเริ่มต้น | คำสั่ง | เมื่อใช้ |
|----------------|---------|-------------|
| **จากเทมเพลต** | `azd init --template <name>` | เรียนรู้ หรือเริ่มแอปใหม่จากตัวอย่างที่ทดสอบแล้ว |
| **จากโค้ดที่มีอยู่ของคุณ** | `azd init` (ในโฟลเดอร์โปรเจกต์ของคุณ) | คุณมีแอปอยู่แล้วและต้องการปรับใช้ |
| **จาก Git repo** | `azd init --from-code` (ใน repo ที่โคลนมา) | นำ azd ไปใช้กับ repository ที่มีอยู่ |

คุณได้ฝึกใช้ตัวเลือกแรกแล้ว บทเรียนนี้จะครอบคลุมตัวเลือกที่สอง—ซึ่งเป็นสถานการณ์ในโลกความจริงที่พบบ่อยที่สุด

---

## ขั้นตอนที่ 1: รัน `azd init` ในโปรเจกต์ของคุณ

เปิดเทอร์มินัล **ภายในโฟลเดอร์โปรเจกต์ที่มีอยู่ของคุณ** และรัน:

```bash
cd my-existing-app
azd init
```

azd จะถามว่าคุณต้องการเริ่มต้นอย่างไร เลือก:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

เลือก **"Use code in the current directory."** จากนั้น azd จะสแกนโฟลเดอร์ของคุณ ตรวจจับภาษาที่ใช้และเฟรมเวิร์ก แล้วแนะนำโฮสต์

### สิ่งที่ azd ตรวจจับ

azd มองหาสัญญาณ เช่น `package.json`, `requirements.txt`, `pom.xml`, `*.csproj` หรือ `Dockerfile` และแนะนำโฮสต์ Azure ที่ตรงกัน:

| แอปของคุณ | โฮสต์ที่น่าจะตรวจจับได้ |
|----------|----------------------|
| Node.js / Python / เว็บแอป .NET | Azure App Service หรือ Container Apps |
| แอปแบบ container (`Dockerfile`) | Azure Container Apps |
| แอปแบบฟังก์ชัน | Azure Functions |
| เว็บไซต์สแตติก (เอาต์พุต React/Vue ที่ถูก build แล้ว) | Azure Static Web Apps |

ยืนยันบริการที่ตรวจจับได้ แล้ว azd จะสร้างโครงสร้างไฟล์ที่คุณต้องการ

## ขั้นตอนที่ 2: เข้าใจสิ่งที่ azd สร้างขึ้น

หลังจาก init คุณจะมีสิ่งใหม่สองอย่างในโปรเจกต์:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — คำนิยามโปรเจกต์

นี่คือหัวใจของโปรเจกต์ azd ไฟล์ขั้นต่ำจะมีลักษณะดังนี้:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

บล็อก `services` คือส่วนสำคัญ: แต่ละรายการเชื่อมโยงโฟลเดอร์โค้ดของคุณกับโฮสต์ Azure หากแอปของคุณมีทั้ง frontend และ API คุณจะมีสองบริการ

### `infra/` — โครงสร้างพื้นฐาน Azure ในรูปแบบโค้ด

โฟลเดอร์ `infra/` เก็บไฟล์ Bicep ที่กำหนดทรัพยากร Azure ที่แอปของคุณต้องการ (เช่น App Service, ฐานข้อมูล ฯลฯ) คุณไม่จำเป็นต้องเขียนด้วยมือ—azd สร้างจุดเริ่มต้นที่ใช้งานได้ คุณ *สามารถ* แก้ไขในภายหลังเพื่อเพิ่มทรัพยากรหรือเพิ่มความปลอดภัย (ครอบคลุมใน [บทที่ 4](../chapter-04-infrastructure/README.md))

> **เคล็ดลับ:** ต้องการดูหรือปรับโครงสร้างพื้นฐานที่สร้างก่อนปรับใช้? รัน `azd infra generate` (มีอีกชื่อว่า `azd infra synth`) เพื่อเขียน IaC ลงดิสก์ให้คุณตรวจสอบและควบคุมเวอร์ชัน

## ขั้นตอนที่ 3: ตั้งค่าคอนฟิกที่จำเป็น

หากแอปของคุณต้องการการตั้งค่าหรือความลับ (เช่น สตริงการเชื่อมต่อ, คีย์ API) อย่ากำหนดไว้ในโค้ด ใช้ค่าจาก environment:

```bash
# สร้างสภาพแวดล้อม
azd env new dev

# ตั้งค่าค่าที่ไม่เป็นความลับ
azd env set API_VERSION 1.0.0
```

สำหรับความลับจริงๆ ให้เก็บใน Key Vault และอ้างอิงจาก infrastructure ของคุณ—ดู [บทที่ 3: การตั้งค่า & การพิสูจน์ตัวตน](../chapter-03-configuration/authsecurity.md)

---

## ขั้นตอนที่ 4: ปรับใช้

ใช้โฟลว์งานเดียวกับที่คุณรู้อยู่แล้ว:

```bash
# การตรวจสอบสิทธิ์ (จำเป็นสำหรับ azd)
azd auth login

# ดูตัวอย่างทรัพยากรที่จะถูกสร้าง
azd provision --preview

# จัดเตรียมโครงสร้างพื้นฐานและปรับใช้โค้ดของคุณ
azd up
```

เมื่อเสร็จแล้ว azd จะแสดง URL ของแอปคุณ ยืนยันโดยวิธีเดียวกับแอป azd ทั่วไป:

```bash
azd show           # แสดง endpoints
azd monitor --logs # ตรวจสอบบันทึกถ้าจำเป็น
```

## ปัญหาที่พบได้บ่อยครั้งในครั้งแรก

| อาการ | สาเหตุที่เป็นไปได้ | วิธีแก้ |
|---------|--------------|-----|
| azd ไม่ตรวจจับแอปของฉัน | ไม่มีไฟล์ Manifest (เช่น `package.json`) | เพิ่มไฟล์ Manifest หรือเลือกโฮสต์ด้วยตนเองตอน `azd init` |
| การ build ล้มเหลวระหว่าง `azd up` | แอปต้องการขั้นตอน build | เพิ่ม `buildCommand`/`outputPath` ใต้ service ใน `azure.yaml` |
| แอปเริ่มแต่คืนค่าข้อผิดพลาด | ขาด config/ความลับ | ตั้งค่าด้วย `azd env set` หรือเชื่อมต่อกับ Key Vault |
| เลือกโฮสต์ผิด | การตรวจจับอัตโนมัติเดา | แก้ไข `host:` ใน `azure.yaml` แล้วรัน `azd up` ใหม่ |

เพิ่มเติมดูที่ [บทที่ 7: การแก้ไขปัญหา](../chapter-07-troubleshooting/README.md)

## สรุป

- `azd init` → **"Use code in the current directory"** เพิ่ม azd ให้กับแอปที่คุณมีอยู่แล้ว
- `azure.yaml` เชื่อมโยงโฟลเดอร์โค้ดของคุณกับโฮสต์ Azure; `infra/` กำหนดทรัพยากรเป็น Bicep
- `azd infra generate` ช่วยให้คุณตรวจสอบหรือปรับแต่งโครงสร้างพื้นฐานที่สร้างขึ้น
- เมื่อเริ่มต้นแล้ว แอปที่มีอยู่ของคุณใช้ workflow เดียวกับแอปจากเทมเพลตทั้ง `azd up` / `azd down`
