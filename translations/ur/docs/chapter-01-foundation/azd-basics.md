## تعارف

یہ سبق آپ کو Azure Developer CLI (azd) سے روشناس کراتا ہے، جو ایک طاقتور کمانڈ لائن ٹول ہے جو آپ کے لوکل ڈیولپمنٹ سے Azure پر تعیناتی کے سفر کو تیز کرتا ہے۔ آپ بنیادی تصورات، مرکزی خصوصیات سیکھیں گے اور سمجھیں گے کہ azd کس طرح کلاؤڈ-نیٹو ایپلیکیشن کی تعیناتی کو آسان بناتا ہے۔

## سیکھنے کے مقاصد

اس سبق کے اختتام تک، آپ:
- سمجھ سکیں گے کہ Azure Developer CLI کیا ہے اور اس کا بنیادی مقصد کیا ہے
- ٹیمپلیٹس، ماحولیاتی نظام، اور سروسز کے بنیادی تصورات سیکھیں گے
- کلیدی خصوصیات دریافت کریں گے جن میں ٹیمپلیٹ پر مبنی ترقی اور کوڈ کے طور پر انفراسٹرکچر شامل ہیں
- azd پروجیکٹ کی ساخت اور ورک فلو کو سمجھیں گے
- اپنے ترقیاتی ماحول کے لیے azd انسٹال اور کنفیگر کرنے کے لیے تیار ہوں گے

## سیکھنے کے نتائج

سبق مکمل کرنے کے بعد، آپ:

- بیان کر سکیں گے کہ ٹیمپلیٹس، ماحولیاتی نظام، اور سروسز کس طرح مل کر کام کرتی ہیں
- azd کے ساتھ کوڈ کے طور پر انفراسٹرکچر کے فوائد کو سمجھ سکیں گے
- مختلف azd کمانڈز اور ان کے مقاصد کو پہچان سکیں گے

## Azure Developer CLI (azd) کیا ہے؟

Azure Developer CLI (azd) ایک کمانڈ لائن ٹول ہے جو آپ کے لوکل ڈیولپمنٹ سے Azure پر تعیناتی کے سفر کو تیز بنانے کے لیے ڈیزائن کیا گیا ہے۔ یہ Azure پر کلاؤڈ-نیٹو ایپلیکیشنز کے تعمیر، تعیناتی، اور انتظام کے عمل کو آسان بناتا ہے۔

### azd کے ساتھ آپ کیا تعینات کر سکتے ہیں؟

azd وسیع قسم کے ورکلوڈز کی حمایت کرتا ہے—اور فہرست مسلسل بڑھ رہی ہے۔ آج، آپ azd کا استعمال کر کے تعینات کر سکتے ہیں:

| ورک لوڈ کی قسم | مثالیں | ایک ہی ورک فلو؟ |
|---------------|----------|----------------|
| **روایتی ایپلیکیشنز** | ویب ایپس، REST APIs، سٹیٹک سائٹس | ✅ `azd up` |
| **سروسز اور مائیکرو سروسز** | کنٹینر ایپس، فنکشن ایپس، متعدد سروسز بیک اینڈز | ✅ `azd up` |
| **AI سے چلنے والی ایپلیکیشنز** | Microsoft Foundry ماڈلز کے ساتھ چیٹ ایپس، AI سرچ کے ساتھ RAG حل | ✅ `azd up` |
| **ذہین ایجنٹ** | Foundry ہوسٹڈ ایجنٹس، متعدد ایجنٹ آرکیسٹریشن | ✅ `azd up` |

اہم بات یہ ہے کہ **azd لائف سائیکل ایک جیسا رہتا ہے چاہے آپ کچھ بھی تعینات کر رہے ہوں**۔ آپ ایک پروجیکٹ شروع کرتے ہیں، انفراسٹرکچر فراہم کرتے ہیں، اپنا کوڈ تعینات کرتے ہیں، اپنی ایپ کی نگرانی کرتے ہیں، اور صفائی کرتے ہیں—چاہے وہ ایک سادہ ویب سائٹ ہو یا ایک پیچیدہ AI ایجنٹ۔

یہ استمرار جان بوجھ کر بنایا گیا ہے۔ azd AI صلاحیتوں کو آپ کی ایپلیکیشن کی ایک اور قسم کی سروس کے طور پر دیکھتا ہے، نہ کہ کوئی بنیادی طور پر مختلف چیز۔ Microsoft Foundry ماڈلز سے چلنے والا چیٹ اینڈپوائنٹ azd کی نظر میں بس ایک اور سروس ہے جسے ترتیب دے کر تعینات کیا جا سکتا ہے۔

### 🎯 AZD کیوں استعمال کریں؟ حقیقی دنیا کی موازنہ

آئیں ایک سادہ ویب ایپ کو ڈیٹا بیس کے ساتھ تعینات کرنے کا موازنہ کریں:

#### ❌ بغیر AZD کے: دستی Azure تعیناتی (30+ منٹ)

```bash
# مرحلہ 1: ریسورس گروپ بنائیں
az group create --name myapp-rg --location eastus

# مرحلہ 2: ایپ سروس پلان بنائیں
az appservice plan create --name myapp-plan \
  --resource-group myapp-rg \
  --sku B1 --is-linux

# مرحلہ 3: ویب ایپ بنائیں
az webapp create --name myapp-web-unique123 \
  --resource-group myapp-rg \
  --plan myapp-plan \
  --runtime "NODE:18-lts"

# مرحلہ 4: کسموس ڈی بی اکاؤنٹ بنائیں (10-15 منٹ)
az cosmosdb create --name myapp-cosmos-unique123 \
  --resource-group myapp-rg \
  --kind MongoDB

# مرحلہ 5: ڈیٹا بیس بنائیں
az cosmosdb mongodb database create \
  --account-name myapp-cosmos-unique123 \
  --resource-group myapp-rg \
  --name tododb

# مرحلہ 6: کلیکشن بنائیں
az cosmosdb mongodb collection create \
  --account-name myapp-cosmos-unique123 \
  --resource-group myapp-rg \
  --database-name tododb \
  --name todos

# مرحلہ 7: کنکشن سٹرنگ حاصل کریں
CONN_STR=$(az cosmosdb keys list \
  --name myapp-cosmos-unique123 \
  --resource-group myapp-rg \
  --type connection-strings \
  --query "connectionStrings[0].connectionString" -o tsv)

# مرحلہ 8: ایپ کی ترتیبات کنفیگر کریں
az webapp config appsettings set \
  --name myapp-web-unique123 \
  --resource-group myapp-rg \
  --settings MONGODB_URI="$CONN_STR"

# مرحلہ 9: لاگنگ فعال کریں
az webapp log config --name myapp-web-unique123 \
  --resource-group myapp-rg \
  --application-logging filesystem \
  --detailed-error-messages true

# مرحلہ 10: ایپلیکیشن انسائیٹس سیٹ اپ کریں
az monitor app-insights component create \
  --app myapp-insights \
  --location eastus \
  --resource-group myapp-rg

# مرحلہ 11: ایپ انسائیٹس کو ویب ایپ سے لنک کریں
INSTRUMENTATION_KEY=$(az monitor app-insights component show \
  --app myapp-insights \
  --resource-group myapp-rg \
  --query "instrumentationKey" -o tsv)

az webapp config appsettings set \
  --name myapp-web-unique123 \
  --resource-group myapp-rg \
  --settings APPINSIGHTS_INSTRUMENTATIONKEY="$INSTRUMENTATION_KEY"

# مرحلہ 12: مقامی طور پر ایپلیکیشن تیار کریں
npm install
npm run build

# مرحلہ 13: ڈیپلائمنٹ پیکیج بنائیں
zip -r app.zip . -x "*.git*" "node_modules/*"

# مرحلہ 14: ایپلیکیشن تعینات کریں
az webapp deployment source config-zip \
  --resource-group myapp-rg \
  --name myapp-web-unique123 \
  --src app.zip

# مرحلہ 15: انتظار کریں اور دعا کریں کہ یہ کام کرے 🙏
# (کوئی خودکار تصدیق نہیں، دستی ٹیسٹنگ ضروری ہے)
```

**مسائل:**
- ❌ 15+ کمانڈز کو یاد رکھنا اور ترتیب وار چلانا
- ❌ 30-45 منٹ دستی کام
- ❌ غلطیاں کرنا آسان (ٹائپوز، غلط پیرامیٹرز)
- ❌ کنکشن سٹرنگز ٹرمینل ہسٹری میں نمایاں
- ❌ ناکامی کی صورت میں کوئی خودکار رول بیک نہیں
- ❌ ٹیم کے دوسرے ممبران کے لیے دوہرانا مشکل
- ❌ ہر بار مختلف (دہرائے جانے کے قابل نہیں)

#### ✅ AZD کے ساتھ: خودکار تعیناتی (5 کمانڈز، 10-15 منٹ)

```bash
# مرحلہ 1: ٹیمپلیٹ سے آغاز کریں
azd init --template todo-nodejs-mongo

# مرحلہ 2: تصدیق کریں
azd auth login

# مرحلہ 3: ماحول تیار کریں
azd env new dev

# مرحلہ 4: تبدیلیوں کا پیش نظارہ کریں (اختیاری لیکن تجویز کردہ)
azd provision --preview

# مرحلہ 5: سب کچھ تعینات کریں
azd up

# ✨ مکمل! سب کچھ تعینات، ترتیب دیا گیا، اور نگرانی کی جارہی ہے
```

**فوائد:**
- ✅ **5 کمانڈز** بمقابلہ 15+ دستی مراحل
- ✅ **10-15 منٹ** کل وقت (زیادہ تر Azure کے انتظار میں)
- ✅ **کم دستی غلطیاں** - مستقل، ٹیمپلیٹ پر مبنی ورک فلو
- ✅ **محفوظ خفیہ ہینڈلنگ** - بہت سے ٹیمپلیٹس Azure کے مینیج کردہ سیکریٹ اسٹوریج استعمال کرتے ہیں
- ✅ **دہرائے جانے والی تعیناتیاں** - ہر بار ایک ہی ورک فلو 
- ✅ **مکمل طور پر دہرائے جانے والی** - ہر بار ایک جیسا نتیجہ
- ✅ **ٹیم کے لیے تیار** - کوئی بھی ایک ہی کمانڈز سے تعینات کر سکتا ہے
- ✅ **کوڈ کے طور پر انفراسٹرکچر** - ورژن کنٹرولڈ بائسپس ٹیمپلیٹس
- ✅ **بلٹ ان مانیٹرنگ** - Application Insights خودکار طریقے سے ترتیب دیا جاتا ہے

## بنیادی تصورات

### ٹیمپلیٹس
ٹیمپلیٹس azd کی بنیاد ہیں۔ ان میں شامل ہیں:
- **ایپلیکیشن کوڈ** - آپ کا سورس کوڈ اور انحصار
- **انفراسٹرکچر کی تعریفیں** - Azure وسائل جو بائسپس یا ٹیرافارم میں تعریف کیے گئے ہیں
- **کنفیگریشن فائلیں** - ترتیبات اور ماحولیاتی متغیرات
- **تعیناتی اسکرپٹس** - خودکار تعیناتی کے ورک فلو

### ماحولیات
ماحولیات مختلف تعیناتی کے مقامات کی نمائندگی کرتی ہیں:
- **ڈیولپمنٹ** - جانچ اور ترقی کے لیے
- **اسٹیجنگ** - پیشگی پیداوار کا ماحول
- **پروڈکشن** - لائیو پیداوار کا ماحول

ہر ماحول اپنی الگ چیزیں رکھتا ہے:
- Azure ریسورس گروپ
- کنفیگریشن کی ترتیبات
- تعیناتی کی حالت

### سروسز
سروسز آپ کی ایپلیکیشن کی تعمیری اکائیاں ہیں:
- **فرنٹ اینڈ** - ویب ایپلیکیشنز، SPAs
- **بیک اینڈ** - APIs، مائیکرو سروسز
- **ڈیٹا بیس** - ڈیٹا اسٹوریج حل
- **اسٹوریج** - فائل اور بلاب اسٹوریج

## کلیدی خصوصیات

### 1. ٹیمپلیٹ پر مبنی ترقی
```bash
# دستیاب ٹیمپلیٹس براؤز کریں
azd template list

# ٹیمپلیٹ سے آغاز کریں
azd init --template <template-name>
```

### 2. کوڈ کے طور پر انفراسٹرکچر
- **بائسپس** - Azure کی مخصوص زبان
- **ٹیرافارم** - کثیر کلاؤڈ انفراسٹرکچر ٹول
- **ARM ٹیمپلیٹس** - Azure Resource Manager ٹیمپلیٹس

### 3. مربوط ورک فلو
```bash
# مکمل تعیناتی ورک فلو
azd up            # فراہمی + تعیناتی یہ پہلی بار سیٹ اپ کے لئے بغیر ہاتھ کے ہے

# 🧪 نیا: تعیناتی سے پہلے انفراسٹرکچر میں تبدیلیوں کا پیش نظارہ کریں (محفوظ)
azd provision --preview    # بغیر تبدیلی کے انفراسٹرکچر کی تعیناتی کا سیمولیشن کریں

azd provision     # اگر آپ انفراسٹرکچر کو اپ ڈیٹ کریں تو Azure وسائل بنائیں
azd deploy        # درخواست کا کوڈ تعینات کریں یا اپ ڈیٹ کے بعد دوبارہ تعینات کریں
azd down          # وسائل کو صاف کریں
```

#### 🛡️ پریویو کے ساتھ محفوظ انفراسٹرکچر کی منصوبہ بندی
`azd provision --preview` کمانڈ محفوظ تعیناتیوں کے لیے انقلابی ہے:
- **خشک دوڑ تجزیہ** - دکھاتا ہے کیا بنایا جائے گا، تبدیل کیا جائے گا، یا حذف کیا جائے گا
- **صفر خطرہ** - آپ کے Azure ماحول میں کوئی اصل تبدیلی نہیں ہوتی
- **ٹیم تعاون** - تعیناتی سے پہلے پریویو کے نتائج شیئر کریں
- **لاگت کا اندازہ** - وعدہ کرنے سے پہلے وسائل کی لاگت سمجھیں

```bash
# مثال کا پیش نظارہ ورک فلو
azd provision --preview           # دیکھیں کیا تبدیل ہوگا
# نتیجہ کا جائزہ لیں، ٹیم سے بات کریں
azd provision                     # اعتماد کے ساتھ تبدیلیاں لاگو کریں
```

**ورک فلو کی وضاحت:**
1. **Init** - ٹیمپلیٹ یا نیا پروجیکٹ شروع کریں
2. **Auth** - Azure سے تصدیق کریں
3. **Environment** - ایک الگ تعیناتی ماحول بنائیں
4. **Preview** - 🆕 ہمیشہ انفراسٹرکچر تبدیلیوں کا پہلے جائزہ لیں (محفوظ طریقہ)
5. **Provision** - Azure وسائل بنائیں/اپ ڈیٹ کریں
6. **Deploy** - اپنی ایپلیکیشن کا کوڈ بھیجیں
7. **Monitor** - ایپ کی کارکردگی کا مشاہدہ کریں
8. **Iterate** - تبدیلیاں کریں اور کوڈ دوبارہ تعینات کریں
9. **Cleanup** - کام مکمل ہونے پر وسائل ہٹائیں

### 4. ماحولیات کا انتظام
```bash
# ماحول بنائیں اور نظم کریں
azd env new <environment-name>
azd env select <environment-name>
azd env list
```

### 5. ایکسٹینشنز اور AI کمانڈز

azd ایکسٹینشن سسٹم کا استعمال کرتا ہے تاکہ کلیدی CLI سے باہر صلاحیتیں شامل کی جا سکیں۔ یہ خاص طور پر AI ورکلوڈز کے لیے مفید ہے:

```bash
# دستیاب توسیعات کی فہرست
azd extension list

# فاؤنڈری ایجنٹس توسیع انسٹال کریں
azd extension install azure.ai.agents

# مینیفیسٹ سے AI ایجنٹ پروجیکٹ کا آغاز کریں
azd ai agent init -m agent-manifest.yaml

# تعینات ایجنٹ کا ٹیسٹ کریں (لیٹینسی اور پہلے بائٹ کے وقت کو ظاہر کرتا ہے)
azd ai agent invoke

# AI معاونت یافتہ ترقی کے لیے MCP سرور شروع کریں (الفا)
azd mcp start
```

**ایجنٹ کا لائف سائیکل، ابتدا سے انتہا تک۔** جب آپ نے `azure.ai.agents` انسٹال کر لیا، تو ایک واحد ورک فلو آپ کو خیال سے چلتے، مانیٹر شدہ ایجنٹ تک لے جاتا ہے۔ آپ کو دن اول پر یہ سب کچھ درکار نہیں—بس جان لیں کہ یہ موجود ہیں:

| مرحلہ | کمانڈ | یہ کیا کرتی ہے |
|-------|---------|--------------|
| **اسکافولڈ** | `azd ai agent init -m <manifest>` | ایک مینیفیسٹ سے ایجنٹ پروجیکٹ بنائیں |
| **ٹیسٹ** | `azd ai agent invoke` | ایجنٹ کو کال کریں اور جواب کا وقت دیکھیں |
| **پیمائش** | `azd ai agent eval generate` | ایجنٹ کے لیے ایک تشخیصی ڈیٹا سیٹ بنائیں |
| **بہتری** | `azd ai agent optimize` | اپنے ڈیٹا کے مطابق ایجنٹ کی ہدایات کو بہتر بنائیں |
| **معائنہ** | `azd ai agent endpoint show` | لائیو اینڈپوائنٹ کی ترتیب دیکھیں |
| **صفائی** | `azd ai agent delete` | ایک ہوسٹڈ ایجنٹ اور اس کے تمام ورژنز کو حذف کریں |

> ایکسٹینشنز کی تفصیل سے وضاحت [باب 2: AI-فرسٹ ڈویلپمنٹ](../chapter-02-ai-development/agents.md) اور [AZD AI CLI کمانڈز](../chapter-08-production/production-ai-practices.md#azd-ai-cli-commands-and-extensions) حوالہ میں ملتی ہے۔

## 🎪 اصل ورک فلو حوالے

### نیا پروجیکٹ شروع کرنا
```bash
# طریقہ 1: موجودہ ٹیمپلیٹ استعمال کریں
azd init --template todo-nodejs-mongo

# طریقہ 2: صفر سے شروع کریں
azd init

# طریقہ 3: موجودہ ڈائرکٹری استعمال کریں
azd init .
```

### ترقی کا سائیکل
```bash
# ترقیاتی ماحول قائم کریں
azd auth login
azd env new dev
azd env select dev

# سب کچھ تعینات کریں
azd up

# تبدیلیاں کریں اور دوبارہ تعینات کریں
azd deploy

# مکمل ہونے پر صفائی کریں
azd down --force --purge # Azure Developer CLI میں کمانڈ آپ کے ماحول کے لیے **سخت ری سیٹ** ہے—خاص طور پر اس وقت مفید جب آپ ناکام تعیناتیوں میں مسئلہ حل کر رہے ہوں، چھوڑے گئے وسائل کی صفائی کر رہے ہوں، یا نئی تعیناتی کی تیاری کر رہے ہوں۔
```

## `azd down --force --purge` کو سمجھنا
`azd down --force --purge` کمانڈ آپ کا azd ماحول اور اس سے منسلک تمام وسائل مکمل طور پر ختم کرنے کا ایک طاقتور ذریعہ ہے۔ یہاں ہر فلیگ کا خلاصہ ہے:
```
--force
```
- تصدیقی پرامپٹ کو چھوڑ دیتا ہے۔
- خودکار عمل یا اسکرپٹنگ کے لیے مفید جہاں دستی ان پٹ ممکن نہ ہو۔
- اس بات کو یقینی بناتا ہے کہ رکاوٹ کے بغیر ماحول ختم ہو جائے، چاہے CLI کو کوئی عدم مطابقت نظر آئے۔

```
--purge
```
تمام متعلقہ میٹا ڈیٹا کو حذف کرتا ہے، بشمول:
ماحولیاتی حالت
مقامی `.azure` فولڈر
محفوظ تعیناتی کی معلومات
azd کو پچھلی تعیناتیوں کو "یاد رکھنے" سے روکتا ہے، جو مسائل پیدا کر سکتا ہے جیسے وسائل کے گروپ کے تضاد یا پرانی رجسٹری حوالہ جات۔

### دونوں استعمال کرنے کی وجہ؟
جب آپ `azd up` کے ساتھ باقی ماندہ حالت یا جزوی تعیناتیوں کی وجہ سے رکاوٹ کا سامنا کرتے ہیں، تو یہ امتزاج ایک **صاف صفحہ** یقینی بناتا ہے۔

یہ خاص طور پر Azure پورٹل میں دستی وسائل حذف کرنے کے بعد یا ٹیمپلیٹس، ماحولیات، یا ریسورس گروپ کے نام کی ترتیب بدلتے وقت مددگار ہے۔

## 🔐 تصدیق اور اسناد

تصدیق کو سمجھنا کامیاب azd تعیناتیوں کے لیے ضروری ہے۔ Azure متعدد تصدیقی طریقے استعمال کرتا ہے، اور azd وہی اسناد چین استعمال کرتا ہے جو دیگر Azure ٹولز استعمال کرتے ہیں۔

### Azure CLI تصدیق (`az login`)

azd استعمال کرنے سے پہلے، آپ کو Azure کے ساتھ تصدیق کرنی ہوتی ہے۔ سب سے عام طریقہ Azure CLI کا استعمال کرنا ہے:

```bash
# انٹرایکٹو لاگ ان (براؤزر کھولتا ہے)
az login

# مخصوص ٹیننٹ کے ساتھ لاگ ان کریں
az login --tenant <tenant-id>

# سروس پرنسپل کے ساتھ لاگ ان کریں
az login --service-principal -u <app-id> -p <password> --tenant <tenant-id>

# موجودہ لاگ ان کی حیثیت چیک کریں
az account show

# دستیاب سبسکرپشنز کی فہرست بنائیں
az account list --output table

# ڈیفالٹ سبسکرپشن سیٹ کریں
az account set --subscription <subscription-id>
```

### تصدیقی فلو
1. **انٹرایکٹو لاگ ان**: آپ کے ڈیفالٹ براؤزر کو تصدیق کے لیے کھولتا ہے
2. **ڈیوائس کوڈ فلو**: اُن ماحول کے لیے جہاں براؤزر دستیاب نہیں
3. **سروس پرنسپل**: خودکاری اور CI/CD کے منظرنامے کے لیے
4. **مینجڈ آئیڈینٹی**: Azure پر مبنی ایپلیکیشنز کے لیے

### DefaultAzureCredential چین

`DefaultAzureCredential` ایک قسم کی سند ہے جو خودکار طور پر متعدد اسناد کے ذرائع کو مخصوص ترتیب میں آزما کر آسان تصدیقی تجربہ فراہم کرتی ہے:

#### اسناد چین کی ترتیب
```mermaid
graph TD
    A[DefaultAzureCredential] --> B[ماحولیاتی متغیرات]
    B --> C[ورک لوڈ شناخت]
    C --> D[مینیجڈ شناخت]
    D --> E[ویژوئل اسٹوڈیو]
    E --> F[ویژوئل اسٹوڈیو کوڈ]
    F --> G[ایزور CLI]
    G --> H[ایزور پاور شیل]
    H --> I[انٹریکٹو براؤزر]
```

#### 1. ماحولیاتی متغیرات
```bash
# سہولت پرنسپل کے لیے ماحولیاتی متغیرات سیٹ کریں
export AZURE_CLIENT_ID="<app-id>"
export AZURE_CLIENT_SECRET="<password>"
export AZURE_TENANT_ID="<tenant-id>"
```

#### 2. ورکلوڈ شناخت (کبرنیٹس/ GitHub Actions)
خودکار طور پر استعمال ہوتا ہے:
- Azure Kubernetes Service (AKS) میں ورکلوڈ شناخت کے ساتھ
- GitHub Actions میں OIDC فیڈریشن کے ساتھ
- دیگر متفقہ شناخت کے منظرنامے

#### 3. مینجد شناخت
Azure وسائل کے لیے جیسے:
- ورچوئل مشینز
- App Service
- Azure Functions
- کنٹینر انسٹینسز

```bash
# چیک کریں کہ آیا ایزور کے وسائل پر منیجڈ شناخت کے ساتھ چل رہا ہے
az account show --query "user.type" --output tsv
# واپسی: "servicePrincipal" اگر منیجڈ شناخت استعمال ہو رہی ہو
```

#### 4. ڈویلپر ٹولز انٹیگریشن
- **Visual Studio**: خودکار طور پر لاگ ان کردہ اکاؤنٹ استعمال کرتا ہے
- **VS Code**: Azure Account ایکسٹینشن کی اسناد استعمال کرتا ہے
- **Azure CLI**: `az login` اسناد استعمال کرتا ہے (زیادہ تر لوکل ترقی کے لیے)

```bash

azd auth login
```

#### لوکل ڈیولپمنٹ کے لیے
#### CI/CD پائپ لائنز کے لیے
#### پروڈکشن ماحول کے لیے

- کوڈ یا کنفیگریشن فائلوں میں اسناد ذخیرہ کرنے سے گریز کریں

## 🙋 اکثر پوچھے جانے والے سوالات

### عمومی سوالات

**س: AZD اور Azure CLI میں کیا فرق ہے؟**

ج: Azure CLI (`az`) انفرادی Azure وسائل کے انتظام کے لیے ہے۔ AZD (`azd`) مکمل ایپلیکیشنز کے انتظام کے لیے ہے:

**اس طرح سوچیں:**
- `az` = انفرادی لیگو اینٹوں پر کام
- `azd` = مکمل لیگو سیٹ کے ساتھ کام

---

**س: کیا AZD استعمال کرنے کے لیے مجھے Bicep یا Terraform جاننا ضروری ہے؟**

ج: نہیں! ٹیمپلیٹس سے شروع کریں:
آپ بعد میں Bicep سیکھ سکتے ہیں تاکہ انفراسٹرکچر کو حسب ضرورت بنایا جا سکے۔ ٹیمپلیٹس کام کرنے والے نمونے فراہم کرتے ہیں جن سے سیکھا جا سکتا ہے۔

---

**س: AZD ٹیمپلیٹس چلانے کی کتنی لاگت آتی ہے؟**

ج: لاگت ٹیمپلیٹ پر منحصر ہے۔ زیادہ تر ترقیاتی ٹیمپلیٹس کی لاگت $50-150 ماہانہ ہوتی ہے:

**پرو ٹِپ:** جہاں دستیاب ہو مفت ٹیرز استعمال کریں:
- ایپ سروس: F1 (مفت) ٹیر
- مائیکروسافٹ فاؤنڈری ماڈلز: Azure OpenAI 50000 ٹوکنز/ماہ مفت
- Cosmos DB: 1000 RU/s مفت ٹیر

---

**س: کیا میں موجودہ Azure وسائل کے ساتھ AZD استعمال کر سکتا ہوں؟**

ج: ہاں، لیکن نیا آغاز کرنا آسان ہے۔ AZD تب بہترین کام کرتا ہے جب یہ مکمل lifecycle کو کنٹرول کرتا ہو۔ موجودہ وسائل کے لیے:

---

**س: میں اپنا پروجیکٹ ساتھیوں کے ساتھ کیسے شئیر کروں؟**

ج: AZD پروجیکٹ کو Git میں کمٹ کریں (لیکن .azure فولڈر کو نہیں):

سب کو ایک ہی انفراسٹرکچر ملتا ہے جو ایک ہی ٹیمپلیٹس سے بنتا ہے۔

---

### مسئلہ حل کرنے کے سوالات

**س: "azd up" آدھے راستے میں ناکام ہوگیا۔ میں کیا کروں؟**

ج: غلطی چیک کریں، اسے درست کریں، پھر دوبارہ کوشش کریں:

**عام مسئلہ:** غلط Azure سبسکرپشن منتخب ہوا ہے

---

**س: میں کوڈ کی تبدیلیاں بغیر دوبارہ انفراسٹرکچر بنانے کے کیسے تعینات کروں؟**

ج: `azd up` کے بجائے `azd deploy` استعمال کریں:

رفتار کا موازنہ:
- `azd up`: 10-15 منٹ (انفراسٹرکچر ترتیب دیتا ہے)
- `azd deploy`: 2-5 منٹ (صرف کوڈ)

---

**س: کیا میں انفراسٹرکچر ٹیمپلیٹس کو حسب ضرورت بنا سکتا ہوں؟**

ج: ہاں! `infra/` میں Bicep فائلیں ایڈٹ کریں:

**ٹپ:** چھوٹے سے شروع کریں - پہلے SKUs تبدیل کریں:

---

**س: میں AZD کے بنائے ہوئے تمام وسائل کو کیسے حذف کروں؟**

ج: ایک کمانڈ سے تمام وسائل حذف ہوتے ہیں:

**ہمیشہ چلائیں جب:**
- کسی ٹیمپلیٹ کی جانچ ختم ہو جائے
- مختلف پروجیکٹ پر سوئچ کریں
- نیا آغاز کرنا ہو

**رقم کی بچت:** غیر استعمال شدہ وسائل کو حذف کرنے سے کوئی چارجز نہیں ہوتے

---

**س: اگر میں نے غلطی سے Azure پورٹل میں وسائل حذف کر دیے تو؟**

ج: AZD کا اسٹیٹ غیر ہم آہنگ ہو سکتا ہے۔ کلین سلیٹ اپروچ استعمال کریں:

ج: ہاں! GitHub Actions کی مثال:

---

**س: میں سیکرٹس اور حساس ڈیٹا کو کیسے ہینڈل کروں؟**

ج: AZD خود بخود Azure Key Vault کے ساتھ انٹیگریٹ ہوتا ہے:

**کبھی بھی کمٹ نہ کریں:**
- `.azure/` فولڈر (ماحول کا ڈیٹا ہوتا ہے)
- `.env` فائلیں (مقامی سیکرٹس)
- کنکشن سٹرنگز
