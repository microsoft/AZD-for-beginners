# Φέρτε Τη Δική Σας Εφαρμογή - Προσθέστε azd σε Υπάρχον Έργο

**Πλοήγηση Κεφαλαίου:**
- **📚 Αρχική Μαθήματος**: [AZD Για Αρχάριους](../../README.md)
- **📖 Τρέχον Κεφάλαιο**: Κεφάλαιο 1 - Θεμέλια & Γρήγορη Εκκίνηση
- **⬅️ Προηγούμενο**: [Το Πρώτο Σας Έργο](first-project.md)
- **➡️ Επόμενο**: [Dev Containers & Codespaces](dev-containers.md)

> Επιβεβαιώθηκε για `azd 1.27.1` τον Ιούλιο του 2026.

## Εισαγωγή

Στο [Το Πρώτο Σας Έργο](first-project.md) αναπτύξατε μια εφαρμογή ξεκινώντας από ένα πρότυπο. Αλλά τις περισσότερες φορές έχετε ήδη *μια εφαρμογή* — ένα Node.js API, μια υπηρεσία Python Flask, μια .NET web εφαρμογή — που βρίσκεται σε έναν φάκελο στον υπολογιστή σας. Αυτό το μάθημα δείχνει πώς να προσθέσετε το azd σε αυτόν τον υπάρχοντα κώδικα ώστε να μπορείτε να το αναπτύξετε με `azd up`, χωρίς να απαιτείται πρότυπο.

## Στόχοι Μάθησης

Μέχρι το τέλος αυτού του μαθήματος, θα:
- Κατανοείτε τους τρεις τρόπους για να ξεκινήσετε ένα έργο azd
- Τρέξετε `azd init` μέσα σε υπάρχοντα κώδικα
- Κατανοείτε τι κάνει το `azure.yaml` και ο φάκελος `infra/` για την εφαρμογή σας
- Ξέρετε πότε να αφήσετε το azd να δημιουργήσει την υποδομή και πότε να γράψετε τη δική σας
- Αναπτύξετε την υπάρχουσα εφαρμογή σας στο Azure με `azd up`

## Αποτελέσματα Μάθησης

Μετά την ολοκλήρωση αυτού του μαθήματος, θα μπορείτε να:
- Αρχικοποιήσετε το azd σε ένα έργο που ήδη έχετε
- Διαβάσετε και επεξεργαστείτε ένα βασικό αρχείο `azure.yaml`
- Δημιουργήσετε αρχική υποδομή με `azd infra generate`
- Επιλέξετε έναν κατάλληλο Azure host για την εφαρμογή σας
- Κάνετε ανάπτυξη και καθαρισμό της δικής σας εφαρμογής

---

## Τρεις Τρόποι για να Ξεκινήσετε Ένα Έργο azd

| Σημείο εκκίνησης | Εντολή | Πότε να χρησιμοποιήσετε |
|----------------|---------|-------------|
| **Από πρότυπο** | `azd init --template <name>` | Εκμάθηση ή εκκίνηση νέας εφαρμογής από δοκιμασμένο δείγμα |
| **Από τον υπάρχοντα κώδικα σας** | `azd init` (στον φάκελο του έργου σας) | Έχετε ήδη εφαρμογή και θέλετε να την αναπτύξετε |
| **Από Git repo** | `azd init --from-code` (σε κλωνοποιημένο αποθετήριο) | Υιοθετώντας το azd για υπάρχον αποθετήριο |

Έχετε ήδη εξασκηθεί στην πρώτη επιλογή. Αυτό το μάθημα καλύπτει τη δεύτερη — το πιο κοινό σενάριο στην πράξη.

## Βήμα 1: Τρέξτε `azd init` στο Έργο Σας

Ανοίξτε ένα τερματικό **μέσα στον υπάρχον φάκελο έργου σας** και τρέξτε:

```bash
cd my-existing-app
azd init
```

Το azd θα ρωτήσει πώς θέλετε να αρχικοποιήσετε. Επιλέξτε:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

Επιλέξτε **"Χρήση κώδικα στον τρέχοντα κατάλογο."** Το azd θα σαρώσει τότε το φάκελό σας, θα ανιχνεύσει τη γλώσσα και το πλαίσιο και θα προτείνει έναν host.

### Τι ανιχνεύει το azd

Το azd ψάχνει για σήματα όπως `package.json`, `requirements.txt`, `pom.xml`, `*.csproj` ή ένα `Dockerfile` και προτείνει έναν αντίστοιχο Azure host:

| Η εφαρμογή σας | Πιθανός ανιχνευμένος host |
|----------|----------------------|
| Node.js / Python / .NET web εφαρμογή | Azure App Service ή Container Apps |
| Εφαρμογή κοντέινερ (`Dockerfile`) | Azure Container Apps |
| Εφαρμογή Function | Azure Functions |
| Στατικός ιστότοπος (εκδοχή React/Vue) | Azure Static Web Apps |

Επιβεβαιώστε την ανιχνευμένη υπηρεσία(ες), και το azd δημιουργεί τα αρχεία που χρειάζεστε.

## Βήμα 2: Κατανοήστε Τι Δημιούργησε το azd

Μετά το init, θα έχετε δύο νέα πράγματα στο έργο σας:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — ο ορισμός του έργου

Αυτό είναι η καρδιά ενός έργου azd. Ένα ελάχιστο δείχνει έτσι:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

Το μπλοκ `services` είναι το κεντρικό μέρος: κάθε εγγραφή αντιστοιχεί σε έναν φάκελο του κώδικά σας σε έναν Azure host. Αν η εφαρμογή σας έχει και frontend και API, θα έχετε δύο υπηρεσίες.

### `infra/` — οι πόροι Azure ως κώδικας

Ο φάκελος `infra/` περιέχει Bicep αρχεία που ορίζουν τους Azure πόρους που χρειάζεται η εφαρμογή σας (App Service, βάση δεδομένων, κλπ.). Δεν χρειάζεται να τα γράψετε εσείς — το azd δημιουργεί ένα λειτουργικό σημείο εκκίνησης. Μπορείτε *να* τα επεξεργαστείτε αργότερα για να προσθέσετε πόρους ή να ενισχύσετε την ασφάλεια (καλυμμένο στο [Κεφάλαιο 4](../chapter-04-infrastructure/README.md)).

> **Συμβουλή:** Θέλετε να δείτε ή να προσαρμόσετε την παραγόμενη υποδομή πριν την ανάπτυξη; Τρέξτε `azd infra generate` (επίσης διαθέσιμο ως `azd infra synth`) για να γράψετε το IaC στο δίσκο ώστε να το εξετάσετε και να το διαχειριστείτε έκδοση.

## Βήμα 3: Ρυθμίστε τις Απαιτούμενες Παραμέτρους

Αν η εφαρμογή σας χρειάζεται ρυθμίσεις ή μυστικά (π.χ. σύνδεσμο σύνδεσης, κλειδί API), μην τα ενσωματώνετε άμεσα στον κώδικα. Χρησιμοποιήστε μεταβλητές περιβάλλοντος:

```bash
# Δημιουργήστε ένα περιβάλλον
azd env new dev

# Ορίστε μια μη μυστική τιμή
azd env set API_VERSION 1.0.0
```

Για πραγματικά μυστικά, αποθηκεύστε τα στο Key Vault και αναφερθείτε σε αυτά από την υποδομή σας — δείτε [Κεφάλαιο 3: Ρύθμιση & Πιστοποίηση](../chapter-03-configuration/authsecurity.md).

## Βήμα 4: Ανάπτυξη

Τώρα χρησιμοποιήστε την ίδια ροή εργασίας που ήδη γνωρίζετε:

```bash
# Πιστοποίηση (απαραίτητη για azd)
azd auth login

# Προεπισκόπηση των πόρων που θα δημιουργηθούν
azd provision --preview

# Παροχή υποδομής και ανάπτυξη του κώδικά σας
azd up
```

Όταν ολοκληρωθεί, το azd εμφανίζει το URL της εφαρμογής σας. Επαληθεύστε με τον ίδιο τρόπο όπως σε οποιαδήποτε εφαρμογή azd:

```bash
azd show           # εμφάνιση τελικών σημείων
azd monitor --logs # ελέγξτε τα αρχεία καταγραφής αν χρειαστεί
```

## Συνήθη Πρώτα Προβλήματα

| Σύμπτωμα | Πιθανή αιτία | Διόρθωση |
|---------|--------------|-----|
| Το azd δεν ανίχνευσε την εφαρμογή μου | Λείπει το manifest (π.χ., `package.json`) | Προσθέστε το manifest, ή επιλέξτε χειροκίνητα τον host κατά το `azd init` |
| Η κατασκευή αποτυγχάνει κατά το `azd up` | Η εφαρμογή χρειάζεται βήμα κατασκευής | Προσθέστε `buildCommand`/`outputPath` κάτω από την υπηρεσία στο `azure.yaml` |
| Η εφαρμογή ξεκινά αλλά επιστρέφει σφάλματα | Λείπει ρύθμιση/μυστικό | Ορίστε τιμές με `azd env set` ή συνδέστε το Key Vault |
| Επιλέχθηκε λάθος host | Αυτόματη ανίχνευση | Επεξεργαστείτε το `host:` στο `azure.yaml` και ξανατρέξτε `azd up` |

Για περισσότερα, δείτε [Κεφάλαιο 7: Αντιμετώπιση Προβλημάτων](../chapter-07-troubleshooting/README.md).

## Καθαρισμός

```bash
azd down --force --purge
```

---

## Περίληψη

- `azd init` → **"Χρήση κώδικα στον τρέχοντα φάκελο"** προσθέτει το azd σε εφαρμογή που ήδη έχετε.
- Το `azure.yaml` αντιστοιχεί τους φακέλους κώδικά σας σε Azure hosts· το `infra/` ορίζει τους πόρους ως Bicep.
- Το `azd infra generate` σας επιτρέπει να δείτε ή να προσαρμόσετε την παραγόμενη υποδομή.
- Μόλις αρχικοποιηθεί, η υπάρχουσα εφαρμογή σας χρησιμοποιεί την ίδια ροή εργασίας `azd up` / `azd down` όπως μια εφαρμογή βασισμένη σε πρότυπο.
