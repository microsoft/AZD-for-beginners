## Εισαγωγή

Στα προηγούμενα κεφάλαια αναπτύξατε μία εφαρμογή—και στο Κεφάλαιο 2 αναπτύξατε έναν μεμονωμένο πράκτορα AI. Αυτό το μάθημα κάνει το επόμενο βήμα: αναπτύσσει ένα **σύστημα πολλαπλών πρακτόρων**, όπου αρκετοί εξειδικευμένοι πράκτορες συνεργάζονται για να λύσουν ένα πρόβλημα που κανένας μεμονωμένος πράκτορας δεν θα μπορούσε να χειριστεί καλά από μόνος του.

Τα καλά νέα για αρχάριους: **δεν χρειάζεστε νέες εντολές.** Μια λύση πολλαπλών πρακτόρων παραμένει ένα έργο azd. Θα κάνετε `azd init`, `azd up`, δοκιμές και `azd down`—ακριβώς τη ροή εργασίας που ήδη γνωρίζετε. Αυτό που αλλάζει είναι η *μορφή* της εφαρμογής στο εσωτερικό.

## Στόχοι Μάθησης

Στο τέλος αυτού του μαθήματος, θα:
- Κατανοήσετε τι σημαίνει "πολλαπλών πρακτόρων" και πότε αξίζει η πρόσθετη πολυπλοκότητα
- Αναγνωρίσετε τους κοινότυπους ρόλους σε ένα σύστημα πολλαπλών πρακτόρων (ορχηστρωτής + ειδικοί)
- Αναπτύξετε ένα πραγματικό, λειτουργικό πρότυπο πολλαπλών πρακτόρων με `azd up`
- Κατανοήσετε τους πόρους Azure που υποστηρίζουν μια εφαρμογή πολλαπλών πρακτόρων
- Γνωρίζετε πώς να επαληθεύετε, να προσαρμόζετε και να αποσυναρμολογείτε τη λύση με ασφάλεια

## Μαθησιακά Αποτελέσματα

Μετά την ολοκλήρωση αυτού του μαθήματος, θα μπορείτε να:
- Εξηγήσετε τη διαφορά μεταξύ ενός μεμονωμένου πράκτορα και ενός συστήματος πολλαπλών πρακτόρων
- Επιλέξετε μεταξύ ενός μεμονωμένου πράκτορα με εργαλεία και ενός πραγματικού σχεδιασμού πολλαπλών πρακτόρων
- Αναπτύξετε και δοκιμάσετε ένα πρότυπο πολλαπλών πρακτόρων από άκρο σε άκρο με azd
- Εντοπίσετε πού τρέχει κάθε πράκτορας και πώς επικοινωνούν
- Καθαρίσετε όλους τους πόρους για να αποφύγετε συνεχιζόμενες χρεώσεις

## Τι είναι ένα Σύστημα Πολλαπλών Πρακτόρων;

Ένας μεμονωμένος πράκτορας AI είναι ένα μοντέλο με ένα σύνολο οδηγιών και (προαιρετικά) κάποια εργαλεία. Αυτό λειτουργεί καλά για εστιασμένα καθήκοντα. Αλλά καθώς ένα έργο μεγαλώνει—έρευνα, μετά συγγραφή, μετά επιμέλεια, μετά επαλήθευση—το να συμπιέζεις τα πάντα σε ένα prompt κάνει τον πράκτορα πιο αργό, λιγότερο αξιόπιστο και πιο δύσκολο στον εντοπισμό σφαλμάτων.

Ένα **σύστημα πολλαπλών πρακτόρων** χωρίζει τη δουλειά σε ειδικούς που ο καθένας κάνει μια δουλειά καλά, συντονισμένους από έναν ορχηστρωτή:

```mermaid
graph TD
    User([Αίτημα χρήστη]) --> Orchestrator[Ορχηστρωτής Πράκτορας<br/>Σχεδιάζει και δρομολογεί την εργασία]
    Orchestrator --> Researcher[Ερευνητής Πράκτορας<br/>Συλλέγει στοιχεία]
    Orchestrator --> Writer[Συγγραφέας Πράκτορας<br/>Συντάσσει περιεχόμενο]
    Orchestrator --> Editor[Επιμελητής Πράκτορας<br/>Επανεξετάζει και βελτιώνει]
    Researcher --> Orchestrator
    Writer --> Orchestrator
    Editor --> Orchestrator
    Orchestrator --> Result([Τελική απάντηση])
```

### Οι δύο ρόλοι που θα βλέπετε πάντα

| Ρόλος | Εργασία | Παράδειγμα |
|------|-----|---------|
| **Ορχηστρωτής** | Αποφασίζει *τι γίνεται στη συνέχεια* και δρομολογεί την εργασία μεταξύ πρακτόρων | "Πρώτα έρευνα, μετά γραφή, μετά επιμέλεια" |
| **Ειδικός** | Εκτελεί μία συγκεκριμένη εργασία και επιστρέφει αποτέλεσμα | Ένας «ερευνητής» που μόνο συγκεντρώνει πληροφορίες |

### Χρειάζεστε πραγματικά πολλούς πράκτορες;

Ξεκινήστε απλά. Επιλέξτε πολλαπλούς πράκτορες **μόνο** όταν ισχύει ένα από τα παρακάτω:

- ✅ Το έργο έχει **διακριτά στάδια** που ωφελούνται από διαφορετικές οδηγίες (έρευνα vs. γραφή vs. ανασκόπηση)
- ✅ Θέλετε οι ειδικοί να τρέχουν **παράλληλα** για εξοικονόμηση χρόνου
- ✅ Διαφορετικά βήματα απαιτούν **διαφορετικά εργαλεία ή πηγές δεδομένων**
- ✅ Χρειάζεστε κάθε βήμα να είναι **ανεξάρτητα ελέγξιμο και αποσφαλματωμένο**

Αν το έργο σας είναι μια απλή ερώτηση-απάντηση ή μια απλή κλήση εργαλείου, ένας **μεμονωμένος πράκτορας με εργαλεία** (Κεφάλαιο 2) είναι απλούστερος, φθηνότερος και πιο εύκολος στη λειτουργία.

> **Συμβουλή για αρχάριους:** «Περισσότεροι πράκτορες» δεν σημαίνει «καλύτερα». Κάθε πράκτορας προσθέτει καθυστέρηση, κόστος και ένα νέο στοιχείο για παρακολούθηση. Προσθέστε πράκτορες μόνο όταν το πρόβλημα διαχωρίζεται σαφώς σε μέρη.

## Δύο Τρόποι για να Δημιουργήσετε Πολλαπλούς Πράκτορες στο Azure

| Προσέγγιση | Τι είναι | Καλύτερο για |
|----------|-----------|----------|
| **Ένας πράκτορας + εργαλεία** | Ένας Foundry πράκτορας που καλεί συναρτήσεις/εργαλεία | Απλές ροές εργασίας, για να ξεκινήσετε |
| **Πολλοί συντονισμένοι πράκτορες** | Πολλοί πράκτορες με έναν ορχηστρωτή | Διακριτά στάδια, παράλληλη εργασία, εξειδίκευση |

Αυτό το μάθημα εστιάζει στη δεύτερη προσέγγιση χρησιμοποιώντας ένα **έτοιμο πρότυπο**, ώστε να δείτε ένα πραγματικό σύστημα πολλαπλών πρακτόρων σε λειτουργία πριν φτιάξετε το δικό σας.

## Πρακτική Άσκηση: Ανάπτυξη Μιας Λειτουργικής Εφαρμογής Πολλαπλών Πρακτόρων

Θα αναπτύξουμε το **Contoso Creative Writer**, ένα επίσημο δείγμα Azure που χρησιμοποιεί πολλούς πράκτορες (ερευνητής, συγγραφέας, επιμελητής) συντονισμένους για να παράγουν ένα άρθρο. Είναι μια εξαιρετική πρώτη εφαρμογή πολλαπλών πρακτόρων επειδή οι ρόλοι είναι εύκολοι να κατανοηθούν.

### Βήμα 1: Αρχικοποίηση του προτύπου

```bash
# Δημιουργήστε έναν φάκελο εργασίας
mkdir creative-writer && cd creative-writer

# Αρχικοποιήστε από το επίσημο πρότυπο πολλαπλών πρακτόρων
azd init --template contoso-creative-writer
```

> Περιηγηθείτε σε περισσότερα πρότυπα πολλαπλών πρακτόρων οποτεδήποτε στη [Συλλογή Awesome AZD AI](https://azure.github.io/awesome-azd/?tags=ai). Άλλες επιλογές φιλικές για αρχάριους περιλαμβάνουν `get-started-with-ai-agents` και `azure-ai-travel-agents`.

### Βήμα 2: Πιστοποίηση

```bash
# Απαιτείται για τις ροές εργασίας azd
azd auth login
```

### Βήμα 3: Δημιουργία περιβάλλοντος

```bash
azd env new dev
```

### Βήμα 4: Προεπισκόπηση, στη συνέχεια ανάπτυξη

```bash
# Δείτε τι θα δημιουργηθεί πριν ξοδέψετε οτιδήποτε (συνιστάται)
azd provision --preview

# Παρέχετε την υποδομή και αναπτύξτε όλους τους agents σε ένα βήμα
azd up
```

`azd up` θα ζητήσει συνδρομή και περιοχή, στη συνέχεια θα προμηθεύσει τους πόρους Azure και θα αναπτύξει την εφαρμογή. Οι αναπτύξεις AI μπορεί να διαρκέσουν περισσότερο από μια απλή web εφαρμογή—αν αναπτύσσετε μεγαλύτερα μοντέλα, μπορείτε να επεκτείνετε το timeout ανάπτυξης:

```bash
azd deploy --timeout 1800
```

> **Προειδοποίηση για κόστος και χωρητικότητα:** Οι εφαρμογές πολλαπλών πρακτόρων αναπτύσσουν μοντέλα AI που καταναλώνουν όριο και επιφέρουν κόστος. Εάν το `azd up` αποτύχει λόγω ορίου μοντέλου, δείτε [Αντιμετώπιση Προβλημάτων AI](../chapter-07-troubleshooting/ai-troubleshooting.md) για διορθώσεις περιοχής και ορίων, και το Κεφάλαιο 6 [Προγραμματισμός Χωρητικότητας](../chapter-06-pre-deployment/capacity-planning.md).

## Κατανόηση αυτού που Αναπτύξατε

Μια τυπική εφαρμογή πολλαπλών πρακτόρων όπως αυτή προμηθεύει ένα σύνολο πόρων Azure που αντιστοιχούν άμεσα στις ευθύνες στο διάγραμμα παραπάνω:

| Πόρος | Γιατί υπάρχει |
|----------|----------------|
| **Microsoft Foundry / Models** | Φιλοξενεί τα γλωσσικά μοντέλα που χρησιμοποιεί κάθε πράκτορας |
| **Azure AI Search** | Παρέχει στον ερευνητή προσβάσιμα δεδομένα για αναζήτηση |
| **Container Apps** (or App Service) | Φιλοξενεί τον ορχηστρωτή και τον κώδικα των πρακτόρων |
| **Cosmos DB** (in some samples) | Αποθηκεύει κοινή κατάσταση/μνήμη που ανταλλάσσεται μεταξύ πρακτόρων |
| **Application Insights** | Παρακολουθεί αιτήματα *διαμέσου* των πρακτόρων ώστε να μπορείτε να αποσφαλματώσετε τη ροή |

### Πώς επικοινωνούν οι πράκτορες μεταξύ τους

Στα περισσότερα δείγματα azd πολλαπλών πρακτόρων, ο **ορχηστρωτής εκτελείται στον κώδικα της εφαρμογής σας** (για παράδειγμα, χρησιμοποιώντας ένα πλαίσιο όπως το Semantic Kernel ή το Microsoft Agent Framework). Ο ορχηστρωτής καλεί κάθε εξειδικευμένο πράκτορα με τη σειρά, προωθεί τα αποτελέσματα και συναρμολογεί την τελική απάντηση. Οι πράκτορες μοιράζονται το πλαίσιο μέσω:

- **Κλήσεις λειτουργιών/εργαλείων** — ο ορχηστρωτής καλεί έναν ειδικό και λαμβάνει ένα αποτέλεσμα
- **Κοινή μνήμη** — μια βάση δεδομένων (συχνά Cosmos DB) αποθηκεύει κατάσταση που μπορούν να διαβάσουν και οι δύο πράκτορες
- **Μηνύματα/γεγονότα** — για χαλαρότερη σύνδεση, οι πράκτορες επικοινωνούν μέσω ουράς ή Service Bus

> **Γιατί αυτό έχει σημασία για τον εντοπισμό σφαλμάτων:** επειδή κάθε βήμα είναι ξεχωριστό, το Application Insights σας δείχνει *ποιος* πράκτορας ήταν αργός ή απέτυχε. Αυτός είναι ένας σημαντικός λόγος για να χωρίσετε τη δουλειά σε πολλούς πράκτορες εξαρχής.

## Επαλήθευση της Ανάπτυξης

Επιβεβαιώστε ότι το σύστημα λειτουργεί πραγματικά πριν προχωρήσετε:

```bash
# Εμφάνιση των αναπτυχθέντων σημείων τερματισμού
azd show

# Άνοιγμα του πίνακα παρακολούθησης της εφαρμογής
azd monitor

# Παρακολούθησε τα αρχεία καταγραφής (logs) αν κάτι φαίνεται λάθος
azd monitor --logs
```

Στη συνέχεια ανοίξτε το URL της εφαρμογής από το `azd show` και δοκιμάστε ένα αίτημα που αξιοποιεί όλους τους πράκτορες (για το Creative Writer, ζητήστε του να γράψει ένα σύντομο άρθρο σε ένα θέμα). Στην **αναζήτηση συναλλαγών** του Application Insights, θα πρέπει να δείτε το αίτημα να διαχέεται στα βήματα ερευνητή, συγγραφέα και επιμελητή.

**Κριτήρια επιτυχίας:**
- ✅ Το `azd show` εμφανίζει έναν προσβάσιμο endpoint
- ✅ Ένα αίτημα παράγει αποτέλεσμα που ξεκάθαρα πέρασε από πολλαπλά στάδια
- ✅ Το Application Insights εμφανίζει ιχνηλατήσεις για περισσότερα από ένα βήματα πρακτόρων

## Προσαρμογή: Προσθήκη ή Προσαρμογή ενός Πράκτορα

Επειδή κάθε πράκτορας είναι απλώς οδηγίες συν εργαλεία, η προσαρμογή είναι υλοποιήσιμη:

1. **Βρείτε τους ορισμούς των πρακτόρων** στο πρότυπο (συχνά ένα σύνολο αρχείων `prompts/`, `agents/`, ή `*.prompty`).
2. **Τροποποιήστε τις οδηγίες ενός πράκτορα** — για παράδειγμα, πείτε στον πράκτορα-επιμελητή να επιβάλει συγκεκριμένο τόνο ή αριθμό λέξεων.
3. **Αναπτύξτε ξανά μόνο τον κώδικα** (η υποδομή παραμένει αμετάβλητη):

   ```bash
   azd deploy
   ```

Για να προχωρήσετε περισσότερο και να δημιουργήσετε πράκτορες από το *δικό* σας manifest, χρησιμοποιήστε το extension πρακτόρων και τον πλήρη κύκλο ζωής του:

```bash
azd extension install azure.ai.agents
azd ai agent init -m agent-manifest.yaml
azd up
azd ai agent invoke      # δοκιμή, με χρονισμό απόκρισης
```

Δείτε [Κεφάλαιο 2: Πράκτορες](../chapter-02-ai-development/agents.md) και την [Αναφορά AZD AI CLI](../chapter-08-production/production-ai-practices.md#azd-ai-cli-commands-and-extensions) για τον πλήρη κύκλο ζωής πρακτόρων (`invoke`, `eval generate`, `optimize`, `delete`).

## Καθαρισμός

Οι εφαρμογές πολλαπλών πρακτόρων εκτελούν πολλαπλές υπηρεσίες που χρεώνονται. Καταργήστε τα πάντα όταν τελειώσετε:

```bash
azd down --force --purge
```

Η σημαία `--purge` επίσης αφαιρεί πόρους AI που έχουν μαλακά διαγραφεί (όπως λογαριασμούς Foundry/Azure AI Services) ώστε να μην εμποδίζουν μια μελλοντική επαναανάπτυξη ή συνεχίσουν να επιφέρουν κόστος.

## Περίληψη

- Ένα σύστημα πολλαπλών πρακτόρων διαχωρίζει την εργασία σε ειδικούς που συντονίζονται από έναν ορχηστρωτή.
- Χρησιμοποιήστε το μόνο όταν το έργο έχει διακριτά στάδια, παραλληλισμό, ή διαφορετικά εργαλεία ανά βήμα—διαφορετικά προτιμήστε έναν μεμονωμένο πράκτορα.
- Η ροή εργασίας azd παραμένει ίδια: `azd init` → `azd up` → δοκιμή → `azd down`.
- Ένα πραγματικό πρότυπο όπως το `contoso-creative-writer` σας επιτρέπει να δείτε και να προσαρμόσετε μια λειτουργική εφαρμογή πολλαπλών πρακτόρων σήμερα.
- Η ιχνηλάτηση του Application Insights ανάμεσα σε πράκτορες είναι ένα από τα μεγαλύτερα πρακτικά οφέλη του σχεδιασμού πολλαπλών πρακτόρων.
