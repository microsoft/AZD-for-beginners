## התחלה

### ש: מהו Azure Developer CLI (azd)?
**ת**: Azure Developer CLI (azd) הוא כלי שורת פקודה ממוקד מפתחים שמאיץ את הזמן שלוקח לך להעביר את היישום שלך מסביבת פיתוח מקומית ל-Azure. הוא מספק שיטות עבודה מומלצות דרך תבניות ועוזר בכל מחזור החיים של הפריסה.

### ש: במה azd שונה מ-Azure CLI?
**ת**: 
- **Azure CLI**: כלי כללי לניהול משאבי Azure
- **azd**: כלי ממוקד מפתח לזרימות עבודה של פריסת יישומים
- azd משתמש ב-Azure CLI באופן פנימי אך מספק הפשטות ברמה גבוהה עבור תרחישי פיתוח נפוצים
- azd כולל תבניות, ניהול סביבה ואוטומציה של פריסה

### ש: האם אני צריך להתקין Azure CLI כדי להשתמש ב-azd?
**ת**: כן, azd דורש Azure CLI עבור אימות וכמה פעולות. התקן את Azure CLI תחילה, ואז התקן את azd.

### ש: באילו שפות תכנות azd תומך?
**ת**: azd אינו תלוי בשפה. הוא עובד עם:
- Node.js/JavaScript/TypeScript
- Python
- .NET/C#
- Java
- Go
- PHP
- אתרי אינטרנט סטטיים
- יישומים מונעי מכולות

### ש: האם אפשר להשתמש ב-azd עם פרויקטים קיימים?
**ת**: כן! באפשרותך:
1. להשתמש ב-`azd init` להוספת תצורת azd לפרויקטים קיימים
2. להתאים פרויקטים קיימים למבנה תבנית azd
3. ליצור תבניות מותאמות אישית המבוססות על הארכיטקטורה הקיימת שלך

## אימות וגישה

### ש: איך מאמתים את עצמך מול Azure עם azd?
**ת**: השתמש ב-`azd auth login` שיפתח חלון דפדפן לאימות מול Azure. לתרחישי CI/CD, השתמש ב-service principals או managed identities.

### ש: האם אפשר להשתמש ב-azd עם מספר מנויים של Azure?
**ת**: כן. השתמש ב-`azd env set AZURE_SUBSCRIPTION_ID <subscription-id>` כדי לציין איזה מנוי לשימוש עבור כל סביבה.

### ש: אילו הרשאות אני צריך כדי לפרוס עם azd?
**ת**: בדרך כלל צריך:
- תפקיד **Contributor** בקבוצת המשאבים או במנוי
- **User Access Administrator** אם מפרסים משאבים שדורשים הקצאת תפקידים
- הרשאות ספציפיות משתנות לפי תבנית והמשאבים המפורסמים

### ש: האם אפשר להשתמש ב-azd בצינורות CI/CD?
**ת**: בהחלט! azd תוכנן לשילוב ב-CI/CD. השתמש ב-service principals לאימות והגדר משתני סביבה להגדרות.

### ש: איך מטפלים באימות ב-GitHub Actions?
**ת**: השתמש בפעולת Azure Login עם אישורי service principal:
```yaml
- uses: azure/login@v1
  with:
    creds: ${{ secrets.AZURE_CREDENTIALS }}
- run: azd deploy --no-prompt
```

## תבניות ופרויקטים

### ש: איפה אפשר למצוא תבניות azd?
**ת**: 
- תבניות רשמיות: [Azure-Samples/awesome-azd](https://github.com/Azure-Samples/awesome-azd)
- תבניות קהילתיות: חפש ב-GitHub "azd-template"
- השתמש ב-`azd template list` כדי לעיין בתבניות זמינות

### ש: איך יוצרים תבנית מותאמת אישית?
**ת**: 
1. התחל ממבנה תבנית קיים
2. שנה את `azure.yaml`, קבצי תשתית וקוד היישום
3. בדוק היטב עם `azd up`
4. פרסם ב-GitHub עם תגיות מתאימות

### ש: האם אפשר להשתמש ב-azd בלי תבנית?
**ת**: כן, השתמש ב-`azd init` בפרויקט קיים כדי ליצור את קבצי התצורה הנחוצים. יהיה עליך להגדיר ידנית את `azure.yaml` וקבצי התשתית.

### ש: מה ההבדל בין תבניות רשמיות לקהילתיות?
**ת**: 
- **תבניות רשמיות**: מתוחזקות על ידי מיקרוסופט, מעודכנות לעתים קרובות, עם תיעוד מקיף
- **תבניות קהילתיות**: נוצרות על ידי מפתחים, בעלות מקרים ספציפיים, איכות ותחזוקה משתנה

### ש: איך מעדכנים תבנית בפרויקט שלי?
**ת**: תבניות לא מתעדכנות אוטומטית. באפשרותך:
1. להשוות ולמזג שינויים מהתבנית המקורית ידנית
2. להתחיל מחדש עם `azd init` באמצעות התבנית המעודכנת
3. לבחור וליישם שיפורים מסוימים מהתבניות המעודכנות

## פריסה ותשתית

### ש: אילו שירותי Azure azd יכול לפרוס?
**ת**: azd יכול לפרוס כל שירות Azure דרך תבניות Bicep/ARM, כולל:
- App Services, Container Apps, Functions
- מסדי נתונים (SQL, PostgreSQL, Cosmos DB)
- אחסון, Key Vault, Application Insights
- רשת, אבטחה ומשאבי ניטור

### ש: האם אפשר לפרוס לאזורי Azure מרובים?
**ת**: כן, הגדר כמה אזורים בתבניות Bicep שלך והגדר את פרמטר המיקום המתאים לכל סביבה.

### ש: איך מטפלים במיגרציות סכימת בסיסי נתונים?
**ת**: השתמש ב-hooks לפריסה ב-`azure.yaml`:
```yaml
hooks:
  postdeploy:
    posix:
      run: ./scripts/migrate-database.sh
    windows:
      run: ./scripts/migrate-database.ps1
```

### ש: האם אפשר לפרוס רק תשתית בלי יישומים?
**ת**: כן, השתמש ב-`azd provision` כדי לפרוס רק את רכיבי התשתית המוגדרים בתבניות שלך.

### ש: איך לפרוס על משאבי Azure קיימים?
**ת**: זה מסובך ולא נתמך ישירות. באפשרותך:
1. לייבא משאבים קיימים לתוך תבניות Bicep שלך
2. להשתמש בהפניות למשאבים קיימים בתבניות
3. לשנות תבניות כדי ליצור או להפנות למשאבים באופן מותנה

### ש: האם אפשר להשתמש ב-Terraform במקום ב-Bicep?
**ת**: כרגע azd תומך בעיקר בתבניות Bicep/ARM. תמיכה ב-Terraform אינה זמינה רשמית, אם כי ייתכן שיש פתרונות קהילתיים.

## הגדרות וסביבות

### ש: איך מנהלים סביבות שונות (פיתוח, ביניים, ייצור)?
**ת**: צור סביבות נפרדות עם `azd env new <environment-name>` והגדר הגדרות שונות עבור כל אחת:
```bash
azd env new development
azd env new staging  
azd env new production
```

### ש: איפה שומרים את התצורות של הסביבות?
**ת**: בתיקיית `.azure` בתוך ספריית הפרויקט. לכל סביבה תיקייה משלה עם קבצי תצורה.

### ש: איך מגדירים הגדרות ספציפיות לסביבה?
**ת**: השתמש ב-`azd env set` כדי להגדיר משתני סביבה:
```bash
azd env set AZURE_LOCATION eastus
azd env set DATABASE_TIER Basic
```

### ש: האם אפשר לשתף תצורות סביבה בין חברי צוות?
**ת**: תיקיית `.azure` מכילה מידע רגיש ואינה צריכה להיכלל בגרסת קוד. במקום זאת:
1. תעד משתני סביבה נדרשים
2. השתמש בסקריפטים לפריסת סביבות
3. השתמש ב-Azure Key Vault להגדרות רגישות

### ש: איך מתעלמים מהגדרות ברירת מחדל של תבנית?
**ת**: הגדר משתני סביבה התואמים לפרמטרים בתבנית:
```bash
azd env set LOCATION "West US 2"
azd env set SKU_NAME "B1"
```

## פתרון בעיות

### ש: למה `azd up` נכשל?
**ת**: סיבות נפוצות:
1. **בעיות אימות**: הפעל `azd auth login`
2. **הרשאות לא מספקות**: בדוק הקצאות תפקיד Azure שלך
3. **התנגשויות בשמות משאבים**: שנה AZURE_ENV_NAME
4. **בעיות מכסה/קיבולת**: בדוק זמינות אזורית
5. **שגיאות בתבנית**: אמת את תבניות Bicep

### ש: איך לנפות כשלים בפריסה?
**ת**: 
1. השתמש ב-`azd deploy --debug` לפלט מפורט
2. בדוק היסטוריית פריסה בפורטל Azure
3. בדוק ביומן פעילות בפורטל Azure
4. השתמש ב-`azd show` להצגת מצב הסביבה הנוכחי

### ש: למה משתני הסביבה שלי לא עובדים?
**ת**: בדוק:
1. שמות משתנים תואמים במדויק לפרמטרים בתבנית
2. הערכים מצוטטות כראוי אם הם כוללים רווחים
3. הסביבה שנבחרה: `azd env select <environment>`
4. המשתנים מוגדרים בסביבה המתאימה

### ש: איך מנקים פריסות שלא הצליחו?
**ת**: 
```bash
azd down --force --purge
```
פעולה זו מסירה את כל המשאבים ואת קובצי התצורה של הסביבה.

### ש: למה היישום שלי לא נגיש לאחר פריסה?
**ת**: בדוק:
1. הפריסה הושלמה בהצלחה
2. היישום פועל (בדוק את הלוגים בפורטל Azure)
3. קבוצות אבטחת רשת מאפשרות תנועה
4. שמות דומיין ו-DNS מוגדרים כראוי

## עלויות וחשבונות

### ש: כמה יעלו פריסות באמצעות azd?
**ת**: העלויות תלויות ב:
- שירותי Azure המפורסמים
- רמות שירות/SKUs שנבחרו
- הבדלים במחיר אזורי
- דפוסי שימוש

השתמש ב-[מחוון המחירים של Azure](https://azure.microsoft.com/pricing/calculator/) להערכות.

### ש: איך שולט על עלויות בפריסות azd?
**ת**: 
1. השתמש ב-SKUs ברמת זול יותר עבור סביבות פיתוח
2. הגדר תקציבים והתראות ב-Azure
3. השתמש ב-`azd down` להסרת משאבים שאינם נדרשים
4. בחר אזורים מתאימים (העלויות משתנות לפי מיקום)
5. השתמש בכלי ניהול עלויות של Azure

### ש: האם יש אפשרויות שכבה חינמית לתבניות azd?
**ת**: שירותי Azure רבים מציעים שכבות חינמיות:
- App Service: שכבה חינמית זמינה
- Azure Functions: מיליון ביצועים חינמיים לחודש
- Cosmos DB: שכבה חינמית עם 400 RU/s
- Application Insights: 5GB ראשונים חינם לחודש

הגדר תבניות כדי להשתמש בשכבות חינמיות במידת האפשר.

### ש: איך מוקצה עלות לפני הפריסה?
**ת**: 
1. סקור את `main.bicep` בתבנית כדי לראות אילו משאבים נוצרו
2. השתמש במחשבון המחירים של Azure עם SKUs ספציפיים
3. פרוס לסביבת פיתוח ראשונית למעקב עלויות בפועל
4. השתמש בניהול העלויות של Azure לניתוח מפורט

## שיטות עבודה מומלצות

### ש: מהם שיטות העבודה הטובות ביותר למבנה פרויקט azd?
**ת**: 
1. שמור קוד יישום נפרד מתשתית
2. השתמש בשמות שירות משמעותיים ב-`azure.yaml`
3. יישם טיפול שגיאות נאות בסקריפטי בנייה
4. השתמש בהגדרות ייחודיות לסביבה
5. כלול תיעוד מקיף

### ש: איך מארגנים מספר שירותים ב-azd?
**ת**: השתמש במבנה המומלץ:
```
├── src/
│   ├── web/          # Frontend service
│   ├── api/          # Backend service  
│   └── worker/       # Background service
├── infra/            # Infrastructure templates
├── azure.yaml        # azd configuration
└── README.md         # Project documentation
```

### ש: האם צריך להעלות את תיקיית `.azure` ל-git?
**ת**: **לא!** תיקיית `.azure` מכילה מידע רגיש. הוסף אותה ל-`.gitignore`:
```gitignore
.azure/
```

### ש: איך מטפלים בסודות והגדרות רגישות?
**ת**: 
1. השתמש ב-Azure Key Vault לסודות
2. הפנה לסודות Key Vault בהגדרות היישום
3. אל תכלול סודות בגרסת קוד
4. השתמש ב-managed identities לאימות בין שירותים

### ש: מה הגישה המומלצת ל-CI/CD עם azd?
**ת**: 
1. השתמש בסביבות נפרדות לכל שלב (פיתוח/ביניים/ייצור)
2. יישם בדיקות אוטומטיות לפני פריסה
3. השתמש ב-service principals לאימות
4. אחסן הגדרות רגישות בסודות/משתנים בצינור
5. יישם שערי אישור לפריסות ייצור

## נושאים מתקדמים

### ש: האם אפשר להרחיב את azd עם פונקציונליות מותאמת?
**ת**: כן, דרך hooks לפריסה ב-`azure.yaml`:
```yaml
hooks:
  predeploy:
    run: ./scripts/custom-setup.sh
  postdeploy:
    run: ./scripts/custom-config.sh
```

### ש: איך משולב azd עם תהליכי DevOps קיימים?
**ת**: 
1. השתמש בפקודות azd בסקריפטים קיימים בצינורות
2. תקן תבניות azd בכל הצוותים
3. שילב עם ניטור והתראות קיימים
4. השתמש בפלט JSON של azd לשילוב בצינורות

### ש: האם אפשר להשתמש ב-azd עם Azure DevOps?
**ת**: כן, azd עובד עם כל מערכת CI/CD. צור צינורות Azure DevOps המשתמשים בפקודות azd.

### ש: איך תורמים ל-azd או יוצרים תבניות קהילתיות?
**ת**: 
1. **כלי azd**: תרום ל-[Azure/azure-dev](https://github.com/Azure/azure-dev)
2. **תבניות**: צור תבניות בהתאם ל[הנחיות התבניות](https://github.com/Azure-Samples/awesome-azd)
3. **תיעוד**: תרום לדוקומנטציה ב-[MicrosoftDocs/azure-dev-docs](https://github.com/MicrosoftDocs/azure-dev-docs)

### ש: מהי מפת הדרך של azd?
**ת**: בדוק את [מפת הדרך הרשמית](https://github.com/Azure/azure-dev/projects) עבור תכונות ושיפורים מתוכננים.

### ש: איך להעביר כלים אחרים לפריסות ל-azd?
**ת**: 
1. נתח את הארכיטקטורת הפריסה הנוכחית
2. צור תבניות Bicep מקבילות
3. הגדר את `azure.yaml` כך שיתאים לשירותים הנוכחיים
4. בדוק ביסודיות בסביבת פיתוח
5. העבר את הסביבות בהדרגה
