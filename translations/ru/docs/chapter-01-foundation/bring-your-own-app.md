> Проверено на `azd 1.25.6` в июне 2026 г.

## Введение

В [Вашем первом проекте](first-project.md) вы развернули приложение, начиная с шаблона. Но чаще всего у вас уже есть приложение — Node.js API, Python Flask сервис, .NET веб-приложение — в папке на вашей машине. Этот урок показывает, как добавить azd к существующему коду, чтобы развернуть его с помощью `azd up`, без использования шаблона.

## Цели обучения

К концу этого урока вы:
- Поймёте три способа начать проект azd
- Выполните `azd init` внутри существовой кодовой базы
- Поймёте, для чего нужны `azure.yaml` и папка `infra/` в вашем приложении
- Будете знать, когда позволить azd сгенерировать инфраструктуру, а когда писать свою
- Развернёте существующее приложение в Azure с помощью `azd up`

## Результаты обучения

После завершения этого урока вы сможете:
- Инициализировать azd в проекте, который у вас уже есть
- Читать и редактировать базовый файл `azure.yaml`
- Сгенерировать начальную инфраструктуру с помощью `azd infra generate`
- Выбрать подходящий хост Azure для вашего приложения
- Развернуть и очистить собственное приложение

## Три способа начать проект azd

| Точка начала | Команда | Когда использовать |
|----------------|---------|-------------|
| **Из шаблона** | `azd init --template <name>` | Обучение или начало нового приложения на проверенном примере |
| **Из вашего существующего кода** | `azd init` (в папке проекта) | У вас уже есть приложение и вы хотите его развернуть |
| **Из Git-репозитория** | `azd init --from-code` (в клонированном репо) | Применение azd к существующему репозиторию |

Вы уже практиковали первый вариант. Этот урок охватывает второй — наиболее распространённый сценарий в реальной жизни.

## Шаг 1: Выполните `azd init` в вашем проекте

Откройте терминал **внутри папки с вашим существующим проектом** и выполните:

```bash
cd my-existing-app
azd init
```

azd спросит, как вы хотите инициализировать. Выберите:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

Выберите **"Использовать код в текущем каталоге."** azd просканирует вашу папку, определит язык и фреймворк и предложит хост.

### Что обнаруживает azd

azd ищет признаки вроде `package.json`, `requirements.txt`, `pom.xml`, `*.csproj` или `Dockerfile` и предлагает соответствующий хост Azure:

| Ваше приложение | Вероятный хост |
|----------|----------------------|
| Веб-приложение Node.js / Python / .NET | Azure App Service или Container Apps |
| Контейнеризованное приложение (`Dockerfile`) | Azure Container Apps |
| Function app | Azure Functions |
| Статический сайт (собранный вывод React/Vue) | Azure Static Web Apps |

Подтвердите обнаруженные сервисы, и azd создаст необходимые файлы.

## Шаг 2: Поймите, что создал azd

После init в вашем проекте появятся две новые вещи:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — определение проекта

Это сердце проекта azd. Минимальный пример выглядит так:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

Блок `services` — ключевая часть: каждая запись сопоставляет папку с вашим кодом и хост Azure. Если у вашего приложения есть фронтенд и API, у вас будет два сервиса.

### `infra/` — ваши ресурсы Azure как код

Папка `infra/` содержит файлы Bicep, которые определяют ресурсы Azure, необходимые вашему приложению (App Service, база данных и т.д.). Вам не обязательно писать их вручную — azd генерирует рабочую стартовую точку. Вы *можете* редактировать их позже, чтобы добавить ресурсы или усилить безопасность (это рассматривается в [Глава 4](../chapter-04-infrastructure/README.md)).

> **Совет:** Хотите увидеть или настроить сгенерированную инфраструктуру до развертывания? Выполните `azd infra generate` (также доступно как `azd infra synth`), чтобы записать IaC на диск и просмотреть/версионировать её.

## Шаг 3: Задайте необходимые параметры конфигурации

Если вашему приложению нужны настройки или секреты (строка подключения, API-ключ), не храните их в коде. Используйте значения окружения:

```bash
# Создать окружение
azd env new dev

# Установить несекретное значение
azd env set API_VERSION 1.0.0
```

Для реальных секретов храните их в Key Vault и ссылайтесь на них из вашей инфраструктуры — см. [Глава 3: Конфигурация и аутентификация](../chapter-03-configuration/authsecurity.md).

---

## Шаг 4: Развертывание

Теперь используйте тот же рабочий процесс, который вы уже знаете:

```bash
# Аутентификация (требуется для azd)
azd auth login

# Просмотреть ресурсы, которые будут созданы
azd provision --preview

# Предоставить инфраструктуру и развернуть ваш код
azd up
```

Когда завершится, azd выведет URL вашего приложения. Проверьте его так же, как любое приложение azd:

```bash
azd show           # показать конечные точки
azd monitor --logs # проверьте логи при необходимости
```

## Распространенные проблемы при первом запуске

| Симптом | Вероятная причина | Решение |
|---------|--------------|-----|
| azd не обнаружил мое приложение | Отсутствует манифест (например, `package.json`) | Добавьте манифест или выберите хост вручную при `azd init` |
| Сборка завершается с ошибкой во время `azd up` | Приложению нужен шаг сборки | Добавьте `buildCommand`/`outputPath` в блоке сервиса в `azure.yaml` |
| Приложение запускается, но возвращает ошибки | Отсутствует конфигурация/секрет | Установите значения с помощью `azd env set` или подключите Key Vault |
| Выбран неверный хост | Автоопределение угадало неправильно | Отредактируйте `host:` в `azure.yaml` и снова запустите `azd up` |

Для дополнительной информации см. [Глава 7: Устранение неполадок](../chapter-07-troubleshooting/README.md).

## Очистка

```bash
azd down --force --purge
```

---

## Резюме

- `azd init` → **"Использовать код в текущем каталоге."** добавляет azd к приложению, которое у вас уже есть.
- `azure.yaml` сопоставляет папки с вашим кодом и хосты Azure; `infra/` определяет ресурсы в виде Bicep.
- `azd infra generate` позволяет просмотреть или настроить сгенерированную инфраструктуру.
- После инициализации ваше существующее приложение использует тот же рабочий процесс `azd up` / `azd down`, что и приложение на основе шаблона.
