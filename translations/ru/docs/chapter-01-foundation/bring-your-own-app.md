## Введение

В [Вашем первом проекте](first-project.md) вы развернули приложение, начав с шаблона. Но чаще всего у вас уже есть готовое приложение — Node.js API, сервис Python Flask, веб-приложение .NET — лежащее в папке на вашем компьютере. Этот урок показывает, как добавить azd к существующему коду, чтобы вы могли развернуть его с помощью `azd up`, без необходимости шаблона.

## Цели обучения

К концу этого урока вы:
- Поймете три способа начать проект azd
- Запустите `azd init` внутри существующей кодовой базы
- Поймете, что делают `azure.yaml` и папка `infra/` для вашего приложения
- Узнаете, когда позволить azd создавать инфраструктуру, а когда писать ее самостоятельно
- Развернете ваше существующее приложение в Azure с помощью `azd up`

## Результаты обучения

После завершения этого урока вы сможете:
- Инициализировать azd в уже имеющемся проекте
- Читать и редактировать базовый файл `azure.yaml`
- Генерировать начальную инфраструктуру с помощью `azd infra generate`
- Выбирать подходящий хостинг в Azure для вашего приложения
- Развертывать и удалять ваше собственное приложение

## Три способа начать проект azd

| Начальная точка | Команда | Когда использовать |
|----------------|---------|------------------|
| **Из шаблона** | `azd init --template <name>` | Изучение или запуск нового приложения на основе проверенного примера |
| **Со своего существующего кода** | `azd init` (в папке проекта) | У вас уже есть приложение и его нужно развернуть |
| **Из Git репозитория** | `azd init --from-code` (в клонированном репозитории) | Внедрение azd в существующий репозиторий |

Вы уже практиковались с первым вариантом. Этот урок посвящен второму — самому распространенному реальному сценарию.

## Шаг 1: Запустите `azd init` в своем проекте

Откройте терминал **внутри папки вашего существующего проекта** и выполните:

```bash
cd my-existing-app
azd init
```

azd спросит, как вы хотите инициализировать проект. Выберите:

```
? How do you want to initialize your app?
> Use code in the current directory
  Select a template
```

Выберите **"Использовать код в текущей директории."** azd просканирует папку, определит язык и фреймворк и предложит хостинг.

### Что azd определяет

azd ищет признаки вроде `package.json`, `requirements.txt`, `pom.xml`, `*.csproj` или `Dockerfile`, и предлагает соответствующий хостинг Azure:

| Ваше приложение | Вероятный определенный хост |
|----------------|--------------------------|
| Node.js / Python / .NET веб-приложение | Azure App Service или Container Apps |
| Контейнеризированное приложение (`Dockerfile`) | Azure Container Apps |
| Функциональное приложение | Azure Functions |
| Статический сайт (сборка React/Vue) | Azure Static Web Apps |

Подтвердите выбранные сервисы, и azd создаст необходимые файлы.

## Шаг 2: Поймите, что создал azd

После инициализации в вашем проекте появятся два новых элемента:

```
my-existing-app/
├── azure.yaml          # ← NEW: tells azd what to deploy and where
├── infra/              # ← NEW: Infrastructure as Code (Bicep by default)
│   ├── main.bicep
│   └── main.parameters.json
├── src/ (your code)    # unchanged
└── ...                 # your existing files, untouched
```

### `azure.yaml` — определение проекта

Это сердце проекта azd. Минимальный пример выглядит так:

```yaml
# azure.yaml
name: my-existing-app
services:
  web:
    project: ./src           # path to your app code
    language: js             # js | python | dotnet | java
    host: appservice         # appservice | containerapp | function | staticwebapp
```

Блок `services` — ключевая часть: каждая запись сопоставляет папку вашего кода с хостом Azure. Если в вашем приложении есть фронтенд и API, у вас будет два сервиса.

### `infra/` — ваши ресурсы Azure как код

Папка `infra/` содержит Bicep-файлы, определяющие ресурсы Azure, необходимые вашему приложению (App Service, база данных и т. д.). Писать их вручную не нужно — azd автоматически создаёт рабочую базу. Позже вы *можете* отредактировать их для добавления ресурсов или улучшения безопасности (рассматривается в [Главе 4](../chapter-04-infrastructure/README.md)).

> **Совет:** Хотите посмотреть или настроить созданную инфраструктуру перед развертыванием? Выполните `azd infra generate` (также доступна команда `azd infra synth`), чтобы записать IaC на диск для проверки и контроля версий.

## Шаг 3: Настройте необходимые параметры

Если вашему приложению нужны настройки или секреты (строка подключения, API-ключ), не прописывайте их напрямую. Используйте значения окружения:

```bash
# Создать окружение
azd env new dev

# Установить значение без секретности
azd env set API_VERSION 1.0.0
```

Для настоящих секретов храните их в Key Vault и ссылайтесь на них из инфраструктуры — см. [Главу 3: Настройки и аутентификация](../chapter-03-configuration/authsecurity.md).

## Шаг 4: Разверните

Теперь используйте знакомый вам рабочий процесс:

```bash
# Аутентификация (требуется для azd)
azd auth login

# Предварительный просмотр создаваемых ресурсов
azd provision --preview

# Подготовка инфраструктуры и развертывание вашего кода
azd up
```

По завершении azd выведет URL вашего приложения. Проверьте его так же, как вы бы проверили любое приложение azd:

```bash
azd show           # показать конечные точки
azd monitor --logs # проверить логи при необходимости
```

## Частые проблемы при первом развертывании

| Симптом | Вероятная причина | Решение |
|---------|------------------|---------|
| azd не обнаружил мое приложение | Отсутствует манифест (например, `package.json`) | Добавьте манифест или выберите хост вручную при `azd init` |
| Сбой сборки во время `azd up` | Приложению нужен этап сборки | Добавьте `buildCommand`/`outputPath` в сервис в `azure.yaml` |
| Приложение запускается, но возвращает ошибки | Отсутствуют настройки/секреты | Установите значения с помощью `azd env set` или подключите Key Vault |
| Выбран неправильный хост | Автоопределение ошиблось | Отредактируйте `host:` в `azure.yaml` и запустите `azd up` снова |

Подробнее см. в [Главе 7: Поиск и устранение неисправностей](../chapter-07-troubleshooting/README.md).

---

## Очистка

```bash
azd down --force --purge
```

---

## Итоги

- `azd init` → **"Использовать код в текущей директории"** добавляет azd в уже имеющееся приложение.
- `azure.yaml` связывает папки кода с хостами Azure; `infra/` определяет ресурсы в Bicep.
- `azd infra generate` позволяет просмотреть или настроить созданную инфраструктуру.
- После инициализации ваше существующее приложение использует тот же рабочий процесс `azd up` / `azd down`, что и приложение на основе шаблона.
