## 1. Введение

[Azure Developer CLI](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/) или `azd` — это инструмент командной строки с открытым исходным кодом, упрощающий рабочий процесс разработчика при создании и развертывании приложений в Azure.

[Шаблоны AZD](https://learn.microsoft.com/azure/developer/azure-developer-cli/azd-templates) — это стандартизированные репозитории, которые включают пример кода приложения, ресурсы _инфраструктуры как кода_ и конфигурационные файлы `azd` для целостной архитектуры решения. Создание инфраструктуры становится таким же простым, как команда `azd provision` — а использование `azd up` позволяет одновременно создать инфраструктуру **и** развернуть приложение!

В результате, начать процесс разработки вашего приложения может быть так же просто, как найти подходящий _AZD Starter шаблон_, который наиболее близок к вашим требованиям к приложению и инфраструктуре — а затем настроить репозиторий под требования вашего сценария.

Прежде чем начать, давайте убедимся, что у вас установлен Azure Developer CLI.

1. Откройте терминал VS Code и введите эту команду:

      ```bash title="" linenums="0"
      azd version
      ```

1. Вы должны увидеть примерно следующее!

      ```bash title="" linenums="0"
      azd version 1.25.6 (commit <current-build>)
      ```

**Теперь вы готовы выбрать и развернуть шаблон с помощью azd**

## 3. Активация шаблона

Давайте попробуем развернуть этот шаблон и убедиться, что он корректен. Мы следуем рекомендациям из раздела [Getting Started](https://github.com/Azure-Samples/get-started-with-ai-agents?tab=readme-ov-file#getting-started).

1. Выберите рабочую среду для репозитория шаблона:

      - **GitHub Codespaces**: Нажмите [на эту ссылку](https://github.com/codespaces/new/Azure-Samples/get-started-with-ai-agents) и подтвердите `Create codespace`
      - **Локальное клонирование или dev-контейнер**: Клонируйте `Azure-Samples/get-started-with-ai-agents` и откройте его в VS Code

1. Дождитесь готовности терминала VS Code, затем введите следующую команду:

   ```bash title="" linenums="0"
   azd up
   ```

Выполните шаги рабочего процесса, которые этим будут запущены:

1. Вас попросят войти в Azure — следуйте инструкциям для аутентификации
1. Введите уникальное имя для среды — например, я использовал `nitya-mshack-azd`
1. Это создаст папку `.azure/` — вы увидите вложенную папку с именем среды
1. Вас попросят выбрать подписку — выберите по умолчанию
1. Вас попросят указать расположение — используйте `East US 2`

Теперь ждите завершения создания инфраструктуры. **Это занимает 10-15 минут**

1. По завершении в консоли отобразится сообщение ОБ УСПЕХЕ, похожее на это:
      ```bash title="" linenums="0"
      SUCCESS: Your up workflow to provision and deploy to Azure completed in 10 minutes 17 seconds.
      ```
1. В вашем портале Azure теперь будет создана группа ресурсов с именем вашей среды:

      ![Infra](../../../../../translated_images/ru/02-provisioned-infra.46c706b14f56e0bf.webp)

1. **Теперь вы готовы проверить развернутую инфраструктуру и приложение**.

## 5. Проверка агента

Azure Container App разворачивает конечную точку, которая подключается к AI агенту, созданному в проекте Microsoft Foundry для этого шаблона. Давайте посмотрим, что это значит.

1. Вернитесь на страницу _Overview_ вашего ресурса в Azure Portal

1. Нажмите на ресурс `Microsoft Foundry` в списке

1. Вы увидите это. Нажмите на кнопку `Go to Microsoft Foundry Portal`. 
   ![Foundry](../../../../../translated_images/ru/04-view-foundry-project.fb94ca41803f28f3.webp)

1. Должна открыться страница Foundry проекта вашего AI приложения
   ![Project](../../../../../translated_images/ru/05-visit-foundry-portal.d734e98135892d7e.webp)

1. Нажмите `Agents` — вы увидите агента по умолчанию, развернутого в вашем проекте
   ![Agents](../../../../../translated_images/ru/06-visit-agents.bccb263f77b00a09.webp)

1. Выберите его — откроются детали агента. Обратите внимание на следующее:

      - Агент по умолчанию использует Поиск по файлам (всегда)
      - В `Knowledge` агента указано, что загружено 32 файла (для поиска по файлам)
      ![Agents](../../../../../translated_images/ru/07-view-agent-details.0e049f37f61eae62.webp)

1. В левом меню найдите и нажмите на опцию `Data+indexes` для деталей. 

      - Вы должны увидеть 32 загруженных файла с данными для базы знаний.
      - Они соответствуют 12 файлам клиентов и 20 файлам продуктов в папке `src/files`
      ![Data](../../../../../translated_images/ru/08-visit-data-indexes.5a4cc1686fa0d19a.webp)

**Вы проверили работу агента!** 

1. Ответы агента основаны на знаниях из этих файлов.
1. Теперь вы можете задавать вопросы, связанные с этими данными, и получать подтвержденные ответы.
1. Пример: `customer_info_10.json` описывает 3 покупки, сделанные "Amanda Perez"

Вернитесь во вкладку браузера с конечной точкой Container App и спросите: `Какие продукты принадлежат Amanda Perez?`. Вы должны увидеть что-то вроде этого:

## 7. Встроенная наблюдаемость

Наблюдаемость — это инструментирование вашего приложения для генерации данных, которые можно использовать для понимания, отладки и оптимизации его работы. Чтобы получить представление об этом:

1. Нажмите кнопку `View Run Info` — вы увидите этот вид. Это пример работы [Agent tracing](https://learn.microsoft.com/en-us/azure/ai-foundry/how-to/develop/trace-agents-sdk#view-trace-results-in-the-azure-ai-foundry-agents-playground). _Этот вид также можно получить, кликнув Thread Logs в верхнем меню_.

   - Получите представление о шагах выполнения и задействованных инструментах агента
   - Поймите общее количество токенов (по сравнению с токенами вывода) для ответа
   - Поймите задержки и где тратится время в процессе выполнения

      ![Agent](../../../../../translated_images/ru/10-view-run-info.b20ebd75fef6a1cc.webp)

1. Перейдите на вкладку `Metadata`, чтобы увидеть дополнительные атрибуты запуска, которые могут быть полезны для отладки проблем в будущем.   

      ![Agent](../../../../../translated_images/ru/11-view-run-info-metadata.7966986122c7c2df.webp)


1. Перейдите на вкладку `Evaluations`, чтобы увидеть автооценки ответа агента. Они включают оценки безопасности (например, самоповреждение) и специфические оценки агента (например, разрешение намерений, соблюдение задачи).

      ![Agent](../../../../../translated_images/ru/12-view-run-info-evaluations.ef25e4577d70efeb.webp)

1. И наконец, нажмите вкладку `Monitoring` в боковом меню.

      - Выберите вкладку `Resource usage` на странице и просмотрите метрики.
      - Отслеживайте использование приложения по затратам (токенам) и загрузке (запросам).
      - Отслеживайте задержку приложения до первого байта (обработка ввода) и до последнего байта (вывод).

      ![Agent](../../../../../translated_images/ru/13-monitoring-resources.5148015f7311807f.webp)

## 8. Переменные среды

Пока что мы проходили процесс развертывания через браузер и проверили, что инфраструктура создана и приложение работает. Но чтобы работать с кодом приложения _в первую очередь_, нам нужно настроить локальную среду разработки с нужными переменными, необходимыми для работы с этими ресурсами. Использование `azd` упрощает этот процесс.

1. Azure Developer CLI [использует переменные среды](https://learn.microsoft.com/en-us/azure/developer/azure-developer-cli/manage-environment-variables?tabs=bash) для хранения и управления настройками конфигурации при развертывании приложений.

1. Переменные среды хранятся в `.azure/<env-name>/.env` — это ограничивает их область до среды `env-name`, используемой при развертывании, и помогает изолировать среды между разными целевыми развертываниями в одном репозитории.

1. Переменные среды автоматически загружаются командой `azd` при выполнении конкретных команд (например, `azd up`). Обратите внимание, что `azd` не читает автоматически переменные среды уровней ОС (например, установленные в оболочке) — вместо этого используйте `azd set env` и `azd get env` для передачи информации в скриптах.


Давайте попробуем несколько команд:

1. Получите все переменные среды, установленные для `azd` в этой среде:

      ```bash title="" linenums="0"
      azd env get-values
      ```
      
      Вы увидите примерно следующее:

      ```bash title="" linenums="0"
      AZURE_AI_AGENT_DEPLOYMENT_NAME="gpt-4.1-mini"
      AZURE_AI_AGENT_NAME="agent-template-assistant"
      AZURE_AI_EMBED_DEPLOYMENT_NAME="text-embedding-3-small"
      AZURE_AI_EMBED_DIMENSIONS=100
      ...
      ```

1. Получите конкретное значение — например, хотите узнать, установлена ли переменная `AZURE_AI_AGENT_MODEL_NAME`

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```
      
      Вы увидите примерно следующее — по умолчанию она не была установлена!

      ```bash title="" linenums="0"
      ERROR: key 'AZURE_AI_AGENT_MODEL_NAME' not found in the environment values
      ```

1. Установите новую переменную среды для `azd`. Здесь мы обновляем имя модели агента. _Примечание: любые изменения сразу отражаются в файле `.azure/<env-name>/.env`.

      ```bash title="" linenums="0"
      azd env set AZURE_AI_AGENT_MODEL_NAME gpt-4.1
      azd env set AZURE_AI_AGENT_MODEL_VERSION 2025-04-14
      azd env set AZURE_AI_AGENT_DEPLOYMENT_CAPACITY 150
      ```

      Теперь мы должны увидеть, что значение установлено:

      ```bash title="" linenums="0"
      azd env get-value AZURE_AI_AGENT_MODEL_NAME 
      ```

1. Обратите внимание, что некоторые ресурсы являются постоянными (например, развертывания моделей) и потребуют больше, чем просто `azd up` для повторного развертывания. Попробуем удалить исходное развертывание и повторно развернуть с измененными переменными среды.

1. **Обновление** Если ранее вы развертывали инфраструктуру с использованием шаблона azd — вы можете _обновить_ состояние локальных переменных среды на основе текущего состояния вашего развертывания Azure с помощью этой команды:

      ```bash title="" linenums="0"
      azd env refresh
      ```

      Это мощный способ _синхронизировать_ переменные окружения между двумя или более локальными средами разработки (например, команда с несколькими разработчиками) — позволяющий развернутой инфраструктуре служить основой для состояния переменных окружения. Члены команды просто _обновляют_ переменные, чтобы снова синхронизироваться.
